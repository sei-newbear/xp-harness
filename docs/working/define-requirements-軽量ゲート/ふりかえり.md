### 深掘り 1: `git reset --hard` の落とし穴 (巻き戻し + branch リーク)
#### 経緯
reframe を再 deploy (apm install) して line17=深掘る を確認した直後、ケース毎リセットの手順どおり `git reset --hard f87d316 && git clean -fd` を実行。この初期 commit は reframe deploy より前に作られており、deploy 済み skill も git tracked で含まれていたため、reset が deploy を旧版へ巻き戻した。case2 の transcript が「軽い/固める」と旧ラベルを出したことで発覚。再 deploy 済み状態を commit した baseline (894bb79) を作り直して再走した。branch リークも同じ穴 (reset --hard は HEAD を戻すだけで base branch に戻さない) で、case3 の「別機能名 branch」表示から気づいた。

#### 分析
harness-verification skill が「再 deploy (Step 2)」と「reset でクリーン化 (Step 5)」を独立記述していて、両者の干渉 (deploy が sandbox に tracked のとき、再 deploy より前の commit へ reset すると deploy が巻き戻る / reset は branch を切り替えない) を surface していなかった。取り決めは 2 つの事実を持つが、その積を欠いていた。別角度では、各ケース前に「対象 skill が本当に deploy されているか」を確認する guard を持っていれば初回で捕捉できた (バグ発覚後に実際この確認を入れた)。両方とも直し先は skill 本文の同じ節。

依頼者フィードバック: 内省の因果連鎖は自走を見ていないので検証できない (= main session が自分で持つべき部分)。アクションは「軽くスキルに書けば十分」= 規律装置最小注入に沿って過剰にしない。

#### アクション
harness-verification skill Step 5 に「`git reset --hard <commit>` の落とし穴」を軽く追記 (実施済): ①再 deploy より前の commit へ reset すると deploy が旧版に巻き戻る → リセット先は再 deploy 済みを commit した baseline にする、②reset は branch を切り替えない → base branch へ戻し不要 branch を消す、③リセット後に対象 skill の deploy を 1 度確認してから流す。

### 深掘り 2: sandbox 忠実度 (case5・6b の濁り)
#### 経緯
ケース5 (そのまま進める → 実装) とケース6b (お任せ+曖昧ゴール → 深掘る) で、sandbox scaffold に「通知設定画面」「ユーザー一覧画面」の実体が無く (README は謳うが placeholder)、セッションが「対象が無い」方向へ逸れ、狙った振る舞い (E2E-spec-first / 深掘る推奨ゲート) の綺麗な観測ができなかった。核となる挙動 (slice-tdd への handoff / ショートカットしない guard) は取れた。

#### 分析
既存機能の変更・改善を投げるケースでは、sandbox にその機能の実体が要る。条件設計 (手順3 のレビュー) の段階で「sandbox は対象機能を実装しているか」を問えば事前に気づけた。深掘り 1 より軽いパスで確認した論点。

#### アクション
検証条件の設計原則「開始状態は検証対象の skill に合わせる」に、既存機能の変更・改善ケースでは sandbox にその機能の実体 (placeholder でない実画面・実 API) があるか条件設計時に確認する旨を軽く追記 (実施済)。
