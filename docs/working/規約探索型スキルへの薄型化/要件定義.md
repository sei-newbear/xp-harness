# 上書き前提スキルの規約探索型への薄型化 — 要件定義

> このドキュメントは進行中の設計メモ。実装後もそのまま残す（振り返り・PR レビューで参照される）。
> 削除のタイミング・要否は依頼者または外側の仕組みで別途判断する（あなたは削除しない）。

## 1. 背景と動機

### フロント + API の一気通貫開発を既定とする判断

consumer のプロジェクトでは front / api のようにディレクトリ（領域）が分かれ、領域ごとに実装規約や E2E の流儀が異なることがある。一方、xp-harness の思想（story-slicing の Small は「ユーザー価値を保ったまま分割可能な最小単位」という価値ベースの定義、outside-in の例外なしルール）は、ユーザーストーリーがフロントと API を縦に貫くこと — すなわち 1 セッションでの一気通貫開発 — を暗黙の前提にしている。層で割る分割（フロントだけ・API だけ）は技術ベースの分割であり、harness 自身が退けている。

壁打ちの結果、**1 セッションでの一気通貫開発を既定とする**と判断した。リポジトリ構成（モノレポか別リポジトリ群か）は運用で対処できる詳細であり（別リポジトリでも追加の作業ディレクトリ指定で 1 セッションに取り込める）、本質的な制約ではない。

代替案として検討した「別セッション + 消費者駆動契約テスト（フロントセッションが消費者として契約を作り、API セッションが契約を満たす）」は、コンテキスト隔離による能力の補償という価値を持つが、ユーザー価値の Done をどちらのセッションでも検証できず、人がセッション間の橋渡しになって自走が途切れる費用が大きい。組織境界（別チーム・API の複数利用者・別リリース周期）が実在する consumer が現れたときに翻訳して採用する拡張候補として保留する。なお契約テスト自体は、テスト経済の装置（境界を契約テストで固めて重い E2E を薄く保つ）として 1 セッション内でも成立するため、セッション分割とは独立に consumer が採用できる。

### 現状のスキル構成が一気通貫を支えられない理由

skill はセッションを開いたディレクトリ直下の `.claude` からしか読み込まれない。領域ごとにスキルを上書きする運用では、領域ごとにセッションを開く必要があり、縦切りのストーリーを 1 セッションで貫けない。共通の親でセッションを開くと、今度は領域別の規約を運ぶ仕組みがない。

規約の適用を「その場で考えさせる」方式は採らない。規律の適用をその瞬間の注意力に依存させると長い自走の後段で漏れることが実証されている（slice-tdd の冒頭に置いた案内が後段で忘れられた検証結果。skill-design-style の「他 skill を参照させるときは効く場所で呼ぶ」原則の根拠）。規約の切り替えは、slice-tdd が手順ごとに implementation を呼び直す既存の構造（リズム）に乗せる。

### スコープ境界の原則: xp-harness は XP の関心ごとに閉じる

本改修の境界原則として「**xp-harness は XP の関心ごと（進め方の規律: TDD・受け入れテスト・リズム・フェーズ）に閉じ、コードやテストの中身の流儀（コメント方針・特定フレームワークの規約・テストの文体）はプロジェクト側に委ねる**」を採用する。

この原則により:

- implementation skill の「コメントの方針」は中身の流儀なので配布既定から外す
- e2e skill の Playwright 固有の流儀（ロールベースセレクタの API 優先順位、日本語ヘルパー関数、GIVEN/WHEN/THEN 構造、テスト独立性の具体手順、仮想時計など）も「テストは仕様書」という哲学ごと配布既定から外し、consumer のプロジェクト側（例: フロントの E2E 流儀スキル）で定義することに委ねる
- 3 つの上書き前提スキル（implementation / e2e / e2e-execution）は「触る範囲から関連する規約・流儀・手順を探して従う」という同型の探索型スキルに揃える

### reviewer subagent はスキルと疎結合にする

e2e-reviewer は「reviewer 自身は規約を持たない。具体の規約は preload された e2e skill から借りる。reviewer の責務は観点を当てること」と自己宣言しているが、本文がその宣言に違反して e2e skill 由来の観点（仕様書性・テスト独立性・ヘルパーの嘘・テスト名連結チェック）をハードコードしている。skill の中身が変わるたびに agent も書き直す状態は、単一の出典の原則に違反する。

reviewer agent は preload された skill（とその探索指示が案内する先のプロジェクト流儀）を読み、**何を点検すべきかの観点をそこから導出する**疎結合な構造に改める。観点の導出はハードコードされたチェック項目より出力が揺れるが、「機械的チェックよりペアプロでの捕捉が筋」という harness の思想と整合する揺れである。reviewer 固有の普遍観点（設計整合・TDD 規律・YAGNI・セキュリティ・Done との双方向照合など、preload skill の中身に由来しない観点）は agent に残す。

## 2. ユーザーストーリー

- **reviewer の疎結合化（順序 1）**: consumer プロジェクトの開発者が、自前で定義したプロジェクト流儀に照らしたレビューを受けるために、reviewer subagent には preload されたスキルとその案内先から点検観点を導出してほしい（スキルの中身を変えても reviewer の定義を書き直さなくて済む状態）
- **スキルの探索型化（順序 2、reviewer の疎結合化の完了が前提）**: consumer プロジェクトの開発者（とその main session）が、front / api など複数領域が同居するプロジェクトでユーザー価値単位のストーリーを 1 セッションで一気通貫に TDD 開発するために、いま触っている範囲に対応するプロジェクト固有の規約・流儀・実行手順をスキルに探させて従わせたい

順序の理由: e2e skill を先に薄型化すると、e2e-reviewer がハードコードしている観点（仕様書性など）の出典が配布物から消え、harness が主張していない流儀でレビューする不整合な中間状態が生まれる。reviewer を疎結合にしてからスキルを薄型化すれば、どの時点でも整合が保たれる。

3 スキル（implementation / e2e / e2e-execution）を 1 ストーリーにまとめる理由: 実装規約だけ探索型になり E2E の流儀が Playwright 固定のまま残ると、領域をまたぐ開発で「探した規約」と「固定の流儀」が衝突する。3 スキルが同型に揃って初めて「触る範囲から規約が効く」価値が成立するため、これ以上の分割は価値を崩す。

## 3. Done（完了基準）

### implementation skill（探索指示の強化）

- 「コメントの方針」section が本文から削除されている
- 本文の探索指示が次の 4 要素を含む:
  - **触る範囲を錨にする**: いま触っているファイルのディレクトリ / 領域を特定し、その範囲に対応する規約スキル / ルールを探す
  - **スキルなら「呼ぶ」**: 探した先がスキルなら（Read で参照するのではなく）スキルとして発火させる
  - **見つからないときの挙動**: 規約が見つからなければ、規約をでっち上げずに一般的な良い設計判断で進む
  - **呼ばれるたびに範囲を特定し直す**: 1 ストーリーの中で触る範囲が領域をまたいで移るため、このスキルが呼ばれるたびに範囲の特定をやり直す

### e2e skill（薄型化）

- Playwright 固有の流儀および「テストは仕様書」哲学が本文から削除され、implementation と同型の探索指示（上記 4 要素を E2E の流儀に置き換えたもの）になっている
- frontmatter description から特定フレームワーク名・API 名・流儀固有の表現が除かれ、薄型化後の責務（E2E spec を書く・編集する・レビューする際に、プロジェクトの E2E 流儀を探して従わせる入口）と整合している
- 削除した Playwright 流儀の本文が、consumer がフロント E2E 流儀スキルを書くときの見本としてリポジトリ内に退避されている（置き場所は基本設計で決める）

### e2e-execution skill（探索指示の強化）

- 本文の探索指示が implementation と同型の 4 要素（対象を E2E の実行手順に置き換えたもの）を含む

### e2e-reviewer subagent（疎結合化）

- e2e skill 由来のハードコード観点（仕様書として読めるか・テスト独立性・ヘルパーの責務と嘘・テスト名連結チェック）が本文・出力フォーマットから削除されている
- 代わりに「preload した skill と、その探索指示が案内する先のプロジェクト流儀を読み、点検観点をそこから導出する」指示が書かれている
- frontmatter description から「仕様書として読めるか」等の流儀固有の表現が除かれている
- reviewer 固有の普遍観点（要件定義の Done と spec の双方向照合、責務範囲、押し付けないルール）は残っている

### code-reviewer subagent（文言調整）

- 「preload された implementation skill でプロジェクトのコード規約・アーキテクチャ方針を知っている」という前提の文言が、「preload した implementation skill の探索指示に従ってプロジェクトの規約を見つけ、それに照らす」という疎結合の表現に調整されている
- reviewer 固有の普遍観点（設計整合・TDD 規律・責務とコード品質・YAGNI・セキュリティ・Refactor 指針）は残っている

### 全体

- 3 スキルの frontmatter description が境界原則（interface には具体ツール名・流儀の中身を入れない）に違反していない
- skill-reviewer による最終レビューを通過している

## 4. スコープ外

- **別セッション + 消費者駆動契約テストによる開発方式の装置化**: 組織境界が実在する consumer が現れたときに翻訳して採用する候補として保留
- **一気通貫開発の実証**: 小さいストーリー 1 本で実際に開発させ、規約の混線・後段でのスキル発火漏れが起きるかを観測する事後評価は、本改修の後続として別途行う
- **別リポジトリ群での配布先問題への対応**: セッションを開く共通の親が git 管理外になるケース（apm の配布先が宙に浮く問題）は痛みが実証されてから検討する
- **consumer 向け導入ガイドの拡充**: 領域別規約スキルの書き方テンプレートや README の大幅な拡充はしない。見本の退避と最小限の参照案内に留める
- **slice-tdd / main instruction の変更**: implementation / e2e を呼ぶ既存の導線は名前が変わらないためそのまま機能する。導線自体の変更はしない
