# 上書き前提スキルの規約探索型への薄型化 — 基本設計

> このドキュメントは進行中の設計メモ。実装後もそのまま残す（振り返り・PR レビューで参照される）。
> 削除のタイミング・要否は依頼者または外側の仕組みで別途判断する（あなたは削除しない）。

## 1. 全体像

探索型化後の構造。slice-tdd（リズム担当）は変わらず、入口スキル 3 つが「触る範囲から探して従う」装置になる。reviewer は preload したスキルから観点を導出する。

```
[ slice-tdd ]──手順ごとに呼ぶ──┬─→ [ implementation ]──探索──┐
                                ├─→ [ e2e ]          ──探索──┤
                                └─→ [ e2e-execution ]──探索──┤
                                                              │
              consumer がセッションルートに登録した           ▼
              ┌──────────────────────────────────────────────────┐
              │ 領域別スキル (例: front 実装規約 / front E2E 流儀) │ → 呼ぶ
              │ スキルでない規約ファイル (領域配下の CLAUDE.md 等) │ → 読む
              └──────────────────────────────────────────────────┘
                                                              ▲
[ code-reviewer ]──preload: slice-tdd + implementation────────┤ (探索指示に従い
[ e2e-reviewer  ]──preload: e2e + slice-tdd───────────────────┘  観点を導出)
```

## 2. 探索節の共通の型（論理設計）

3 スキルに同一構造で展開する。差は対象の語だけ:

| スキル | 対象の語 |
|---|---|
| implementation | 実装の規約・アーキテクチャ方針 |
| e2e | E2E の流儀（spec の書き方・構造・命名） |
| e2e-execution | E2E の実行手順 |

型（`<対象>` を上記の語で置換して展開する）:

```markdown
## 関連する<対象>を探して従う

このスキルが呼ばれるたびに、以下を行う（1 つの作業の中で触る範囲が
別の領域に移ったら、その都度やり直す）:

1. **触る範囲を特定する**: いま読み書きしようとしているファイルの
   ディレクトリ / 領域を特定する（例: front 配下と api 配下で規約が
   分かれている場合、どちらの領域の作業かを特定する）
2. **範囲に対応する<対象>を探す**:
   - セッションに登録されているスキル（このスキル自身を除く）にその範囲の
     <対象>を扱うものがあれば、（参照して読むのではなく）スキルとして呼ぶ
   - スキルでない規約ファイル（その領域のディレクトリ配下の CLAUDE.md
     やドキュメント等）があれば、読む
3. **見つけた<対象>に従って作業する**
4. **見つからなければ、<対象>をでっち上げない**: 一般に良いとされる
   判断で進める
```

設計判断の根拠:

- **時系列の手順型にする**: 4 要素は順序を持つ（範囲を特定しないと探せない、探さないと従えない）。手順として上から流す構造にすることで、規律をその場の注意力でなく構造に乗せる（slice-tdd が手順ごとにこれらのスキルを呼び直すリズムと噛み合い、領域をまたいだ瞬間に振り分けが再評価される）
- **例を 1 つだけ入れる**: この型の存在理由が複数領域対応なので、典型例（front / api）を 1 つ示す。冗長ではないと判断
- **スキルは「呼ぶ」、ファイルは「読む」の二枝**: セッションに読み込まれていないスキルは呼べないため、スキルとして提供される規約はセッションルート登録が前提（要件定義の前提と制約を参照）。スキルでない規約ファイルは読み込み制約を受けないため「読む」枝で拾う

## 3. 各ファイルの変更構成

### .apm/skills/implementation/SKILL.md

- 「コメントの方針」section を削除
- 既存の「関連する規約を探して従う」section を上記の型で置き換え
- 本文構成: 前文（1-2 行、責務の宣言）+ 探索節
- description: 「命名やコメントの方針を判断する」の言及を除去し、変更後の本文の責務（プロジェクト固有の規約・アーキテクチャ方針を探して従わせる入口）と整合させる。文面は skill-design-style の手順 4 に従い実装フェーズで確定

### .apm/skills/e2e/SKILL.md

- Playwright 固有の流儀・「テストは仕様書」哲学・APM 機構言及（last-installed-wins の説明）をすべて削除
- 本文構成: 前文 + 探索節（implementation と同型）
- description: Playwright / test.use / getByRole / page.locator / 仕様書ルールの言及を除去し、抽象表現（E2E spec を書く・編集する・レビューする際に発火、プロジェクトの E2E の流儀を探して従わせる入口）に書き換え

### .apm/skills/e2e-execution/SKILL.md

- 既存の「関連する実行手順を探して従う」section を上記の型で置き換え
- description: 変更後の本文と矛盾しないか実装フェーズで確認（現 description は抽象表現なので大きな変更は不要見込み）

### .apm/agents/e2e-reviewer.md

**残す**（reviewer 固有・preload 由来でないもの）:

- 役割（E2E spec を書いた直後のペアプロ相手）、preload 構成（e2e + slice-tdd のまま）
- 要件定義の Done と spec の双方向照合（カバレッジ / 超過の両方向）
- 責務範囲（実装コードは code-reviewer、要件・設計は pre-implementation-reviewer、read-only、実行しない）
- 押し付けないルール、重要度の使い分け

**置き換える**:

- ハードコード観点（仕様書として読めるか・テスト独立性・ヘルパー責務・テスト名連結チェック）と出力フォーマット内の「テスト名連結チェック」section を削除
- 「呼び出される時点の前提」section の「preload された e2e skill がプロジェクト固有の規約（ロールベースセレクタ、natural language ヘルパー、GIVEN/WHEN/THEN 構造、テスト独立性、ファイル命名・配置など）を持っている」を、疎結合の表現（preload した e2e skill が流儀の探し方を持つ）へ書き換え。「specs/ 配下」というディレクトリ前提も「レビュー対象 spec のパス（git diff または呼び出し時指定で取得）」程度の抽象表現にする
- 「観点の導出」section を新設: **preload した skill を読み、点検観点をそこから導出して当てる。skill が流儀の探し方（探索の案内）を持つ場合は、レビュー対象 spec の触る範囲に対応するプロジェクトの E2E 流儀をその案内に従って探して読み、観点に加える**（探索指示の存在に依存しない文面にする — 疎結合化を先に実装する中間状態では e2e skill がまだ探索指示を持たないため）。指摘時は流儀のどの記述に基づくかを引用する（現行「preload された skill を根拠にする」ルールの一般化）

**流儀が見つからないときの床**（探索節の型の 4 番目の要素と対称）:

- 流儀をでっち上げて「プロジェクト規約違反」と指摘しない
- Done との双方向照合と、一般に良いとされるテストの質の判断だけで観点を立てる

description: 「仕様書として読めるか」の言及を除去し、「preload した skill とその案内先の流儀から観点を導出してレビューする」趣旨に調整。

### .apm/agents/code-reviewer.md

- 前提の文言「preload された `implementation` skill でプロジェクトのコード規約・アーキテクチャ方針を知っている」→「preload した `implementation` skill の探索指示に従ってプロジェクトの規約を見つけ、それに照らす」
- 観点 7「実装規約との整合」の規約列挙（命名・構造・責務分離・コメント方針など）→ 探索で見つけた規約に照らす表現へ
- 規約が見つからないときの床を追加（e2e-reviewer と対称）: 規約をでっち上げて「プロジェクト規約違反」と指摘しない。設計整合・TDD 規律・SOLID 等の普遍観点だけで指摘する
- 上記以外の普遍観点（設計整合・TDD 規律・責務とコード品質・YAGNI・セキュリティ・Refactor 指針）は変更しない

## 4. 削除する Playwright 流儀の退避

- 退避先: **本リポジトリ外** `/home/sei/oss/e2e-playwright-skill/SKILL.md`（スキルの形 = frontmatter + 本文のまま保存）
- 本リポジトリでは管理しない。利用者側リポジトリでの領域別 E2E 流儀スキルとしての定義は依頼者に委ねる
- 退避ファイルは受け渡しの一時置きであり、永続的な管理場所は依頼者が後日決める。本リポジトリの完了条件は退避ファイルの存在まで（なお削除前の本文は本リポジトリの git 履歴からも復元可能）
- リポジトリ内に見本を置く案（examples/ 新設、docs/examples/、配布同梱）は検討の上で不採用: このリポジトリの責務（XP の関心ごとの harness）の外にある内容を管理し続けることになるため
