# 規約探索型スキルへの薄型化 — skill-reviewer レビューの判断記録

> 要件定義の Done「skill-reviewer によるレビューを受け、指摘への対応 / 非対応の判断が記録されている」に対応する記録。
> pre-implementation-reviewer のレビュー（要件段階 1 回 + 要件 + 基本設計セット 1 回）への判断は、依頼者との対話で確定し要件定義.md / 基本設計.md への反映 commit に記録済み。

## skill-reviewer の指摘と判断（2026-06-07）

### 指摘: reviewer の動詞が「読む」で、探索節の「呼ぶ」と割れている → 対応（「呼ぶ」に揃えた）

subagent が Skill tool で名指しスキルを発火できることは検証済み（skill-design-style 原則 4）で、e2e-reviewer / code-reviewer はどちらも tools に Skill を持つ。よって「探した先がスキルなら呼ぶ、スキルでないファイルなら読む」の二枝を reviewer 側にも展開し、探索節と動詞を揃えた。補助ファイル取りこぼしのリスクも「呼ぶ」で解消される。

### 指摘: e2e-reviewer に `src/ 配下` が残存し ROADMAP の TODO 7 完了注記と食い違う → 対応（除去した）

「実装コード（src/ 配下）はレビューしない」から括弧の例示を外し、「実装コードはレビューしない」とした。プロジェクト構成に依存しない表現になり、TODO 7 完了注記（プロジェクト固有値を除去）と実体が一致した。

### 指摘: e2e-execution の探索節手順 1 が「差は対象の語だけ」を厳密に満たさない → 非対応（意図的な逸脱として基本設計に注記）

E2E の実行はファイルの読み書きではないため、「いま読み書きしようとしているファイルの」を機械的に展開すると不自然になる。「いま動かそうとしている E2E の」への調整は対象の性質に合わせた妥当な逸脱と判断し、基本設計の完了の定義に注記を追加して設計と実装の主張を一致させた。

### 指摘: 「設計 md の Done」と「要件定義の Done」の用語ずれ → 対応（「要件定義の Done」に統一）

Done の在処は要件定義.md なので、e2e-reviewer の既存 section（見出し・本文・振る舞いのルール）を「要件定義の Done」「Done と spec の双方向照合」に統一した。

### 将来判断として保留: e2e skill の description が slice-tdd を名指している件

発火タイミング（When）の表現であり、done-verifier が兄弟 reviewer を名指す既存パターンとも揃っているため今回は対応しない。interface 純化の棚卸しの機会に見る。
