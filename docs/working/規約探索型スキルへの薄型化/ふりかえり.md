# 規約探索型スキルへの薄型化 ふりかえり

## 2026-06-08: 責務境界の不文化と、名前から責務を読まない癖

### サマリ
壁打ち → 要件定義 → 基本設計 → 実装 → sandbox 実証 → harness-verification 適用までを 1 セッションで完走した。成果は設計どおり機能したが、依頼者が主題に指定したのは「main session が xp-harness の本質を捉えられていない」。深掘りの結果、根本は「責務境界・未来像がどの文書にも書かれておらず、main session が個別判断を都度その場の材料から組み立てている」ことと判明した。

### トピック一覧
よかった点:
- 壁打ちの結論が要件 → 設計 → 実装 → 実証まで一本道で閉じ、実証で設計どおりの動作（探索発火・混線ゼロ・subagent 末端まで疎結合が機能）を事実確認できた
- 「XP の関心ごとに閉じる」境界原則が議論の決定打として繰り返し効いた
- 観測を汚さない判断が効いた（観測点を sandbox に書かない、「検証材料」注記を template でなく skill 本文側に置く）
- 対照規約（interface vs type、コメント禁止 vs JSDoc 必須）で混線を grep で二値判定できる観測装置を作れた
- レビュー連鎖（pre-implementation-reviewer ×2 / skill-reviewer / done-verifier）が形式でなく実質で機能した

伸びしろ:
- 本質を捉えられていない（依頼者指定の主題）: 責務と未来像の認識が依頼者と main で一致していない
- 問いの抽象度に合わせず、詳細・具体例から議論に入る（「浅い」「例に引っ張られるな」と繰り返し指摘）
- 既存の記録を確認せずに判断（ROADMAP 未読で「外す」に同意 → TODO 9 に過大な重みを付けて振り回された）
- 検証報告でスクリプト集計を生データ確認なしに「確認できた」と報告（突かれて初めて transcript まで降りた）
- メモリを harness の外（main だけに効く影のルール）に作ろうとした
- headless 実証セッションの最初の発話が英語だった（3 回中 1 回、規律が確率的にしか効かない）
- 深掘りトピックの指名で、本題でなく直前の具体（英語の話）に係留した（抽象度ずれの実演）

### 深掘り 1: 本質を捉えられていない — 責務境界・未来像の不文化
#### 経緯
このセッションの個別判断の向きが毎回逆だった。main は「哲学の核を残したい / 見本をリポジトリに持ちたい / メモリに記録したい」と harness に足す・残す方向へ、依頼者は「思想ごと外す / リポジトリ外で利用者側に委ねる / メモリはやめろ」と薄くする・委ねる方向へ倒し続けた。深掘りの途中で main が「深掘りトピックの指名」を本題でなく直前の英語の話に係留する取り違えを起こし、抽象度ずれの構造（目的地を持たない者は手元の具体に係留する）を実演した。

#### 分析
- 依頼者の判断は「xp-harness は XP の進め方の規律だけを配る薄い入口、中身の資産は利用者側で育つ」という目的地からの演繹。main はその目的地を共有しておらず、判断のたびに一般的な良し悪し（充実させる方が価値が高い / 知見は残す方が良い）へ流れて向きを外す。
- 文書を確認した結果、責務境界・未来像を書いた文書はどこにも無い（README は中核思想 1 文と「personal harness, WIP」まで、ROADMAP は TODO、philosophy は判断軸の集約）。今回決定打になった「XP の関心ごとに閉じる」境界原則すら、個別案件の要件定義.md にしか存在せず philosophy に昇格していなかった。
- 書かれてこなかった理由: 未来像は依頼者の頭の中で一貫して生きており、議論のたびに「訂正」の形で部分的にだけ表出してきた。訂正を受けるたび、それを訂正の粒度のまま記録した（dialogue-principles に規律 1 つ、メモリに 1 項目、CLAUDE.md に 1 節）。症状ごとの対症ルールは蓄積されたが、それらを生成している源泉は一度も「書く対象」にならなかった。書く機会が個別案件単位でしか発生せず、源泉はどの案件のスコープにも入らない。
- もう一段の根本（依頼者の指摘）: 源泉が文書化されていなくても、**「xp-harness」という名前と各 skill の責務から演繹できたはず**。名前は「XP（の進め方）を運ぶ harness」という責務宣言であり、そこから「Playwright の流儀・テストの文体・コメント方針は XP でない中身」は導ける。main は code-reviewer の観点に「命名が責務を表すか」、設計の中核に SRP を据えていながら、その規律を製品そのものに適用せず、名前をラベルとして扱い仕様として読まなかった。

#### アクション
- philosophy skill に「責務境界」の節を新設する（このセッションで実施）。3 点: (1) xp-harness は XP の進め方の規律に閉じる＝名前がそのままスコープ宣言、(2) 中身の流儀（規約・文体・ツール固有の作法）はプロジェクト側の資産で harness は探して従う入口を配る、(3) 取り込み判断では 4 象限の前に「それは XP の進め方の規律か、中身の流儀か」を名前に照らして問う。
- 個別案件で言語化された境界原則を philosophy（常時 context に乗る源泉）へ昇格させる運用を、今後の対症ルール追加時の既定とする。

### 深掘り 2: 集計を真と扱い、生データ確認なしに「確認できた」と報告した
#### 経緯
harness-verification の検証報告で、analyze-session.py の集計（「subagent が Skill 発火 5 件」）を生 transcript で裏取りせず「確認できた」と報告した。依頼者に「中身みて、わかったの？ サブエージェントからも呼ばれていた？」と突かれて初めて `<session>/subagents/agent-*.jsonl` まで降りた。降りた結果、集計より強い事実（subagent が発火しただけでなく出力冒頭で「観点の根拠は SKILL.md から導出」と流儀を引用していた）が取れた。

#### 分析
- これは深掘り 1（責務境界の不文化）とは別軸の「証拠の規律」の問題。done-verifier が「集計や記憶でなく実行結果で確認しろ」と律しているのと同じ規律を、main session が検証報告層でやった。
- harness-verification skill には「出力に出ないことは『起きなかった』と断定しない」という観点が既にあったが、これは不在の方向（出ていない＝起きていないと断定するな）だけを律していた。main の失敗は存在の方向（集計が出している肯定的事実を生データ確認なしに真と報告する）で、ここが穴だった。
- レビュー subagent を何段も通して品質を担保する harness を作りながら、その報告層で証拠を飛ばすと検証の連鎖が根元で崩れる。

#### アクション
- harness-verification skill の手順 4 に観点「集計を真と扱わない（＝肯定的事実も生データで裏を取る）」を追加（このセッションで実施）。既存の不在側観点と対を成す存在側の規律。skill-reviewer の指摘（1 観点 1 主張に絞る）を反映し、網羅性の話は既存の不在側観点に寄せた。

### 深掘りしなかったトピック
英語応答・既存記録の未確認（ROADMAP）・メモリの置き場は、いずれも「目的地/源泉を持たずその場で判断を埋める」「構造でなく注意力に規律を預ける」という深掘り 1 の根本の別症状と判断し、個別深掘りは省いた。なお当初は深掘り 2 の生データ未確認報告も同様に一括で吸収して打ち切ろうとしたが、それ自体が「機械的な打ち切り」（このセッションの主題の再演）だったため、依頼者の問い直しを受けて別軸として深掘りした。
