# ROADMAP のかんばん化 — 要件定義

> このドキュメントは進行中の設計メモ。実装後もそのまま残す（振り返り・PR レビューで参照される）。
> 削除のタイミング・要否は依頼者または外側の仕組みで別途判断する（あなたは削除しない）。

## 1. 背景と動機

`ROADMAP.md` が 1 ファイル集約で肥大化した（約 1,200 行 / 97KB、TODO 57 件 + 欠番記録）。肥大化により次の痛みが出ている:

- **状況確認しづらい**: 「次なにやるか」を判断するとき、巨大ファイルをスクロールして tier1 や生きてる項目を拾い出す必要がある
- **編集衝突しやすい**: 1 ファイルなので、複数の改修を並行すると同じファイルを取り合う
- **機械的な集計ができない**: ステータス・優先度別の俯瞰がツールで出せない

単にファイルを小さくするだけ（完了分のアーカイブ分離等）でも肥大化は緩むが、「状況確認できるようにしたい」というゴールには **1 項目 1 ファイルのかんばん化 + 状況表示ツール** の方が直接効く（一覧性・並行編集・機械集計が同時に得られる）。

## 2. ユーザーストーリー

- 改修者が、次にやることを素早く判断するために、改修バックログをかんばん（1 項目 1 カード）で一覧・更新したい

## 3. Done（完了基準）

- `ROADMAP.md` の各項目が `kanban/` ディレクトリのカード（1 項目 1 ファイル）に分割されている
- カードのファイル名から、ステータス（TODO / DONE）と優先度（tier1 / backlog）が一目で分かる
- 状況確認スクリプトを引数なしで叩くと tier1 の TODO 一覧が出る。`todo` で TODO 全体、`done` で DONE 一覧が出る
- `kanban` skill が「次なにやるか」「これ完了にして」のような改修バックログの話題で発火し、規約に沿ってカードを確認・操作できる
- 既存の番号参照（`release` skill の「ROADMAP TODO 38」、README の「ROADMAP #3」等）がタイトルベースの参照に貼り替えられ、リンク切れがない

## 4. スコープ外

- **tier2 / tier3 の lane**: 今 Tier の実体があるのは tier1 のみ。tier2/3 は実際に振り分けたくなったとき足す（無い lane を先に作らない）
- **フィルタ機能**（特定 Tier だけ・特定対象だけ等）: 欲しくなってから足す
- **カード変更の自動化をスクリプトに持たせること**: スクリプトは読み取り専用。カードの追加・完了・Tier 変更は改修者（エージェント）が git 操作で透過的に行う
- **カード間の相互参照・安定 ID**: かんばんはカード独立が基本。ID で依存させない
- **consumer への配布**: `kanban` skill / スクリプトは改修バックログ管理の内部ツール。XP の進め方の規律ではないので配布しない（`.claude/` 直コミット、`.apm/` には置かない）
