# AgentTeam調査スパイク ふりかえり

## 2026-07-17: 調査スパイクの進め方と、ふりかえり自体の規律

### サマリ

AgentTeam 調査スパイク (2026-07-15〜16、調査記録は同ディレクトリ) を終えてのふりかえり。「調査という作業クラスの扱い」と「ふりかえり中の裏取り規律の崩れ」の 2 本を深掘りした。

### トピック一覧

よかった点:

- カード着手時にフェーズゲートが機能した (git-workflow → kanban → define-requirements のモードゲートの順で通り、「カードは要件確定ではない」が効いた)
- harness-verification の「集計を真と扱わない」が 3 回機能し、検証セッションの自己報告と実態のズレ (通常サブエージェントへの落ち・チーム参加の実態・並行検証の担当がチーム外だった件) をすべて生 transcript で捕捉した
- 実走前の条件レビュー関門 (ノーヒント / 反証可能性 / 依頼者への条件提示) を毎回通した
- 検証で出た偶然の発見 (組み込み worktree 分離、ポート競合の自力回避) を流さず、導入カードの設計論点に書き戻した

伸びしろ:

- 完了を 2 回早く宣言した (スパイクの Done を依頼者と観測可能な形で固めず、カードの 3 つの問い = Done と固定した)
- 用語の日本語化が 2 回崩れた (teammate / lead を固有名詞だから残してよいと自己判定したが、依頼者に伝わらない語彙だった)
- 構図の認識合わせが後手に回った (依頼者側から 2 回「伝わってる？」の確認が必要だった)
- かんばんカードの扱いを 3 回打診して依頼者のペースを急かした
- 調査結果の引き継ぎがフローに組み込まれていなかった (→ 深掘り 1)
- ふりかえり中に裏取りせず「規定が無い」と断言した (→ 深掘り 2)

### 深掘り 1: 調査フローに引き継ぎが組み込まれていなかった

#### 経緯

調査記録は会話の時系列でひたすら追記され (チャット報告の控え扱い)、導入カードも最後の事後処理として書かれた。引き継ぎの点検 (handoff-verifier) は依頼者に促されて初めて実施した (判定は Yes だったが、用語の割れ・スコープのすれ違いという会話文脈の漏れが指摘された)。

#### 分析

- スパイクはストーリーと成果物の性質が違う。普通のストーリーは成果物がコードで、下流に渡すのは着手に足りる最小情報 (ゴールと完了条件) でよい。スパイクは**成果物が情報そのもの**で、次工程の入力に引き継がれて初めて価値になる。だからスパイクの Done は「問いに答えた」ではなく「**答えが次工程の入力として使える形で引き継がれている**」まで含む
- main session はスパイクをストーリーと同型に扱い、作業の性質 (成果物が何で、誰の入力になるか) から Done とフローを立てる判断が抜けた
- 取り決め側の背景: 要件定義・基本設計・実装にはフェーズ skill があるが、調査/スパイクという作業クラスには持ち家が無い。handoff-docs の description の一般条項 (「別セッションへ渡して作業を継続させる成果物」) は調査記録をカバーしていたが、発火しなかった
- 検討して棄却した案: handoff-docs の description の例示に「調査記録」を足す。例示リストの増殖は「例示に無いものは対象外」という読み方を強化するので筋が悪い

#### アクション

- **調査/スパイク skill の新設**をかんばんカードに切る。skill が持つべきもの (問いの立て方、Done の型 = 知見の引き継ぎまで、調査記録の書式、handoff-docs との接続) はカードに論点として残し、中身は要件定義フェーズで固める

### 深掘り 2: ふりかえり自体の裏取り規律が崩れた

#### 経緯

深掘り 1 の途中、main session が handoff-docs の description を確認せずに「取り決めにどこにも規定が無い」を「事実」ラベルで出した (自分の発火失敗を問わずに済む、都合のいい因果)。依頼者の「skill の description に書いてない？」で発覚した。

#### 分析

- 正直さの規律は部分的に効いていた (失敗を隠さない / 事実と推測のラベリング / 「一本道すぎる」の自己疑義まではやった)。しかし**本人が事実と思い込んでいる誤りは、意図の正直さでは検出できない**。自己弁護の引力は出力の段階で嘘をつかせるのではなく、上流の「何が事実に感じられるか」を歪める
- retrospective skill はこの構造を織り込み済みで、機械的な裏取り動作 (「自分の語りを信じず記録で裏を取る」) を対抗装置として持っている。故障点はその**発火条件の狭さ**: 「発火・タイミングなど挙動の主張」と例示されており、「取り決めの内容・有無」という主張クラスを覆っていなかった (例示照合で一般条項を取りこぼす、深掘り 1 と同じ失敗の形)
- context に常時載っている description は**既読感** (確認した気分) を作るが、長いセッションでは注意が薄れており「確認した」の代わりにならない。実際このセッション内で対照が取れた: Read したものへの主張は正確、載っているだけのものへの主張が外れた。確認は「ファイル実体を新鮮に読み直す動作」でだけ成立する
- 裏取りの手段 (Read / grep) は既にあり、欠けていたのは発火。なおスクリプト化は「規律の矯正装置」ではなく「**何度もやる作業の手助けの道具**」として評価する (board.sh と同型)。ふりかえりの観測 (「利用可能だった skill を一通り当てる」) は毎回やる反復作業なので、道具の対象として最初から正当。ただしクロスエージェント対応を考えると skill の配置パスに依存する難しさがあり、スクリプトの置き方 (skill 同梱か、道具非依存の指示に留めるか) は設計論点

#### アクション

- **retrospective skill の裏取り条項の拡張**をかんばんカードに切る: 裏取り対象を「改善の判断に効く主張全般 (取り決めの内容・有無を含む)」に広げ、「『無い』という否定主張は context の記憶でなくファイル実体を Read / grep してから事実として出す」を明記する。あわせて観測の「skill を一通り当てる」を支える一覧スクリプト (board.sh 型) を検討する (クロスエージェントの配置パス問題が設計論点)
