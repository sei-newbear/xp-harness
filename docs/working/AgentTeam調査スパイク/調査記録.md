# AgentTeam 調査スパイク — 調査記録

> このドキュメントは調査スパイクの記録。実装後もそのまま残す (振り返り・導入判断で参照される)。
> 削除のタイミング・要否は依頼者または外側の仕組みで別途判断する。

実施日: 2026-07-15〜16。対象: Claude Code v2.1.209 の実験的機能 Agent Teams (`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS`)。

用語の対応 (本記録は検証の時系列で呼称が揺れているため、ここで固定する):

- **親セッション** = lead = トップ (依頼者と対話し、ストーリーを分割・委任する側)
- **チームメンバー** = teammate (チームに参加する部下エージェント)。本記録の「世話役」「ストーリー担当」「担当エージェント」は、ストーリーを 1 本任される役割の呼び名で、チームメンバー形態でも通常サブエージェント形態でもありうる
- **サブエージェント** = subagent (従来からある使い捨ての部下。レビュアー等)

## 目的とスコープ

かんばんカード「AgentTeam 導入検討」の tier1 スコープ (= 調査スパイク限定) を実施した記録。潰す問いは 3 つ:

1. teammate モードの実際の挙動 (lead 固定 / ネスト不可 / 1 セッション 1 チームの制限が xp-harness の使い方に効くか)
2. 既存 agent 定義を teammate role で再利用したとき `skills:` プリロードが無効になる問題の回避策があるか
3. 「subagent はペアプロ相手」哲学と teammate モードが両立するか (worker 模式に倒れないか)

本格導入・試験導入の判断はこのスパイクの範囲外 (スパイク結果を踏まえて philosophy の 4 象限で再判定する)。

## 調査方法

- 公式ドキュメント調査: code.claude.com/docs の agent-teams / subagents / changelog (v2.1.178〜v2.1.209)
- 実地検証: harness-verification skill の流儀に沿い、最小構成 sandbox (`xp-harness-test/agent-team-spike`、xp-harness を apm install 済 + 実験フラグを sandbox の `.claude/settings.json` の `env` に設定) で interactive セッション 3 本を claude-launcher.sh で自駆動し、transcript (メインセッションの jsonl と `subagents/agent-*.jsonl`、`~/.claude/teams/` のチーム設定) で裏取りした
- プリロード検証には harness-verification のプローブ一式 (skill-probe subagent + probe-preloaded / probe-target skill、合言葉 `PROBE_TARGET_FIRED_OK`) を使用

## 機構の全体像 (実地で確定した事実)

- チームはフラグ有効時にセッション起動と同時に自動作成される (`~/.claude/teams/session-<id>/config.json`、初期メンバーは team-lead のみ)。チーム作成の専用ツールは v2.1.178 で削除済み
- **teammate の正体は Agent tool の `name` パラメータ**。`name` を付けて spawn するとチームに参加する teammate になり、外すと従来の通常 subagent になる。両者は同一セッション内で共存できる
- teammate の実態は transcript の meta で判別できる: teammate は `"taskKind": "in_process_teammate"` + `"teamName"` を持ち、通常 subagent はどちらも持たない (`spawnDepth: 1` のみ)
- teammate 間通信は SendMessage tool (deferred 状態のため各 agent が ToolSearch でスキーマを読み込んでから使う)。メールボックスは `~/.claude/teams/<team>/inboxes/*.json`
- チーム設定・メールボックスはセッション終了時に掃除される。teammate の transcript は `~/.claude/projects/<project>/<session-id>/subagents/agent-a<名前>-*.jsonl` に残る

## 問い 1: 制約の実際 — 3 つとも実在し、回避不能

- **lead 固定**: 公式ドキュメントに「メインセッションがセッション生存中ずっと lead。昇格・移譲不可」と明記。lead の権限を弱める仕組み (委譲モード等) は存在しない。タスク割り当て・計画承認・終了要求は lead が握る
- **ネスト不可**: teammate に「更に teammate を参加させて」と頼む実験で、プラットフォームの実エラーを確認:
  > Teammates cannot spawn other teammates — the team roster is flat. To spawn a subagent instead, omit the `name` parameter.
  ただし teammate が名前なしの通常 subagent を配下に持つことは可能 (エラーメッセージ自身が案内している)
- **1 セッション 1 チーム**: 別名のチームを作らせる実験で、チーム名の指定は無視され、spawn した teammate は既存の単一チームに合流することを確認

xp-harness の使い方への影響: 「ユーザーストーリーごとにチームを作る」構想は 1 ストーリー = 1 セッションで運用する限り 1 セッション 1 チーム制限と衝突しない。lead 固定は次の「問い 3」の通り運用でしか弱められない。

## 問い 2: プリロード無効の回避策 — あり (ただし罠つき)

対照実験の結果 (同一の skill-probe agent 定義を 2 つの形態で spawn):

| 形態 | probe-preloaded の扱い | 観測事実 |
|---|---|---|
| 通常 subagent (`name` なし) | プリロード適用 | Skill 発火は probe-target の 1 件のみ (= preload 済みなので発火不要)。合言葉出力 |
| teammate (`name` あり) | **プリロード非適用** | **probe-preloaded を Skill tool で自分から発火**してから probe-target を発火。合言葉出力 |

確定した事実:

- 公式明記どおり、`skills:` (と `mcpServers:`) frontmatter は teammate では適用されない。一方 `model:` は適用される (meta の `"model": "sonnet"` で確認)。permission mode は lead から継承
- **回避策は有効**: teammate は project の skill を Skill tool で発火できる (通常セッションと同じ扱い)。今回は spawn プロンプトに skill 名が無くても、skill の description と依頼文の合致から teammate が自力で正しい skill を発火した。ただし自力発火は保証がないので、**確実にやるなら spawn プロンプトで skill を名指しして発火させる**
- **罠 (実観測)**: チーム有効セッションでは、lead が「Task ツールの subagent として起動して」と明示指示されても `name` 付き spawn (= teammate) に倒れることがあった。teammate に倒れるとプリロードが黙って落ちる。xp-harness の reviewer 系 subagent (code-reviewer / e2e-reviewer 等) はプリロード前提なので、**チーム有効環境では「規律注入が黙って消える」事故がありうる**

## 問い 3: ペアプロ哲学との両立 — 能力はある、デフォルトは worker 模式に倒れる

teammate 2 人に運用方針の議論・合意・文書化を頼む実験:

- **デフォルト (通し方を指定しない)**: lead がラウンド制の議事進行を設計し、両 teammate の主張を lead が中継 (teammate 同士の直接メッセージはゼロ)、最終文書も lead 自身が Write した。議論の中身は本物 (両者が論点で譲歩・立場交代する場面もあった) だが、**構造は完全に lead 中心のハブ型 = worker 模式**
- **「lead を介さず直接議論して」と明示した追試**: teammate A→B、B→A の双方向の直接メッセージが記録され、2 人で合意してから片方が lead に結果だけ報告した。**対等な横の対話は機構として機能する**

含意: philosophy の「lead 固定 = 中央集権感が強い → ペアプロ的に弱める方向で使う」は**運用 (規律注入) 次第で成立する**。ただし放置すると重力は worker 模式に向かう。規律の注入経路はプリロードが使えないため、spawn プロンプト・CLAUDE.md・project の skill (teammate が Skill tool で発火) のいずれかになる。

## 痛み (実地で観測した運用上の摩擦)

- teammate の素の text 出力は lead に届かない。SendMessage を使わない teammate に lead が「返信は必ず SendMessage で」と教育する場面が実際に発生した
- メッセージの未達・再送が発生した (公式の既知問題「タスク完了が記録されず後続が詰まる」と同系)
- teammate は依頼完了後も常駐する。放置すると待機し続けるため、shutdown の管理が lead (= 運用) 側に発生する
- SendMessage は deferred tool のため、各 agent が使う前に ToolSearch でのスキーマ読み込みが 1 手挟まる
- トークン消費は大きい (公式明記のとおり teammate 数にほぼ線形。今回の 3 セッションで teammate/subagent 計 7 体、teammate 1 体あたり 2 万トークン超も観測)
- セッション再開 (`/resume`) で teammate は復元されない (公式の既知制限)

## 追試: teammate 配下の subagent にはプリロードが効く (2026-07-15)

依頼者のイメージ (= ユーザーと対話したセッションがストーリーを分割し、分割されたストーリーを別の「チーム」= 世話役エージェントとその配下の subagent 群が担う。チームの数は動的に増やしたい) を機構に写像すると、「1 つのチーム」は Agent Teams の用語では「teammate 1 人 + その配下の subagent 群」に対応する。この構図の成立条件「teammate が spawn した subagent に `skills:` プリロードが効くか」を追試した。

結果: **効く**。teammate に skill-probe を名前なし subagent として起動させたところ、subagent の transcript には probe-target の発火 1 件のみが記録された (= probe-preloaded はプリロード済みで発火不要。meta は `parentAgentId` が teammate、`spawnDepth: 1`、taskKind なし = 通常 subagent)。teammate 自身のプリロード非適用 (前述の対照実験) と合わせ、プリロードの生死は「teammate か subagent か」だけで決まり、spawn 元がチーム内かどうかには依らない。

これにより依頼者イメージの構図は次の形で成立する:

```
ユーザーと対話するセッション (= lead。ストーリー分割と割り振り)
  ├─ teammate A (= ストーリー 1 の世話役。lead・他の世話役と直接対話できる)
  │    └─ 配下の通常 subagent (実装・レビュー等。プリロード = 規律注入が効く)
  ├─ teammate B (= ストーリー 2 の世話役)
  │    └─ ...
  └─ ストーリーが増えたら teammate を動的に追加 (数の上限なし、コストは線形増)
```

制限は「階層は lead → 世話役 → subagent の 2 段まで」(世話役が更に teammate を持つことは不可) の 1 点。lead 固定はこの構図では痛くない (ユーザーと対話するセッションこそが分割と割り振りの主体なので、移譲の欲求が生じない)。

依頼者と認識合わせした結果、この構図は xp-harness のモード境界とも噛み合う:

- 世話役には依頼者への質問ツール (AskUserQuestion) が無い = 依頼者と対話できない。しかし構想では「ストーリーを切るのはトップ (親セッション)」であり、世話役に渡るのはゴールが固まった自走モードの仕事だけ。対話モードの仕事 (要件定義・基本設計) は親セッションに残る — 「対話で固めた前提を持って自走する」という philosophy のモードがそのまま機構に写る
- チーム全員が同じ作業ディレクトリを共有するため、並行ストーリーには worktree 分離が要る。これは git-workflow skill が既に「worktree を切って移動してから作業する」を規律化しているので、新しい発明は不要
- ~~未検証の留保: 「世話役が自分で worktree を切って移動する」動作は実測していない~~ → 追試で実測済み (次項)

## 追試 2: 世話役は自分で worktree を切って移動できる (2026-07-15)

「git-workflow skill の worktree 手順に従って worktree を切り、その中で README に追記して commit する」というゴールを世話役に自走で任せる実験。transcript で観測した時系列:

1. 世話役が **git-workflow skill を Skill tool で自分から発火** (指示文で skill 名を案内)
2. skill の手順どおりセッション開始時の同期確認 (fetch / branch / status / worktree list) を実施
3. `git worktree add -b docs/readme-sandbox-note ../<リポジトリ名>-worktrees/<ディレクトリ名>` — 規約どおりの外部配置・branch 命名
4. **EnterWorktree (path 指定) で worktree へ移動成功**。ただし `.claude/worktrees/` の外への移動は「permission-root relocation」の許可プロンプトになり、**lead セッションの画面に確認が出る** (auto mode でも自動承認されない)。人間 (またはそれを見ている親セッションの操作者) が承認して初めて通る
5. worktree 内で Write / `git -C` により追記・commit を完遂 (世話役の Bash は呼び出し間で cwd がリセットされる挙動があり、世話役は絶対パスと `git -C` で自力回避した)
6. worktree の絶対パス・branch 名・作業方法を SendMessage で lead に報告

結論: **依頼者イメージの構図の最後のつなぎ目 (世話役の worktree 分離) は成立する**。運用上の注意 2 点:

- worktree 移動の許可プロンプトが親セッション側に都度出る。ストーリー並行数が増えると承認の手間が乗る (permissions 設定での事前許可を検討する余地)
- 世話役が worktree に移動すると、その世話役の transcript の保存先プロジェクトディレクトリも worktree 側に付け替わる。transcript 解析 (harness-verification) は worktree 側のプロジェクトディレクトリを見る必要がある

検証で作られた worktree / branch は削除済み (sandbox は baseline のまま)。

なお同追試で、teammate が同じ subagent を 2 回起動する挙動を観測した (1 回目の結果を取りこぼして再実行したと見られる)。teammate 経由の多段委譲は往復の取りこぼし・コスト増の摩擦が乗ることは織り込んでおく。

## 追試 3: ストーリー 1 本の通し検証 — 「このセッションと同じ構成」が世話役の中で再現された (2026-07-16)

アプリ付き sandbox (`xp-harness-test/agent-team-story`、front: Vite + React / api: Hono / e2e: Playwright、xp-harness deploy 済) に小ストーリー「API 稼働状態表示」の要件定義・基本設計を仕込み、lead に「teammate を 1 人チームに参加させてゴールを渡し、自走で任せて」とだけ依頼する通し検証。ノーヒント (md にもプロンプトにも worktree / TDD / レビュアー / skill の言及なし。規律は deploy 済みの CLAUDE.md / skill 側から発火するかを見る)。

結果: **世話役 (Haru) の中に、通常のメインセッションと同じ規律構成がほぼ完全に再現された**。transcript の事実:

- 世話役の Skill 発火 6 件: git-workflow / slice-tdd / front-implementation / e2e / e2e-playwright-front / e2e-execution (フェーズ skill + 探索型 skill の連鎖が全部通った)
- 世話役が起動した subagent 4 体: code-reviewer × 2 (TDD サイクル毎) / e2e-reviewer (spec 直後) / done-verifier (完了報告前)。**reviewer へのプリロード注入も実在を確認** (code-reviewer の transcript 先頭に slice-tdd 本文、e2e-reviewer に e2e 本文が注入されていた)
- TDD リズムの実証拠: Playwright 実行結果が「1 failed → 1 passed → 2 passed → 3 passed」と推移 (レッド → グリーンの反復、最終 3 spec 全通過)。commit は正常系・異常系の 2 スライスで刻まれた
- 「止まる」判断も再現: 世話役は「front に unit test 層を持たせるか」をプロジェクト構造レベルの判断として自己判断せず、lead 経由で依頼者に相談として上げた (対話と自走の境界がチーム越しに機能)
- 完了フローも再現: 統合方法 (main へ統合 / push / 保留 / PR) を依頼者への確認として上げてから閉じた

逸脱・注意点:

- 世話役は worktree でなく **branch 運用を選択** (lead が委任時に「master 直でなく branch で」と添えたため)。git-workflow の worktree 推奨は AskUserQuestion での確認が前提で、世話役は依頼者に聞けないため軽量側に倒れた。並行ストーリー運用では「worktree で」を委任時に明示する運用になる (worktree 自体が動くことは追試 2 で実証済み)
- code-reviewer の 1 体が front-implementation skill を Skill 発火でなく Read で済ませた (既知の失敗モード。チーム機構固有ではなく通常セッションでも起きる)
- lead 自身は skill を 1 件も発火しなかった (純粋な調整役に徹した)。トップセッション側の規律発火は今回の検証対象外

## 追試 4: 複数ストーリーの並行通し検証 — 実ユースケースの再現 (2026-07-16)

依頼者の実ユースケース (= トップが分割した複数ストーリーを、複数の担当エージェントが並行でやる) を再現する検証。アプリ付き sandbox に 1 機能・ユーザーストーリー 2 本 (「サーバー時刻表示」「バージョン表示」、両方 `api/src/index.ts` と `front/src/App.tsx` に触る = 統合時に conflict が起きうる現実的構成) の要件定義・基本設計を仕込み、lead に「ストーリーごとに teammate を 1 人ずつ、worktree で分離して並行で」とだけ依頼した。

結果: **並行実行・規律再現・統合まで通しで成立**。transcript とファイルシステムの事実:

- 2 体が別 worktree・別 branch で同時に走り、互いの作業を踏まなかった。両ストーリーとも規律構成 (git-workflow / slice-tdd / 領域別実装 skill / e2e 系の発火、code-reviewer・e2e-reviewer・done-verifier の起動、reviewer へのプリロード注入) が再現された
- **予想したポート競合は実際に問題化し、エージェント自身が検知・回避した**: この sandbox の Playwright は固定ポート (5173/3001) + 起動済みサーバー再利用 (`reuseExistingServer: true`) のため、並行 E2E は「相手の worktree の dev サーバーで自分の spec を走らせる」取り違えが起きうる。担当エージェントは実行前にポート空きを確認 (`lsof -i :3001 -i :5173` 等) してから E2E を走らせ、「CI 化するなら worktree ごとのポート分離を検討」と申し送りまで残した。**並行運用の実痛みとして「E2E 実行環境の分離 (ポート等)」を設計課題に積む**
- 設計の穴 (api の package.json に version フィールドが無い) を担当エージェントが発見し、値の暫定対応 + バージョン運用方針を依頼者判断として申し送り (「止まる」判断の再現)
- 統合フェーズ: lead が 2 branch を順に master へ統合し、予告どおりの conflict (route 登録の並記 / front の state・effect の結合) を解消、統合後に E2E 全件 (3 spec) を再実行して全通過を確認した

機構面の重要な発見 2 つ:

- **lead は 2 体を teammate ではなく「名前付き + `isolation: "worktree"` の通常 subagent」として起動した** (meta に teamName 無し = チーム機構を使っていない)。つまり並行ストーリー実行だけなら、Agent tool の組み込み分離オプションで足りる。チーム機構が固有に足すのは「担当同士の直接対話」「共有タスクリスト」「常駐 (完了後も残って追加指示を受けられる)」の部分
- **「subagent の中ではハーネスの skill / subagent が使えないのでは」という懸念は、実測では否定された**: ストーリー担当 2 体は lead の通常 subagent (深さ 1) でありながら、skill を 6〜7 件発火し、配下の subagent (深さ 2) として reviewer 系を Agent tool で起動できていた (プリロード注入も実在)。効くかどうかの分かれ目は「subagent か否か」ではなく **agent 定義のツール制限** — ストーリー担当が汎用型 (Skill / Agent を含む全ツール持ち) だったから全部効いた。reviewer のような制限付き定義 (Agent tool なし) で担当を立てると配下のレビュアーを呼べなくなるため、**「ストーリー担当は汎用型で立てる」が運用ルール**になる。
  - 「subagent は subagent を呼べない」という従来の制約は**実在した**が、**v2.1.172 で公式に解除された** (changelog: "Sub-agents can now spawn their own sub-agents (up to 5 levels deep)")。深さ上限は 5 段で固定 (設定不可)。subagent 内の Skill tool 使用も公式ドキュメントに明記あり。どちらも agent teams の実験フラグとは無関係の正式仕様 (出典: code.claude.com/docs の sub-agents / changelog)。なお「teammate が teammate を作れない」(チームのネスト不可) は別の制約で、こちらは現在も有効
- 組み込みの worktree 分離は repo 内 `.claude/worktrees/` に自動作成され、**許可プロンプトが出ない** (追試 2 の外部 worktree + EnterWorktree 経路では都度プロンプトが出たのと対照的)。git-workflow skill は「自動作成機構は使わない (repo 内配置だから)」と定めており、規律と組み込み機構が衝突している。並行ユースケースを本採用するなら、git-workflow の worktree 規約をエージェント委任時にどちらへ寄せるか (外部配置の規律を守らせるか、組み込み分離を許容するか) の設計判断が要る

## 価値 (xp-harness の構想との照合)

- 「要件定義から agent ペアで意思決定」: teammate 同士の直接対話が機構として成立するので、対等なペア議論の土台はある。独立 context 同士が根拠を出し合って合意する動きは実験でも観測できた (dialogue-principles の健全なコンフリクトと同型)
- 「ユーザーストーリーごとにチーム」: 1 セッション 1 チームで足りる
- subagent との使い分け: 規律注入 (プリロード) が要る役割は通常 subagent、対話・並行作業が要る役割は teammate、と住み分けるのが現状の機構に素直

## 制約を構想に照らした最終整理 — 実質的な懸念は消える

依頼者と認識を揃えた結論。依頼者の構想では、チームメンバー (= ストーリー担当) は「親セッションがそのままチームメンバーになる」イメージであり、reviewer のような専門職定義の使い回しではない。この前提で制約一覧を引き直すと:

- **チームメンバーにプリロードが効かない → 無関係**。チームメンバーには親セッションと同じ規律注入経路 (CLAUDE.md + skill の発火条件) がそのまま効く。親セッション自身もプリロードは使っていない。チームメンバーがヒントなしで CLAUDE.md を読み、フェーズ skill を連鎖発火させ、レビュアーを呼び、構造判断で止まることは追試 3 で実測済み
- **チームの入れ子不可 → 無関係**。担当の配下はサブエージェントで足りる (追試 3・4 で実測済み)
- **道具を絞った定義の制限 → 無関係**。レビュアーは末端なので絞られていて正しい

残るのは注意点 2 つ (どちらも導入設計の一行で対処可能):

1. レビュアーを誤ってチームメンバーとして招き入れるとプリロードが黙って落ちる (drift を実測) → 「レビュアーはサブエージェントとして呼ぶ」を委任規約に書く
2. 並行 E2E のポート衝突 (追試 4 で実痛みを確認) → E2E 実行環境の分離を設計する

## 本格導入判断への論点 (スパイクの出力、決定はしない)

- 導入するなら「teammate へ規律を届ける経路」の設計が先 (プリロード不可の代替: spawn プロンプト規約 / CLAUDE.md / Skill tool 名指し発火)
- worker 模式への重力に抗う規律 (lead の振る舞い規約) を skill 化できるかが、ペアプロ哲学と両立させる鍵
- 実験的機能につき仕様変動リスクあり (チーム作成ツールの削除など、直近の changelog でも機構が動いている)。導入判断は「痛みが明確化してから」の原則 (philosophy の 4 象限) で据え置きも選べる
