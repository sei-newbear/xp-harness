# ふりかえりのスキル反映候補提案 — 基本設計

> このドキュメントは進行中の設計メモ。実装後もそのまま残す（振り返り・PR レビューで参照される）。
> 削除のタイミング・要否は依頼者または外側の仕組みで別途判断する（エージェントは削除しない）。

要件は同ディレクトリの `要件定義.md` を参照。改修対象は `.apm/skills/retrospective/SKILL.md`（利用者向け配布物。`.claude/skills/retrospective` は同実体への symlink なので実体側だけ編集すればよい）。

この設計は依頼者の委任のもと、skill-reviewer subagent との壁打ちを経て固めた（依頼者は「skill-reviewer とペアプロしながら進めて」と委任済み）。

## 1. 流れの構造（段の配置）

現行の流れ「観測 → トピック選択 → 深掘り → アクション → [別トピックがあればトピック選択に戻る]」のループを抜けた後、成果物記録の前に、独立の段 **「取り決めへの書き戻し候補」** を新設する。

```
観測 ──→ トピック選択 ──→ 深掘り ──→ アクション
              ▲                          │
              └──── 別トピックを深掘り ───┘
                         │ もう深掘りしない
                         ▼
           取り決めへの書き戻し候補 (新設)
                         │
                         ▼
                    成果物の記録
```

### 配置の根拠

- 深掘りループ内の各段は「深掘りトピック単位」で回るのに対し、この段は「セッション全体のスイープ」で粒度が違う。ループの外に置くのが単一責任に沿う
- きっかけの実体験（セッションの終わりに「スキルに反映した方がいいことない?」と聞いたら候補が出た）とも位置が一致する

### 意識的なトレードオフ: 収穫は深掘りに回さない

終端配置にすると、収穫した候補を深掘りトピックに回す道は閉じる。これは意識的な割り切り: 知識・手順の収穫は根本原因分析ではなく収集なので、深掘りは不要。候補に議論の余地があれば、依頼者が追加・削除・修正する対話の中で自然に扱える。

### 却下した代案

- **観測段に第 3 の軸として混ぜる案**: 候補を深掘りトピックにも選べる利点はあるが、観測の発散軸（よかった点 / 伸びしろ）が濁り、既に密度の高い観測段がさらに肥大化する
- **アクション段を拡張する案**: 段を増やさない利点はあるが、アクションは「深掘りしたトピック単位」の出力で、収穫は「セッション全体のスイープ」なので性質が違い、段の単一責任を壊す
- **記録を「書き戻し候補」1 セクションに集約し直す案**（skill-reviewer 壁打ちで出た「一出力・二経路」構造の記録版）: 深掘り由来のアクションもスイープ由来の候補も 1 箇所に集め、反映する人が一望できる利点がある。しかし既存の「深掘りセクション内にアクションを置く」記録構造を作り変えることになり、要件の制約（既存の流れを壊さず統合する、規律装置最小注入）とぶつかるので採らない。代わりに「一出力・二経路」は概念として採用し、新段の本文で「深掘りのアクションと合わせて、このセッションの書き戻し候補の全量になる」と枠づける（下記 2 参照）

## 2. 新段の論理設計

### 責務

セッション全体をスイープし、「どの取り決め（どの skill / どこ）に、何を書き戻すか」の形で候補を提案する。main session が先に候補を出し、依頼者が追加・削除・修正する（観測段と同じ共同編集の型）。

拾う対象は両方（要件で合意済み）:

- 依頼者から受けた指摘・修正・好みの伝達の蓄積
- エージェント自身が作業中に見つけた手順・調べ方・ノウハウ（うまくいったやり方）

### 観測段のスイープとの峻別（レンズが違う）

観測段も「利用可能だった skill / subagent / ルールを一通り当てる」というセッション全体のスイープを要求しており、main session が 2 つのスイープを機械的に同一視して片方に寄せる危険がある（skill-reviewer 壁打ちの指摘）。新段の冒頭に峻別の一文を置く:

- 観測は **評価のレンズ**（よかった点 / 伸びしろ）
- この段は **収穫のレンズ**（次セッションに残すべき知識・手順）

### 既存アクション段との境界（重複の交通整理）

- 深掘りのアクションとして既に取り決めへの接続が出ているものは、この段で **再掲しない**
- この段の候補と深掘りのアクションを合わせたものが、このセッションの「取り決めへの書き戻し候補」の全量になる（分断ではなく補完）
- この境界は段の冒頭に据えて目立たせる（両方に書かれる / どちらにも書かれない事故を防ぐ。skill-reviewer 壁打ちの指摘）

### 既存流儀への相乗り（再説明しない = 規律装置最小注入）

以下はアクション段と同じ流儀であることを一言で示し、再説明しない:

- 接続の形式（どの取り決めに、何を）
- ここで実装・反映はしない（出す場であって直す場ではない）
- 0 件は「(なし)」と明示（0 件も価値ある情報）

### 語彙

- 段名は「取り決めへの書き戻し候補」。動詞は本文の既存語彙「書き戻す」（skill の核の一文「取り決めに書き戻さないと改善は次までに消える」と直結）に揃える。「落とす」「収穫する」等の新語を段名に持ち込まない（内部語彙を増やさない。skill-reviewer 壁打ちの指摘）
- consumer 展開後も自然に読める汎用語（「取り決め」「skill」）のみ使う。APM 機構の語彙は持ち込まない

### 責務文の微修正

「なぜこの skill があるか」の責務文（「気づきを引き出して改善案を残すまでがこの skill の責務」）を一句だけ広げ、知識・手順の書き戻し候補も責務に含まれると読めるようにする。段落追加はしない（同セクションに既にある「気づきを取り決めに書き戻さないと改善は消える」との重複説明を作らない。skill-reviewer 壁打ちの指摘)。

### frontmatter description

変更しない。発火条件は変わらず、書き戻し候補の提案は「呼ばれた後の振る舞い」なので本文の責務（境界原則: description は発火条件、本文は発火後の振る舞い）。

## 3. 記録フォーマットの差分

`ふりかえり.md` の日付セクション内、深掘りセクション群の後に「取り決めへの書き戻し候補」セクションを追加する:

```markdown
### 深掘り 2: [別のトピックの名前]
...

### 取り決めへの書き戻し候補        ← 新設
- [どの取り決め（どの skill / どこ）] に [何を書き戻すか]
- ...
（0 件なら「(なし)」）
```

記録の規則に追記する内容:

- 「取り決めへの書き戻し候補」は深掘りセクション群の後に置く
- 深掘りのアクションに出たものは再掲しない
- 0 件は「(なし)」と明示

## 4. 既存資産への影響

- 変更するファイルは `.apm/skills/retrospective/SKILL.md` の 1 つだけ（`.claude/skills/retrospective` は symlink なので追随）
- 変更箇所: (a) 責務文の一句、(b) 「流れ」に新段を 1 つ追加、(c) 成果物のフォーマット例と記録の規則に候補セクションを追記
- 「通底する構え」「観測」「トピック選択」「深掘り」「アクション」の各段、および frontmatter は変更しない
- 他 skill / agent / instruction への影響なし（retrospective を preload する subagent は存在しない）

## 5. 完了の定義（実装フェーズが満たすべきもの）

要件定義の Done をこの設計の語彙で具体化したもの:

- [ ] retrospective skill 本文の「流れ」に、深掘りループの後・成果物記録の前の段として「取り決めへの書き戻し候補」が規定されている
- [ ] その段に、拾う対象として「依頼者から受けた指摘・修正の蓄積」と「エージェント自身が見つけた手順・調べ方・ノウハウ」の両方が明記されている
- [ ] 候補の形式が「どの取り決め（どの skill / どこ）に、何を書き戻すか」であることが、アクション段の流儀への相乗りとして書かれている
- [ ] 観測段とのレンズの違い（評価 / 収穫）と、アクション段との境界（再掲しない・全量は両者の和）が段の本文にある
- [ ] 0 件時の「(なし)」明示が書かれている
- [ ] 成果物のフォーマット例と記録の規則に「取り決めへの書き戻し候補」セクションが追加されている
- [ ] ここで実装・反映はしないことが本文から読み取れる
- [ ] 責務文が一句広がり、知識・手順の書き戻し候補も責務に含まれると読める
- [ ] frontmatter description は変更されていない
