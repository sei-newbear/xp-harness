# ふりかえりのスキル反映候補提案 — 基本設計

> このドキュメントは進行中の設計メモ。実装後もそのまま残す（振り返り・PR レビューで参照される）。
> 削除のタイミング・要否は依頼者または外側の仕組みで別途判断する（エージェントは削除しない）。

要件は同ディレクトリの `要件定義.md` を参照。改修対象は `.apm/skills/retrospective/SKILL.md`（利用者向け配布物。`.claude/skills/retrospective` は同実体への symlink なので実体側だけ編集すればよい）。

## 設計の経緯

当初は「深掘りループの後・成果物記録の前に独立の段を新設する」設計で skill-reviewer subagent と壁打ちして実装まで進めた。しかし依頼者レビューで「深掘りしてからではなく、情報収集（観測）のフェーズで追加の一覧として出すイメージだった」と認識ズレが判明し、**観測段への統合**に設計変更した。変更後の形の方が筋がよいと再評価した根拠は「配置の根拠」に記す。当初案は「却下した代案」に理由つきで残す。

## 1. 流れの構造（候補一覧の置き場）

新しい段は作らない。既存の**観測段**で、main session が出すリストに第 3 の一覧 **「取り決めへの書き戻し候補」** を追加する。よかった点 / 伸びしろと同じタイミングで提示し、依頼者が追加・削除・修正する（観測の共同編集の型にそのまま乗る）。

```
観測 ──────────────→ トピック選択 ──→ 深掘り ──→ アクション
 ├ よかった点              ▲                          │
 ├ 伸びしろ                └──── 別トピックを深掘り ───┘
 └ 取り決めへの書き戻し候補 (新設、深掘りトピックにも選べる)
                                     │ もう深掘りしない
                                     ▼
                                成果物の記録
```

### 配置の根拠

- **スイープが 1 回で済む**: 終端の独立段だとセッション全体のスイープが観測と終端の 2 回になり、「2 つのスイープを機械的に同一視して片方に寄せる」危険への防御（レンズ峻別の明文）が必要だった。観測に統合すればこの問題自体が構造的に発生しない。防御装置が要る設計より、問題が起きない設計の方が規律装置最小注入に合う
- **候補を深掘りに回す道が開く**: 「この候補は本当に取り決めに入れるべきか」と議論の余地がある候補を、そのまま深掘りトピックに選べる
- **共同編集の場が既にある**: 観測は「main session が先に出し、依頼者が追加 / 削除 / 修正する」段。候補一覧も同じ型に乗るだけで、新しい対話の型を増やさない

### 却下した代案

- **深掘りループ後の独立段案**（当初実装した案）: セッション終端の「スキルに反映することはないか」という実体験と位置が一致する利点はあるが、(a) スイープが 2 回になり峻別の防御装置が要る、(b) 候補を深掘りに回す道を閉じる割り切りが要る、(c) 依頼者のイメージ（情報収集の場で一覧に出す）と合わない。依頼者レビューで置き換え
- **アクション段を拡張する案**: 段を増やさない利点はあるが、アクションは「深掘りしたトピック単位」の出力で、収穫は「セッション全体のスイープ」なので性質が違い、段の単一責任を壊す
- **記録を「書き戻し候補」1 セクションに集約し直す案**（skill-reviewer 壁打ちで出た案）: 深掘り由来のアクションもスイープ由来の候補も 1 箇所に集め、反映する人が一望できる利点がある。しかし既存の「深掘りセクション内にアクションを置く」記録構造を作り変えることになり、要件の制約（既存の流れを壊さず統合する、規律装置最小注入）とぶつかるので採らない

## 2. 観測段に追記する内容の論理設計

### 責務

観測の出力に第 3 の一覧「取り決めへの書き戻し候補」を加える。候補は「どの取り決め（どの skill / subagent / ルール）に、何を書き戻すか」の形。依頼者に「スキルに反映することはないか」と聞かれるのを待たず、観測の一部として必ず出す（依頼者の聞き忘れで知識が消えるのを防ぐ = 要件の核）。

拾う対象は両方（要件で合意済み）:

- 依頼者から受けた指摘・修正・好みの伝達の蓄積
- エージェント自身が作業中に見つけた手順・調べ方・ノウハウ（うまくいったやり方）

### 評価の 2 軸との関係

よかった点 / 伸びしろは**評価のレンズ**、書き戻し候補は**収穫のレンズ**（次セッションに残すべき知識・手順）。レンズが違うので、評価の 2 軸に混ぜず独立の一覧として並べる。観測段の既存文「軸は基本 2 軸」は評価の軸の話としてそのまま残し、候補一覧は軸の変更ではなく追加の一覧として書く。

### 深掘り・アクションとの関係

- 候補に議論の余地があれば、他の観測トピックと同じく深掘りトピックに選べる。収穫は基本そのまま記録に進んでよい（根本原因分析は不要な、収集の性質のため）
- 当初案にあった「深掘りのアクションに出たものは再掲しない」という交通整理は**不要になった**: 記録の「トピック一覧」は元々「観測の結果の全リスト（深掘りしなかったものも含む）」という規則で、候補一覧もその流儀に乗る。深掘りされた候補は深掘りセクションにも現れる（既存の観測トピックと同じ扱い）

### 既存流儀への相乗り（再説明しない = 規律装置最小注入）

以下はアクション段と同じ流儀であることを一言で示し、再説明しない:

- 接続の形式（どの取り決め（どの skill / subagent / ルール）に、何を書き戻すか）
- ここで実装・反映はしない（出す場であって直す場ではない）
- 0 件は「(なし)」と明示（0 件も価値ある情報）

### 語彙

- 一覧名は「取り決めへの書き戻し候補」。動詞は本文の既存語彙「書き戻す」（skill の核の一文「取り決めに書き戻さないと改善は次までに消える」と直結）に揃える。「落とす」「収穫する」等の新語を一覧名に持ち込まない（内部語彙を増やさない。skill-reviewer 壁打ちの指摘）
- consumer 展開後も自然に読める汎用語（「取り決め」「skill」）のみ使う。APM 機構の語彙は持ち込まない

### 責務文の微修正

「なぜこの skill があるか」の責務文（「気づきを引き出して改善案を残すまでがこの skill の責務」）を一句だけ広げ、知識・手順の書き戻し候補も責務に含まれると読めるようにする。段落追加はしない（同セクションに既にある「気づきを取り決めに書き戻さないと改善は消える」との重複説明を作らない。skill-reviewer 壁打ちの指摘）。配置変更の影響を受けないのでそのまま維持。

### frontmatter description

変更しない。発火条件は変わらず、書き戻し候補の提案は「呼ばれた後の振る舞い」なので本文の責務（境界原則: description は発火条件、本文は発火後の振る舞い）。

## 3. 記録フォーマットの差分

`ふりかえり.md` の「トピック一覧」セクションに、よかった点 / 伸びしろと並ぶ第 3 の一覧として追加する:

```markdown
### トピック一覧
よかった点:
- ...

伸びしろ:
- ...

取り決めへの書き戻し候補:
- [どの取り決め（どの skill / subagent / ルール）] に [何を書き戻すか]
（0 件なら「(なし)」）
```

記録の規則に追記する内容:

- 「取り決めへの書き戻し候補」も「トピック一覧」の一部（= 観測の結果の全リスト。深掘りに選ばれた候補も一覧に残る）
- 0 件は「(なし)」と明示

## 4. 既存資産への影響

- 変更するファイルは `.apm/skills/retrospective/SKILL.md` の 1 つだけ（`.claude/skills/retrospective` は symlink なので追随）
- 変更箇所: (a) 責務文の一句、(b) 観測段に候補一覧の追記、(c) 成果物のフォーマット例と記録の規則に候補一覧を追記
- 「通底する構え」「トピック選択」「深掘り」「アクション」の各段、および frontmatter は変更しない
- 他 skill / agent / instruction への影響なし（retrospective を preload する subagent は存在しない）

## 5. 完了の定義（実装フェーズが満たすべきもの）

要件定義の Done をこの設計の語彙で具体化したもの:

- [ ] retrospective skill 本文の観測段に、第 3 の一覧「取り決めへの書き戻し候補」が規定されている（依頼者に聞かれるのを待たず必ず出す、と読める）
- [ ] その一覧に、拾う対象として「依頼者から受けた指摘・修正の蓄積」と「エージェント自身が見つけた手順・調べ方・ノウハウ」の両方が明記されている
- [ ] 候補の形式が「どの取り決め（どの skill / subagent / ルール）に、何を書き戻すか」であることが、アクション段の流儀への相乗りとして書かれている
- [ ] 評価の 2 軸とのレンズの違いと、候補を深掘りトピックに選べることが本文にある
- [ ] 0 件時の「(なし)」明示が書かれている
- [ ] 成果物のフォーマット例の「トピック一覧」と記録の規則に「取り決めへの書き戻し候補」が追加されている
- [ ] ここで実装・反映はしないことが本文から読み取れる
- [ ] 責務文が一句広がり、知識・手順の書き戻し候補も責務に含まれると読める
- [ ] frontmatter description は変更されていない
- [ ] 当初案の「深掘りループ後の独立段」が本文から除去されている
