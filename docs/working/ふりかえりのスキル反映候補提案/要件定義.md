# ふりかえりのスキル反映候補提案 — 要件定義

> このドキュメントは進行中の設計メモ。実装後もそのまま残す（振り返り・PR レビューで参照される）。
> 削除のタイミング・要否は依頼者または外側の仕組みで別途判断する（エージェントは削除しない）。

## 1. 背景と動機

改修対象は `retrospective` skill（`.apm/skills/retrospective/SKILL.md`、利用者向け配布物）。

きっかけになった実体験: コードレビューのセッションで、依頼者があれこれ指摘を重ねた後、依頼者の側から「スキルに反映した方がいいことない?」とエージェントに聞いたところ、セッション中の指摘の蓄積から「これは skill / 規約に落とすべき」という候補がきちんと出てきた。つまりエージェントは、聞かれれば知識・手順の抽出ができる。しかし現状はその抽出が依頼者の発話に依存していて、依頼者が聞き忘れたセッションでは知識が残らず、同じ指摘を次のセッションでも繰り返すことになる。

これは `retrospective` skill が既に前提として持つ「学びが人に残らない — エージェントは学びを次セッションに持ち越せない。気づきを取り決め（skill / subagent / CLAUDE.md・ルール）に書き戻さないと、改善は次までに消える」の一形態だが、現在の流れ（観測 → トピック選択 → 深掘り → アクション）では拾いきれない:

- 観測の軸は「よかった点 / 伸びしろ」の 2 軸で、問題・改善を起点にしている
- 依頼者の指摘の蓄積や、エージェント自身が見つけたうまくいく手順・ノウハウは、失敗にも伸びしろにも分類されないことがあり、深掘りトピックに選ばれなければ言語化されないまま流れる
- 既存のアクション段には「気づきを『どの取り決めを、どう直すか』に接続する」規律があるが、そこに到達するのは深掘りされたトピックだけ

そこで、ふりかえりの中で「スキルに落とした方がいい知識・手順」の候補を、依頼者が促さなくてもエージェントの側から提案する形に改修する。

## 2. ユーザーストーリー

- ふりかえりをする依頼者が、セッション中に出た知識や手順（依頼者が重ねた指摘の蓄積・エージェント自身が見つけた手順やノウハウ）を次セッション以降も効く形で残すために、「スキルに反映した方がいいことない?」と自分から聞かなくても、ふりかえりの中でスキルに落とす候補を提案してほしい

ポイント 2 つ:

- **誰が・何のために**: 依頼者が、知識を次セッションに残すため（エージェントは学びを持ち越せない、という retrospective skill の既存前提にそのまま接続する）
- **聞かなくても**: きっかけの核心。依頼者の聞き忘れに依存しない形にする

### 拾う対象の範囲（合意済みの決定）

**依頼者の指摘由来とエージェント発見由来の両方を拾う。**

- 依頼者の指摘由来: セッション中に依頼者から受けた指摘・修正・好みの伝達の蓄積
- エージェント発見由来: エージェント自身が作業中に見つけた手順・調べ方・ノウハウ（うまくいったやり方）

却下した代案: 「依頼者の指摘由来だけに絞る」（きっかけの再現に忠実な案）。却下理由: 拾い漏らしを減らす価値を優先した。候補が増えて提案が薄まるリスクは認識した上での決定。

## 3. Done（完了基準）

harness 機構の E2E テスト枠組みは未確立のため、Done は skill 本文とふりかえり成果物の状態として観測可能な形に置く（事前縛り最小化、事後評価許容）:

- ふりかえりの流れの中に、依頼者が促さなくても「スキルに落とす候補（= 知識・手順）」を提案する段が retrospective skill 本文に規定されている
- 拾う対象として「依頼者の指摘の蓄積」と「エージェント自身が見つけた手順・ノウハウ」の両方が本文に明記されている
- 候補は「どの取り決め（どの skill / どこ）に、何を落とすか」の形で提示される（既存のアクション段の「取り決めに接続する」流儀と揃える）
- 候補が 0 件のときは「(なし)」と明示する（既存の「アクション 0 件 OK」の流儀と揃える）
- `ふりかえり.md` の記録フォーマットに候補の置き場があり、記録の規則が書かれている
- ここで実装はしない（「気づきを出して残す場であって直す場ではない」という既存の責務境界を保つ）ことが本文から読み取れる

## 4. スコープ外

- 提案された候補を実際に skill / 規約へ反映する作業そのもの（ふりかえりは「気づきを出して残す場」、反映は別セッション / 人間判断のまま）
- ふりかえり以外の場面（作業中のリアルタイム）で知識を収集・提案する仕組み
- retrospective の発火条件の変更（いつ発火するかは今のまま）

## 5. 制約（Constraints）

- `retrospective` は配布物（`.apm/skills/`）なので、consumer 展開後も自然に読める表現で書く（APM 機構や xp-harness 固有の語彙を skill 本文に持ち込まない）（種類: 組織・配布規約）
- 既存の流れ（観測 → トピック選択 → 深掘り → アクション）と「通底する構え」（責任追及でなく改善の場 / 正直さ / 共創・発散と収束 / 既存フレームワーク不採用）を壊さず、そこに統合する形にする（規律装置最小注入、肥大化させない）（種類: 設計方針）
- 利用者影響のある変更なので、完了時は main 直統合ではなく branch を push して PR を出しレビューを通す（種類: 組織・Git 運用）
