### 深掘り 1: 検証の体験と、検証をどう残すか（バケツ1）
#### 経緯
サンドボックスで痩せ版 instruction の発火・振る舞いを実走検証した。依頼者から「対話フェーズは依頼者役を演じないと再現できない」「一気通貫でやって」「n=1 は揺らぎでは、要件完成まで複数回」と正され、検証の精度と体験が上がった。story-slicing の発火は新旧とも約 50〜67% でバラつく（揺らぎ・痩せ非依存）と実測で確定できた。一方で「今回の検証をどう残すか」が課題として残った（サンドボックス自体は throwaway で git に入れたくない）。

#### 分析
残すべきはサンドボックスそのものでなく **再現できるレシピ（目的→条件→結果）**。条件（setup-sandbox の flavor / 仕込んだ状態 / ノーヒント依頼の全文 / 前後比較の仕組み / 試行数）を再現粒度で記録すれば、サンドボックスは `setup-sandbox.sh` ＋条件で再生成でき、コミット不要。この記録は「結果報告」と「再現手順」を兼ね、カバレッジの証跡・次の変更での参照にもなる。harness-verification の検証レポート書式に近いが「条件を再現粒度で書く」「検索できる構造化ヘッダを持つ」が今は明示されていない。検索は grep で足り、ベクトル/キーワード検索は記録が増えて grep が辛くなってから（痛み顕在化後）。スクリプト（board.sh の検証版）はトークン節約・再現性向上のため軽く足してよい（YAGNI を効かせすぎない）。

#### アクション（harness-verification は非配布なので実装まで行う）
- harness-verification の検証レポート書式に「**条件（再現レシピ粒度）**」section と「**検索可能な構造化ヘッダ**（目的 / 対象 / 条件 / verdict / 日付）」を足す
- 検証記録を **references/ に定義フォーマットで貯める**規約（サンドボックスは入れない・再現レシピで代替）
- **軽い query スクリプト**（目的 / 条件で grep 検索、board.sh の検証版）
- ベクトル/キーワード検索は将来（記録増加で grep が辛くなったら）
- （実装前に format / references 置き場 / script スコープの たたき台を合意する）

### 深掘り 2: 急いでゲートを飛ばす（バケツ2）— 記録のみ
#### 経緯
1 セッションの中で、フェーズをまたいで「急いで締め/次へ進み、その手前のゲート・合意・裏取りを飛ばす」が複数回起きた: worktree ゲート（開始時）/ pre-impl の「議論して決める」（要件）/ 実 section 棚卸し（設計）/ retrospective の締め（終了時）/ story-slicing の n=1 早合点。

#### 分析
取り決め（git-workflow の冒頭確認 / pre-impl の議論ルール / harness-verification の実 section 棚卸し / retrospective の終わり発話発火 / dialogue-principles「機械的処理に流れない」）は**すべて存在するのに、main session が「ルーチン/自明」と分類して飛ばした**＝機械的処理に流れる・規律の decay。このセッション序盤で同種の decay 対策を「また発生したら出す」と保留したが、**同セッション内で 5 回再顕在化**した。既に tier1 カード「対話モード中の対話原則の継続注入」があるが、今回の 5 回は対話原則に収まらず **別々の skill のゲート・確認全般**にまたがる（＝既存カードのスコープより広い可能性）。

#### アクション
記録のみ（このセッションを、保留していた decay / 機械的処理の痛みが強く再顕在化した証拠として残す）。既存カード「対話モード中の対話原則の継続注入」の材料。スコープを「対話原則」から「ゲート・確認全般の継続遵守」に広げるかは、そのカードの要件定義で別途判断する。
