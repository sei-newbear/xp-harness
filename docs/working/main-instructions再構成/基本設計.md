# main.instructions.md のゼロベース再構成 — 基本設計

> このドキュメントは進行中の設計メモ。実装後もそのまま残す（振り返り・PR レビューで参照される）。
> 削除のタイミング・要否は依頼者または外側の仕組みで別途判断する（あなたは削除しない）。

## 1. 設計方針

要件定義のルーブリック（consumer に届く価値である / 配布 skill が single source として担える〈テキスト重複でなく役割重複・常時性で判定〉/ 常時 context の重量を正当化できる）に沿って、既存構造に縛られず新しい骨格を第一原理で組む。

新しい骨格は 3 つの core 責務に落ちる（A / B / C）。加えて、そのどれにも属さないものは骨格から外す（REMOVE）。

REMOVE / SLIM の判断は「skill が担う」を口先で言わず、その規律が single source と主張する配布 skill の**本文に実在するかを引用付きで確認**してから下す（要件定義の完了の定義 (c)）。「skill が発火する ≠ その skill が当該規律を保持している」を混同しない。

## 2. 新しい骨格（core 責務）

### A: 薄い XP フレーム宣言

「XP のエンジニアとして振る舞う」という全体の構えを一行〜数行で宣言する。consumer は xp-harness を入れた＝ XP を求めており、この構えは他のどこ（philosophy は非配布）でも供給されないため、instruction が唯一の供給源。

旧「開発スタイル」ブロックにあった XP の価値・プラクティスの長い列挙は圧縮する（どの skill も直接参照しておらず、振る舞いを駆動していない飾りに近く、常時の重量を正当化できないため）。

### B: 常時効かせる横断規律

skill が発火する *前* に効いている必要がある load-bearing な規律。書式は「**operative な芯を一行で常時置く＋ how の詳細は担当 skill を呼ばせる**」。芯をポインタに置き換えないこと（「詳細は skill 参照」だけだと、その規律は skill が発火するまで効かず、B の存在意義〈発火前に効く〉が消える）。背後に担当 skill が無いものは芯の一行で完結。

skill を呼ばせるときは「参照」でなく「呼ぶ」と書く（skill-design-style 原則 4: 「参照」だと補助ファイルを取りこぼす）。

含める芯:

- **迎合しない**: 対話フェーズでは依頼者に迎合しない（即同意も迎合）。進め方の詳細は `dialogue-principles` を呼ぶ
- **エラーを黙って直さない / 長処理の前に一言**: エラーが出たら黙って直さず、何のエラーか・どう直すかを報告してから対処する。長時間かかる処理の前に、何を・なぜ・どのくらいかを一言伝えてから実行する（背後に専用 skill が無い芯のみの規律。責務境界はグレーだが、常時層でしか供給されず、盲目実行を止める協働規律として残す）
- **既存のツール設定を確認してから動く**: プロジェクトに既存のツール設定・エージェント定義があれば、必ず確認してから動く（起動時に効く常時規律。single source 無し。「開発者として振る舞う」スタンスの一部）
- **新しいライブラリ・ツールは思想・作法を調べてから使う**（仮置き）: 導入前に思想・ワークフローを調べてから実装に入る。常時性は弱い（導入時のみ）が、消えると困るため受け皿ができるまで仮置き。行き先は「調査スパイク skill 新設」— その skill 新設時に取り込み、instruction 側は移設（または routing 化）する（カードに移設メモを記録済み）
- **Why を意識する（モード次第で掘る / 進む）**: 依頼を受けたら、表面の解決策でなく Why（誰に何の価値か・真の課題）を意識する。毎回深掘りするのではなく、疑って対話するか・そのまま進めるかはモード合意で決める。詳細は `define-requirements` を呼ぶ
- **モード合意ゲート**: 設計・実装・変更など帰結ある行動の手前で、対話モード / 自走モードを依頼者と合意する（合意は再来しうる）。詳細は `define-requirements` を呼ぶ
  - この「Why を意識する」と「モード合意ゲート」は同じ軸（Why をどこまで掘るかがモードで決まる）なので、B の中で隣接して書く（実質つながった一かたまり）
  - **常時層に残す理由（load-bearing の所在）**: `define-requirements` はほぼあらゆる依頼で発火するため、how はそちらが供給する。芯が本当に効くのは `define-requirements` が **発火しない**ケース — 定型作業と判定して skip したときに、合意なしで帰結ある行動へ滑り込むのを止める backstop として。だから常時要る
- **応答は依頼者の言語で**: 読んだ資料の言語に引っ張られず、依頼者の発話言語で応える（芯のみ。責務境界グレーだが常時供給源が他に無く残す）
- **対話に入ったら dialogue-principles を呼ぶ（routing の芯）**: 依頼者と対話・議論するときは `dialogue-principles` を呼ぶ。違和感を感じたら再読する。段階的開示・発散収束・チャット/AskUserQuestion の使い分けなど対話の機構そのものは常時層に持たず（フェーズ固有で実装中は効かないため）、対話に入った瞬間に skill が発火することで担保する
- **code-review の Security 指摘は例外なく対応**（芯一行）: severity 処理の詳細（Critical / Warning / Suggestion の扱い）は `slice-tdd` が single source なので REMOVE するが、**Security だけは芯一行を常時層に残す**。理由: 処理は実装中（slice-tdd 発火後）に効くので原則 REMOVE だが、長セッションで規律が減衰したときの実害が最も高い（セキュリティ指摘の握りつぶし）ため、最小の保険として常時層に置く。詳細は `slice-tdd` を呼ぶ

### C: routing map

skill 横断でここにしかない「どの skill / subagent をいつ呼ぶか」の地図。

- **フェーズ → skill の対応**: 要件定義 → `define-requirements`、基本設計 → `basic-design`、実装 → `slice-tdd`、E2E spec → `e2e`、ループの締め（ふりかえり）→ `retrospective`。旧フェーズフローの ASCII 図は捨て、この対応情報だけをテキストで残す
- **フェーズ判定の起こし方**: フェーズは依頼冒頭で自動判定して依頼者に確認し、slash コマンド（`/define-requirements` 等）で明示的に上書きもできる。これは「どの skill を発火させるか」を決める起点で single source が無いため、フェーズ → skill 対応の隣に短く残す
- **複数案が必要な場面 → propose-options を呼ぶ**: 設計判断・アプローチが 2 つ以上ありうる場面の routing
- **成果物の置き場**: 要件定義・基本設計の成果物は `docs/working/<title>/` に置く（この規約は routing として残す）
- **レビュー・検証 subagent の表**: どの subagent をいつ呼ぶか（`pre-implementation-reviewer` / `code-reviewer` / `e2e-reviewer` / `done-verifier` / `handoff-verifier`）。ここにしかない routing 情報なので表を残す。ただし **表（どの subagent いつ）だけ** を残し、詳細な指摘処理ルールは下記のとおり分離する

#### subagent ブロックの内訳分解（criterion 3 を内訳に適用）

裏取りの結果、指摘処理ルールは配布 skill が single source を持ち、いずれもその skill 発火後に効く（＝常時層でなくてよい）ことが確認できた:

- **code-review の severity 処理ルール**（Security / Critical / Warning / Suggestion の扱い）→ `slice-tdd` 本文が single source（実装中に効く）。REMOVE。ただし Security の一行だけは B 芯に残す（上記 B、decay ×実害の保険）
- **上流レビュー（pre-implementation-reviewer）指摘の処理ルール**（依頼者と議論する材料として扱う・main が機械的に対応/非対応を決めない）→ `basic-design` / `define-requirements` 本文が single source（要件・設計フェーズで効く）。REMOVE
- 詳細な「例の運び方」テンプレは例示なので一緒に落とす

### ふりかえりの扱い（決定）

ループの締めとして C の routing map に一行入れる（→ `retrospective` を呼ぶ）。太いフロー図に戻すのではなく、routing の一行のみ。retrospective skill が機構の single source。

判断理由: C は既にフェーズ → skill を並べる場所で、ループの締めを足すのは一貫して安い。philosophy の最終形「1 ストーリー走り切り＝ 1 ループ」でふりかえりがループを閉じる。ふりかえりの発火トリガー（セッション終わりの発話）は明示的な作業依頼より subtle で見落としやすく、routing の一行が発火を助ける。

## 3. 旧ファイルの実 section → 新のマッピング

現行 `main.instructions.md` の実 section（`##` / `###` を全列挙）を突き合わせた。頭の中の「8 ブロック」要約でなく実ファイルベース。

| └ その他（既存ツール設定確認 / 新ライブラリ調査） | KEEP | B 芯（既存ツール設定＝残す。新ライブラリ調査＝仮置き、行き先は調査スパイク skill） |

| ├ フェーズ判定はハイブリッド（冒頭で判定・確認 / slash 上書き） | KEEP | C（フェーズ判定の起こし方。single source 無し・load-bearing） |

## 4. 既存資産への影響

- **触るファイル**: `.apm/instructions/main.instructions.md`（consumer 配布物、symlink 経由で `.claude/` にも同一実体）
- **例外的に触る配布 skill**: `dialogue-principles` に「議論はチャット / クローズは AskUserQuestion / 迷ったらチャットに倒す」の使い分けメタ則を追記する（instruction の REMOVE の受け皿。要件のスコープ例外として明記済み）
- **その他の skill 側は変更しない**: 各フェーズの振る舞い詳細・severity 処理・対話の機構は既に各 skill が single source
- **kanban**: 「調査スパイク skill 新設」カードに、新ライブラリ調査の芯を将来この skill へ移設する旨のメモを追記
- **発火への影響**: instruction がフェーズ判定・skill 誘導に効いていた分を痩せさせるため、痩せさせた後に各フェーズ skill が従来どおり発火するかを事後観測する（下記「完了の定義」）
- **改修者向けドキュメント（CLAUDE.md / philosophy 等）は対象外**: 非配布なので触らない

## 5. 完了の定義（実装フェーズが満たすべきもの）

- [ ] 新しい骨格（A 薄い XP フレーム / B 常時規律 / C routing map）だけで `main.instructions.md` が再構成され、REMOVE 対象（各フェーズ振る舞い詳細・ASCII フロー図・severity 処理の Security 以外・上流レビュー処理ルール・対話の型の機構）が消えている
- [ ] B の各芯が「operative な芯＋担当 skill を呼ぶ」形式になっている（芯がポインタに置き換わっていない。背後に skill が無いものは芯のみ）
- [ ] 三方向の取りこぼし照合の記録がある: (a) 旧の実 section を全列挙して「意図的に落とした / 拾い直した」に仕分け、(b) 新に残った各ブロックに「skill との重複が残っていない / 残すなら常時性が load-bearing な理由」＋グレーな芯は責務境界の判断も記録、(c) REMOVE / SLIM した各規律の single source が配布 skill 本文に実在することを引用付きで確認（実在しなければ B 芯に拾い直す）。あわせて削除・スリム化の項目と理由の一覧（before/after が追える形）
- [ ] 影響を受けうる skill の発火を事後観測している: 影響 skill を列挙し、各フェーズにノーヒントの通常依頼をサンドボックスで **複数回** 投げ、**痩せ後だけ再現的に発火しない** ものを拾い直す（単発の非発火＝揺らぎでは判定しない。harness の E2E 枠組みは未確立なので保証でなく事後評価）
- [ ] `dialogue-principles` にチャット/AskUserQuestion 使い分けメタ則が追記され、instruction 側はその routing 芯（対話に入ったら呼ぶ）でカバーされている
- [ ] ふりかえりの扱いが反映されている: C の routing map にループの締めとして一行入っている
- [ ] 「調査スパイク skill 新設」カードに、新ライブラリ調査の芯の移設メモが入っている

## 6. 実装フェーズへの注意

- instruction に「dialogue-principles を呼ぶ」等の一行を書いても、それで **発火が担保されたと過信しない**。担保するのは事後観測（ノーヒント・複数試行）であって一行ではない。一行は nudge、実発火は各 skill の description に依存する
- REMOVE は必ず single source の実在確認（完了の定義 (c)）を通してから。マッピング表は計画であって完全性の保証ではない
