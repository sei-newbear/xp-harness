## 2. (b) 残存監査 — KEEP した各芯の残す理由（常時性 ＋ 責務境界）

新に残したブロックが「配布 skill との重複を常時層に残していないか」を、常時性の理由つきで監査する。責務境界がグレーなものは XP 規律か中身の流儀かの判断も記録する。

- **Why を意識する / モード合意ゲート**: `define-requirements` はほぼ常に発火するが、定型作業と判定して skip したとき合意なしで帰結行動に滑り込むのを止める backstop。発火前に効く必要があり常時層でしか担保できない
- **迎合しない**: `dialogue-principles` にテキストは重複するが、skill 発火前（対話に入る前）に効かせる必要がある役割重複。テキスト重複を根拠に削らない
- **対話に入ったら dialogue-principles を呼ぶ / 違和感で再読**: routing の芯。機構自体は skill が single source、instruction は発火させる nudge のみ
- **エラーを黙って直さない / 長処理の前に一言**（責務境界グレー）: XP の進め方というより一般的な協働の行儀。ただし背後に専用 skill が無く常時供給源が他に無い＋盲目実行を止めるので残す
- **既存の設定を確認してから動く**: 「開発者として振る舞う（作業者でなく）」スタンスの一部で XP 隣接。起動時に効く常時規律、single source 無し
- **新しいライブラリ・ツールは思想を調べてから使う**（常時性が弱い・仮置き）: 常時層の資格は弱い（導入時のみ）が消えると困るため仮置き。行き先は「調査スパイク skill 新設」（カードに移設メモ済み）
- **応答は依頼者の言語で**（責務境界グレー）: 一般的な協働の行儀寄りだが常時供給源が他に無いため残す
- **セキュリティ指摘は例外なく対応**（Security 一行）: severity 処理は `slice-tdd` が single source で原則 REMOVE だが、長セッションでの規律減衰の実害が最も高いため最小の保険として芯一行のみ常時層に残す

## 3. (c) REMOVE / SLIM の single source 実在確認（引用付き）

REMOVE / SLIM した規律が、single source と主張する配布 skill の本文に実在することを行番号付きで確認する。「skill が発火する ≠ その skill が当該規律を保持している」を混同しない。実在するもののみ REMOVE した。

- **各フェーズの振る舞い（要件の不明点確認 等）** → `define-requirements` 171 行「#### 不明点を推測で埋めない」173 行「不明点が出たらその場で質問。依頼者の沈黙を『OK』と解釈しない」に実在。基本設計・実装フェーズの振る舞いも各 skill 本文に実在
- **code-review の severity 処理（Security 以外）** → `slice-tdd` 179 行「**Security**: 例外なく対応」以下、Critical / Warning / Suggestion の処理が実在。実装中（slice-tdd 発火後）に効くので常時層でなくてよい
- **上流レビュー指摘の処理ルール** → `basic-design` 337 行「レビュー結果は依頼者と一緒に議論する材料として扱う（main session が機械的に対応/非対応を決めない）」に実在。要件・設計フェーズ（該当 skill 発火後）に効く。`define-requirements` も pre-implementation-reviewer 提案を持つ
- **議論はチャット / クローズは AskUserQuestion の使い分け** → REMOVE 時点では配布 skill に受け皿が無かったため、本ストーリーで `dialogue-principles` 195 行「### 議論はチャット、クローズな選択は質問ツール (AskUserQuestion)」に移設した（スコープ例外、要件定義に明記）。移設後は skill が single source
- **フェーズフロー ASCII 図** → single source を要さない（過剰情報として削除）。フェーズ→skill の対応情報だけ C にテキストで残した
