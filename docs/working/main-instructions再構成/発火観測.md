# main.instructions.md 痩せ版 — 発火事後観測レポート

要件定義の完了の定義（サンドボックスでの発火事後観測）に対応する記録。痩せ版 instruction を deploy したサンドボックスで、ノーヒントの通常依頼を投げ、各フェーズ skill が発火するか・全フローの振る舞いが崩れないかを観測した。

## 観測条件

- サンドボックス: `xp-harness-test/lean-rebuild-verify`（`--bare`、痩せ版 54 行を `.claude/rules/main.md` に deploy）
- 方式: 要件定義・基本設計は**対話フェーズ**なので、検証者が依頼者役として対話を回し、実装フェーズで自走に切り替わる実フローを 1 本通した（1 発プロンプトの発火チェックでは対話フェーズを再現できないため）。retrospective は別セッションで観測
- ノーヒント: プロンプトに skill 名・期待挙動を書かず、「注文レビュー画面を作りたい」という自然な新機能依頼から開始（git の worktree/branch を切らない自走前置きのみ付与）

## 全体結論

**PASS（温度感: 厚め）**。痩せ版 instruction は、全フェーズ skill の発火・対話フェーズの振る舞い・対話→自走のモード切替・自走実装の TDD リズムのいずれも損なわなかった。痩せ後だけ発火しなくなった skill は無し（＝拾い直しは不要）。

温度感の但し書き: フルフロー 1 本＋ retrospective 1 本での実観測（複数試行はしていない）。全対象がマージナルでなく明確に発火したため確信は持てるが、確率的揺らぎの厚みまでは詰めていない。

## シナリオ詳細

### フルフロー 1 本（要件定義→基本設計→実装）

「注文レビュー画面を新しく作りたい」から開始し、依頼者役で対話を回した。観測された発火（flow セッション）:

- **Skill 発火 9 件**: `define-requirements` / `dialogue-principles` / `handoff-docs` / `basic-design` / `propose-options` / `slice-tdd` / `e2e` / `e2e-execution` / `implementation`
- **subagent 起動 4 件**: `handoff-verifier`（要件・設計の引き継ぎ点検 ×2）/ `code-reviewer`（Cycle 1）/ `e2e-reviewer`（E2E spec）
- **成果物**: `要件定義.md` / `基本設計.md` / `feat(注文レビュー): 注文一覧の表示（Cycle 1）` コミット

検証対象ごとの結果:

- **define-requirements（#1）PASS**: 新機能依頼で発火。対話モードを提案→ Why を 1 問ずつ掘る→ YAGNI を尊重しつつ「全注文目視はスケールしない」と正直にリスク指摘（迎合しない）→ ストーリー 1 文化・永続化を要件として明示・書き起こし前に確認。痩せてもフェーズ判定と対話規律が効いている
- **basic-design（#2）PASS**: 「設計を進めて」で発火。設計トピックの合意を先に取り、propose-options で技術スタック・モジュール分割・データモデル・競合時挙動・並び順を複数案＋推奨で提示、書き起こし前に確認
- **slice-tdd（#3）PASS**: 「実装を進めて」で発火し**自走に切替**。TaskCreate で 5 サイクルに分割、outside-in（E2E→実装）、グリーンで code-reviewer / e2e-reviewer を起動、Cycle 1 を完了・コミット
- **dialogue-principles（#4）PASS**: 対話フェーズで発火。1 問ずつ・段階的開示・迎合しない が対話に乗っていた
- **モード切替 PASS**: 対話（要件・設計）→自走（実装）の切替が機能。実装フェーズは確認を求めず自走した

### retrospective（#5）

- **暴発しない（依頼者の懸念への回答）**: フルフローを Cycle 2 の途中で「続きはまた今度」と中断したとき、retrospective は**発火せず**「全ストーリー未完了なので今日はやらない、次の区切りで」と明示的に見送った。＝ 途中での暴発はしない（routing は agent の認識にあり、実トリガー〈全ストーリー完了・終わり発話〉に照らして正しく判断）
- **発火する（別セッション）PASS**: 別セッションで「今日のふりかえりをしよう」と明示すると `retrospective` が発火（＋ `dialogue-principles`）。セッションの伸びしろを列挙し、深掘りトピックを聞く正しい振る舞い

## 付随観測（この痩せの範囲外）

- **story-slicing がフルフローで未発火**: 要件定義完成直後に story-slicing が発火しなかった（retrospective 自身も伸びしろとして観測）。ただし instruction の routing に story-slicing は元々無く、発火は story-slicing 自身の description が司るため、**この痩せの回帰ではない**（単一ストーリーで不要と判断された可能性）。別途 story-slicing の発火安定性として見るべき論点

## 結論

痩せ版 instruction は全フェーズ skill の発火と全フローの振る舞いを保っており、発火事後観測の Done を満たす。拾い直し（instruction への差し戻し）は不要。
