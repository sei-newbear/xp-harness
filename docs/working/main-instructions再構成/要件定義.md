# main.instructions.md のゼロベース再構成 — 要件定義

> このドキュメントは進行中の設計メモ。実装後もそのまま残す（振り返り・PR レビューで参照される）。
> 削除のタイミング・要否は依頼者または外側の仕組みで別途判断する（あなたは削除しない）。

## 1. 背景と動機

`.apm/instructions/main.instructions.md` は APM 経由で consumer に配布され、consumer 環境で **常時 context に乗る** 運用ルール。現状は 1 ファイルに 8 ブロック（開発スタイル / 開発者として振る舞う / フェーズフロー図 / 各フェーズの振る舞い / 横断ルール / レビュー subagent 表 / 対話の型 / 実装中のルール）が集約されている。

このうち多くが配布 skill 側（define-requirements / basic-design / slice-tdd / propose-options / dialogue-principles 等）と記述が重複しており、context の二重課金と、instruction と skill が食い違う drift の源になっている。加えて、重複でなくても「常時 context に乗せるほどの価値がない」情報（例: フェーズフローの ASCII 図）が混ざっている。

### なぜ今やるか（予防的投資）

現時点で「これが原因で事故った」という顕在化した痛みはない。それでも早めにやる理由:

- **常時 context に乗る負債** は、痛みが顕在化する前から毎セッション課金され続けている
- **利用者が増えてきた**。後になるほど instruction への依存が広がり、変更の波及が重くなる。「前は良かったのに」という後戻りを生まないよう、ロックインされる前の今のうちに整える

これは philosophy の「投資フェーズでは運用コスト最適化より価値命題の充実を優先」と衝突しない。今回は価値命題（consumer に届く運用ルール）を削らず、負債（重複・過剰情報）だけを落とすのが条件。

### ゼロベースで作り直す方針

既存構造の section を 1 つずつ削る incremental な進め方ではなく、「**この常時 instruction は本来何を積むべきか**」を第一原理から組み直す。既存の `main.instructions.md` は参照（入力）であって骨格ではない。既存構造に引っ張られると sunk-cost で無駄が残るため、あくまで参考にとどめる。

ゼロベースは「骨格を組み直す」ことであって「新機能を足す」ことではない。基調は減算（重複・過剰情報を落とす）で、加算の検討は **ふりかえりの是非に閉じる**。ふりかえりだけを名指しで検討対象にする理由は、既に tier1 カード「フェーズフローにふりかえり追加」があり、他の削減系カードの方針待ち（このストーリー待ち）で宙に浮いていた経緯から、骨格を組み直す本ストーリーで一緒に判断するのが自然なため。それ以外の「新たに積むべき core」の洗い出し・追加は本ストーリーのスコープ外とし、別ストーリーで扱う（ゼロベースを口実に無制限に加算しない歯止め）。

## 2. ユーザーストーリー

- xp-harness を install した開発チーム（人間 + コーディングエージェント）が、無駄な後戻りや誤誘導なく XP の進め方に沿って開発するために、常時 context に乗る運用ルールを **重複・過剰情報なく、正確な状態で** 受け取りたい。
- 併せて改修者が、instruction と skill の記述が食い違う drift の保守コストを下げたい（instruction を痩せさせ、各フェーズ内の振る舞いは skill を single source にする）。

## 3. Done（完了基準）

### 残す / 削るの判定ルーブリック

再構成後の instruction に載る 1 ブロックは、次を **すべて** 満たすものだけとする。1 つでも欠ければ削除またはスリム化する。

1. **consumer に届く価値** である（改修者向けの内輪情報ではない）
2. **配布 skill が single source として担える**（その規律を配布 skill 本文が持っており、かつ **担当 skill が発火した後で間に合う** ＝常時層でなくてよい）。ここで「重複」を **テキスト重複** と **役割重複** に分ける: すべての skill は発火型で常時 context に乗るのは instruction だけなので、**常時層でしか果たせない役割（担当 skill が発火する前・フェーズ判定の局面で効かせる必要がある規律）は、テキストが skill と重なっていても重複に数えない**。テキスト重複だけを根拠に削らない。なお philosophy のような “consumer に配布されない” ものとの重なりは、そもそも重複に数えない（← 下記「制約」の供給源ルール）
3. **常時 context の重量を正当化できる**（「あれば親切だが常時乗せるほどではない」ものは落とす。skill 横断でここにしかない routing map のようなものは残す）

過剰削除への歯止め: 「一見冗長だが、その一文が振る舞いを軌道に乗せている」規律（例: 迎合しない / エラーを黙って直さない / Why が分からないままやらない）は、**たとえ配布 skill とテキストが重複していても、常時層でしか効かない（担当 skill が発火する前の局面で効く必要がある）なら残す側に倒す**。削る前に「これが無くて、担当 skill が発火する前の局面で振る舞いが崩れないか」を必ず一度問う。

### 完了したと言える状態

- 旧構造に縛られず、上記ルーブリックを満たすブロックだけで `main.instructions.md` が再構成されている
- **取りこぼし照合パス（三方向）が通っている**:
  - (a) 旧 instruction と新 instruction を突き合わせ、「旧にあって新にない」もの 1 つ 1 つを「意図的に落とした（ルーブリック不合格）」か「拾い直した（資格あり）」に仕分ける（＝価値を落としていないことの確認）。棚卸しは頭の中の要約でなく、現行ファイルの実 section を全列挙して行う
  - (b) **新に残った各ブロックについても、配布 skill との重複が残っていないこと（残す判断なら常時性が load-bearing である理由）を記録する**。責務境界がグレーな芯（エラーを黙って直さない / 長処理告知 / 応答は依頼者の言語で 等）は、常時性だけでなく「責務境界（XP の進め方の規律か / 中身の流儀か）でも残す」判断も一言記録する（＝入口の判定ミスを事後に拾う、残存側の監査）
  - (c) **REMOVE / SLIM する各規律について、single source と主張する配布 skill の本文に実在することを引用付きで確認する**。実在しなければ B 芯に拾い直す（skill 側にその規律を足すのは別ストーリー）。「skill が発火する ≠ その skill が当該規律を保持している」を混同しない
  - あわせて **削除・スリム化した項目と各理由の一覧（before/after が追える形）** を成果物として残し、PR レビュアーが痩せた効果を追える状態にする
- 再構成で影響を受けうる skill の **発火が事後観測されている**（harness の E2E 枠組みは未確立なので「保証」はしない、事後評価に倒す）: 影響を受けうる skill を列挙し、各フェーズに **ノーヒントの通常依頼**（検証したい挙動をプロンプトに書かない ← 自己成就を避ける）を **複数回** 投げて、痩せ前 / 痩せ後で対象 skill が発火するかを事後観測する。判定は **痩せ後だけ再現的に（複数回）発火しない** ものを拾い直す対象とする（1 回の非発火＝揺らぎでは判定しない）。落ちていれば instruction 側に拾い直す
- **ふりかえりの扱いが決まっている**: 再構成後の instruction のフェーズモデル / routing に、ふりかえりを含めるか・どこに置くかが判断され反映されている（含める / 含めないのどちらでも、判断した結果が残っていればよい）

## 4. スコープ外

- **section 分割による常時 context の重量配分の最適化**（1 ファイルを複数 instruction ファイルに割り、配布時の配置を最適化する仕組みを活かす）は後続ストーリーに回す。中身が痩せる前に割ると割り直しになるため、「削減・整形 → 分割」の順とする。今回は 1 ファイルのまま中身を再構成することに閉じる
- **ふりかえり以外の「新たに積むべき core」の洗い出し・追加**は本ストーリーのスコープ外（ゼロベースを口実に無制限に加算しない。加算検討はふりかえりの是非に閉じる）
- **各 skill 本体の改修**（skill 側の記述を直す・description を短縮する）は本ストーリーの対象外。既存の「既存 skill description 短縮」カード等が別途扱う。**例外**: 「議論・自由記述はチャット / クローズな選択は AskUserQuestion」の使い分けメタ則だけは、instruction から REMOVE する受け皿が配布 skill に無いため、今回 `dialogue-principles` に移設する（この 1 件のみのスコープ拡張）
- **reviewer subagent の description にパスを渡す前提を明示するカード群**（code-reviewer / e2e-reviewer / pre-implementation-reviewer）は別ストーリー
- 改修者向けドキュメント（project root の `CLAUDE.md` / `.claude/` 配下 / `docs/` / `kanban/`）の再構成は対象外。今回触るのは配布物である `.apm/instructions/main.instructions.md` のみ

## 5. 制約（Constraints）

- **供給源ルール（組織・設計）**: philosophy skill 等は改修者向けで consumer に **配布されない**。consumer が受け取るのは `main.instructions.md` + 配布 skill だけ。したがって「philosophy と重複しているから消せる」は成り立たない。真の重複は **配布される skill 本文との重複** に限る。この区別を削る / 残すの判定の土台にする
- **配布物ゆえレビューを通す（運用）**: `main.instructions.md` は consumer 影響のある配布物。main へ直接統合せず、branch を push して PR を出しレビューを通す（改修者向け CLAUDE.md の git-workflow 上書きに従う）。編集は worktree / branch を切ってから
- **発火を落とさない（非機能）**: 再構成で instruction を痩せさせても、フェーズ skill の発火・XP の進め方の規律が consumer 環境で従来どおり効くことを損なわない
- **独断で削らない（運用）**: consumer 配布物なので、置き換え前に「残す / スリム / 削除 ＋ 理由」を依頼者に通してから編集に入る
