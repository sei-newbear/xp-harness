# 本番セッション分析 ふりかえり

## 2026-07-06: 初の本番セッション分析と、そこから出た harness 拡張・自身の working style

### サマリ
初めて本番の実運用セッション（他者の自走実装）を transcript から分析した。その「分析の仕方（craft）」を harness-verification に「本番分析モード」として書き戻した。あわせて、自身の working style（暴走↔全確認の往復・英語混じり・発散/収束の混同）が長いセッションで規律をすり抜けた事実を観測した。

### トピック一覧

よかった点:
- reviewer（skill-reviewer / 会話文脈なしの独立監査 subagent）を気軽に流し、自分の見落とし（配布設定の未登録・境界線上の固有名詞）を独立に捕捉できた
- 分析で記憶に頼らず記録から機械抽出して事実で裏を取った（数え間違いも、指摘後は記録確認に切り替えた）
- すり抜ける規律を構造/ツールで強制する改修を回せた（優先度確認を選択肢ツールの明示回答強制に、公開前の固有名詞点検を独立 subagent 化）

伸びしろ:
- 暴走↔全確認の往復（認識が曖昧なまま進む／叱られると全確認に振れる、を行き来）
- 英語混じり（日本語の場で英語の概念語を出し、依頼者が判断できない）
- 発散/収束の混同・見せ方（表・時系列を並べ、結論・事実・仮説を一度に混ぜる）
- 依頼者の発言の取りこぼし（先に言われた前提を、後で自分の発見として出す）
- 抽出の机上断定（表層の grep 結果を鵜呑みにして誤計測、指摘後に再計測）

### 深掘り 1: 分析の仕方を skill に残す

#### 経緯
本番セッションの分析は初めてで、その「やり方（transcript から振る舞いを復元し、あるべき姿に照らす）」を skill に残すべきか議論した。当初「新スキル」「別物」と過大に描いたが、対話で「craft 自体は harness-verification と同じ／本当の違いは"本番は的が開いていて、何を見るかを自分で立てる"こと」に収束した。

#### 分析
- transcript を読んで振る舞いを復元する craft は harness-verification に既にある（sandbox 版）。本番はその craft を、仕込み・起動を飛ばして既存 transcript に当てるだけ。
- 唯一の本質的な差は「的が開いている」こと。sandbox は改修が的を決める（既知）が、本番は何を見るか自体が未定。だから「理想を先に立てる」が最重要ステップになる。
- 理想は分析専用でなく「作る際（実装時）に見るもの」でもある。だから理想そのものは実装スキルに宿るべきで、harness-verification は「対象を統べる skill から理想を読む」というリファレンスで足りる。

#### アクション
- harness-verification に「本番の実運用セッションを見る場合」の一節を追加（実施済み）。核 = 的が開いているので、まず対象 skill の本文から理想を立て（発散/収束で合意、固まる前に照合しない）、各理想項目を lens に「できてる/できてない」を証拠付きで照合、事実/仮説を分ける。description を 2 文脈化し、「評価しない」を「スコアリングはしない（できてる/できてないの事実確認はする）」に明確化した。
- 理想そのものを別途書く必要はなかった（対象の実装スキルに既に反映済みで、分析の照合結果がそのスキル強化の設計基盤として使われた）。

#### 残（深掘りせず観測にとどめた）
- 「暴走↔全確認の往復」は今回の根に近いと見ているが、本セッションでは深掘りせず次回以降に回した。他の伸びしろも同様に観測のみ。

## 2026-07-18: 自走セッションを4観点で分析し harness-verification を拡張、判断委任の落としを深掘り

### サマリ
実運用の自走セッション（レビュー指摘対応、3リポジトリ）を分析し、そこで立てた分析手法を harness-verification に書き戻した（良し悪しへの拡張・自走の質4観点の型・タイムライン復元の落とし穴・時間内訳・サイクル構造+skill発火の2ビュー）。ふりかえりは「判断委任の枝を base デフォルトに落とす」傾きと、その修正場所の取り違え（改修者側の問題を配布側に出そうとした境界越え）を深掘り。

### トピック一覧
よかった点:
- harness-verification の本番分析手順が深い分析を導け、出た手法を skill に書き戻せた（再現する形にできた）
- skill-reviewer が正典 drift・カテゴリずれ等の実問題を毎回捕捉（ペアプロが機能）
- disclosure-guard → auditor → push の順を守り公開前ゲートが効いた
- 「分析→発見→skill化→レビュー→修正」の増分ループが回った
- jsonl 監視（行数ベースのアイドル検知）が誤検知を避けて機能した

伸びしろ:
- 同じ指摘を skill-reviewer に2回受けた（正典サイクルを書き下して drift）＝「同じ失敗の再演」を自分で止められず reviewer 頼み
- 設計議論に AskUserQuestion を使い、依頼者に「対話の原則見てる？」と指し直された
- 複数案が「案A/案B のスコープずらし」＝ dialogue-principles 規律4（別角度）に反した
- 「remote-control だから」と未検証の因果を断定（後で AskUserQuestion の tool_result 由来と判明、依頼者が捕捉）＝自己分析の捏造
- 序盤「一気に話しすぎ」＝段階的開示を効かせられなかった
- skill-reviewer を重い判断として躊躇＝「気軽に呼ぶ」を内面化していなかった

### 深掘り 1: skill-reviewer を気軽に呼べない（＝判断委任の枝を base デフォルトに落とす、対話の原則の件も同根）

#### 経緯
skill-reviewer を「通しますか？」と依頼者に許可を取る形にし、気軽に呼べなかった。依頼者から「スキルレビューは気軽にしていい」「コードレビューは何度も通せるのにこの違いは？」と問われて掘った。

#### 分析
- code-reviewer は「Refactor 前に必ず呼ぶ」という高頻度・機械的トリガーに乗るので反射的に何度でも呼べる。skill-reviewer の「気軽に（任意）」は自分の判断で起こす枝で、そこを起こさず汎用の許可取りデフォルトに落とした。
- 共通根（対話の原則の件と同じ形）: **機械的に義務化された動作は守れるが、「気軽に／判断で起こす」と委ねられた部分は base モデルのデフォルト（許可取り・簡潔・選択肢化）に落ちる**。設計議論をチャットでやる（判断が要る規律）も AskUserQuestion に落ちた。
- 修正場所の取り違え（二次の気づき）: この改修者作業で出た私の傾きを、当初「配布物である dialogue-principles に1行足す」で直そうとした＝**利用者向け／改修者向けの境界を踏み越えた**。改修者側で出たものを配布側に出すと、利用者全員にモデルの癖のパッチを背負わせることになる。philosophy が最初に問う責務境界を、修正提案そのものが破っていた。

#### アクション
- 正しい場所は改修者向けの skill-design-style（既に「気軽に呼ぶ」はあるが弱く、適用できなかった）。**まず「skill-reviewer を気軽に呼ぶ＝許可取りにしない（『通しますか？』と委ねない）」を skill-design-style で補強する**（本ふりかえりでは記録、実装は別ステップ）。
- 「判断委任の枝が base デフォルトに落ちる」共通根そのものは base モデルの傾きで、配布ルールで潰す話ではない（ペアプロが捕まえる設計解）。今回は一番具体な skill-reviewer の一手だけ補強し、共通根の一般化・追加注入は再発を見てから判断（過剰注入を避ける）。
