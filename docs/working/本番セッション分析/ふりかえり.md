# 本番セッション分析 ふりかえり

## 2026-07-06: 初の本番セッション分析と、そこから出た harness 拡張・自身の working style

### サマリ
初めて本番の実運用セッション（他者の自走実装）を transcript から分析した。その「分析の仕方（craft）」を harness-verification に「本番分析モード」として書き戻した。あわせて、自身の working style（暴走↔全確認の往復・英語混じり・発散/収束の混同）が長いセッションで規律をすり抜けた事実を観測した。

### トピック一覧

よかった点:
- reviewer（skill-reviewer / 会話文脈なしの独立監査 subagent）を気軽に流し、自分の見落とし（配布設定の未登録・境界線上の固有名詞）を独立に捕捉できた
- 分析で記憶に頼らず記録から機械抽出して事実で裏を取った（数え間違いも、指摘後は記録確認に切り替えた）
- すり抜ける規律を構造/ツールで強制する改修を回せた（優先度確認を選択肢ツールの明示回答強制に、公開前の固有名詞点検を独立 subagent 化）

伸びしろ:
- 暴走↔全確認の往復（認識が曖昧なまま進む／叱られると全確認に振れる、を行き来）
- 英語混じり（日本語の場で英語の概念語を出し、依頼者が判断できない）
- 発散/収束の混同・見せ方（表・時系列を並べ、結論・事実・仮説を一度に混ぜる）
- 依頼者の発言の取りこぼし（先に言われた前提を、後で自分の発見として出す）
- 抽出の机上断定（表層の grep 結果を鵜呑みにして誤計測、指摘後に再計測）

### 深掘り 1: 分析の仕方を skill に残す

#### 経緯
本番セッションの分析は初めてで、その「やり方（transcript から振る舞いを復元し、あるべき姿に照らす）」を skill に残すべきか議論した。当初「新スキル」「別物」と過大に描いたが、対話で「craft 自体は harness-verification と同じ／本当の違いは"本番は的が開いていて、何を見るかを自分で立てる"こと」に収束した。

#### 分析
- transcript を読んで振る舞いを復元する craft は harness-verification に既にある（sandbox 版）。本番はその craft を、仕込み・起動を飛ばして既存 transcript に当てるだけ。
- 唯一の本質的な差は「的が開いている」こと。sandbox は改修が的を決める（既知）が、本番は何を見るか自体が未定。だから「理想を先に立てる」が最重要ステップになる。
- 理想は分析専用でなく「作る際（実装時）に見るもの」でもある。だから理想そのものは実装スキルに宿るべきで、harness-verification は「対象を統べる skill から理想を読む」というリファレンスで足りる。

#### アクション
- harness-verification に「本番の実運用セッションを見る場合」の一節を追加（実施済み）。核 = 的が開いているので、まず対象 skill の本文から理想を立て（発散/収束で合意、固まる前に照合しない）、各理想項目を lens に「できてる/できてない」を証拠付きで照合、事実/仮説を分ける。description を 2 文脈化し、「評価しない」を「スコアリングはしない（できてる/できてないの事実確認はする）」に明確化した。
- 理想そのものを別途書く必要はなかった（対象の実装スキルに既に反映済みで、分析の照合結果がそのスキル強化の設計基盤として使われた）。

#### 残（深掘りせず観測にとどめた）
- 「暴走↔全確認の往復」は今回の根に近いと見ているが、本セッションでは深掘りせず次回以降に回した。他の伸びしろも同様に観測のみ。
