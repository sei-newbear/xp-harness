## ユーザー向け変更 (= consumer 影響あり、リリースノートに記載)

### 対話の出力規律 5 項目を追加 (dialogue-principles skill)

`.apm/skills/dialogue-principles/SKILL.md` に **「出力の規律」section を新設**:

- 省略しない、内容で書く (= 指示語 / 要件定義書のナンバリング / 案番号を対話で使わない)
- 英語の概念用語を混ぜない (= 日本語の場面では固有名詞以外は日本語化)
- 即同意しない (= 賛成も即同意になりうる、自分の独立判断を持つ)
- 別角度から発想する (= 1 つの案をずらすだけにしない)
- 機械的処理に流れない (= 「これはルーチン作業」と分類した瞬間に立ち止まる)

git-workflow skill と同じ system prompt 上書きパターン (= 「コーディングエージェントのデフォルトを override する」を明示) で強く書かれている。consumer の Claude の対話振る舞いが変わる (= positive、対話の質が上がる方向)。

### 違和感時の dialogue-principles 再読トリガー (main.instructions)

`.apm/instructions/main.instructions.md` の「対話の型」section に 1-2 行追加:

- 「対話・議論中に違和感を感じたら / 機械的処理に流れている兆候を感じたら、`dialogue-principles` skill を再度読み込む」

consumer の Claude が「忘れかけたら再読」を自己観察するトリガー。

### 要件定義書 / 基本設計書のナンバリング禁止 (define-requirements / basic-design)

両 skill のドキュメント執筆ルールに追加:

- 「Done (= 完了条件) に番号 (D1 / D2 等の識別子) を振らない」
- 「内容で参照する」(= 「D5」のような番号ではなく「『呼び忘れがない構造』の Done」のような中身で書く)

依頼者の指摘: 「対話で番号を出されると認知負荷が高い、md は Claude 自身の context 再現用、依頼者は見ない」。既存 consumer の文書 (= 既に番号を振っている要件定義書) は drift 発生、consumer の判断で対処。

### 全 skill から `name` フィールド削除 (= 機能影響なし)

12 ファイルの skill SKILL.md から `name:` 行を削除:

- 公式 docs で skill の `name:` は **省略可能** (= ディレクトリ名がデフォルトで使われる)
- subagent の `name:` は **必須** (= 残す)
- 機能変更なし、重複管理を解消するための整理
