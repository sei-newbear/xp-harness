## 1. 背景と動機

xp-harness の改修フローで、main session の Claude が「機械的処理に流される」失敗を再発させた (`docs/working/retrospective-skill/ふりかえり.md` 深掘り 1、2026-05-17 観測)。

ふりかえり.md の root cause 分析:

- **規律装置の不在**: 規律を「判断軸 / 観点 / 概念」として持っているだけで、具体 action / 強制チェックポイントが無い。CLAUDE.md は判断軸の集合であり、TDD のような action リズム (= 「N. 〜する → M. 〜する」の手順) になっていないため、判断時に起動しない場面がある
- **規律間の関係不在**: CLAUDE.md の skill 設計関連 section (`skill のメンタルモデル` / `skill 設計の境界原則` / `Description の書き方` / `出力前に立ち止まる`) が複数 section に独立して並んでおり、規律間の関係 / 優先順位 / 統合された全体像が見えない。個別判断時に「まずどの規律を適用するか」が曖昧で、機械的処理に流れる選択コストを上げている

xp-harness の中核思想との接続:

- 「**価値で導きつつ規律装置を最小注入**」が xp-harness の中核。規律装置の質が xp-harness の価値に直結する。harness 自身の規律装置が崩れると、その負債が consumer 配布の skill に波及する
- 「**ペアプロ哲学**」と整合させると、機械的なチェックよりも subagent によるペアプロでの捕捉が筋。閾値調整 (字数制限など) で本質を捉えられない問題 + 抽象度や既出判定のような文脈依存の規律違反は機械的なチェックでは捕捉不能 — ペアプロの相手としての subagent なら捕捉可能

### 依頼者の真のゴール (= 上位の Why)

これらの背景の上位に、依頼者の本質的な要望がある:

> **改修者が「やりたいこと」を伝えたら、main session (コーディング Agent) が xp-harness の harness 機構 (= skill / CLAUDE.md / subagent) に適切に落とし込めるよう、ある程度自走して任せられる状態にしたい。**

現状、main session の harness 理解が浅く、機械的処理に流れる失敗が大きな痛みになっている。改修者が都度説明しないと整理できない状態が続くと、`.apm/` 配下 (= 利用者向け本題) の改修が全く進まない。本要件は手段としての規律装置整備だが、**上位ゴールは「自走できる状態の実現」** である。

### 本 TODO の position (ROADMAP TODO 44 との関係)

本 TODO は ROADMAP TODO 44 を実体化するが、TODO 44 起草時の「観察期間 → 再発したら subagent 投入」という段階的な判断手順は踏まない。理由は 3 つ:

- **事実関係**: TODO 44 自体が 2026-05-17 のふりかえりセッション内で派生 TODO として記録された (= TODO 44 が要求した観察期間は一度も始まっていない)。その派生のきっかけとなった失敗を、本 TODO の根拠として再利用している
- **subagent の前提シフト**: TODO 44 起草時の subagent は「同じ失敗が再発したら入れる保険的な対策」(1 用途) だった。一方本要件で想定する subagent は「設計中の壁打ち / 改修後の最終レビュー / main session が判断に迷ったときの第三者視点」の 3 用途に拡張されている (= ペアプロ哲学との接続)。価値命題が変わっているため、段階的な判断手順自体が前提から外れる
- **原因層と症状層の相補性**: 「機械的処理に流れる」という原因層は CLAUDE.md 整理 + skill-design-style skill の新設 (ROADMAP TODO 47/49) で対処し、個別の違反パターン (description の具体名混入 / 本文の APM 機構言及 / 抽象度の取り違え 等) という症状層は skill-reviewer (TODO 44) で対処する。後者は文脈依存の判断を機械的なチェックで捕捉できないため、subagent (ペアプロの相手としての第三者) でしか届かない。両者は競合せず並走するため、CLAUDE.md 大幅書き換えとセットで本 TODO を進めるのが自然

## 2. ユーザーストーリー

> **xp-harness 上で稼働するコーディング Agent (main session、現時点では Claude Code 等) が、skill / CLAUDE.md 改修時に「機械的処理に流される」失敗を起こしても、規律違反の再発確率を構造的に下げる仕組みを使いたい。**

- **誰が**: main session のコーディング Agent (現時点では Claude Code 等。将来 Cursor 等の他コーディング Agent も対象、cross-agent 対応の ROADMAP TODO 3 と整合)
- **何のために**: skill / CLAUDE.md 改修時の規律違反 (description に具体名混入 / 本文に APM 機構言及 / 既出情報の重複 / 抽象度の取り違え 等) の再発確率を下げるため。上位ゴールは「改修者の指示を harness に自走して落とし込める状態の実現」(§1 参照)
- **何をしたい**: 「自発的に規律を思い出す」運用から脱して、**規律違反の再発確率を構造的に下げる構造**を入れたい (= 違反ゼロを強制する gate ではなく、ペアプロの第三者視点で確率を下げる介入)

## 3. Done (完了基準)

**Done の前提 (= 検証粒度について)**: 本 TODO は xp-harness の harness 機構 (skill / CLAUDE.md / subagent) を改修する作業で、harness 機構の E2E テスト枠組みは未確立。各 Done は観測可能性を保ちつつ、**厳密な機械的テストではなく事後評価での実証** を許容する形で書く (= 事前に形式や数値で縛らず、やってみて「これは解決した」と判断できる形)。これは「投資フェーズの仕組み作りだから事前縛りを最小化する」という xp-harness の判断軸と整合する。

### 規律 / 判断軸の集約 (新規 skill: `skill-design-style`)

- **skill-design-style skill の新規作成**: `.claude/skills/skill-design-style/SKILL.md` (改修者専用、`.apm/` 配下に置かない、philosophy skill と同じパターン) が新規作成されている。以下を含む:
  - skill 設計の規律 (= 既存 CLAUDE.md の `skill のメンタルモデル` / `skill 設計の境界原則` / `Description の書き方` / `出力前に立ち止まる` から移行 + 整理 + 重複解消)
  - 判断軸 (= 下記「議論で固まった判断軸」の内容)
  - skill 改修ワークフロー (= 改修時の手順、skill-reviewer 参照 / 立ち止まりポイント / etc)
- **議論で固まった判断軸が skill-design-style に書かれている**: 以下の判断軸が skill-design-style 本文に書かれている。
  - **harness 機構の E2E テスト枠組みは未確立** → harness 機構を作る作業の Done は事前縛り最小化、事後評価許容
  - **投資フェーズの仕組み作り** では運用コスト最適化より価値命題の充実を優先
  - **subagent はペアプロ的な第三者** として「壁打ち / レビュー / 第三者視点」の 3 用途で使う (= 強制 gate ではなく助言)
  - **スコープ境界の判別**: 対象が利用者なら `.apm/` 配下、改修者なら `.claude/` 配下に直接コミット (= philosophy skill と同じパターン)。`.claude/` 配下の symlink (= `.apm/` 配下と同じもの) はいじらない。判別は `ls -la` で symlink でないことを確認
  - **機械的処理に流れる failure mode の認識** (= 自己観察項目):
    - 自分の発想軸に縛られていないか (= 別角度から発想できるか、1 つの案をずらすだけになっていないか)
    - 確認を skip していないか (= 依頼者に確認する前に自分で読んで確認しているか)
    - 省略していないか (= 要件定義書のナンバリング / 案番号 / 略号で語らない、対話では内容で語る — 依頼者は要件定義書を見ていない前提)
    - 英語混じりにしていないか (= 日本語の場面で英語の概念用語を混ぜない、固有名詞以外は日本語化)
    - 即同意していないか (= 依頼者の発話に乗っかって判断を組み立てない、自分の独立判断を持つ、賛成も即同意になりうる)
    - 機械的処理に流れていないか (= 「これはルーチン作業」と分類した瞬間に立ち止まる)

### CLAUDE.md の整理 (ROADMAP TODO 47 + 49)

- **CLAUDE.md の skill 設計関連 section が skill-design-style に移行**: CLAUDE.md の skill 設計関連 section (`skill のメンタルモデル` / `skill 設計の境界原則` / `Description の書き方` / `出力前に立ち止まる`) が **skill-design-style skill に移行** されている。CLAUDE.md は `@.claude/skills/skill-design-style/SKILL.md` で transclude (or 参照) する形に書き換わっている。重複解消、single source of truth (= 唯一の真の出典) が成立

### skill レビュー subagent 新設 (ROADMAP TODO 44)

- **skill-reviewer subagent の新規作成**: `.claude/agents/skill-reviewer.md` (改修者専用、`.apm/` 配下に置かない、philosophy skill と同じパターン) が新規作成されている。frontmatter の preload skill に `skill-design-style` を指定 (= 起動時に規律 / 判断軸 / ワークフローが context に注入される)。skill SKILL.md (description / 本文) をレビューする
- **skill-reviewer の呼び忘れがない構造**: skill 改修時に skill-reviewer の **呼び忘れがない構造** になっている (= 自動呼び出しでも規律ベースの呼び出しでもよい、形式は問わない)。発火定義は CLAUDE.md (改修者専用 instruction) または skill-design-style skill 本文に書く
- **違反パターン検出機能**: skill-reviewer が違反パターン (description の具体名混入 / 本文の APM 機構言及 / 抽象度の取り違え / 既出情報の重複 等) を **検出する機能を持つ** (= 検出数の具体目標は事後評価、やってみて既存 skill の違反を発見できれば実証)
- **3 用途を満たす対話設計**: skill-reviewer は **何度かやり取りしながら使える** 設計 (= 1 回呼びきりで終わらない) で、以下 3 つの使い方を満たす:
  - **設計中の壁打ち / 議論** (任意・対話的、何度でも)
  - **改修後の最終レビュー** (必須)
  - **main session が判断に迷ったときの第三者視点** (任意、依頼者に確認する前のステップとして気軽に呼ぶ)
- **severity スコアなしの日本語の文章出力**: skill-reviewer の指摘は **severity スコアを持たず、日本語の文章で対応必要性を伝える** 形式とする (= code-reviewer の Security / Critical / Warning / Suggestion のようなヘッダーを付けない、ペアプロ哲学 + 規律装置最小注入と整合)。文章内で「これは consumer 配布で skill が機能しなくなる」「これは表現改善の余地」のように対応必要性を文脈で表現する。1 subagent で機械的な違反パターン検出 + 創造的なディスカッションの両方をやる (= 区別しない統合型)
- **既存 reviewer との責務切り分け**: 既存 reviewer (`code-reviewer` / `e2e-reviewer` / `pre-implementation-reviewer` / `done-verifier`) と skill-reviewer の **責務切り分け** の最小定義が基本設計.md に書かれている (= skill-reviewer は SKILL.md 専門 / code-reviewer は通常コード前提 / ダブった呼び出しが起きても main 裁量でスルー)。痛みが見えてから構造的対処に移行する経路を残す (= 破壊的変更許容)
