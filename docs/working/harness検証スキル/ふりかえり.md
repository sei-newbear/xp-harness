# harness検証スキル ふりかえり

## 2026-06-21: 検証の条件設計が下手だった件

### サマリ
slice-tdd 自走強化（申し送り）の実機検証を通じて、検証の「条件」（プロンプト・仕込み・開始状態）を何度も外し、やり直しが多発した。原因と対策を harness-verification skill と philosophy に反映した。

### トピック一覧
よかった点:
- 検証が実際に skill の穴（自走強化で報告漏れ）を発見し、申し送り機構で塞いで再検証で確認まで一周回った
- 申し送りの設計が skill-reviewer の指摘で「記録（気づいた場所）＋完了関門（必ず通る所で報告）」の二点留めに洗練された
- 最終的にノーヒントの忠実な再現条件（外部 API を WireMock でモックする実ケース）を組めた

伸びしろ:
- 依頼者の intent を何度も取り違えた（仮想シナリオをでっち上げ、依頼者が実際に観測したケースを使わなかった）
- 要件定義（Why / Done）を固めず手を動かし始めた（「テストするだけ」で作業者モードに入った）
- プロンプトに検証対象の振る舞いを漏らした（条件汚染）を繰り返した
- 確認せず高コストな検証ランを走らせた
- 自分の報告に英語が混じった

### 深掘り 1: 検証の条件が下手だった
#### 経緯
検証のたびに条件を外し、依頼者の指摘（「プロンプト微妙じゃね？」「また条件悪いのでは？」「全然違う」）で何度も仕切り直した。最終的に依頼者の実観測ケース（外部 API モックのリクエストマッチが緩く、主要パラメータしか検証していない）を、ノーヒント・設計が終わった開始状態で再現して、ようやく有効な検証になった。

#### 分析
- 「条件の良し悪しを判断する基準」を持たずに走った。プロンプト・仕込みに検証対象の振る舞いやヒントを書くと自己成就して因果が測れない、という原則を持っていなかった。
- 自己レビューは同じ盲点で素通りした。実際に汚染を捕まえたのは依頼者（外部の目）。
- 根っこは「検証の要件定義（Done = 何が観測できたら成功か / 何が条件を汚すか）を固めず、作業者モードで手を動かした」こと。依頼者に「要件定義スキル呼びなさい」と指摘されて初めて立ち止まった。

#### アクション（反映済み）
- harness-verification skill に「検証条件の設計原則」を明文化（ノーヒント / 実観測ケースを使う / 開始状態は検証対象 skill に合わせる）
- harness-verification skill の手順に「走らせる前に条件をテスト観点で外部レビューする関門」を追加（自己レビューは盲点が同じで効かない）
- 上位原則「振る舞いは skill / 規約に宿らせる（プロンプト依存にしない）」を philosophy の判断軸に昇格

## 2026-07-07: 検証セッションの待ち検知が繰り返し破綻した件

### サマリ
本番作業（handoff-docs の PR レビュー対応 + 追加依頼の検証駆動）で、検証セッションの「入力待ち（idle）検知」が何度も外れ、依頼者の指摘で毎回自己修正した。深掘りの結果、`claude-launcher.sh` の `wait-idle` を transcript mtime → pty 画面(log) mtime 方式に作り直し、quiet を実測に基づき固定、timeout を状態つきで返すフォールバックにした。sandbox 実走で全ケース検証済み。

### トピック一覧
よかった点:
- skill-reviewer を各改修段で通し指摘（原則の橋渡し / PRD 例整合）を反映できた
- 「検証条件が薄い」の指摘に迎合せず検証を組み直した（会話文脈を stress する条件へ再設計）
- 公式を claude-code-guide で確認し、subagent の長さ基準を記憶で断言せず「独自基準」と明示できた
- 引き継ぎ書の漏れ議論を philosophy（責務境界・最小注入）と整合させ generic に一般化できた

伸びしろ:
- harness-verification の待ち検知が繰り返し破綻（本深掘り）
- 序盤 EnterWorktree していなかった（絶対パス編集で成果物は正しく置けたが、依頼者に「移動している？」と聞かれて気づいた）
- 検証条件の初期設計が薄かった（会話文脈が漏れないかを stress する観点が最初抜けていた）

### 深掘り 1: 待ちの検知が繰り返し破綻した
#### 経緯
検証セッションを FIFO 駆動する中で、ターン完了（入力待ち）の検知に自作の待機ロジックを継ぎ足しで作り、長考・巨大トークン生成・AskUserQuestion で終わるターンで false-negative / false-positive を連発。行数 baseline を応答後に取って固着し、セッションを 2.5 分放置したこともある。自己修正の 3 回はいずれも依頼者の指摘（「適切に待ててる？」「また待てないな」「なんか待ってんの？」）が起点。

#### 分析
- 技術的根本: transcript mtime は thinking 中に書かれないので、長考を idle と原理的に区別できない。行数増加は AskUserQuestion 終わりで増えないことがある。log の grep は残骸で誤検知。正しい信号は pty 画面(log)ファイルの mtime（処理中はスピナーが毎フレーム書き、idle で止まる。thinking / subagent 実行中も画面は生きる＝実測で確認）。
- 働き方の根本: 失敗①で立ち止まって検知を設計し直すべきところ、「この検証ランを前に進める」局所最適を優先し、その場の bash で回避策を継ぎ足した（＝判断軸5「自走とは認識を揃えて実行、止まらず手を動かすことではない」の誤読）。回避策を main loop の improvisation に書いたので再現せず同じ失敗を再演した（＝判断軸8「振る舞いは取り決めに宿らせる」の逆）。
- 自己検証の欠如: 自作 waiter の "IDLE" を実状態と突き合わせず信用した（＝ふりかえり skill「集計を真と扱わない・挙動は記録で裏を取る」の失敗）。依頼者の「安定したん？」で AskUserQuestion ケースを試して初めて残バグ（NOSTART 誤判定）を発見できた。
- 設計を絞れた: 「send にも安全ゲート」案は依頼者の「wait だけであかんの？」で却下（send ゲートは wait-idle と同じ log-mtime 信号で、同じポーズに同じく騙され独立防御にならない。wait-idle を信頼できる値にすれば "wait→send" で割り込みは起きない）。過剰設計が 1 個削れた。

#### アクション（実装・検証済み）
- `claude-launcher.sh wait-idle` を pty 画面(log) mtime 方式に作り直し（transcript を捨てる）。quiet=6s 固定（stress 実測: 作業中ポーズ最大 2s に 4s 余裕）。timeout は名前付きフラグで呼ぶ側が指定、超過時は「まだ処理中 / 想定外停止」を切り分けた状態つき TIMEOUT を返す（放置でなく引き戻す）。「起動時から静止」は既に idle として IDLE 返却（AskUserQuestion 終わり等）。
- harness-verification SKILL.md 手順3 を「transcript 更新停止で判断」→「wait-idle を使う」に更新。
- sandbox 実走で全ケース検証: IDLE / TIMEOUT-still-processing / subagent 中 log 生存 / AskUserQuestion idle / 91s 重い作業で false-fire なし。

#### 広い教訓の扱い（アクションにしない判断）
「ツールがタスク中に破綻したら、その場の回避策で凌がず取り決めを直してから使う」は、判断軸5（自走の誤読）＋判断軸8（振る舞いは取り決めに宿らせる）の instance なので、新ルールは philosophy に注入せず本記録に留める（判断軸1 最小注入に沿う）。
