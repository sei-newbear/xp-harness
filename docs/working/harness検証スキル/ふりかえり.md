# harness検証スキル ふりかえり

## 2026-06-21: 検証の条件設計が下手だった件

### サマリ
slice-tdd 自走強化（申し送り）の実機検証を通じて、検証の「条件」（プロンプト・仕込み・開始状態）を何度も外し、やり直しが多発した。原因と対策を harness-verification skill と philosophy に反映した。

### トピック一覧
よかった点:
- 検証が実際に skill の穴（自走強化で報告漏れ）を発見し、申し送り機構で塞いで再検証で確認まで一周回った
- 申し送りの設計が skill-reviewer の指摘で「記録（気づいた場所）＋完了関門（必ず通る所で報告）」の二点留めに洗練された
- 最終的にノーヒントの忠実な再現条件（外部 API を WireMock でモックする実ケース）を組めた

伸びしろ:
- 依頼者の intent を何度も取り違えた（仮想シナリオをでっち上げ、依頼者が実際に観測したケースを使わなかった）
- 要件定義（Why / Done）を固めず手を動かし始めた（「テストするだけ」で作業者モードに入った）
- プロンプトに検証対象の振る舞いを漏らした（条件汚染）を繰り返した
- 確認せず高コストな検証ランを走らせた
- 自分の報告に英語が混じった

### 深掘り 1: 検証の条件が下手だった
#### 経緯
検証のたびに条件を外し、依頼者の指摘（「プロンプト微妙じゃね？」「また条件悪いのでは？」「全然違う」）で何度も仕切り直した。最終的に依頼者の実観測ケース（外部 API モックのリクエストマッチが緩く、主要パラメータしか検証していない）を、ノーヒント・設計が終わった開始状態で再現して、ようやく有効な検証になった。

#### 分析
- 「条件の良し悪しを判断する基準」を持たずに走った。プロンプト・仕込みに検証対象の振る舞いやヒントを書くと自己成就して因果が測れない、という原則を持っていなかった。
- 自己レビューは同じ盲点で素通りした。実際に汚染を捕まえたのは依頼者（外部の目）。
- 根っこは「検証の要件定義（Done = 何が観測できたら成功か / 何が条件を汚すか）を固めず、作業者モードで手を動かした」こと。依頼者に「要件定義スキル呼びなさい」と指摘されて初めて立ち止まった。

#### アクション（反映済み）
- harness-verification skill に「検証条件の設計原則」を明文化（ノーヒント / 実観測ケースを使う / 開始状態は検証対象 skill に合わせる）
- harness-verification skill の手順に「走らせる前に条件をテスト観点で外部レビューする関門」を追加（自己レビューは盲点が同じで効かない）
- 上位原則「振る舞いは skill / 規約に宿らせる（プロンプト依存にしない）」を philosophy の判断軸に昇格
