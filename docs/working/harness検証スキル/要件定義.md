# harness 検証スキル — 要件定義

> このドキュメントは進行中の設計メモ。実装後もそのまま残す（振り返り・PR レビューで参照される）。
> 削除のタイミング・要否は依頼者または外側の仕組みで別途判断する（あなたは削除しない）。

## 1. 背景と動機

xp-harness の skill / subagent を改修したとき、「改修が実際に効いているか」(skill が Skill tool で発火するか、subagent が呼ばれるか、preload が効くか、期待した振る舞いが起きるか) を確認する検証作業が繰り返し発生している。

過去 2 回 (2026-05-25 と 2026-06-07 のセッション) で同じ検証を実施したが、いずれも検証のやり方をその場でゼロから組み立てていた:

- 検証環境 (テスト用プロジェクト) の準備・整理を手作業で実施
- `claude -p` で headless 起動するコマンド・フラグをその場で調査
- transcript (セッションの jsonl) の場所の特定、解析の grep / python ワンライナーをその場で手書き
- プローブ用の検証 skill / subagent 一式をその場で設計・作成・削除

検証の手順とノウハウが過去セッションの中にしか残っておらず、検証のたびに再発明している。これが痛み。

検証の構造は毎回同じ: **状態を仕込む → Claude Code を起動して走らせる → transcript を解析して発火・振る舞いを観測 → 後片付け**。仕込む状態の大きさ (検証用 skill 一式だけ置くか、要件定義・基本設計 md を置いて実装フェーズを走らせるか) が検証の目的に応じて変わるだけで、全部この同型のプローブ検証である。

なお、ここで言うのは「検証」(仕組みが意図どおり動いたかの事実確認、二値で観測可能) であって「評価」(出来の良し悪しの測定、スコアリング・比較) ではない。評価方法論は別判断 (明確な課題が見えてから着手) で、本件のスコープ外。

### 制約: headless 起動の従量課金

2026-06-15 から `claude -p` (headless 起動) が従量課金になる。main session が `claude -p` を直接叩く構成にはせず、**改修者 (人間) に Claude Code を起動してもらう**構成にする。main session は起動ディレクトリと渡すプロンプトを提示し、走り終わったセッションの transcript (`~/.claude/projects/<パスをエンコードしたディレクトリ>/` 配下の jsonl) を解析する。

## 2. ユーザーストーリー

- **xp-harness の改修者が、skill / subagent の改修が実際に効いているか事実確認するために、headless ではなく自分で Claude Code を起動して走らせ、transcript を解析する検証手順を、毎回掘り直さずに最低限の手数で回したい**

## 3. Done（完了基準）

- **検証 skill が改修者向けとして `.claude/skills/` 配下に直接コミットされている** (philosophy / release と同じパターン、非配布、`.apm/` には置かない)
- **skill は本文 + 補助ファイル (スクリプト・リファレンス) で構成され、決定的にできる部分はスクリプト化されている**:
  - 検証用の sandbox 環境の構築がスクリプトで実行できる
  - transcript の場所特定と解析 (Skill tool 発火の一覧化、Read で済まされたかの判別、main と subagent の区別) がスクリプトで実行できる
- **skill 本文には判断が要る部分の手順が書かれている**: 検証目的に応じた途中状態の仕込み方 / 改修者に起動してもらう案内 (起動ディレクトリと渡すプロンプトの提示、main session 自身は `claude -p` を叩かない) / 解析結果をどの観点で読むか / 後片付け
- **実際の検証を 1 回、この skill の手順だけで回せる** (過去セッションを掘り直さずに済む)

## 4. スコープ外

- **評価 (良し悪しの測定)**: 発火精度のスコアリング、ベンチマーク、skill の書き方の比較検証。明確な課題が見えてから別途検討
- **main session が `claude -p` を自動で叩く仕組み**: 従量課金対応で人間起動に倒したため、自動 headless 起動の動線は作らない
- **CI 組み込み・自動回帰テスト化**: harness 機構の E2E テスト枠組みの確立そのものには踏み込まない。検証はあくまで改修者が手動で発火させる
- **利用者向けへの配布**: この skill は改修者向け専用
