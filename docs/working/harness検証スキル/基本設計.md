# harness 検証スキル — 基本設計

> このドキュメントは進行中の設計メモ。実装後もそのまま残す（振り返り・PR レビューで参照される）。

## 1. 全体構成

改修者向け skill として `.claude/skills/harness-verification/` に直接コミットする (philosophy / release / skill-design-style と同じパターン、非配布)。

```
.claude/skills/harness-verification/
├── SKILL.md                 # 判断が要る部分の手順 (検証フロー)
└── scripts/
    ├── setup-sandbox.sh     # sandbox 環境の構築 (決定的)
    ├── setup-probe.sh       # プローブ一式の配置と削除 (決定的)
    └── analyze-session.py   # transcript の特定と解析 (決定的)
```

ディレクトリ構成は Claude Code 公式の skill 構成例 (SKILL.md + `scripts/` に実行可能スクリプト) に準拠する。スクリプトは「実行する」用途 (= 中身を context に読み込まない) で、SKILL.md から相対パスで案内する (参照は一段階まで)。

## 2. 検証フロー (SKILL.md に書く手順)

検証の構造は同型のプローブ検証 1 種類: **仕込み → 人間が起動 → 解析 → 後片付け**。

1. **検証目的の確認**: どの skill / subagent の、どの発火・振る舞いを確認したいかを特定する
2. **仕込み**:
   - sandbox が必要なら `setup-sandbox.sh <名前>` で構築 (既存 sandbox の使い回しも可、改修者と判断)
   - 検証目的に応じた途中状態を置く:
     - skill の連鎖発火だけ安く確認したい → `setup-probe.sh` でプローブ一式を配置
     - 実フェーズの発火・振る舞いを確認したい → `docs/working/<title>/` に要件定義.md / 基本設計.md を置いて途中状態を作る
3. **起動案内**: 改修者に「このディレクトリで Claude Code を起動して、このプロンプトを貼ってください」と起動ディレクトリと貼り付け用プロンプトを提示する。**main session 自身は `claude -p` を叩かない** (2026-06-15 から従量課金)
4. **解析**: 走り終わったら `analyze-session.py <sandbox パス>` で発火事実を一覧化し、観点に沿って読んで事実として報告する
5. **後片付け**: プローブを配置した場合は `setup-probe.sh <sandbox パス> --clean` で削除。sandbox 自体の削除要否は改修者と判断

## 3. スクリプト仕様

### setup-sandbox.sh

- 引数: sandbox 名 (例: `memo-app`)
- 動作: `<xp-harness の親ディレクトリ>/xp-harness-test/<名前>/` を作成し、以下を実施
  - `git init`
  - `apm.yml` を生成 (ローカルの xp-harness を依存指定、targets: claude、includes: auto)
  - `.gitignore` を生成 (node_modules / apm_modules / test-results 等)
  - `apm install` を実行 (skill / agent が `.claude/` 配下に deploy され、CLAUDE.md が生成される)
  - 初期 commit (検証セッションが何を変更したか後で `git status` / `git diff` で観測できるようにする)
- パスはスクリプト自身の位置から導出する (ハードコードしない)

### setup-probe.sh

- 引数: sandbox パス、`--clean` オプション
- 動作 (配置): sandbox に以下のプローブ一式を書き込む
  - `.claude/agents/skill-probe.md`: 検証用 subagent。`probe-preloaded` skill を preload し、その指示に厳密に従う
  - `.claude/skills/probe-preloaded/SKILL.md`: preload される skill。`probe-target` skill を Skill tool で呼ぶよう指示する
  - `.claude/skills/probe-target/SKILL.md`: 呼ばれた証拠として合言葉 `PROBE_TARGET_FIRED_OK` を最終出力に含めるよう指示する
- 動作 (`--clean`): 上記 3 つを削除する
- 仕掛けの出典: 2026-05-25 の検証セッションで確立したプローブパターン

### analyze-session.py

- 引数: sandbox パス (必須)、`--session <ID>` (任意、省略時は最新セッション)、`--list` (セッション一覧)、`--grep <文字列>` (合言葉などの出現確認)
- transcript の場所: `~/.claude/projects/<sandbox 絶対パスの英数字以外を - に置換したディレクトリ>/` 配下
  - main セッション: `<セッションID>.jsonl`
  - subagent: `<セッションID>/subagents/agent-*.jsonl` + 対の `agent-*.meta.json` (agentType が入る)
- 出力 (発火・振る舞いの事実):
  - main の Skill 発火と subagent 起動の時系列 (tool_use の name=Skill / name=Agent)
  - 各 subagent (agentType 別) 内の Skill 発火
  - SKILL.md を Read しただけのケース (= Skill tool で発火せず Read で済まされた失敗モードの検出)
  - `--grep` 指定時: 合言葉等の出現箇所 (main / subagent の別)

## 4. SKILL.md の frontmatter 方針

- `name` は省略 (skill はディレクトリ名がデフォルト)
- `description` は抽象レベルで What + When のみ (具体ツール名・配布管理の内部情報は入れない)

## 5. 検証した事実 (設計の根拠)

- 対話起動セッションの transcript 形式は本設計の通り (2026-06-07 の memo-app 検証セッションの実データで確認済)
- 公式 docs の skill 構成例は `SKILL.md` + 補助 md + `scripts/` (実行可能スクリプト同梱は公式推奨、実行時はスクリプト本体を context に読み込まない)
- sandbox の構造 (xp-harness-test/ 配下に検証案件ごとの git repo + apm install) は既存の運用 (memo-app / todo-add / upgrade-v020-v030) と同じ
