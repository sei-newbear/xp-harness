# 実装 skill / E2E 実行 skill の新設 — 要件定義

> このドキュメントは進行中の設計メモ。実装後もそのまま残す（振り返り・PR レビューで参照される）。
> 削除のタイミング・要否は依頼者または外側の仕組みで別途判断する（あなたは削除しない）。
>
> 対応 ROADMAP TODO: 53 (本体) / 35 (吸収) / 36 (吸収) / 27 (本格版は残す)

## 1. 背景と動機

consumer が自分の project で「実装規約」「E2E 実行手順」を skill として用意していても、実装フェーズの Agent (main session) が **まったく参照しなかった** という痛みが実際に起きた。

原因は、Agent が slice-tdd の TDD ループ (テスト先書き → 最小実装 → リファクタ → コミット) に集中してしまい、「他の skill を見に行く」という発想自体が起きなかったこと。skill 自体の発火条件 (description) の問題ではなく、TDD に注意が偏って受け渡しが起きなかった構造の問題。

さらに reviewer 系の subagent (code-reviewer / e2e-reviewer) は `tools:` が allowlist 方式で Skill tool を持たないため、preload した skill しか使えない。実装規約 skill が別の skill (例: architecture skill) を名指しで参照していても、subagent はそれを追えず詰む。consumer が実装規約とアーキテクチャ規約を別 skill に分けている構成は普通にありえる。

結果として、せっかく書いた project 固有規約が、実装にもレビューにも反映されない。

この課題は「規約を書き入れる空間がない」ではなく **「用意された規約 skill が、実装ループ・レビューの中で参照されない」= 発見・参照の問題** である。したがって解決策は「skill を呼ぶように明示的に書く」+「consumer が上書きできる枠を用意する」+「subagent が名指し参照を追えるようにする」の組み合わせになる。

## 2. ユーザーストーリー

> 実装フェーズの Agent が、TDD ループに没頭していても project 固有の実装規約・E2E 実行手順を確実に参照でき、reviewer も同じ規約でレビューできるように、xp-harness に「実装 skill」「E2E 実行 skill」のデフォルト枠を用意し、slice-tdd から呼び出しを促し、code-reviewer に実装 skill を preload し、reviewer が名指し参照された skill を追えるようにし、consumer がこれらを project 固有内容で上書きできるようにする。

ポイントは 3 つがセットで効くこと:

1. **デフォルト枠を置く** — xp-harness 標準で薄い実装 skill / E2E 実行 skill を用意し、上書きの土台にする
2. **呼び出しを促す** — slice-tdd から「実装中はこの skill も参照せよ」と明示。注意が TDD に偏っても拾える
3. **上書きできる** — 既存 e2e skill と同じ「上書き前提」パターン。consumer が同じ枠を project 固有内容で差し替える

どれか 1 つでは痛みが消えない。「デフォルト枠を作る」と「slice-tdd から呼ぶ」は特にセット。

### INVEST 点検の結論 (1 ストーリーのまま進める)

Small の字面では 4 つの価値単位 (実装規約が実装ループで効く / E2E 実行手順が実装ループで効く / レビューでも実装規約が効く / reviewer が名指し参照 skill を追える) に分割可能。だが分割しない。理由:

- **価値の確定は holistic**: 実装ループで全部が揃って実際に参照されるのを動作確認する 1 点で価値が発揮される。実装 skill だけあってレビューが揃わない半端な状態は中途半端な価値しか出さない
- **Small が効く前提が弱い**: 並列開発の必要も、段階的に価値を出す必要も無い小さく coherent な harness 改修
- **保険はストーリー分割なしで保たれる**: 分割で守りたかった「Skill tool が効かなかったときの保険」は Done に「効かなければ preload + 同名上書きに落ちる」として既に含まれている

### 実装順序 (slice-tdd 向けの目安)

1. 実装 skill のデフォルト + slice-tdd 参照 (土台)
2. E2E 実行 skill のデフォルト + slice-tdd 参照 (実装規約と並列可)
3. code-reviewer に実装 skill を preload (確実に効く)
4. code-reviewer + e2e-reviewer に Skill tool 追加 + 名指し参照追従の本文指示 + done-verifier / pre-implementation-reviewer の preload only 明示
5. 最小の動作確認 (subagent の名指し skill 呼び出し)。効かなければ preload + 同名上書きに落ちる

## 3. Done（完了基準）

> harness 改修は E2E テスト枠組みが未確立なので、成果物 (skill ファイル + 参照指示) の存在と、振る舞い (実装・レビューで参照される) を事後評価で見る。

### 実装 skill のデフォルトが存在する

- 「コメントは基本書かない (コードを綺麗にして名前で意図を伝える)、コメントは Why のみ OK」をサンプル的な具体方針として持つ
- 「関連する実装規約 skill (architecture / project 固有等) があれば名指しで参照する」案内を持つ
- consumer が project 固有のコード規約 / architecture で上書きする想定が本文に明示されている

### E2E 実行 skill のデフォルトが存在する

- 「project 固有の E2E 実行手順 skill (環境セットアップ / 実行コマンド / CI 統合 / 失敗時手順等) があれば参照する」案内を持つ
- 既存 e2e skill (E2E spec の書き方・仕様書性・セレクタ・ヘルパー) とは description で責務が明確に区別されている
- consumer が project 固有の E2E 実行手順で上書きする想定が本文に明示されている

### slice-tdd が両 skill を参照するよう本文に明示されている

- 実装フェーズ (TDD ループ) の中で、実装 skill と E2E 実行 skill を参照するよう指示が入っている

### code-reviewer が実装規約を拾える

- 実装 skill を preload に追加する (現在 preload している slice-tdd に足す形、置き換えではない)
- `tools:` に Skill を追加する
- 本文に「レビュー対象の文脈で名指し参照された skill を追って読み、それに基づいてレビューする」指示を追加する

### e2e-reviewer が名指し参照を追える

- `tools:` に Skill を追加する
- 本文に「名指し参照された skill を追う」指示を追加する (今回新設する E2E 実行 skill とは別系統で、E2E spec 規約 / test framework 規約 / project 固有 E2E spec 規約の動的参照が狙い)

### subagent の名指し skill 呼び出しが動作確認されている

- 最小の動作確認で、subagent が preload 以外の名指し skill を実際に呼べることを 1 回確かめる
- 万一効かなければ、信頼できる土台 (同名上書き + preload) は残るので、保証ラインはそこに落ちる

### done-verifier / pre-implementation-reviewer は preload only を意図した設計と明示されている

- 将来「これらにも Skill を入れるか」が再燃しないよう、preload only を意図した設計である旨を本文に明示する (Skill tool は追加しない)

### (振る舞い・事後評価)

- 実装フェーズで実装 skill / E2E 実行 skill が参照され、レビュー時も実装規約 + 名指し参照先が効く

## 4. スコープ外

- **skill 名の確定**: 設計フェーズで propose-options で議論する。要件段階では枠の存在と責務だけ固める (名前に中身を引きずらせない)
- **実装規約・architecture の本格的な作り込み**: ROADMAP TODO 27 (E2E と対になる実装 skill の本格版、規約 + architecture 込み) に残す。本件 MVP は薄いデフォルト + 上書き枠まで
- **done-verifier / pre-implementation-reviewer への Skill tool 追加**: TODO 36 で「追加しない (preload only 維持)」と決定済み。証拠検証専属 / 観点を当てる責務で、他 skill 動的発見の場面が無いため
- **subagent が文脈から自律的に skill を発見する高度な動的発見**: 保証ラインは「名指し参照された skill を追う」まで。「カタログから自分で探す」は subagent の context に全 skill description が載るか不確実なので過剰約束しない
- **リリース作業そのもの**: 別タスクの引き継ぎメモ (CLAUDE.md 規律改善) と束ねて別途リリース
