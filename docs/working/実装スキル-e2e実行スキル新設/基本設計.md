## 1. アーキテクチャ (skill / agent の関係)

新設する 2 skill は実装フェーズの参照対象として slice-tdd・reviewer と次のように繋がる。

```
[ 実装フェーズ (main session) ]
        │
        ├── slice-tdd (TDD のリズム / 分割)
        │      │ 本文から参照を促す
        │      ├──► implementation  (コード規約 + architecture)
        │      │         │ 名指し参照 (任意)
        │      │         └──► consumer の architecture skill 等
        │      └──► e2e-execution   (E2E の実行手順)
        │
[ レビュー (subagent) ]
        ├── code-reviewer
        │      ├─ preload: slice-tdd + implementation
        │      └─ Skill tool: 本文が名指しする skill を追う
        └── e2e-reviewer
               └─ Skill tool: 本文が名指しする skill を追う

[ consumer 上書き ]
   implementation / e2e-execution を同名で差し替え
   → main session は差し替え後を発火、code-reviewer は preload で差し替え後を取得
   → 上書きの「機構」は README に記載 (skill 本文には書かない)
```

## 2. 新設・変更する構成要素

### 新設 skill (`.apm/skills/` 配下、symlink で `.claude/skills/` に展開)

**`implementation`**
- 発火条件 (description): 実装中にコード規約 / architecture を参照する場面。framework・言語非依存の表現で書く
- 本文の中身:
  - コメント方針 (コメントは基本書かない、コードを綺麗にして名前で意図を伝える、コメントは Why のみ) を **サンプル的な具体方針** として置く
  - 「関連する実装規約・architecture skill があれば名指しで参照する」案内

**`e2e-execution`**
- 発火条件 (description): E2E を実行する手順 (環境構築 / 実行コマンド / CI 統合 / 失敗時手順) が必要な場面。spec の書き方を担う `e2e` skill とは責務が別であることを description で区別
- 本文の中身: 「project 固有の E2E 実行手順 skill があれば参照する」案内

### 変更 skill

- **`slice-tdd`**: 本文に「実装中は `implementation` / `e2e-execution` skill を参照する」指示を追加 (置く section は実装時に判断)

### 変更 agent

- **`code-reviewer`**: preload (`skills:`) に `implementation` を追加 (現在の `slice-tdd` に足す) / `tools:` に `Skill` を追加 / 本文に「文脈で名指し参照された skill を追って読み、それに基づいてレビューする」指示を追加
- **`e2e-reviewer`**: `tools:` に `Skill` を追加 / 本文に「名指し参照された skill を追う」指示を追加
- **`done-verifier` / `pre-implementation-reviewer`**: 本文に「preload only を意図した設計 (Skill tool は持たない)」を明示 (将来の再燃防止)

### 変更 doc

- **README**: skill 上書きの機構 (project 固有 skill での差し替え方) を記載。skill 本文側には上書き機構を書かない

## 3. 発火・参照フロー (シーケンス)

```
実装時:
  main session が実装フェーズに入る
    → slice-tdd 発火 (TDD のリズム)
    → slice-tdd 本文の指示で implementation / e2e-execution を参照
    → implementation が architecture skill を名指し → main session が追う
      (main session は起動時に全 skill の description カタログを持つので名指し / 自律発見の両方ができる)

レビュー時:
  main session が code-reviewer を起動
    → code-reviewer は implementation を preload 済 (consumer 差し替え後の規約)
    → 本文が名指しする skill (architecture 等) を Skill tool で発火して追う
      (subagent は preload 外 skill を Skill tool で名指し発火できる。公式仕様で確認済:
       code.claude.com/docs/en/sub-agents 409-426 行
       「Subagents can still invoke unlisted project, user, and plugin skills through the Skill tool」)
    → 規約 + 名指し参照先に基づきレビュー
```

## 4. 論理設計 (責務境界と上書きモデル)

### 責務境界

- **slice-tdd**: 実装の進め方 (リズム / 分割)。「何を満たすか」は持たない
- **implementation**: コードの規約 + architecture (「何を満たすか」)。framework・言語非依存、project 固有は上書き
- **e2e-execution**: E2E の実行手順。spec の書き方は持たない
- **e2e**: E2E spec の書き方 (仕様書性 / セレクタ / ヘルパー / 独立性)。実行手順は持たない

slice-tdd と implementation は直交する (進め方 vs 満たすべき規約)。同時発火は正常。

### 上書きモデル

- **信頼できる土台 = 同名上書き + preload**: consumer が `implementation` / `e2e-execution` を同名で差し替えると、main session は差し替え後を発火し、code-reviewer は preload で差し替え後を取得する。両経路で確実に効く (preload は起動時に名前で解決されるため、Skill tool の有無に依存しない)
- **補助 = 名指し参照の追従**: consumer が規約を複数 skill (例: `implementation` + `architecture`) に分けた場合、本文の名指し参照を Skill tool で追う。これにより reviewer subagent でも分割構成を辿れる
- **保証ライン**: 「本文が名指しした skill を Skill tool で追う」まで。この名指し発火は公式仕様で確認済 (code.claude.com/docs/en/sub-agents 409-426 行「Subagents can still invoke unlisted project, user, and plugin skills through the Skill tool」)。一方「subagent が context のカタログから自律的に skill を発見する」は docs に明記が無いグレーゾーンなので約束しない (= 名指しまでが保証ライン)
- **文書の置き場所**: 上書きの機構は README、skill 本文は機能的な中身のみ (skill 本文は展開後も自然に読める APM 非依存表現に保つ)

### 退避 (前提が実地で覆ったときのみ)

名指し発火は公式仕様で裏付け済なので、設計の主路はこれを前提にする。ただし実装時の観測で code-reviewer が名指し skill を実際に発火しないと判明した場合に限り、信頼できる土台 (preload + 同名上書き) に退避する。土台は残るので、名指し参照追従が効かなくても実装規約はレビューに届く。
