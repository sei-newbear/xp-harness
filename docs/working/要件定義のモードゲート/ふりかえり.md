### 深掘り 1: 完了の節目でのバックログ同期漏れ
#### 経緯
Constraints 改修を v0.13.0 までリリースしたのに、対応するかんばんカードを DONE 化せず、独立して shipped したモードゲート化もカードに残さなかった。依頼者が「かんばん変えたりした?」と気づいて発覚。
#### 分析
- 事実 (記録で裏取り): このセッションで kanban に触れたのは `bca89b2` (カードに再開起点を追記) の 1 本のみ。DONE 化・新規 DONE カードなし。Constraints カードは TODO のまま残っていた。
- 根本: `kanban` skill は description に「完了の場面で使う」とあるが、完了・リリースの節目で発火させなかった。設計・実装・検証に没頭し、「バックログを現実に合わせる」という一段上の視点が節目で抜けた。
- これは下の「優先順位の逆転」と同じ構造 (節目で全体 / 一段上を見ない) の別の現れと見ている。
#### アクション
- `release` skill に Step 9「かんばん同期」を追加。リリースの締めで必ず `kanban` を呼び現実同期する、を規律化 (実装済、`e055418`)。カード化の判断・操作は kanban に一本化 (single source)。
- `kanban` skill に「完了済みを直接 DONE 記録する」操作を追加 (TODO を経ず shipped した独立項目の受け皿、実装済)。
- 残る穴 (正直): リリースを伴わない dev-facing のみの完了はこの規律を通らない。そこは kanban の完了トリガー＋人間判断に残る。「節目で一段上を見る」は完全には機械化しきれない。

### 深掘り 2: 優先順位の逆転 (観測のみ・未深掘り)
#### 経緯
主目的は「かんばんの要件定義を強化するやつ」＝ Constraints カードで、深掘り確認の強度調整は「ついで」だった。しかし設計議論が盛り上がる中で main session が「ついでが本体になった」と勝手に位置づけ直し、モードゲートを作り込んで Constraints 本体を据え置いた。依頼者が「それがメインの対応だけどやってないの?」と指摘して発覚。
#### 分析
今回は深掘りせず (依頼者判断。改善を別途入れたので一旦ここまで)。上の「バックログ同期漏れ」と同根 (節目で主目的 / 全体を見失う) と見ている。ただし今回入れた `release` Step 9 は「完了の DONE 同期」を直すもので、この「作業中に主目的を見失う」失敗そのものは直していない ＝ **開いた伸びしろのまま**。
