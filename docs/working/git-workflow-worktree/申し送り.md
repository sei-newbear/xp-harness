# git-workflow → worktree 化 — 別セッションへの申し送り

## このレポートの位置づけ

git-workflow skill を「ブランチ作成」から「worktree 作成」に変える作業の **入力**。
別セッション(xp-harness ルートで起動)で進めること。要件定義・設計はそのセッションで
define-requirements / philosophy / skill-design-style を通す。本レポートは spike で
確定した事実と、設計上の地雷を渡すだけ。

関連: `worktree-scope-spike-メモ.md`(worktree 機構の詳細な検証記録)

---

## spike で確定した worktree の事実(実測ベース)

1. **外部ディレクトリ配置がクリーン**
   - `.claude/worktrees/`(EnterWorktree 自動作成の既定)→ メイン側 `git status` に `?? .claude/` が漏れる → gitignore 必須(トリッキー)
   - リポジトリの外(兄弟ディレクトリ)に作る → 漏れゼロ・gitignore 不要・完全分離

2. **Claude 自身が切れる(2 手)**
   - `git worktree add -b <branch> <外部path>` → `EnterWorktree(path: <外部path>)`
   - EnterWorktree 自動作成(`name`)は置き場所が `.claude/worktrees/` 固定。外部に置くには `path` 形式を使う

3. **EnterWorktree は必ずリポジトリルートに入る**
   - サブディレクトリ(例 app/hoge)から呼んでも worktree のルートに着地。サブディレクトリ着地は不可(引数も設定も無い)
   - `path` にサブディレクトリを渡すと「not a registered worktree」で拒否

4. **cwd が worktree に完全移動 → 元 repo はデフォルト視界外**
   - EnterWorktree 後、セッションの作業ディレクトリは worktree。元 repo は兄弟で cwd の外 → デフォルトの検索・操作に入らない
   - 保護は権限の壁でなく「視界(cwd)の分離」。auto mode で権限が開いていても、明示的に元 repo を指さない限り触らない
   - 戻るには ExitWorktree(keep/remove)

5. **リポジトリルート起動 + 外部 worktree = 隔離はクリーン**
   - これが「Claude は worktree で作業 / ユーザーは元 repo で別作業 / git は混ざらない」の成立形
   - ただし「スコープ(無関係 dir を見せない)」は worktree では得られない(別問題、下記)

6. **スコープ(fuga 等の除外)は worktree と別軸**
   - EnterWorktree がルート入りなので、worktree セッションは全ツリーが見える
   - add-dir は足し算で引き算できない / `/add-dir` の永続化は絶対パス + settings.local.json(worktree に追従しない)
   - 除外したいなら Read deny ルール か sparse-checkout(git-workflow の責務外の可能性が高い)

## git-workflow 改修の設計論点(別セッションで詰める)

### 地雷 1: EnterWorktree は Claude Code 固有ツール
- skill-design-style 原則 2: skill 本文は agent 非依存で書く(配布後も自然に読める)。`EnterWorktree` を本文に直書きできない
- `git worktree add` は汎用 git だが、「セッションが worktree に入る」操作は agent 固有
- → 「worktree を作って、その worktree で作業する(入り方はインストール先 agent の仕様に従う)」程度の抽象表現にとどめる設計が要る

### 地雷 2: consumer 配布スキルである
- git-workflow は全 consumer に配布。worktree 化を既定にするのか、opt-in / project override にするのか
- 現状の「ブランチ提案」を置き換えるのか、worktree を選べるようにするのか

### 地雷 3: philosophy との整合
- philosophy の 4 象限で「自動 worktree」は **「痛みが明確化していない / スコープ外(Git 運用が固まってから)」** に置かれている
- worktree 化はこの過去判断を上書きする変更 → Why を立て、philosophy 側の記述更新も検討

### 地雷 4: 「branch でなく worktree」ではなく「branch + worktree」
- worktree は必ずブランチを伴う(`worktree add -b <branch>`)
- 既存の git-workflow のブランチ命名・提案ロジックを捨てるのでなく、worktree 作成に統合する形になる

### 地雷 5: 配置・命名規約
- 外部ディレクトリのどこに、どんな名前で切るか。consumer 環境で一貫する規約が要る(例: `../<repo>-worktrees/<branch>` 等)
