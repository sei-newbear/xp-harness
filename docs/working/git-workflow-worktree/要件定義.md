## 1. 背景と動機

### 痛み

Claude が自走でコード作業をしている間、利用者の手は空くことが多く、その時間に別作業を進めたい場面が多々ある。ところが同じマシン・同じ repo で別作業をしようとすると踏み合う。1 つの作業ツリーを共有しているため、ブランチ切替・作業中ファイル・index が衝突する。結果として「Claude が動いている間、利用者は手を止めて待つ」しかなくなり、空いているはずの時間を別作業に使えない。

過去の philosophy 判断（「自動 worktree = 痛みが明確化していない / Git 運用が固まってから」）が条件にしていた点については、git-workflow skill が既に配布され、ブランチ・commit・push の運用が確立している現状で「Git 運用が固まった」は満たされていると判断する。並行作業の痛みも上記のとおり実体験として顕在化しており、過去判断を上書きする条件が揃っている。

### 解決の方向

git-worktree を使うと、`.git`（オブジェクト・refs・config・stash）を共有したまま、作業ツリー + index + ブランチだけを別ディレクトリに分離できる。Claude が worktree 側、利用者がメイン側で作業すれば、両者の git が踏み合わない。これにより「Claude の自走中も利用者が並行して別作業を進める」が成立する。

### 責務境界（harness が配る規律か）

philosophy は「4 象限の前に、まず『XP の進め方の規律か、中身の流儀か』を名前に照らして問う」を最初の関門に置く。worktree 化はこの問いを通す。

開発においてコード変更に伴う Git 運用（ブランチ・commit・push）は必須の進め方であり、xp-harness は既にそれを git-workflow skill として「Git 運用の規律」の形で配布している。worktree 化は、その git-workflow が持つ「ブランチを切る」ステップを worktree 作成に置き換えるもの（worktree は必ずブランチを伴う）であって、新しい中身の流儀（コード規約・テストの文体・framework の作法）を持ち込むわけではない。よって既存の git-workflow 責務の延長として、harness が配る規律の内側に収まる。

### philosophy の過去判断を上書きする

xp-harness の philosophy では、これまで「自動 worktree」は 4 象限の「痛みが明確化していない / スコープ外（Git 運用が固まってから）」に置かれていた。本件は、上記の並行作業の痛みを Why として、その過去判断を上書きする変更にあたる。基本設計フェーズで philosophy 側の記述更新も検討する。

## 2. ユーザーストーリー

xp-harness を使う開発者が、Claude の自走中も自分は手を止めず並行して別作業を進めるために、Claude の作業を別 worktree に隔離して、自分は元 repo で作業を続けたい。

- **誰が**: xp-harness（git-workflow skill）を使う開発者（利用者）
- **何のために**: Claude が自走している間も、自分は手を止めず並行して別作業を進めるため
- **何をしたい**: Claude の作業を別 worktree に隔離し、自分は元 repo で作業を続けたい

## 3. Done（完了基準）

### worktree をブランチと同じノリで提案する

git-workflow が、これまで「ブランチを切りますか」と提案していたのと**同じ場所・同じ軽さ**で、worktree を切る提案を出す。worktree は気軽に選べる選択肢として扱い、軽量で済ませたいときはブランチも選べる。「毎回必ず worktree」でも「特別な opt-in 操作」でもなく、現在のブランチ提案の延長で worktree を選ぶ形にする。

ブランチ命名・提案のロジックは捨てず、worktree 作成に統合する（worktree は必ずブランチを伴うため）。

### worktree は repo の外に作り、Claude がそこで作業する

worktree を選んだら、repo の外（兄弟ディレクトリ）に worktree が作られ、Claude がその worktree 側で作業する状態に入る。repo 内（`.claude/worktree(s)/` 等）に作るとメイン側の git status に漏れて gitignore が必要になるため、外部配置を採る。

### 利用者は元 repo で並行作業できる

利用者は元の repo でそのまま別作業を続けられ、Claude の worktree 側の作業と踏み合わない。「踏み合わない」は skill が引き起こす観測可能な状態で判定する：worktree を選んだ後、Claude の作業対象（cwd）が repo 外の worktree になっていること、および Claude が worktree 側で作業している間、元 repo 側の git status がクリーンに保たれていること。「worktree という git 機構そのものが踏み合わない」ことの証明（git 仕様の確認）には倒さない。

### タスク完了時はトランクベースの統合フローを既定にする（基本設計フェーズで追加されたスコープ）

基本設計の議論（後始末の設計）の中で、worktree のライフサイクルの終わり側が「タスク完了時の統合フロー」に格上げされ、要件のスコープが一段広がった。経緯が追えるようここに記録する。

理想の流れ: worktree 作成 → 開発完了（テスト含む）→ 利用者に報告 → worktree の成果を main に統合 & worktree / branch を削除 → push。

- 完了時、利用者に確認を取った上で **main へ統合（トランクベース）を既定** とする。「branch のまま push（PR 運用）」「保留」も選べる
- project 固有の instruction に Git 運用ルール（PR 必須等）があれば、そちらが優先（既存の部分上書き機構と同じ）
- この完了フローは worktree 作業に限らず、**branch 作業でも統一** する（「どこで作業するか」と「成果をどう取り込むか」は独立した判断のため）
- これに伴い、git-workflow の中核哲学「PR 作成・merge は依頼者の責務（してはいけない）」のうち merge 側を「利用者の確認を取った統合は守備範囲内」に変更する。**PR 作成は引き続き依頼者の責務のまま**
- slice-tdd skill に直書きされている push 操作の案内は、git-workflow の完了フローへの委譲に書き換える（git 操作の規律を git-workflow に集約する既存方針に合わせる）

### add-dir された場所が worktree 化されないことを知らせる

worktree を切る提案時に、`/add-dir` で追加された作業ディレクトリが存在する場合のみ、どのディレクトリが隔離の外側になるかを具体的に挙げて利用者に知らせる（add-dir が無いときは知らせない）。add-dir された場所はそこで Claude が作業すると利用者と踏み合いうる穴になるため、利用者が分かった上で判断できるよう透明性を担保する。穴を塞ぐ（add-dir 先も worktree 化する / 作業を禁止する）ことはしない。

## 4. 前提（隔離の性質）

この隔離は二層に分けて考える。

- **git レベル（ブランチ・HEAD・index）**: worktree の仕組みで確実に防げる。worktree でのコミットは worktree 側のブランチに乗り、元 repo の HEAD / index は無傷（git の仕様で保証される）。
- **ファイルレベル（作業ツリーへの書き込み）**: 「cwd（視界）の分離」に依存する。worktree に入ると Claude の作業対象が worktree になり、元 repo は視界の外に出るため、普段の動作では元 repo を触らない。ただし権限では保証しない。auto mode で権限が開いていると、Claude が明示的に元 repo を指せばファイルは書けてしまう。さらにこの視界分離が auto mode で実際に守られるかは spike では未検証（調査セッションが親ディレクトリを working dir に持つ交絡があり、クリーンに確認できていない）。

「権限の壁ではなく視界の分離」という性質は前提として残す。元 repo への書き込みを権限で遮断することは今回やらない（スコープ外）。

また、worktree を作る操作は純粋な git 操作で agent に依存しないが、worktree に入って作業する状態に移るのはインストール先 agent の機構に依存する。当面は Claude Code を前提とし（Claude Code は worktree に入る機構を持つ）、それで成立すればよい。skill 本文を agent 非依存な表現で書くこと自体（特定ツール名を本文に直書きしない）は基本設計で扱う。

## 5. スコープ外

- **改修者側 dogfooding**: xp-harness 自身を worktree で並行開発できるようにする環境整備（worktree を切ると consumer 向け skill / agent の symlink が来ないため `scripts/setup-dev.sh` の再実行が要る、という固有事情への対応）。ロードマップ TODO 56 に保留。
  - なお「consumer の worktree でも配布された skill / agent が認識されないのでは」という懸念は本案件のスコープ外で問題ない。consumer は APM install で **実体ファイル**が `.claude/` に deploy され、それを git 管理するため、git 追跡された実体は worktree に追従し worktree 側でも skill / agent が効く。worktree に来ない問題が起きるのは、`scripts/setup-dev.sh` で symlink を張りつつ gitignore している **xp-harness 固有の dev 構成だけ**であり、それが上記 TODO 56。
- **add-dir 先・無関係ディレクトリの隔離 / スコープ除外**: add-dir された場所や無関係ディレクトリを worktree に取り込む / 隔離する / 作業を禁止すること。穴は塞がず、知らせるだけにとどめる。
- **元 repo への書き込みの権限遮断**: 上の前提のとおり、隔離は視界の分離で担保する。権限の壁は今回設けない。
- **cross-agent 対応**: 当面は Claude Code 前提で動けばよい。他 agent（Cursor 等）への対応はスコープ外（ロードマップ TODO 3）。
