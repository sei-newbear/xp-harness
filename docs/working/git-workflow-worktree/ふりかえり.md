# git-workflow-worktree ふりかえり

## 2026-06-11: 対話原則の発火漏れと「自走モード以外は全部対話」の境界

### サマリ
git-workflow worktree 化案件（要件定義 → 基本設計 → 実装 → 検証 → PR）を完遂したセッションのふりかえり。最も重い伸びしろとして「セッション序盤に dialogue-principles を一度も発火させていなかった」を深掘りし、根が「自走モード以外は全部対話」という境界の欠如にあると突き止めた。

### トピック一覧

よかった点:
- Why の引き出しと軌道修正（申し送りに Why が無いことを特定、「推測できるはず」の指摘に迎合せず根拠付きで自分の読みを出し当てた）
- 対話原則の仕切り直しが効いた（読み直し後、小さいラリーが最後まで機能）
- 別セッションとの spike 連携（調査依頼を 2 回出し、設計判断を実測ベースで固めた）
- reviewer 群の活用と取捨（pre-implementation-reviewer 2 回 / skill-reviewer、指摘を全採用せず議論で取捨）
- 実機検証で逸脱を発見（harness-verification で end-to-end + add-dir 肯定側を観測、文書レビューで出ない逸脱を炙り出した）
- 依頼者の指摘での設計転換が機能（remote 経由統合の撤回、rebase 統合、削除タイミング、README 再確認）

伸びしろ:
- done-verifier 文言を確認なしで変更した（「push 前に直したい」を逸脱 2 件両方への承認と拡大解釈 → 指摘されて取り消し）
- 序盤の対話原則違反（複数論点を番号付きで 1 メッセージに詰め込み、仕切り直しが必要に）
- README「書いてある」の早合点（grep の断片だけで判断、全文を読み直して一覧表の古さが見つかった）
- 検証の運転ミス（until ループの bash 構文エラー、FIFO の制御文字混入で add-dir 検証を 2 回仕切り直し）
- セッションエラーで再開不能 → 別セッションで完遂（ワークフロー上の事実）

### 深掘り 1: 対話原則違反（dialogue-principles の発火漏れ）

#### 経緯
セッション序盤、申し送り.md を読んでの分析返し・Why 推測・スコープ詰め・ROADMAP 追記・要件対話の開始まで、すべて dialogue-principles を一度も発火させないまま進めた。複数論点を番号付きで 1 メッセージに詰め込む違反が出て、依頼者の「対話の原則読み直しな」で初めて発火させた（＝「読み直し」ではなく初回発火だった）。発火後は同じく文脈の濃い基本設計の各論点でも 1 論点ずつのラリーを保てた。

#### 分析
複数の要因を順に検討した。

- 検証できる事実: このセッションで dialogue-principles の Skill 発火は依頼者の指摘より後。それまで発火は git-workflow 程度。main.instructions には「議論・対話を進める場面で必ず dialogue-principles を発火」が起動時から context にあったのに発火させなかった。
- 発火しなくてもエラーにならない: 対話規律は出力スタイルなので、忘れても出力自体は生成できてしまう。slice-tdd / git-workflow のように「物が残らない」検知が無く、違反が顕在化する経路は依頼者の違和感だけ。
- dialogue-principles は常時 context に乗っていない（philosophy / skill-design-style は CLAUDE.md から常時埋め込みだが、対話規律は発火待ち）。「違和感を感じたら再読」は違和感を検知できる前提のトリガーで、検知器が働かない場面では動かない。
- 発火していないときのデフォルト出力が違反方向を向いている（土台の「短く簡潔に」バイアス）。「発火漏れ」と「違反」は独立でなく、発火させない限りデフォルトで違反方向に出る。
- リッチコンテキストが「一気に出す」圧力をかける（basic-design 冒頭の警告と同型）。濃い入力ほど完成形で出したくなる。ただし規律を乗せた後は同じ圧力下で抑えられた＝決定要因でなくバイアス。

正直の監査: 最初に出した「作業だと分類したから」は検証できない内部状態の推測（後付けの物語の可能性）。

「なぜ作業だと思ったか」を掘ると、依頼者の入力（「@申し送り.md を見て、なにをやりたいか書いてある？書いてないなら伝えるから教えて」）が **read & report の表面構造**を持っていた。中身は Why の所在を問う要件定義の入り口で、「伝えるから教えて」は共創の合図だったが、形で分類して中身が指す先を見なかった。同じ型がセッション内に反復: README を grep 断片（形）で「書いてある」と判断 / done-verifier を「push 前に直したい」（形）で「両方承認」と解釈。**根は「表面の形で分類して、中身が指している先を見ない」**。

さらに踏み込むと、根本は「**自走モード以外は全部対話**」という境界が無かったこと。dialogue-principles の発火を「議論っぽい場面か？」と形で毎回判定するから取りこぼす。正しい境界は二値: 自走モード（実装フェーズ = slice-tdd の責務範囲）でない限り、要件定義・基本設計・レビュー共有・判断相談・ふりかえりは**すべて対話**。判定の向きを「対話だと分かったら発火」から「自走でない限り発火（デフォルト ON / 例外 OFF）」へ反転させれば、形での誤分類の余地が消える。

#### アクション
- dialogue-principles の発火条件（description）の判定の向きを反転させる: 「対話の場面で発火」という形ベースの列挙ではなく、「**自走モード（実装フェーズ）に入っている時以外は対話として発火する**」というデフォルト ON / 例外 OFF の境界で書く。具体文言は実際に改修するときに詰める（このふりかえりは気づきの記録まで。改修は別途）。
  - 補足: 常時 context への埋め込みは採らない（自走モードでは対話規律は不要で、context の無駄かつ自走の邪魔）。philosophy / skill-design-style とは性質が違う。あくまで「対話の場面で確実に発火する」を境界の引き方で担保する。
- 「形で分類して中身を見ない」を一般原則として philosophy 等に昇格させる案は**見送り**（抽象規律は効きが不確かで規律装置最小注入に反する。今回は具体的な発火境界の修正に落とす）。
