# git-workflow の worktree 化 — 基本設計

> このドキュメントは進行中の設計メモ。実装後もそのまま残す（振り返り・PR レビューで参照される）。
> 削除のタイミング・要否は依頼者または外側の仕組みで別途判断する（あなたは削除しない）。
>
> 関連資料:
> - `要件定義.md`（同ディレクトリ。Why / Done / スコープ外）
> - `worktree-scope-spike-メモ.md`（worktree 機構の実測 spike 記録。本設計の git 挙動の根拠は全てここの【実測】に基づく）

## 1. 全体フロー

```
[ コード変更を伴う依頼 / セッション開始 ]
            │
            ▼
   セッション開始時の同期確認 (既存: fetch / branch / status)
   + worktree 点検 (新規: stale worktree があれば削除提案)
            │
            ▼
   作業場所の提案 (AskUserQuestion)
   「worktree を切る (推奨)」「branch を切る」「そのまま作業」「違う名前」
            │ worktree を選択
            ▼
   worktree 作成 (../<repo>-worktrees/<branch名平坦化>/) → worktree に入る
            │
            ▼
   開発 (slice-tdd 等。高頻度 commit、push しない)
            │ 開発完了 (テスト含む)
            ▼
   done-verifier 通過 → 完了報告 + 統合確認 (AskUserQuestion)
            │
   ┌────────┼──────────────┐
   ▼        ▼              ▼
 main に統合  branch のまま   保留
 (既定・推奨)  push (PR運用)  (worktree 残す)
   │        │              │
   ▼        ▼              ▼
 統合手順    push のみ      何もしない
 + 後始末    (worktree残)   (次セッションの
 + push                     worktree 点検が拾う)
```

## 2. 作業場所の提案（branch 提案の置き換え）

現行 git-workflow の「ブランチ運用」section の AskUserQuestion を拡張する。affirmative な選択を取る規律（沈黙を同意としない、auto mode でも例外なし）は無変更で維持。

main / master にいる場合の例:

```
question: "main にいます。この session どう進めますか？"
header:   "作業場所"
options:
  - "worktree を切る (推奨)"
    description: "repo の外に別ツリーを作りそこで作業。あなたは元 repo で並行作業できる。提案名: feat/<inferred>"
  - "branch を切る"
    description: "同じツリーで軽量。並行作業しないならこちらでも十分"
  - "そのまま main で作業"
    description: "branch 切らず続行 (commit / push 注意)"
  - "違う名前で切る"
    description: "選んだら名前をチャットで教えてください"
```

- worktree を推奨にする（並行作業の価値が出る既定。ただし branch も同じ軽さで選べる、強制しない）
- 別 branch / worktree にいる場合（前セッションの続き）は、現行と同じく「そのまま続ける」を先頭にした形
- セッション中のタスク切り替えも現行構造のまま（main にいれば提案、別の場所にいれば利用者の意図を尊重して何もしない）
- branch 名の生成規約（`feat/` `fix/` 等 + English ケバブケース 5-7 語）は無変更。worktree のディレクトリ名はこの branch 名から導出する

### add-dir の通知

worktree の選択肢を出す前に、セッションに追加の作業ディレクトリ（add-dir）があるか確認する。**ある場合のみ**通知する:

- 基本: worktree 選択肢の description に含める（add-dir は worktree を選ぶときだけ意味を持つ情報のため）。例: 「注意: 追加作業ディレクトリ ../shared-libs は worktree に含まれず隔離されません」
- add-dir が多くて description に収まらない場合: description には「追加作業ディレクトリ N 件は隔離されません」と件数のみ、どのディレクトリかは直前のチャットメッセージで出す
- add-dir が無ければ何も出さない

検知は設定ファイルを読む方式ではなく、セッションが持つ環境情報（追加 working directory の一覧）から判断する（環境差で壊れにくい）。

## 3. worktree の作成と配置・命名規約

### 配置・命名

```
../<repo名>-worktrees/<branch名のスラッシュを - に変えたもの>/
```

例: repo が `myapp`、branch が `feat/order-review` → `../myapp-worktrees/feat-order-review/`

- repo の外（兄弟階層）に置く理由: repo 内（`.claude/worktrees/` 等）だとメイン側 `git status` に漏れて gitignore が必要になる（spike 実測）。外部なら漏れゼロ・完全分離
- `<repo名>-worktrees/` で束ねる理由: 複数 worktree が兄弟に散らからない、どの repo の worktree か一目で分かる、丸ごと消せば掃除が簡単
- ディレクトリ名を branch 名から機械的に導出する理由: 名前体系を増やさない。スラッシュは `-` に平坦化（中間ディレクトリを作らない）

### 作成と入り方（2 手）

1. `git worktree add -b <branch名> ../<repo名>-worktrees/<dir名>` — 純粋な git 操作
2. 作った worktree に入って作業対象を移す（Claude Code では `EnterWorktree` の `path` 指定）

注意点（spike 実測に基づく）:

- EnterWorktree の `name` 形式（自動作成）は置き場所が `.claude/worktrees/` 固定で repo 内に作られてしまうため**使わない**。必ず「`git worktree add` で外部に作る → `path` 指定で入る」の 2 手
- worktree に入ると必ず repo ルートに着地する（サブディレクトリ着地は不可）
- 外部 worktree 同士の行き来は「ExitWorktree でメインに戻る → 再度入る」で可能（直接の乗り換えは不可）

### agent 依存の扱い

当面は Claude Code 前提で確実に動くことを優先する（現行本文に既に `AskUserQuestion` の直書きがあり、汎用化は既存の将来課題）。ただし文の主語は「worktree に入る」「作業対象を移す」という操作にし、Claude Code のツール名は具体例として添える書き方にする（将来の他 agent 対応時にツール名の差し替えだけで済む）。

## 4. 完了フロー（トランクベース統合が既定）

作業場所（worktree / branch）に関わらず完了フローは統一する。「どこで作業するか」と「成果をどう取り込むか」は独立した判断のため。例外は「そのまま main で作業」を選んだ場合のみ（統合の概念がないので、done-verifier 通過 → push 確認 → push で終わり。統合確認は出さない）。

### 完了報告 + 統合確認

開発完了（テスト含む）→ done-verifier 通過 → 完了報告と統合確認を出す。出す前に 2 つ確認する:

1. **project 固有の Git 運用ルール**: project の instruction（CLAUDE.md / rules 等）に Git 運用の記載があれば選択肢と推奨をそれに合わせる。例えば「main へ直 push 禁止 / PR 必須」とあれば「main に統合」を出さない（または最下位に落とし）、「branch のまま push」を推奨にする。部分上書き機構（project ルールが skill 既定より優先）の適用
2. **メイン側の状態**: worktree の中から `git -C <メインrepoパス> status --short` と `git -C <メインrepoパス> branch --show-current` で確認。結果を質問文に添える

メイン側がきれい（main にいて未コミット変更なし）な通常ケース:

```
question: "開発完了しました (branch: feat/order-review)。成果をどう取り込みますか？
           メイン側: main、変更なし (統合可能な状態です)"
header:   "統合方法"
options:
  - "main に統合して push (推奨)"
    description: "main へ取り込んで push し、worktree と branch を削除します"
  - "branch のまま push (PR 運用)"
    description: "branch を push して終了。PR 作成はお願いします。worktree は残ります"
  - "何もしない (保留)"
    description: "worktree をこのまま残します。後で統合できます"
```

メイン側が汚れている / 別 branch にいる場合: 状態を質問文に明示し、推奨を「保留して報告のみ」に入れ替える（利用者の作業を踏まないことを優先）。「それでも main に統合する」を選んだ場合は変更の退避（stash）を明示してから行う。

「保留」を必ず選択肢に置く: 並行作業中の利用者が「今は手が離せない」と言える出口を常に確保する。worktree は残り、次のセッション開始時の worktree 点検が拾う。

### 統合手順（main に統合を選んだ場合）

前提: done-verifier は統合手順に入る**前**に通過済みであること（検証は「外に出る前の最後の砦」として、branch が生きていてまだ何も統合していない状態で行う。統合・削除の後に検証が来る順序にはしない）。統合手順中に rebase conflict を手で直した場合の再確認はテスト再実行（軽量）であり、done-verifier の再起動までは要らない。

1. **メイン側の状態を再チェック**: `git -C <メインrepoパス> status --short`。完了報告時点の「きれい」確認から統合を選ぶまでの間に、利用者の並行作業で汚れた可能性があるため（並行作業はこの機能の前提そのもの）。汚れていたら統合に進まず報告し、保留を提案し直す
2. 手元の main を最新化: worktree の中から `git -C <メインrepoパス> pull`
3. worktree 内で `git rebase main` — branch の commit を最新 main の上に乗せ直す
4. rebase で conflict したら利用者に状況共有（黙って解決しない）。解決して進めた場合は**テストを再実行**してから次へ（conflict が無ければテスト再実行は不要 — モノリポ等で触る範囲が分かれている前提。テスト済みの成果をそのまま乗せる）
5. ExitWorktree(keep) でメイン repo に戻り、`git merge --ff-only <branch名>` — rebase 済みなので必ず fast-forward。履歴は直線に保たれる。fast-forward できない場合は何かが想定とずれているサインなので、無理に進めず利用者に報告
6. 後始末: `git worktree remove <worktreeパス>` → `git branch -d <branch名>`（統合済みなので即消せる。push を待たない — 成果はローカル main の履歴に入っており、push が失敗しても失われない）
7. `git push origin main`

rebase を worktree 側で行う理由: main はメイン repo で checkout 中のため worktree から checkout できない（同一 branch の二重 checkout 禁止）。またメイン側で cherry-pick する方式（GitHub の Rebase and merge 相当）は、conflict 解決が利用者のメイン作業ツリーで起きる + commit が複製され `git branch -d` が拒否される（`-D` が必要になる）ため採らない。worktree 側 rebase なら衝突解決は worktree 内で完結し、メイン側は fast-forward と push だけの最小介入で済む。

### branch のまま push（PR 運用）を選んだ場合

従来どおり `git push origin <branch名>`。PR 作成は依頼者の責務のまま。worktree は残し、統合・削除はしない（PR マージ後、次のセッション開始時の worktree 点検が拾う）。

### 完了報告の文言

統合経路:

```
main に統合して push 完了 (branch: feat/xxx は統合済みのため削除)
worktree ../<repo>-worktrees/feat-xxx も削除しました。
```

PR 経路:

```
push 完了 (branch: feat/xxx)
PR 作成はお願いします。worktree ../<repo>-worktrees/feat-xxx は残っています
(マージ後の次セッションで掃除を提案します)。
```

## 5. セッション開始時の worktree 点検（stale 掃除、補助）

既存の「セッション開始時の同期確認」（fetch / branch / status / 同期差分）に worktree 点検を足す:

- `git branch --merged main` と `git worktree list --porcelain` の突き合わせで、main に取り込まれ済みの branch を持つ worktree（stale）を列挙
- 見つかったときだけ AskUserQuestion で削除を提案する。**消す対象の worktree 名（branch 名とパス）を質問文に列挙**し、一括削除でよい
- 削除提案には「worktree のみ消す / branch も一緒に消す」の選択を含める（`git worktree remove` では branch は消えないため）

「作業中の別セッションがいないか」を利用者に問う文言は**入れない**。利用者にもどの worktree にセッションがいるかの判断材料はなく、答えられない問いを聞いても防御にならない。代わりに機械的な安全装置で守る:

1. main に取り込まれ済みの branch しか提案しない（消しても成果は失われない）
2. 未コミット変更がある worktree は `git worktree remove` 自体が拒否する（spike 実測）。作業中のセッションがいれば通常は変更が残っているのでここで止まる。拒否されたら force せず報告（削除の安全規律どおり）
3. 残るのは「取り込み済み・変更なし・中にセッションが座っている」ケースだけで、消しても失うものがなく、宙吊りになったセッションも worktree から抜ける操作で復旧できる（spike 実測）

統合経路が既定になったので worktree は通常タスク内で消える。この点検は「PR 経路・保留経路で残った worktree」「中断されたタスクの worktree」の拾い忘れ掃除という補助の位置づけ。

制約として記録: stale 判定の `--merged` は「branch tip が main から到達可能」の判定なので、**squash merge された branch は検出できない**。検出漏れは worktree が残るだけで害は小さく、利用者の手動削除で補える。設計上許容する。

## 6. 削除まわりの安全規律（spike 実測に基づく）

- **worktree の削除は必ずメイン repo 側から実行する**。worktree の中から自分を消すと git は拒否せず消してしまい、cwd が宙吊りになる（実測）。worktree 内のセッションが消す場合は ExitWorktree(keep) でメインに戻ってから
- **branch 削除は `git branch -d`（安全側）限定**。`-D`（強制）は skill からは使わない。`-d` が拒否される = 未統合の何かが残っているサインなので、止まって利用者に報告
- **worktree 削除の `--force`**（未コミット変更があると `git worktree remove` が拒否する）も同様に、勝手に使わず利用者確認

## 7. 例外時の振る舞い

事前に分かっている分岐（project ルール）は確認する、事後にしか分からない例外は既存の汎用規律（報告して相談）に任せる — という線引き。

- 既存の「動作しないとき」section に一行追加: 「main への push が拒否された（保護されている等）→ 原因を共有して対処を相談」
- 復旧の台本（branch 立て直し → PR 切替等）は skill に書かない。起きたその場で状況から組み立てる（エンジニアとして振る舞う、規律装置最小注入）
- remote 無し repo の分岐も事前には書かない。push が無いだけで統合と後始末は同じ

#### branch 単独利用者への既定変更（影響として明記）

完了フロー統一により、worktree を使わず branch だけで作業する利用者の完了体験も変わる:

- 今まで: 作業完了 → push の確認 → push → 「PR 作成はお願いします」で終わり

従来どおりにしたい利用者は「branch のまま push」を選べば操作は一手のまま。PR 必須等のルールが project の instruction（CLAUDE.md 等）に書かれていれば統合の選択肢自体が出ない（部分上書き機構）。

上書き方法の案内は新設しない: README の「Skill management > 同名衝突時の挙動」に部分カスタマイズの一般機構（`.apm/instructions/<custom>.md` に追加ルールを書く）が既にあり、git-workflow skill 本文の「Project 固有ルールでの上書き」section にも PR 必須運用の例を書く（本設計の変更対象に含む）。重複案内は足さない。既定変更の周知はリリース時の release notes が担う。

### `.apm/agents/done-verifier.md`（consumer 配布）

description / 本文の「`git push` の前に必ず呼ぶ」「push 前」の表現を「完了フロー（統合・push）に入る前」に揃える。「完了系発話前」は無変更。

## 9. 完了の定義（実装フェーズが満たすべきもの）

harness 機構の E2E テスト枠組みは未確立のため、事前縛りは最小化し事後評価を許容する（検証は harness-verification skill の sandbox 実走を使える）。観測条件:

- [ ] コード変更を伴う依頼のとき、作業場所の AskUserQuestion に「worktree を切る (推奨)」が含まれて出る
- [ ] worktree を選ぶと `../<repo名>-worktrees/<branch名平坦化>/` に branch 付き worktree が作られ、Claude の作業対象（cwd）がその worktree に移る
- [ ] セッションに追加の作業ディレクトリがある場合のみ、worktree 提案に「その場所は隔離されない」旨が含まれる（無い場合は出ない）
- [ ] Claude が worktree 側で作業している間、元 repo 側の git status がクリーンに保たれる（利用者の並行作業を踏まない）
- [ ] 開発完了時、done-verifier 通過後に統合確認（main に統合 (推奨) / branch のまま push / 保留、メイン側の状態添え）が出る
- [ ] project の instruction に PR 必須等の Git 運用ルールがあるとき、統合確認の選択肢と推奨がそれに従って変わる
- [ ] 統合を選ぶと「main 最新化 → worktree 内 rebase → fast-forward 統合 → worktree / branch 削除 → push」が実行され、統合経路の完了報告が出る

- [ ] メイン側に未コミット変更がある状態で完了すると、統合確認の推奨が「保留して報告のみ」に入れ替わる
- [ ] fast-forward できない状況で、統合を強行せず利用者に報告して止まる
- [ ] セッション開始時、main に取り込まれ済みの branch を持つ worktree があるとき、対象の worktree 名を列挙した削除提案が出る
