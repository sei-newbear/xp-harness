# 引き継ぎ書: 引き継ぎ可能な自己完結ドキュメント案件

> 想定読者は会話文脈を持たない別セッションのエージェント。この文書だけを読んで残タスクに着手できることを目指して書く。前セッションの会話は前提にしない。

## この案件は何か

xp-harness に「別セッションのエージェントへ引き継ぐ成果物を自己完結に書く skill（handoff-docs）」と「文脈ゼロで引き継ぎ可能性を点検する subagent（handoff-verifier）」を新設し、既存の define-requirements / basic-design に連携動線を追加する案件。書き手側（自己完結に書かせる規律）と点検側（文脈ゼロで検証する subagent）の両面を1案件として実装した。

## 決定と却下（再検討で時間を溶かさないために）

- **会話の漏れ対策は「問題が顕在化してから」で着地**。会話で決めたのにドキュメントへ書き落とす漏れは、handoff-verifier（文脈ゼロ＝会話を知らない）では構造的に検出できない。専用装置（都度メモ等）は足さない。理由: 都度メモは対話テンポとトークンのコストが大きくデメリットが上回る、既存の handoff-docs 規律（Why・決定根拠・却下案まで漏らさず書く）で実際に相当カバーできている（検証で書き手は漏らさなかった）、philosophy の「規律装置最小注入・痛みが顕在化してから」と整合。実際に書き手が漏れで事故ったケースが出たら、そのとき専用の仕組みを検討する。
- **穴あきドキュメントでの No 検出テスト、および書き手自己点検の対照テストは実施しない**。理由: 検証で仕込んだ穴（認可の記述を削除）は「着手を止める穴」ではなく「スコープが狭い＝妥当性の穴」で、handoff-verifier が責務境界どおり正しく Yes を出した（点検器は正常動作）。書き手自己点検の対照は「消した版を書き手に渡す」ので、自然な書き漏れの見逃しの再現にならない（書き手は書いたはずと気づく）。

## この案件の設計判断（背景を再構成できるように）

- **handoff-verifier を新設した Why（既存との差分）**: pre-implementation-reviewer は「決定の妥当性」を、done-verifier は「コードの Done 達成」を見る。handoff-verifier は「文脈ゼロで下流タスクに着手できるか＝ドキュメントの自己完結性」だけを見る。fresh context で起動する骨格は共通だが、問いも対象範囲（ドキュメント全般）も違うので、責務分離のため分けた。
- **skill と subagent の責務分担**: 1つの handoff-docs skill が「書き手規律」と「点検を呼ぶ動線」を束ねる（slice-tdd が done-verifier 動線を持つのと同型で単一責任は破らない）。subagent は文脈ゼロ点検に限定。
- **文脈ゼロの定義と呼び出し規律**: ゼロにするのは会話文脈であって、プロジェクトの既存資産（コード・他ドキュメント）は読んでよい。呼び出し側（main）は成果物のファイルパスと下流タスクだけを渡し、会話の背景・要約を渡さない（渡すと文脈ゼロ点検が成立しない）。この規律は handoff-verifier の description と本文の両方に明記してある。

## 前提・制約

- `.claude/` 配下の symlink は gitignore（`scripts/setup-dev.sh` が各環境で生成）。commit 対象は `.apm/` 配下の実体のみ。

- harness 改修は E2E テスト枠組み未確立。検証は harness-verification（sandbox 実走）で行う。done-verifier はテスト・build 前提なのでこの案件には素直に当たらない。

## 検証セッション駆動の運用メモ（次に検証するとき用）

- `claude-launcher.sh` は改善済み: `launch <name> <dir> [session-id]`（session-id で --resume 復帰）、`send`（内部で dialog 閉じ→クリア→Enter で submit 保証、ただし送信後に通知が被ると未送信になることがあるので log で処理開始を確認する）、`wait-idle <name> [quiet] [timeout]`（main+subagent の transcript mtime を見る。処理が始まらなければ NOSTART を返す＝送信失敗）、`stop`（script プロセスも落とす）。
- 画面ログ（`log`）は端末の生ダンプで過去の "Pollinating…" 等が残骸として残る。アイドル判定を log の grep でやると誤検知するので、待機は必ず `wait-idle`（transcript mtime）を使う。
- 前セッションでは main session 側のツール出力に不正文字列の連発バグが起き、そのため PR 確認まで到達できなかった。作業成果（実装・検証・PR）には影響していない。
