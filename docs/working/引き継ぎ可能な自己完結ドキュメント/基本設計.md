# 引き継ぎ可能な自己完結ドキュメント — 基本設計

> このドキュメントは進行中の設計メモ。実装後もそのまま残す（振り返り・PR レビューで参照される）。
> 削除のタイミング・要否は依頼者または外側の仕組みで別途判断する（あなたは削除しない）。
>
> **想定読者はこのリポジトリを引き継ぐコーディングエージェント（別セッション）**。会話文脈ゼロで読んで実装に着手できるコンテキストの完全性を優先して書いている（長さより漏れのなさ）。要件は同ディレクトリの `要件定義.md` にある。

## 0. 設計の要約

要件で決めた「書き手側（自己完結に書かせる規律）＋ 点検側（文脈ゼロで引き継ぎ可能性を検証する subagent）」を、次の構成で実装する:

- **新規の横断 skill 1つ**（仮称 handoff-docs）: 別セッションへ引き継ぐ成果物を書く場面で発火し、自己完結性の規律（書き手側）と、書き終える箇所で点検 subagent を呼ぶ動線を持つ。
- **新規の点検 subagent 1つ**（仮称 handoff-verifier）: 会話文脈を持たずに起動し、成果物だけから「文脈ゼロの別セッションが下流タスクに着手できるか」を Yes/No ＋ 穴の具体指摘で返す read-only な点検者。
- **既存 define-requirements / basic-design の改修**: 重複する自己完結規律を除いて新 skill を呼ぶ動線に置き換え、成果物完成箇所での動線順序（妥当性が先・引き継ぎ可能性が後）を明記する。

仮称の確定は実装フェーズで skill-design-style の description 設計手順に従う（名前は発火精度に直結するため、役割が固まった実装段で詰める）。本設計では役割ベースで「新 skill」「点検 subagent」と呼ぶ。

## 1. アーキテクチャ（コンポーネント配置と動線）

### 配置

- 新 skill: `.apm/skills/<name>/SKILL.md`（利用者向け配布物）。`scripts/setup-dev.sh` が `.claude/skills/<name>` への symlink を作って改修者環境でも認識させる。
- 点検 subagent: `.apm/agents/<name>.md`（利用者向け配布物）。同様に `.claude/agents/<name>.md` への symlink。
- 既存 define-requirements / basic-design: `.apm/skills/` 配下の本文を改修（利用者向け配布物）。

すべて利用者向け（`.apm/` 配下）なので、この案件は PR を出してレビューを通す対象。

### 動線図

```
[ 別セッションへ引き継ぐ成果物を書く場面 ]
        │
        ├─ 引き継ぎ書 / PRD / README 等（横断的な成果物）
        │       │
        │       ▼
        │   [ 新 skill 発火 ]───── 書き手規律（自己完結性の単一出典）
        │       │                  で書く
        │       │
        │       ▼ 成果物を書き終える箇所
        │   [ 点検 subagent を呼ぶ動線 ]
        │
        └─ 要件定義.md / 基本設計.md（特定フェーズの成果物）
                │
                ▼
            [ define-requirements / basic-design ]
                │  成果物完成箇所
                ├─(1) pre-implementation-reviewer を提案（妥当性・規模で任意）  ← 先
                └─(2) 新 skill を呼ぶ（自己完結性 + 引き継ぎ点検）             ← 後
                        │
                        ▼
                    [ 新 skill の点検動線 ]
                        │
                        ▼
                [ 点検 subagent（会話文脈ゼロ・read-only）]
                        │
                        │ 成果物ファイルのみ + プロジェクト既存資産を読む
                        ▼
                    Yes / No + 穴の具体指摘
                        │
                        ▼
                [ main session がループを回す ]
                  No → 成果物を直して再点検 → … → Yes で完了
```

要点は、**点検動線を新 skill に集約している**こと。既存 define-requirements / basic-design は点検 subagent を直接呼ばず、新 skill を呼ぶ。これで既存 skill 本文に subagent 動線が二重にぶら下がらない（レビュー指摘「成果物完成箇所に2つの subagent 動線が並ぶ」への対応）。既存 skill に残るのは「pre-implementation-reviewer 提案」と「新 skill 呼び出し」の2動線で、問いが違う（妥当性 / 引き継ぎ可能性）ので順序を明記して区別する。

## 2. データモデル

該当なし。この案件は harness 機構（skill / subagent / instruction）の改修で、永続データ・DB スキーマを持たない。

## 3. シーケンス（引き継ぎドキュメント作成から点検ループまで）

```
[ main session ] 別セッションへ引き継ぐ成果物を書く
     │
     ▼ 新 skill の書き手規律に沿って成果物を書く
     │   （Why / 決定根拠 / 前提 / 制約 / スコープ境界 / 却下案 を漏らさず、
     │    文脈なしで読める表現で）
     │
     ▼ 書き終える箇所で点検 subagent を呼ぶ
[ main session ] ── Task(点検 subagent) を起動 ──▶ [ 点検 subagent ]
     │                                                  │ 会話文脈を持たず起動
     │  prompt に渡すのは：                             │
     │   - 成果物のファイルパス                         │ 成果物ファイルを Read
     │   - 下流タスクの指定                             │ 必要ならプロジェクト
     │     （例「この要件定義から実装に着手できるか」）  │ 既存資産も Read/Grep
     │  prompt に書かないもの：                         │
     │   - 会話の背景・要約・意図の補足                 │ 下流タスクへの着手を試行
     │                                                  │ 止まる箇所・穴を検出
     │                          ◀── Yes/No + 穴 ────────┘
     ▼
[ main session ] 判定を処理
     │  No → 穴を直す → 再度 Task(点検 subagent) → …（Yes まで繰り返す）
     │  Yes → 引き継ぎ可能として完了
     ▼
```

エラー・境界のケース:

- **点検 subagent への prompt に会話文脈が漏れ込む**: 文脈ゼロが崩れ、点検の意味が失われる最大の失敗。これを防ぐため「呼び出し側は成果物パスと下流タスクだけを渡す」を subagent の description と本文の両方に規律として明記する（下記4節）。
- **No が延々と続いて Yes に収束しない**: main が同じ穴を直しきれていないサイン。穴の具体指摘を1つずつ潰す。収束しない場合は成果物の構造自体を見直す（要件・設計の欠落が根因の可能性）。

## 4. 論理設計（責務分離）

### 新 skill の責務

「引き継ぎ可能な自己完結ドキュメントを作らせる」の1責務。内訳は "どう書くか（書き手規律）" と "書けたら点検を呼ぶ（動線）" で、これは1責務の裏表。slice-tdd skill が「TDD で実装させる」責務の中に done-verifier を呼ぶ動線を持つのと同じ構造なので、単一責任を破らない。点検動線は書き手規律が効く場所（書き終える箇所）に置く（skill-design-style の「他 skill / subagent を参照させるときは効く場所で呼ぶ」）。

### 点検 subagent の責務

「引き継ぎ可能性＝文脈ゼロで下流タスクに着手できるか」だけを点検する。read-only（tools は Read / Grep / Glob / Bash、Edit / Write を持たない）。出力は Yes/No ＋ 穴の具体指摘。成果物は直さない（直すのは書き手＝ main session）。

### 既存 reviewer 群との責務差分（新設の Why、レビュー最重要指摘への回答）

新規 subagent を新設する根拠は、既存と問いが違うこと:

- **pre-implementation-reviewer**: 対象は要件定義.md / 基本設計.md に限定。問いは「決定の**妥当性**」（Why が弱くないか、Done が検証可能か、設計に無理がないか、既存資産と整合するか、リスクはないか）。ドメインの中身に踏み込んで批判する。自己完結性の崩れも見るが6観点のうちのおまけ。
- **done-verifier**: 対象はコードと要件の Done。問いは「宣言の真偽」（テスト再実行・build・TODO 残存で Done が実行結果として達成されたか）。
- **点検 subagent（新設）**: 対象はドキュメント全般（要件・設計に限らず引き継ぎ書・PRD・README）。問いは「妥当性ではなく、**文脈ゼロで下流タスクに実際に着手できるか**」。妥当性は問わない。実証されたやり方は「成果物だけ渡して次のタスクをやらせ、止まるかを見る」。

「fresh context で読む」骨格は既存にもあるが、問い（妥当性 / 引き継ぎ可能性）と対象範囲（特定成果物 / ドキュメント全般）が違う。責務が違うものを subagent 数を惜しんで混ぜると単一責任が崩れ、description が曖昧化して発火精度が落ちる。よって分ける。

### 書き手規律の単一出典

「別セッションのエージェントが読んで自走するための自己完結性」の規律は新 skill が単一出典として持つ:

- コンテキストの網羅: Why / 決定根拠 / 前提 / 制約 / スコープ境界 / 却下案 を漏らさず書く。
- 文脈なしで読める表現。

既存 define-requirements / basic-design が持っていた「自己完結する文章にする」相当は新 skill に寄せ、既存はそこを呼ぶ。既存に残すのは要件定義.md / 基本設計.md **固有の書式ルール**（「省略記号を持ち込まない」「Done に番号を振らず内容で参照する」など、その成果物固有のもの）。これで自己完結性の規律が2か所に重複しない（skill-design-style の「重複させない・単一の出典」）。

### 動線の順序（妥当性が先・引き継ぎ可能性が後）

成果物完成箇所で pre-implementation-reviewer（妥当性）と 新 skill 経由の点検（引き継ぎ可能性）が両方効きうる。順序は妥当性が先。理由: 中身の妥当性がまだ固まっていないのに引き継ぎ可能性を点検しても、中身が変われば書き直しになる。既存 skill 本文にこの順序と役割の違いを明記して、main session が迷わないようにする。

### 文脈ゼロの定義と呼び出し側規律

- **ゼロにするのは会話文脈**であって、プロジェクトの既存資産ではない。引き継ぎ先の別セッションも、会話履歴は見えないがプロジェクトのコード・他ドキュメントは読める。点検 subagent も同じ条件（会話履歴なし・プロジェクト資産あり）で、引き継ぎ先の実際の条件を再現する。
- **呼び出し側規律**: main が点検 subagent を呼ぶとき、prompt に渡すのは成果物のファイルパスと下流タスクの指定だけ。会話の背景・要約・意図の補足を prompt に書かない。これを subagent の description と本文の両方に明記する。技術的な文脈ゼロ（fresh context 起動）は既存で成立済みなので、新しく作るのは仕組みではなくこの呼び出し規律。

### ループの回し手

main session が回す。点検 subagent は read-only で成果物を直せない。subagent は Yes/No ＋ 穴を返すだけ、main が穴を直して再点検を呼び、Yes まで繰り返す（既存 reviewer と同じ「subagent は判定材料を返す、main が処理する」構造）。

## 5. 既存資産への影響

- **改修する既存ファイル**:
  - `.apm/skills/define-requirements/SKILL.md`: 重複する自己完結規律を除去し新 skill を呼ぶ動線に置換、成果物完成箇所の動線順序（妥当性→引き継ぎ可能性）を明記。
  - `.apm/skills/basic-design/SKILL.md`: 同上。
- **新規追加**:
  - `.apm/skills/<name>/SKILL.md`（新 skill）と `.claude/skills/<name>` symlink。
  - `.apm/agents/<name>.md`（点検 subagent）と `.claude/agents/<name>.md` symlink。
  - `scripts/setup-dev.sh` の symlink 対象に新 skill / subagent を追加（既存の symlink 生成の仕組みに合わせる）。
- **変更しない既存資産**: pre-implementation-reviewer / done-verifier の本文は変えない。新旧の責務差分は新 subagent 側の description / 本文に書く（既存を触らずに境界を立てる）。
- **main.instructions への影響**: レビュー / 検証用 subagent の一覧表に点検 subagent を1行足すか要検討（実装時に判断。表に載せると main session が起動時に存在を認識できる）。
- **migration / permission 影響**: なし（永続データ・権限機構を持たない）。
