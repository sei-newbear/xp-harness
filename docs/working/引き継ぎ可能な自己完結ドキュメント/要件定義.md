# 引き継ぎ可能な自己完結ドキュメント — 要件定義

> このドキュメントは進行中の設計メモ。実装後もそのまま残す（振り返り・PR レビューで参照される）。
> 削除のタイミング・要否は依頼者または外側の仕組みで別途判断する（あなたは削除しない）。
>
> **想定読者はこのリポジトリを引き継ぐコーディングエージェント（別セッション）**。人間向けの簡潔さではなく、会話文脈ゼロで読んで自走できるコンテキストの完全性を優先して書いている（長さより漏れのなさ）。この方針自体が、本案件で作ろうとしている skill / subagent のテーマと同じもの。

## 0. この案件が何か（一段の要約）

エージェントが「別セッションへ引き継ぐ前提の成果物」を書くとき、自己完結性の穴に自分で気づけない。これを **書き手側（自己完結に書かせる規律）** と **点検側（文脈ゼロで引き継ぎ可能性を検証する subagent）** の両面で解決する。成果物は次の3つ:

- 新規の横断 skill 1つ（書き手側の規律 + 点検 subagent を呼ぶ動線）
- 新規の点検 subagent 1つ（会話履歴を渡さず、成果物だけで引き継ぎ可能性を Yes/No 判定）
- 既存の define-requirements / basic-design 本文への連携動線追加（成果物を書き終える箇所で点検を効かせる）

## 1. 背景と動機

### 解こうとしている痛み

エージェントが「別セッションへ引き継ぐ前提の成果物」（要件定義・基本設計・引き継ぎ書・PRD・README 等）を作ると、書いた本人の主観で「完成・自己完結している」と宣言してしまいがち。

原因は、**書き手自身の会話の文脈が "のり" になっている**こと。ドキュメントに書かれていない前提を、書き手は脳内で補完しながら読む。だから本人には完成して見える。この「のり」があるかぎり、書き手本人も、文脈を持ったままの自己点検も、穴を見つけられない。穴は「文脈ゼロの読み手」にしか見えない。

引き継ぎ先の別セッションは、まさにその「文脈ゼロの読み手」。ドキュメントに前提・判断の経緯・制約が漏れていると、設計確認で止まる（引き継ぎ先が書き手に問い合わせられない）か、設計意図とズレた実装に流れる。

### 想定読者がエージェントであることの含意

これらの成果物の読み手は人間ではなく、別セッションのコーディングエージェント。人間向けの読みやすさを意識して情報を削ると、別セッションで前提・判断の経緯が欠落する。

したがって前提が反転する: **簡潔さより文脈の完全性を優先する。長くてよい。** むしろ、別セッションでそのまま動くために必要なコンテキスト（Why / 決定の根拠 / 前提 / 制約 / スコープ境界 / 却下した案とその理由）を漏れなく書き起こす方向に倒す。

### なぜ xp-harness に取り込むか（責務境界の判断）

この機能を xp-harness に取り込むか、Claude Code 本体への上流フィードバックに留めるかを、philosophy の責務境界（XP の進め方の規律か / 中身の流儀か）に通した:

- 「引き継ぎ可能性を文脈ゼロで点検する subagent」は、既存の done-verifier / pre-implementation-reviewer と同じ**ペアプロ的レビューの系譜**。観測可能な Done（別セッションが止まらず進めるか）を担保する装置で、XP の「受け入れテスト」「観測可能な完了」の考え方に乗る。
- 「別セッションのエージェントが自走できる自己完結ドキュメントを書く規律」も harness が配る「進め方の規律」側であって、コード規約・命名のような中身の流儀ではない。

したがって「XP の進め方の規律」側と判断し、xp-harness に取り込む（philosophy の4象限では「直接採用 OK」寄り）。上流フィードバックに留めるなら skill / subagent を作らないので、取り込む＝作る、と表裏。

### 実証されたやり方（この機構の裏付け）

「文脈ゼロ」のサブエージェントを立て、成果物だけ（会話履歴なし）を渡して次をやらせる方法が、実セッションで有効だった:

1. 成果物だけを頼りに下流タスクに着手できるか
2. 矛盾・抜け・曖昧点はどこか
3. 「文脈ゼロの別セッションが、設計確認で止まらず進められるか」の Yes/No

そして直して再実行し、Yes になるまで回す。実セッションでは、これが「完了」として出てしまうはずだった具体的な穴を捕まえた:

- セクション間でフィールドの置き場が食い違っていた
- 必須の列挙集合が書かれていなかった
- 仕様が上書き層に散らばっていて、再構成に4手順の脳内パッチが要った

1回目は No、修正後は Yes になった。肝は「点検者が会話の文脈をゼロにする（成果物だけ渡す）」こと。実際の引き継ぎ条件を再現できるのはそれだけだから。

この機構は done-verifier（コードが要件の Done を達成したか実行で確認する）とは別物。点検するのは「文脈ゼロの読み手にとってのドキュメントの自己完結性」= 引き継ぎ可能性であって、コードの Done 達成ではない。

## 2. ユーザーストーリー

> **別セッションのエージェントに成果物（設計書・引き継ぎ書・PRD・README 等）を渡して作業を継続させたい開発者が、引き継ぎ先が設計確認で止まらず・意図とズレずに自走できるように、自己完結なドキュメントを引き継ぎたい**

実際に手を動かすのは開発者の代理である main session エージェント。

点検（文脈ゼロ subagent、Yes になるまで回す）は、この「自己完結なドキュメントを引き継げる」を**どう担保するかの手段**であって、ストーリー本体には含めない。手段をストーリーに混ぜると手段に縛られるため、意図的に外している。点検は Done と実現手段の側で扱う。

## 3. Done（完了基準）

観測可能な形で書く。harness 改修なので事前縛りは最小化し、事後評価を許容する（philosophy: harness 機構の E2E テスト枠組みは未確立）。

### 書き手側

- 開発者が「別セッションに引き継ぐ成果物を書きたい」場面で新 skill が発火し、自己完結性の規律に沿ってドキュメントが書かれる。
- 自己完結性の規律とは、別セッションのエージェントが読んで自走するために漏らしてはいけないコンテキスト項目を書き切らせること（Why / 決定の根拠 / 前提 / 制約 / スコープ境界 / 却下した案とその理由 など）。

### 点検の動線

- skill が、成果物を「自己完結だ」と宣言する直前に、文脈ゼロ点検 subagent を呼ぶよう促す。

### 点検側

- 点検 subagent が、会話履歴なし・成果物だけを渡され、「文脈ゼロの別セッションが設計確認で止まらず進められるか」を **Yes/No + 穴の具体指摘** で返す。
- No の指摘は、どこが自己完結していないか（食い違い・欠落・散らばり等）を具体的に挙げる。

### 回すループ

- No なら成果物を直して再点検し、Yes になるまで回る。

### 検証

- harness-verification で、skill の発火と subagent の文脈ゼロの振る舞いが sandbox 実走で確認できる。
- 「文脈ゼロで点検する」を**どう技術的に担保するか**（会話履歴を渡さない呼び出し方）は実現方法なので、この要件では「文脈ゼロで点検できること」だけを Done に置き、実現方法は基本設計フェーズに回す。文脈ゼロが技術的に難しいと判明した場合は要件に跳ね返る可能性があるため、基本設計で最初に検証する論点として送る（下記5節）。

## 4. スコープ

### スコープ内

- **新規の横断 skill 1つ**。「別セッションのエージェントが読んで進める前提の成果物」全般 — 要件定義・基本設計に限らず引き継ぎ書・PRD・README なども含む — を対象にする。担うのは (書き手側の規律) と (点検 subagent を呼ぶ動線) の両方。
- **新規の点検 subagent 1つ**。文脈ゼロの点検側。責務は「引き継ぎ可能性 = ドキュメントの自己完結性」に限定する。
- **define-requirements / basic-design 本文への連携動線の追加**。要件定義.md / 基本設計.md を書き終える箇所（＝動線が効く場所）で、点検 subagent を呼ぶ／自己完結性の規律を効かせる。skill-design-style の「他 skill を参照させるときは効く場所で呼ぶ」に従う。

### スコープ外

- **retrospective ほか他 skill への連携動線**。retrospective は改善を引き出す skill で、「別セッションへ引き継ぐ成果物を書く」性質ではないため、連携対象にしない。連携対象は成果物を生む define-requirements / basic-design の2つに絞る。
- **文脈ゼロ点検 subagent の実装詳細**（会話履歴を渡さない呼び出し方の具体）。基本設計フェーズの範囲。
- **引き継ぎ書・PRD・README など成果物種別ごとの具体テンプレートの量産**。skill は「何を漏らさず書くか」の規律を配るのが責務で、種別ごとのテンプレ整備までは踏み込まない。
- **harness-verification での実走検証の実施そのもの**。Done には「検証できること」を含めるが、検証を回す作業自体は実装後の別ステップ。

## 5. 基本設計フェーズに送る論点

要件では決めず、基本設計で詰める（ここに残すのは引き継ぎ先が設計確認で止まらないため）:

- **文脈ゼロ点検をどう実現するか**。会話履歴を渡さず成果物だけを subagent に渡す呼び出し方。ここが技術的に成立するかを最初に検証する（成立しなければ Done に跳ね返る）。
- **「Yes になるまで回す」ループを誰が回すか**。main session が回すのか、subagent 自身が回すのか。
- **新規横断 skill と既存 reviewer 群の責務切り分け**。点検 subagent は done-verifier（コードの Done 達成を実行確認）とも pre-implementation-reviewer（要件・設計の妥当性レビュー）とも別物で、引き継ぎ可能性に限定する。この線引きを設計に明記する。
- **新規横断 skill と define-requirements / basic-design の責務の重複回避**。自己完結性を書く規律は新 skill に集約し、既存 skill は動線で呼ぶだけにするなど、単一責任を壊さない構造にする。
- **新規横断 skill と点検 subagent を1案件で作ることの単一責任**。skill が (書き手規律) と (点検動線) の両方を束ねる構造が単一責任を破らないか。

## 6. 判断の経緯（引き継ぎ先が背景を再構成できるように）

- **書き手側と点検側は分割できない1ストーリー**。書かせても点検しないと穴に気づけない／点検だけでは書き手を導けない。片方だけでは価値が立たないため、ユーザーストーリーとして分割せず1案件として扱う（元は tier1 の2カード「成果物の引き継ぎ可能性を保証する」= 点検側 と「設計ドキュメントをエージェント向けに自己完結化」= 書き手側 だったものを統合した）。
- **既存 skill 本文を触るか否かの線引きは一度反転した**。当初「独立した横断 skill にするから既存 skill 本文は触らない」とスコープ外に置いたが、成果物を書き終える箇所で点検動線を効かせる必要があるため、define-requirements / basic-design 本文への連携動線追加はスコープ内に修正した。ただし連携対象は成果物を生む2 skill に限定し、retrospective は性質が違うので外した。
