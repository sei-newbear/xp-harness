# slice-tdd: アウトサイドインとリズム強化 — 要件定義

## Why（何の改善か、なぜ要るか）

ある実プロジェクトの実装フェーズを数時間まるごと自走させた検証で、成果（ストーリーの Done）は達成できる一方、slice-tdd の TDD 規律が自走で系統的に崩れることが確認された。崩れは3つ:

1. **アウトサイドインが per-cycle になっていない** — E2E を最初に「書く」だけで動かさず、実装は最内層から積む内→外（ボトムアップ）になった。外側の層から書く選択肢が浮かびすらしなかった。
2. **リズム（小サイクルの回し方）が欠ける** — refactor が全サイクル0、skip 理由の記録も無し。「緑になったら code-reviewer」が「substantial な節目で1回」に化け、後半のコードが無レビュー。
3. **E2E を早く頻繁に流す規律がない / e2e-execution 未発火** — E2E を終盤に1回溜めた結果、層をつなぐ統合の不備が最後にまとめて噴出。

再現性を上げるには harness 側（＝振る舞いを宿す skill 本文）の手当てが要る。中身の流儀（何をテストするか／E2E＝仕様・ユニット＝中心 等）は別論点でこのカードには含めない。

## 診断（依頼者と議論して固めた）

3つとも「ルールが無い」のではなく「ルールはあるのに**理解が乗っていない**（mental model を持っていない）」。本文を読むと refactor も code-reviewer も E2E 実行も既に書いてある。それでも数時間の自走で崩れた。

- アウトサイドイン: 本文がモック2サイクル型に特殊化しすぎて、「各サイクルがその層で外→内」という**一般原理を教えていない**。テンプレに合わない状況で拠り所を失いボトムアップに落ちた。
- refactor 全サイクル0・skip 理由すら無し: refactor を「拍の一部」として持てていない（視界に入っていない）。
- E2E 終盤溜め: 「早く頻繁に流すと速くフィードバックが返る」価値が腑に落ちていない＋本文が「最初に書いて動かす」を**一度きりの初手**として書いていた。

この診断が手当ての向きを決める（後述「装置化を主軸にしない理由」）。

## Done（観測可能な完了条件 — harness は E2E 枠組み未確立のため事後評価許容）

- slice-tdd 本文に、アウトサイドインの一般原理（一番外側のまだ無い層から、各サイクル外→内）とアンチパターン（ボトムアップ）が明示されている。
- モック2サイクル型が「唯一の型」ではなく「一例」に降格され、層が増えればサイクルが増える旨が書かれている。
- 緑ごとの code-reviewer / E2E 実行のトリガーが、Step 2 の効く手順（per-cycle の位置）に置かれている。
- 文言を盛らず（規律装置最小注入と整合）、肥大化していない。
- 最終的な効き目の確認は harness-verification の自走検証で事後に評価する（このカードの検証と同じ経路）。

## スコープ

- 対象: `.apm/skills/slice-tdd/SKILL.md`（利用者向け → PR レビューを通す）。
- 3つの崩れを1回でまとめて手当てする（依頼者合意）。
- 含めない: 中身の流儀（何をテストするか）、装置化（言い訳対応表 / Red Flags）の全面導入。
