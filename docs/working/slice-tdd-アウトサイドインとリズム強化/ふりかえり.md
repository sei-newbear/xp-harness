- 合意済み用語（拍・緑）を提案で再導入 ← 未深掘り
- 初期診断が装置寄り（装置 vs 理解の切り分けを最初に外した）← 未深掘り
- 検証条件の realism を最初に詰めなかった（docker / DB / 重さを依頼者に押されて後追い）← 未深掘り
- リリースノートを内部 how で書いた ← 深掘り済（下記）

### 深掘り 1: リリースノートの書きっぷり（user-facing を内部 how で書いた）

#### 経緯
v0.11.0 のノートで、user-facing に内部詳細（書き直しの詳細・行数・用語統一）を書き、依頼者に「細かすぎ、どう振る舞うようになったかでは？」と指摘され behavior-focused に直した。

#### 分析
`release` skill の Step 4 は「配布先での分類（ユーザー向け / 開発者向け）」は導くが、「**user-facing は consumer が観測する挙動で書く**」という内容の書きっぷりを書いていなかった。加えて、style 参考にした前回 v0.10.0 のノートは実は behavior-focused だったのに、format だけ真似て "挙動で書く" パターンを抽出しなかった。学びは次セッションに残らないので、durable な当て所は skill 側。

#### アクション
`release` skill Step 4 に「内容の書きっぷり: ユーザー向けは観測可能な挙動の変化で書く（内部の how は書かない）」を良い / 悪い例つきで追記（実施済み）。
