# slice-tdd: アウトサイドインとリズム強化 — 基本設計

## 崩れの整理（実走記録の裏取り後）

自走検証の実走記録（ロール／権限のアクセス制御を1スライスで通す実装、実スタックの重い E2E）で、崩れは以下に整理できた。

- **エージェントは自分が slice-tdd / outside-in に従っているつもりだった**。E2E を「Red north-star」と呼び、節目レビューを「per slice-tdd」と述べていた。崩れは意図的逸脱でなく、「E2E を最初に書いた」「節目でレビューした」を規律遵守と自己解釈した結果で、スキルの文言がその退化した読みを許した。

## 3 つのコア（＋補足）

書き直しはこの3コアを軸にする。文言を盛るのでなく、欠けていた芯（入れ子モデル）を入れ、退化した読みを潰す。

### コア1: サイクルは入れ子（マトリョーシカ）

1 サイクル = レッド → グリーン → レビュー → リファクタ → コミット（分割しない一続き）。大きいサイクル（ストーリーを E2E で駆動）の中に小さいサイクル（層ごと）が入れ子で並び、**小サイクルも大サイクルと同じ形**。実装は「大きいサイクル1本」でなく、中の小サイクルを1つずつ開いて各々回す。一番外の人形だけ開けて中を開けない＝大サイクル1本に潰す、が崩れ。レビュー / リファクタは各サイクルに乗る。

### コア2: どのサイクルも常に outside-in

outside-in は **外側の層から内側の層へ進む方向** のこと。入れ子のどのサイクルにも毎回かかる。アンチパターンはボトムアップ、特に **形だけ outside-in**（最上位で E2E を書くだけ、中は内側から積む）。

**境界: outside-in とテストは別物**。outside-in は進む向きだけを言い、テストの書き方とは無関係。TDD だからテストから書くが、そのテストが何を担保するか・どういう役割か（設計を洗練させる道具か、仕様の受け入れか）・どの粒度で書くかは project 固有（`implementation` / `e2e` skill が探す）。slice-tdd はここに踏み込まない。当初「その層の外側テスト（インターフェース）」と書きかけたのは、outside-in をテストの書き方に混線させた誤りで、向きだけに締め直した。

### コア3: E2E をちゃんと流す

- e2e-execution を**ちゃんと使う**（実行手順を自前で再発明しない）。再発明は無駄な探索・トークン浪費で、実行を実際以上に重く見せて「流さない」言い訳を生む。
- リズム: 作業中は**今触っている層に関係する E2E だけ**を各サイクルで流し、**最後に全件**流す。関係分だけなら軽いので「全件は重い」が「何も流さない」の言い訳にならない。
- 重さは2原因（docker compose 等で本当に重い／軽い経路を使わず自前で重くした）。両方を上のリズムで捌く。プロジェクト側に抑止系（実行ポリシー）があれば従う。

### 補足（コア外・最小の装置）: 退化した読みを潰す

「失敗時の規律注入」（philosophy 軸1、失敗モードが大きいと実証された所にだけ）。退化した読みが"合格"にならない文言を要所に置く:

- 「E2E を書いても走らせず内側から積んだら outside-in ではない」
- 「大きな節目で溜めて1回はグリーンごとのレビューではない」。あわせて「小さいサイクルで領域は重なるので、別サイクルで再レビューになるのは問題ない（別観点が出るのはむしろ価値）」を添え、"レビュー済みだから飛ばす" という台帳管理の発想を持ち込ませない
- 「あとの単位でまとめて直すは refactor ではない」

## 設計判断: グリーンごとレビューのコストは価値観で解く

入れ子が深いほどグリーンごとの code-reviewer 呼び出しが積み、この強制コスト自体が「サイクルを潰して呼び出しを減らす」誘因になりうる（skill-reviewer が指摘した緊張）。これを「意味のあるグリーンだけレビュー」に緩めると、記録にあった「節目で1回に化けて以降が無レビュー」の抜け穴を再び開ける。

解き方は enforcement でも「様子見で観測」でもなく、**価値観を伝える**こと: 小さく都度レビューする方が大きな手戻りを防いで結果的に安定して速い、それがペアプロの価値そのもの。だから呼び出しはコストでなく投資で、減らすためにサイクルを潰すのは本末転倒。これをペアプロ section の「なぜグリーンごとに呼ぶか」に明記した。xp-harness の「価値で導く／ペアプロ哲学」と整合。

## 実装した変更点

`.apm/skills/slice-tdd/SKILL.md`（＋ 記録先ルールの整合で `.apm/instructions/main.instructions.md` 1 箇所）。

- 「実装方針: outside-in TDD」を入れ子モデルに書き換え（サイクルの定義 → マトリョーシカ → 常時 outside-in → アンチパターン[形だけ outside-in 含む]）。モック2サイクル例は「入れ子の一例（層が2つの姿）」に位置づけ。
- Step 2 の自問を入れ子・3コアに対称化（もう一段小さく開けるか／外側の層から進めたか／関係する E2E を流したか／グリーンで code-reviewer／サイクルを閉じたか）。
- リファクタ段を「レビューの指摘を入力に、グリーンの安全網で小さく片付ける／後回し・まとめてはリファクタでない」に書き換え（旧「綺麗にする・skip しない」から深化）。
- E2E section を core3 のリズムに書き換え（e2e-execution を再発明しない／関係分を各サイクル・最後に全件／重さを言い訳にしない／抑止系に従う）。
- code-reviewer section にグリーンごと規律の退化ガードを追記。
- 用語統一: 造語「拍」を一般語「サイクル」に。TDD ステップ名を全てカタカナに（レッド／グリーン／レビュー／リファクタ／コミット、コード例図だけ英語 `Red:/Green:` を保持、git の commit / commit message は git 用語として英語）。散文の英語概念語も日本語化（normal→普通、unit test→ユニットテスト 等）。
- refactor skip / Critical 非対応の理由の記録先を **commit message に統一**（コード内コメントに残さない）。同ルールが main.instructions.md にもあったため整合。
- **各サイクルは自分のテストでレッド→グリーンを取る**を Red 段と E2E section に明示（E2E の厚みで内側サイクル自身のテストが埋もれる／E2E で代替される、を防ぐ）。

## スコープ外（記録に出たが別扱い）

- テスト配置が全層＝中心集中でない → 中身の流儀（org 固有）、カード明示除外。触らない。
- git-workflow の同期確認飛び → git-workflow の別カード。
- implementation skill が領域切替で再発火しなかった → 置き場所の論点。今回は自問「その層の外側から書いたか」で間接的に触れるに留め、独立の手当ては別途判断（過剰注入を避ける）。
