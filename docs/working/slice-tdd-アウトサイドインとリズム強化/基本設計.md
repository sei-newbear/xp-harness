# slice-tdd: アウトサイドインとリズム強化 — 基本設計

## 採用方針

「分かる形に書き直す」を、文言を盛らずに3本柱で実現する。

### 1. 原理先行（背骨）

欠けていた一般 mental model を主文に置き、特殊例を降格する。

- アウトサイドインの section を「outside-in とは（原則）」に書き換え、「どのサイクルでも一番外側のまだ実装されていない層から、各サイクル外→内」を主文化。
- **アンチパターン: ボトムアップ（内→外）** を明示し、「まず内側のロジックから」と感じたら外側を見落としているサイン、と崩れの起点を名指し。
- モック2サイクル型は「複数サイクルの一例」に降格。「2サイクルは層が2つのときの姿にすぎない」を追記。

### 2. per-cycle の置き場所（リズム）

skill-design-style 原則4（冒頭に置くと skill 発火時に1回消費されて後段で忘れられる）に従い、トリガーを効く手順に置く。

- Step 2 冒頭に「1サイクル = Red→Green→Refactor→Commit の **1つの拍**、分割できない」と拍の概念を置き、次の単位に移る前の**自問**を per-cycle に配置。自問は3つの診断に対称化する（外側から書いたか／このサイクルで E2E を走らせたか／この緑で code-reviewer 呼んだか／拍は閉じたか）— 最も重い「E2E を一度も走らせなかった」を関門から落とさない。
- Green 手順に「緑になった直後、毎回 code-reviewer を呼ぶ。節目で1回ではなくこの緑で」をインライン化（呼び方・処理ルールは後段 section に残す）。
- Red 手順に「E2E は書いたら e2e-execution で走らせる。溜めず外側のサイクルごとに」をインライン化。後段 E2E section の「最初の段で呼ぶ＝一度きり」枠組みも「走らせるのは毎サイクル」に補正。

### 3. 失敗機序は補助（主軸にしない）

E2E を溜めると統合不備が終盤に噴出、ボトムアップは外側が最後まで未検証、という**なぜ崩れるか**を一文程度で添え、価値を腑に落とす補助に留める。

## 装置化を主軸にしない理由（philosophy 軸1 の精緻化）

done-verifier 並みの言い訳対応表 / Red Flags のような装置は「**分かっているのに自走中に言い訳して飛ばす**」失敗（done-verifier がまさにそれ）に効く道具。「**そもそも mental model を持っていない**」失敗には効かない（理解していない相手に「飛ばすな・言い訳一覧」を足しても、機械的に従うか言い訳をすり抜けるだけ）。

今回の診断は後者なので、理解を伝える書き直しを主軸にし、装置は最小に留めた（per-cycle の軽い自問のみ、重い装置は入れない）。これは philosophy「失敗モードが大きい規律にだけ装置を注入」を、「失敗モードが大きい」だけでなく「**理解はあるが言い訳で飛ばす型か**」で切り分ける形に精緻化したもの。philosophy への昇格を別途検討する。
