# description純化 ふりかえり

## 2026-07-21: description 純化 (v0.17.0) の一括改修セッション

### サマリ

かんばん tier1 の description 純化系 5 枚を一括で処理し、PR レビュー・前後比較検証・リリースまで通したセッション。検証で dialogue-principles の発火回帰を検出し、根本原因 (呼ぶ動線の不在) まで掘って対処した。伸びしろは検証の条件設計と運用手順に集中している。

### トピック一覧

よかった点:

- 検証の条件設計に依頼者の指摘 (前後比較で手厚く / 実ユースケースに寄せる / モデルを固定する) が入るたびに作り直し、実走前に条件を固め切った
- 「パス受け渡し規約は旧版でも 31/31 だった」という、自分の PR の価値を下げる事実を隠さず報告した (検証しなければ誤った成果報告をしていた)
- dialogue-principles の回帰で、description 短縮を犯人にせず「呼ぶ動線が元から無い」構造まで掘り、対症療法 (description に発火語彙を戻す) を選ばなかった
- 検証記録に手順上の失敗 (実行中スクリプト編集、tag のサイレント失敗、自動回答による無効化) を残した
- PR レビュー指摘への回答で、主張 (skip 禁止は多層で担保されている) を検証で裏付けてから報告した

伸びしろ:

- `harness-verification` 発火直後の初回条件設計が薄く、依頼者に 3 回差し戻された (反証可能性・実観測ケースの原則を発火直後に通せていなかった)
- 検証セッションのモデルを確認せずに走らせようとした (依頼者の指摘で停止)
- 実行中の bash スクリプトを編集し、走っていたランを 1 本壊した
- `git tag -f <tag> -q` のサイレント失敗に気付かず、baseline 欠落で 1 ラン無効化
- 自走中のランを TIMEOUT で早計に停止した (`harness-verification` 本文に「TIMEOUT 時は状態を見てから判断」と書いてあるのに従えていない)
- 検証レポートを skill の書式と違う形で出し、指摘されて出し直した
- 質問ウィジェットの自動回答を作り込んで 1 本無効化 (手駆動なら確実と分かっていたのに自動化に倒した)

取り決めへの書き戻し候補 (仕分け結果つき):

- `harness-verification` に「前後比較・試行数の要否を検討して提案する」 → **本文に昇格 (適用済み)**
- `harness-verification` に「検証セッションのモデルを固定し実測値を記録する」 → **本文に昇格 (適用済み)**
- `harness-verification` に「ドライバ実行中の質問も回答は手で送る」 → **本文に昇格 (適用済み)**
- `harness-verification` に「走っているドライバスクリプトを編集しない」 → **本文に昇格 (適用済み)**
- `harness-verification` に「baseline は tag で打つ」 → **本文に昇格 (適用済み、skill-reviewer の指摘で追加)**
- `git-workflow` または `harness-verification` に「tag / branch 作成後の存在確認」 → **昇格せず検証記録に留置** (道具固有の落とし穴であり、`git-workflow` は配布物かつ「XP の進め方の規律」でなく道具の使い方なので責務外)

仕分けの基準として「次の検証でも必ず効く規律か / 今回の道具に固有の落とし穴か」を使った。この基準自体を skill に明文化するかは議論したが、**まだ 1 回しか使っていないので言葉にしない**と判断 (次に同じ場面が来れば自然に再燃するため、覚えておくためのメモも作らない)。

### 深掘り 1: 取り決めへの書き戻しの仕分け

#### 経緯

ふりかえりの観測で書き戻し候補が 6 件出た。依頼者から「非配布のものはこの場で修正までしたい、どこまで適用するか決めたい」と要望があり、適用範囲を対話で決めた。

#### 分析

全件を skill 本文に足すのは規律装置最小注入に反する。検証記録 (`references/`) に個別事例として既に残っているものと、本文に昇格すべき規律は性質が違う。前者は「今回の環境・道具で踏んだ穴」、後者は「次の検証でも条件設計・手順に効く規律」。

配布物 (`git-workflow`) への追記は、責務境界 (XP の進め方の規律か / 道具の使い方か) で外れると判断した。

#### アクション

- 昇格 5 件を `harness-verification` に適用済み (commit ea25369)
- skill-reviewer の指摘で 2 点を追加修正: 前後比較の太字の当たり方が「必ずやれ」に読める問題 (太字を提案側だけに寄せた)、手順 3 冒頭の実走前レビューが設計原則の項目列挙を写していて追随できていなかった問題 (単一出典への参照に変更)
- 仕分け基準の明文化は見送り (痛みが 1 回では一般化しない)
