# 自走時の git 最新化支援 — 基本設計

> このドキュメントは進行中の設計メモ。実装後もそのまま残す（振り返り・PR レビューで参照される）。
> 削除のタイミング・要否は依頼者または外側の仕組みで別途判断する（あなたは削除しない）。

要件は同ディレクトリの `要件定義.md` を参照。本書は「どう作るか」を固めたもの。

## 1. アーキテクチャ

支援 hook 2 つ（発火タイミングが違う）＋スクリプト本体 1 つ＋既存 skill への規律追加、で構成する。

```
[ Claude Code セッション ]
        │
        ├─ セッション開始 ──────────→ SessionStart hook ─┐
        │                                                 │
        ├─ 依頼ごと ────────────────→ UserPromptSubmit hook ─┤
        │                                                 │
        │                          （両方とも同じスクリプトを呼ぶ）
        │                                                 ▼
        │                                    [ git 最新化支援スクリプト ]
        │                                       ├─ git fetch（timeout 付き）
        │                                       ├─ 差分/経過を判定
        │                                       └─ 支援メッセージを stdout に注入
        │
        └─ push 直前（PR 運用経路）── git-workflow 規律「push 前 fetch」で担う（hook でなく skill）
```

hook は「気づきを注入する支援」に徹し、エージェントの動作をブロック・矯正しない（常に終了コード 0）。

## 2. データモデル

DB は無い。判定に使う状態は git のローカルファイル 1 つ:

- **`FETCH_HEAD` の更新時刻**（前回 fetch 時刻）。現在時刻との差で「前回同期からの経過」を機械的に算出する。ローカルの stat なのでオフラインでも読める。
- **閾値**: 24 時間（1 日）。既定固定。カスタマイズ機構は今回作らない（要件のスコープ外）。

## 3. シーケンス

### SessionStart（開始時・無条件）

```
発火
 └→ git fetch（timeout 付き）
      ├─ 成功 → origin/現ブランチ（または origin/main）との差分を算出
      │          ├─ 差分あり → 「origin/… が N コミット進んでいます」を注入
      │          └─ 差分なし → 黙る（うるさくしない）
      └─ 失敗（オフライン/remote 無し/timeout）
                 └→ フォールバック: FETCH_HEAD の経過時間を見る
                      ├─ 古い（24h 超）→ 「前回の同期から N 経過、git fetch で最新化してください」を注入
                      └─ 新しい → 黙る
```

### UserPromptSubmit（依頼ごと・間が空いた時だけ）

```
発火
 └→ FETCH_HEAD の経過時間を見る（まずローカルの安い判定）
      ├─ 24h 以内 → 黙る（毎依頼で出さない）
      └─ 24h 超 → git fetch（timeout 付き）
                    ├─ 成功 → 差分を注入（SessionStart と同じ）
                    └─ 失敗 → フォールバック: 「前回の同期から N 経過、git fetch で最新化してください」を注入
```

SessionStart は無条件 fetch、UserPromptSubmit は「古い時だけ」fetch、という非対称。理由は UserPromptSubmit が毎依頼で発火するため、毎回ネットワークを叩かないよう、まずローカルの経過判定で絞る。

## 4. 論理設計

### 責務分離

- **hook スクリプト**: 「同期の古さを検知して支援メッセージを注入する」責務のみ。fetch 実行・経過判定・メッセージ整形を持つ。エージェントの動作制御（ブロック）は持たない。
- **git-workflow skill 規律**: 「push 直前の同期確認」の責務。push はエージェントが git 操作を明示的に実行している最中なので、規律（skill 本文）が効きやすい（この非対称性が hook でなく規律に委ねる根拠）。
- **philosophy skill**: 「hook の使い分け原則」を思想の源泉として記録する責務。

### 支援に徹する（矯正しない）

- hook は stdout に支援メッセージを出すだけ。終了コードは常に 0。エージェントの動作をブロックしない。最新化するかの最終判断はエージェントに委ねる。
- 既存の block-claude-headless.sh（`exit 2` でブロックする矯正 hook）とは性格が違う。あちらは安全・コストのハードストップ（従量課金の誤発火防止）の例外ケース。

### 異常系

- fetch 失敗（オフライン・remote 未設定・timeout）は、沈黙ではなくフォールバックで `FETCH_HEAD` の経過時間を見て、古ければ「git fetch で最新化してください」と案内する（失敗そのものには触れず、やるべきことを案内する）。
- `FETCH_HEAD` すら無い・読めない場合は、静かに終了コード 0 で抜ける（セッションを止めない）。

### メッセージ整形

- fetch 成功・差分あり: remote との乖離を事実として注入（例「origin/main が N コミット進んでいます」）。
- fetch 失敗・古い: やるべきことを案内（「前回の同期から N 経過、git fetch で最新化してください」）。失敗の報告はしない。
- 差分なし・古くない: 何も出さない。

## 5. 既存資産への影響

### 新規追加

- `.apm/hooks/`（新規ディレクトリ・配布物）: SessionStart 用と UserPromptSubmit 用の hook 定義 JSON。apm の `includes: auto` が拾い、`apm install` で consumer の `.claude/settings.json` に自動登録される（検証済み）。
- hook スクリプト本体（配布パッケージ内に 1 つ）。apm の PLUGIN_ROOT 相対の正確な配置パスは実装時に apm の実物で確定する。

### 変更

- `.apm/skills/git-workflow/SKILL.md`（配布物）: 「branch のまま push（PR 運用）」経路（現状 `git push origin <branch名>` のみ）に「push 直前に fetch して同期確認する」を追加する。main 統合経路は既に `git pull` + rebase で同期されるため追加不要。
- `.claude/skills/philosophy/SKILL.md`（改修者向け・非配布）: 「取り込み / 取り込まない 4 象限」の hook 記述を差し替える。「hook 全面採用＝衝突」→「矯正・強制する hook は衝突／気づきを注入する支援 hook は整合。ただし安全・コストのハードストップは矯正でも正当な例外（block-claude-headless が実例）。支援 hook も『失敗モードが大きい節目に限る・痛みが顕在化してから』のガード付き」。
- `.claude/settings.json`（改修者向け・git tracked）: 改修者環境で効かせるため、hook エントリを直接コミットで追加（block-claude-headless の隣）。JSON マージは symlink できないため、直接コミットで握る。
- `scripts/setup-dev.sh`（改修者向け）: hook スクリプトを `.apm/hooks/…` から `.claude/hooks/` へ symlink する処理を追加（skills / agents の symlink と同じ方式）。

### 配布と統合経路

- `.apm/` 配下（hook 定義・git-workflow skill）は利用者影響のある配布物なので、この branch は main 直統合せず push して PR を出しレビューを通す。
- consumer は apm 経由で自動登録、改修者環境は symlink（スクリプト）＋ settings.json 直コミット（登録）の二本立て。

## 6. 完了の定義（実装フェーズが満たすべきもの）

- [ ] SessionStart hook が、セッション開始時に無条件で fetch し、remote と差分があれば支援メッセージを注入する。差分が無ければ黙る。
- [ ] UserPromptSubmit hook が、前回 fetch から 24 時間を超えた依頼のときだけ fetch し支援メッセージを注入する。24 時間以内は黙る。
- [ ] fetch 失敗・オフライン・remote 無しのとき、hook がフォールバックで FETCH_HEAD の経過時間を見て、古ければ「git fetch で最新化してください」と案内する。
- [ ] hook はいずれの経路でも終了コード 0 で、エージェントの動作をブロックしない。
- [ ] git-workflow skill の PR 運用 push 経路に「push 直前に fetch」規律が書かれている。
- [ ] philosophy skill の hook 記述が「矯正／支援」の軸（＋安全・コストのハードストップ例外＋ガード）に置き換わっている。
- [ ] `.apm/hooks/` の hook が apm install で consumer の settings.json に自動登録される（テスト用 consumer 環境で確認）。
- [ ] 改修者環境（この repo）で、setup-dev.sh の symlink ＋ settings.json の登録により hook が実際に発火する。
