# 自走時の git 最新化支援 — 基本設計

> このドキュメントは進行中の設計メモ。実装後もそのまま残す（振り返り・PR レビューで参照される）。
> 削除のタイミング・要否は依頼者または外側の仕組みで別途判断する（あなたは削除しない）。

要件は同ディレクトリの `要件定義.md` を参照。本書は「どう作るか」を固めたもの。pre-implementation-reviewer のレビュー指摘（比較基準・メッセージ到達検証・push 非対称の論拠・二源管理・gitignore 整合ほか）を反映済み。

## 1. アーキテクチャ

支援 hook 2 つ（発火タイミングが違う）＋スクリプト本体 1 つ＋既存 skill への規律追加、で構成する。

```
[ Claude Code セッション ]
        │
        ├─ セッション開始 / resume ──→ SessionStart hook ─┐
        │                                                 │
        ├─ 依頼ごと ────────────────→ UserPromptSubmit hook ─┤
        │                                                 │
        │                          （両方とも同じスクリプトを呼ぶ）
        │                                                 ▼
        │                                    [ git 最新化支援スクリプト ]
        │                                       ├─ git fetch（timeout 付き）
        │                                       ├─ 未取り込みコミット数を算出
        │                                       └─ 支援メッセージを stdout に注入
        │
        └─ push 直前 ── git-workflow の「push 確認ゲート」に紐付けて fetch（hook でなく規律）
```

hook は「気づきを注入する支援」に徹し、エージェントの動作をブロック・矯正しない（常に終了コード 0）。

## 2. データモデル

DB は無い。判定に使う状態は git のローカル情報のみ:

- **前回 fetch 時刻**: `FETCH_HEAD` の更新時刻。現在時刻との差で「前回同期からの経過」を機械的に算出する。ローカルの stat なのでオフラインでも読める。
  - 注意: `FETCH_HEAD` は **worktree ごと**（`.git/worktrees/<name>/FETCH_HEAD`）で、worktree 間で共有されない。別 worktree が直前に fetch していても、当 worktree は「古い」と判定して fetch しうる（無害・冗長なだけ）。
- **未取り込みコミット数**: fetch 後に `git rev-list --count HEAD..origin/main`（= origin/main にあって現在の HEAD に無いコミット数 = 自分がまだ取り込んでいない main 側の前進）。
  - `HEAD` 基準にする理由: main で作業中は `HEAD` = ローカル main なので「リモート main がどれだけ進んだか」と一致する。feature branch では、自分の branch の commit では鳴らず（HEAD に含まれる）、既に最新 main へ rebase 済みなら 0（黙る）。誰かが main に push した時だけ鳴る。ahead/behind や origin/現ブランチ基準だと feature branch で恒常的に誤検知して狼少年化するため採らない。
- **閾値**: 24 時間（1 日）。既定固定。カスタマイズ機構は今回作らない（要件のスコープ外）。

## 3. シーケンス

### SessionStart（開始 / 明示 resume・無条件）

matcher は startup と resume を対象にする。ただし resume は `/resume` 等の**明示操作**のときだけ発火する（公式仕様）。**タブをクリックして開きっぱなしのセッションに戻るだけでは発火しない**。したがって「放置して後から戻る」パターンは SessionStart では拾えず、UserPromptSubmit が担う（下記）。SessionStart の役割は「新規起動・明示 resume」に絞られる。

```
発火
 └→ git fetch（timeout 付き）
      ├─ 成功 → git rev-list --count HEAD..origin/main
      │          ├─ >0 → 「origin/main が N コミット進んでいます（未取り込み）」を注入
      │          └─ 0  → 黙る（うるさくしない）
      └─ 失敗（オフライン/remote 無し/timeout）
                 └→ フォールバック: FETCH_HEAD の経過時間を見る
                      ├─ 24h 超 → 「前回の同期から N 経過、git fetch で最新化してください」を注入
                      └─ 24h 以内 → 黙る
```

### UserPromptSubmit（依頼ごと・間が空いた時だけ）

```
発火
 └→ FETCH_HEAD の経過時間を見る（まずローカルの安い判定）
      ├─ 24h 以内 → 黙る（毎依頼で出さない）
      └─ 24h 超 → git fetch（timeout 付き）
                    ├─ 成功 → 未取り込みコミット数を算出して注入（SessionStart と同じ）
                    └─ 失敗 → フォールバック: 「前回の同期から N 経過、git fetch で最新化してください」を注入
```

UserPromptSubmit を持つ根拠: 依頼者はセッションを開きっぱなしで放置し、後から同じセッションで作業を続けることが常態（実運用で確認済み）。この「タブに戻って続ける」動線は SessionStart の resume では発火しない（公式仕様、明示 resume のみ）ため、**UserPromptSubmit がこの主要パターンを拾う唯一の機構**。よって「開いたまま古くなる」痛みは顕在化しており、これは 2 本目の hook の要否を疑う余地なく必須。philosophy の「支援 hook も痛みが顕在化してから」のガードと矛盾しない（離席して戻る話であり、フル自走ではない）。

hook の timeout は UserPromptSubmit の既定 30 秒より短く取る（fetch がネットワークで固まってもプロンプト処理を止めないため）。

SessionStart は無条件 fetch、UserPromptSubmit は「古い時だけ」fetch、という非対称。理由は UserPromptSubmit が毎依頼で発火するため、毎回ネットワークを叩かないよう、まずローカルの経過判定で絞る。

## 4. 論理設計

### 責務分離

- **hook スクリプト**: 「同期の古さを検知して支援メッセージを注入する」責務のみ。fetch 実行・経過判定・未取り込み数の算出・メッセージ整形を持つ。エージェントの動作制御（ブロック）は持たない。
- **git-workflow skill 規律**: 「push 直前の同期確認」の責務。後述の通り push の確認ゲートに紐付ける。
- **philosophy skill**: 「hook の使い分け原則」を思想の源泉として記録する責務。

### push 前 fetch を hook でなく規律に置く根拠（非対称）

push 前 fetch は git-workflow の規律として置く。ただし前回レビューの「なぜ push なら skill が発火するのか」への答えは「push はエージェントが git 文脈にいるから」ではない（それは論証にならない。自走中に raw `git push` が git-workflow を経由せず走ることはありうる）。

本物の非対称は「**push には依頼者確認ゲートがあり、人が立ち会う**」こと。git-workflow の完了フローも、この repo の CLAUDE.md も、push 前に必ず確認を取る。セッション開始はエージェントが単独で始める（人が居ない）ので hook で機械的に固めたが、push は人が居るので規律で足りる。

具体的には、fetch を**確認ゲートの瞬間に紐付ける**（統合 / push の確認を出す前に fetch して差分を添える）。best-effort だが、人の確認ゲートが backstop になる。

### 支援に徹する（矯正しない）

- hook は stdout に支援メッセージを出すだけ。終了コードは常に 0。エージェントの動作をブロックしない。最新化するかの最終判断はエージェントに委ねる。
- 既存の block-claude-headless.sh（`exit 2` でブロックする矯正 hook）とは性格が違う。あちらは安全・コストのハードストップ（従量課金の誤発火防止）の例外ケース。

### 異常系

- fetch 失敗（オフライン・remote 未設定・timeout）は、沈黙ではなくフォールバックで `FETCH_HEAD` の経過時間を見て、古ければ「git fetch で最新化してください」と案内する（失敗そのものには触れず、やるべきことを案内する）。
- `FETCH_HEAD` すら無い・読めない場合は、静かに終了コード 0 で抜ける（セッションを止めない）。

### メッセージ整形（作業前の同期を促す）

弱い通知（「取り込みを検討してください」）だと、エージェントは明示タスクを優先し古いまま作業して後で気づく（実測で確認）。そこで、同期を「タスクと張り合う注意」でなく「**依頼を正しく叶えるための前提**」として位置づける。生コマンド（`git pull` 等）は決め打ちしない — 手元の変更の有無で安全な手順が変わるため、git-workflow skill の「state を変える git 操作の前に手元を整理する」手順に従わせる（調査・実装いずれの作業前にも同じ）。

害の説明は「コンフリクト・手戻り」だけにしない。それは変更（mutate）の害なので、**調査（読むだけ）のタスクは「自分は commit/push しないから関係ない」と読み飛ばせてしまう**（Pattern D で実測。「調査・実装いずれも」と書いても無視された）。そこで **読むこと自体の害＝「古い状態のコード・ドキュメントを読んで判断すると誤った結論を出す」** を明示し、「調査・回答・実装のいずれであれ」と名指しする。これで調査タスクでも作業前に同期することを再実測で確認（前段は stale な README を読んで「内容が空」と誤結論 → 修正後は「最新を取り込まないと正しく答えられない」と自ら判断し ff 同期してから正答）。

- fetch 成功・未取り込み >0: 事実＋「古い状態を読んで判断すると誤った結論になる（実装すればコンフリクト・手戻りにもなる）」＋「調査・回答・実装のいずれであれ、取り掛かる前に git-workflow の同期手順に従って取り込め」を注入。
- fetch 失敗・古い: git-workflow の同期手順で最新化するよう案内（前回同期からの経過を添える）。失敗の報告はしない。
- 未取り込み 0・古くない: 何も出さない。

## 5. 既存資産への影響

### 新規追加

- `.apm/hooks/`（新規ディレクトリ・配布物）: SessionStart 用と UserPromptSubmit 用の hook 定義 JSON。apm の `includes: auto` が拾い、`apm install` で consumer の `.claude/settings.json` に自動登録される（検証済み）。
- hook スクリプト本体を**パッケージルート直下** `hooks/scripts/git-freshness.sh` に置く（配布物）。apm は `${PLUGIN_ROOT}` をパッケージルート起点で解決し **`.apm/` 配下は見に行かない**ため、スクリプトは `.apm/` の外（パッケージルート）に置く必要がある（実物の apm install で確認済み）。`.apm/hooks/git-freshness.json` が `${PLUGIN_ROOT}/hooks/scripts/git-freshness.sh` で参照し、apm install で consumer の `.claude/hooks/<pkg>/hooks/scripts/` に配置・登録される。

### 変更

- `.apm/skills/git-workflow/SKILL.md`（配布物）: 完了フローの「push / 統合の確認」を出す直前に fetch して差分を添える手順を追加する。「branch のまま push（PR 運用）」経路（現状 `git push origin <branch名>` のみ・fetch 欠落）が主対象。main 統合経路は既に `git pull` + rebase で同期されるため追加不要。
- `.claude/skills/philosophy/SKILL.md`（改修者向け・非配布）: 「取り込み / 取り込まない 4 象限」の hook 記述を差し替える。「hook 全面採用＝衝突」→「矯正・強制する hook は衝突／気づきを注入する支援 hook は整合。ただし安全・コストのハードストップは矯正でも正当な例外（block-claude-headless が実例）。支援 hook も『失敗モードが大きい節目に限る・痛みが顕在化してから』のガード付き」。あわせて、この支援 hook を「痛みが顕在化して象限4（痛み明確化後の採用）から採用に移った実例」（自動 worktree と同じ構図）として位置づけ、表と地の文（象限3 は踏みとどまる／象限4 は痛みが見えてから）が差し替え後も矛盾しないよう通しで整える。

### 改修者環境（self-host）での効かせ方

- hook スクリプトはパッケージルート `hooks/scripts/git-freshness.sh` に置く（配布・self-host 共通の実体）。**symlink は作らない**。改修者環境の `.claude/settings.json`（git tracked）に hook エントリを直接コミットし、コマンドは `$CLAUDE_PROJECT_DIR/hooks/scripts/git-freshness.sh` を直接指す。
  - symlink を `.claude/hooks/` に生やさない理由: `.claude/hooks/` は gitignore 対象外（block-claude-headless.sh が tracked 実ファイル）で、そこに symlink を作ると git status が汚れる。`.apm/` の実ファイルを直接指せば symlink も setup-dev.sh 変更も不要で、汚れも出ない。
- したがって setup-dev.sh（skill / agent の symlink 生成）は今回いじらない。

### 配布と統合経路、二源管理のギャップ

- `.apm/` 配下（hook 定義・git-workflow skill）は利用者影響のある配布物なので、この branch は main 直統合せず push して PR を出しレビューを通す。
- hook の登録経路は 2 つになる: consumer は `.apm/hooks/*.json` を apm が自動登録、改修者環境は `.claude/settings.json` の手書きエントリ。スクリプト本体は `.apm/` の単一源なのでズレない。ズレうるのは登録エントリだが、一度書けば滅多に変わらない（普段編集するのはスクリプト中身）ためリスクは小さい。
- ただし日常の dogfooding では手書き経路を使い、consumer が受け取る `.apm/hooks/*.json` の登録経路を通らない。このギャップを受容した上で、出荷物の経路（`.apm/hooks/*.json` → apm install → settings.json 登録）を **テスト用 consumer で一度検証する**（完了条件に含める）。

## 6. 完了の定義（実装フェーズが満たすべきもの）

- [x] （実装の最初のゲート）固定文字列を出すだけの trivial hook で、SessionStart と UserPromptSubmit の stdout（exit 0）注入が実際にエージェントの context に surface することを実測できている。**検証済み**: 隔離 sandbox で interactive claude を起動し、両イベントの目印が transcript の `hookEvent` content として context 注入されることを確認（リコールテストでエージェントが注入値を実際に読んで回答したことも確認、ブロックされず通常応答）。
- [x] **作業前の同期挙動（実装・状態別）**: apm install した本番同等構成＋stale ローカル（上流 3 コミット先行）で、git 動詞を含まないノーヒントの実装タスクを投げ、手元の状態別に実測。きれい／未コミット変更あり／untracked のみ の 3 パターンとも、エージェントが作業前に同期する（未コミット変更は破棄せず衝突確認、untracked は ff で直接 pull）ことを確認。最新（遅れ 0）の陰性対照では hook が黙り余計な同期をしないことも確認。
- [x] **作業前の同期挙動（調査）**: 調査（読むだけ）のタスクでも作業前に同期することを再実測。当初は「調査は git 操作をしないので同期を読み飛ばす」失敗（stale README で誤結論）だったが、メッセージに「読んで判断すると誤った結論を出す」という読みの害を足したことで、エージェントが自ら「最新を取り込まないと正しく答えられない」と判断し ff 同期してから正答するよう反転した。
- [ ] SessionStart hook（matcher: startup / resume）が、開始・resume 時に無条件で fetch し、`HEAD..origin/main` の未取り込みコミットが 1 つ以上あれば支援メッセージを注入する。無ければ黙る。
- [ ] UserPromptSubmit hook が、前回 fetch から 24 時間を超えた依頼のときだけ fetch し支援メッセージを注入する。24 時間以内は黙る。
- [ ] fetch 失敗・オフライン・remote 無しのとき、hook がフォールバックで FETCH_HEAD の経過時間を見て、古ければ「git fetch で最新化してください」と案内する。
- [ ] hook はいずれの経路でも終了コード 0 で、エージェントの動作をブロックしない。
- [ ] git-workflow skill の完了フローで、push / 統合の確認を出す直前に fetch して差分を添える手順が書かれている。
- [ ] philosophy skill の hook 記述が「矯正／支援」の軸（＋安全・コストのハードストップ例外＋ガード＋象限4 実例の位置づけ）に置き換わり、表と地の文が矛盾していない。
- [ ] `.apm/hooks/*.json` が apm install で consumer の settings.json に自動登録されることを、テスト用 consumer 環境で確認できている。
