# 自走時の git 最新化支援 — 要件定義

> このドキュメントは進行中の設計メモ。実装後もそのまま残す（振り返り・PR レビューで参照される）。
> 削除のタイミング・要否は依頼者または外側の仕組みで別途判断する（あなたは削除しない）。

## 1. 背景と動機

xp-harness はチーム開発・複数マシン・複数人が同じ repo を触る前提の harness。自走するコーディングエージェントが git の最新化を飛ばし、stale（remote より古い）なローカル状態のまま作業を始めると、次の実害が出る。

- 古い前提のまま進めて手戻り・トークン浪費（調査で古いコードを掴むのも含む）
- 認識が噛み合わない（古い情報を土台に依頼者と議論してズレる）
- 連番ファイル（DB migration など）を新規追加したときに既存と番号衝突し、振り直しになる

git-workflow skill 本文には既に「セッション開始時に `git fetch` を必ず最初に叩く」「`git status` の up to date は前回 fetch 時点の比較で、fetch しないと古いまま up to date を返す」と書いてある。**それでも自走で落ちた**。skill 本文は、skill が発火して読まれないと効果がない。自走だと skill の発火・実行段階で同期ラリーが飛ぶ。

このセッション自体が実例になった。セッション開始時に fetch・branch 確認・status 確認・作業場所確認を丸ごと飛ばして本題に入り、ローカルが remote より 3 コミット古い（別マシンで push 済みの kanban カードが未取得）まま改修バックログを何ターンも議論した。最新化に気づいたのは依頼者の「最新になってる?」という指摘。harness 側でなく依頼者が捕捉した。

規律を skill 本文で強める方向は、今まさに落ちた手段なので同じ穴に落ちる。**エージェントの規律記憶に頼らず、機械的な支援で最新化に気づかせる**必要がある。

### hook の使い分け原則（この案件で言語化、philosophy に昇格させる）

エージェントを **矯正・強制する** hook（動作をブロックする・矯正する）は、xp-harness の「価値で導く・コーディングエージェントの能力を最大化する」思想と衝突する。一方、**気づきを注入して支援する** hook（外部アラートとして知らせ、判断はエージェントに委ねる）は思想と整合する。時間計算や fetch のような機械が得意な作業を機械にやらせ、エージェントは本来の判断に集中させるのは「能力を最大化する」と同じ向き。

今回導入するのは後者（支援 hook）。タイマーで人間に通知して気づかせるのと同じ位置づけ。

## 2. ユーザーストーリー

自走するコーディングエージェント（および、それを使う改修者・チーム）が、stale な状態で作業を始めて手戻りや認識ズレを起こさないために、節目（セッション開始 / 長く間が空いた依頼 / push 直前）で git の最新化に確実に気づける。

## 3. Done（完了基準）

harness 機構は E2E テスト枠組みが未確立のため、事前縛りは最小化し、観測可能なものは観測可能に、それ以外は事後評価を許容する。

### セッション開始時

- セッション開始時、エージェントの規律記憶や skill 発火に頼らず、最新化（fetch）に気づける支援メッセージが hook によって注入される。
- 古い・新しいに関わらず、**開始時は無条件**で最新化が促される（閾値判定ではない。開始時点で必ず最新化）。

### 間が空いた依頼

- 前回 fetch から 1 日（24 時間）を超えて間が空いた状態で依頼が来たとき、hook が「同期が古い」ことに気づける支援メッセージを注入する。
- 前回 fetch から日が浅い通常の依頼では、hook は黙っている（毎依頼で警告を出してうるさくしない）。
- 「前回 fetch からの経過」は機械が判定する（エージェントに毎回時間計算させない）。

### push 直前

- git-workflow skill 本文の push 手順に「push 直前に fetch して同期確認する」規律を**新規追加する**。現状この規律は存在しない（PR 運用の push 経路は `git push origin <branch名>` のみで fetch が無い）。要件で「既に書かれている」と前提しない。
- push 前の最新化は hook ではなく git-workflow 規律で担う。根拠は、push がエージェントの git 操作（git-workflow の完了フロー）を**明示的に実行している最中**の動作であること。「作業前に思い出す」セッション開始の前提儀式と違い、エージェントは既に git-workflow の文脈にいるので規律が効きやすい（この非対称性が委譲の根拠）。

### 支援に徹する（矯正しない）

- hook は警告・案内の注入に留まり、エージェントの動作をブロック・矯正しない。最新化するかどうかの最終判断はエージェントに委ねる。

### 配布と異常系

- consumer に配布できる形（`.apm/` 配下の配布物）で提供される。配布の実現可能性は検証済み（APM は `.apm/hooks/<name>.json` を `apm install` 時に consumer の `.claude/hooks/` へ配置し、`.claude/settings.json` へ自動登録する。既存の settings.json は壊さずマージ、uninstall も綺麗）。
- hook のコマンドはプロジェクトルート相対パスで登録されるため、起動ディレクトリ依存に注意する（基本設計でパス解決を確認する）。
- remote 未設定・オフライン・fetch 失敗などの異常系で、hook が暴発（誤警告・エラーで止まる）せず、静かに何もしない。

### 思想の記録

- philosophy skill に「支援 hook は思想と整合 / 矯正 hook は衝突」の境界原則が昇格して書かれている（現状の「hook 全面採用＝衝突」という粗いまとめを、強制か支援かの軸に置き換える）。ただし「支援なら何でも採用してよい」にならないよう、支援 hook も「規律装置最小注入・失敗モードが大きい節目に限る・痛みが顕在化してから」というガードを併記する（hook 乱立の白紙委任を防ぐ）。

## 4. スコープ外

- エージェントの動作を強制・ブロックする矯正型 hook。今回は支援に限定する。
- 閾値（1 日）をユーザーがカスタマイズする機構。既定 1 日で始め、調整が必要になったら別途。
- 「skill 本文に書いた規律が自走で落ちる」という一般問題の全面解決。slice-tdd の自走時の振る舞い崩れ（実装が層ごとの外→内にならない、refactor やレビューのリズムが落ちる）と同根だが、今回は git の最新化に閉じる。
- 作業中に別マシンが push したことで起きる stale（例: migration 連番ファイルの新規追加時の番号衝突）。これはセッション開始・時間経過・push 前のどの節目でも捕まらない（ファイル作成の瞬間に古くなる）。今回は受容し、作業中のファイル作成ごとに割り込む支援は入れない。動機に挙げた連番衝突は、この案件では部分的にしか緩和されない（push 前 fetch で push 時点では検知できるが、作成時点では防げない）。
- 「hook が fetch を実際に走らせるか／案内の注入に留めるか」「支援を『古さの surface』までに留めるか、具体的な乖離（N コミット遅れ）の提示や fetch 実行まで含めるか」などの実装方式。いずれも支援の範囲内の選択で、基本設計フェーズで決める。
