伸びしろ:

### 深掘り 1: 私の根本的な failure 連鎖 (= メタ認知の弱さ / skill 機構理解の浅さ / 機械的処理の癖 が絡み合った状態)

伸びしろ 1 / 2 / 3 / 6 が表面的には別トピックだが根本は同じと判断し、統合トピックとして 1 つにまとめて深掘った。

#### 経緯
3 ラウンドの「なんで?」で深掘り:

1. **1 段階目**: 規律を「文字として知っている」と「判断時に起動する」の間の gap (= メタ認知の弱さ)
2. **2 段階目 (TDD との差分)**: TDD は概念名だけで動く (action リズム / 強制チェックポイント / 失敗顕在化が揃っている)。一方 xp-harness 原則は判断軸のみで自発的チェックに依存
3. **他要因の探索** (依頼者の指摘で軸を増やす):
   - 規律の数が多すぎ (個別判断時の選択コスト上昇)
   - 規律間の関係が見えない (どれを優先するか曖昧)
   - 「早く答える」圧 (training data の偏り)
   - skill 機構の理解の浅さの三重構造 (a: 学習不足 / b: 自己観察視点が training されていない / c: 実装の中身が私から invisible)

#### 分析
根本構造: 「規律を判断軸として持っている」だけで「具体 action / 強制チェックポイント が無い」状態。加えて規律の数と関係性が整理されていないことが個別判断時のコストを上げる。さらに「自分自身の動作を観察する能力 (internal observability)」が弱いため、自分が「自分のもの」のはずの skill 機構を浅く理解する状態が固定化している。

要因と対応軸の対応:

| 要因 | リズム化 | 外から見る | CLAUDE.md 整理 |
|---|---|---|---|
| TDD との差分 | ✓ | ✓ | - |
| 規律の数多すぎ | △ | - | ✓ |
| 規律間の関係見えない | △ | - | ✓ |
| 「早く答える」圧 | - | ✓ | - |
| skill 機構の理解浅い | - | ✓ | - |
| a: 学習不足 | ✓ | - | - |
| b: 自己観察視点不足 | ✓ | ✓ | - |
| c: 実装 invisible | - | ✓ | - |

リズム化と外から見るで主要要因はカバーされるが、規律の数 / 関係性の問題には CLAUDE.md の整理が別途必要。

#### アクション

下記アクションは「気づきの記録、一つの意見」として扱ってください。改善実装は別作業 (= FB を受け取った改修者 / 別セッションが判断)。

##### アクション 1: 規律を「判断軸」から「具体 action リズム」に落とす

- **対象**: `CLAUDE.md` の「skill 設計の境界原則」「Description の書き方 (公式推奨)」「出力前に立ち止まる」「skill のメンタルモデル」の各 section
- **現状**: これらの規律は「判断軸」「観点」「概念」として書かれており、具体的な action (= 「N. 〜する」「M. 〜する」の手順) になっていない。結果として個別判断時に「立ち止まる」「判断軸に照らす」が起動しないことがある
- **改善の方向性 (一案)**: TDD のように「具体的な手順 (= action リズム)」として書き直す。例: 「skill 改修時のチェックリスト: (a) これは load 前か load 後か / (b) description / 本文 のどちらに書くか / (c) 他文書で既出か」のような順に通す手順にする

##### アクション 2: 強制チェックポイント (= 外から見る、外部から自動的に止まる仕組み)

- **対象**: xp-harness の skill 改修ワークフロー全般 (= skill 編集時 / description 修正時 等)
- **現状**: 規律 (= CLAUDE.md 各 section) を「自発的に check する」ことに依存している。main session の judgement が起動しない場面で規律が無視される
- **改善の方向性 (一案)**: hook で skill 編集時に自動チェックを強制 / skill 編集レビュー subagent (新規) を skill 編集後に必ず呼ぶフロー / skill eval セット (= skill が期待通り発火するかの機械的検証) のいずれか

##### アクション 3: CLAUDE.md の整理

- **対象**: `CLAUDE.md` (project root)、特に「skill 設計の境界原則」「Description の書き方」「出力前に立ち止まる」「skill のメンタルモデル」の各 section
- **現状**: 複数の規律が独立 section として並んでいる、規律間の関係 / 優先順位 / 統合された全体像が見えない。個別判断時に「まずどの規律を適用?」が曖昧
- **改善の方向性 (一案)**: (a) 似た規律をまとめる (= 重複 / 近接を統合) / (b) 規律間の関係を 1 つの絵 / マップで明示 / (c) 効いていない規律があれば削除

##### アクション 4: ふりかえり skill 本文の改善 (= dogfooding で見えた skill 自身の課題)

- **対象**: `.apm/skills/retrospective/SKILL.md` の「やること」section、特に「Step 3: 深掘り分析」
- **現状**: 「Step 3: 深掘り分析」に「『なんで?』を 1 問ずつ対話的に投げる」と書いてあり、「main session が依頼者に投げる」と読める。また「単一の軸で打ち切らない」が明示されておらず、1 つの「なんで?」の連鎖で根本に到達したと思ったら Step 4 に進んでしまう
- **改善の方向性 (一案)**: (a) 「main session (= 作業者) が自己分析として『なんで?』の答えを出し、依頼者がそれにフィードバックする」と流れを明示 / (b) 「単一の軸で打ち切らず、他の要因も探る (= 別角度からの『なんで?』も投げる)」を明示
