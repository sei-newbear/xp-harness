# ふりかえり skill — 基本設計

> このドキュメントは進行中の設計メモ。実装後もそのまま残す（振り返り・PR レビューで参照される）。
> 削除のタイミング・要否は依頼者または外側の仕組みで別途判断する。

ROADMAP TODO 25 (Tier 1) に対応する skill の基本設計。要件は `要件定義.md` 参照。

## 1. skill の構成

ふりかえり skill は単一の SKILL.md ファイルとして実装する。

- 配置: `.apm/skills/retrospective/SKILL.md`
- 構造: frontmatter (interface 層) + 本文 (実装層) のセット (CLAUDE.md の「skill のメンタルモデル」と整合)

### frontmatter (description) の内容

```
作業をふりかえって改善案を引き出し、記録する。セッション内のすべてのストーリーが完了したとき、依頼者の終わり発話 (作業を区切る発話) を観測したとき、または依頼者が「ふりかえろう」「retrospective」と明示したときに発火。同セッション内で 1 度 skip された場合は再発火しない。
```

### 本文 section の構成

| section | 内容 |
|---|---|
| なぜこの skill があるか | ふりかえりの意義 (使い手の作業 / 使っている skill / 取り組み方の継続的な改善) と責務範囲 (改善案の記録まで、実装は別) |
| やること | 進行プロセス Step 1-5。各 Step 内に対話の流儀・規律 (`dialogue-principles` と整合 / 情報収集 2 軸の柔軟性 / 既存フレームワーク非採用 等) を含める |
| 成果物 | 記録先 / フォーマット / 累積方式 |

xp-harness 既存 skill の構成パターンを参考にしつつ、ふりかえり skill 固有の責務に合わせて section を選んだ。「次のフェーズへの誘導」「規模・複雑度で reviewer を呼ぶ提案」「成果物の細かい執筆ルール」は含めない (= ふりかえりは最後、reviewer 対象外、ふりかえり md は要件定義 / 基本設計よりラフでよい)。

「やること」section の Step 1-5 形は、ふりかえりがループ含みの逐次プロセスのため、既存 skill (`define-requirements` / `basic-design` 等) の「N. 〜を引き出す」というオープンな項目並びとは異なる構成を選んだ。

## 2. 発火モデル

### 観測モデル

**段階的観測**: main session が文脈から「セッション内のすべてのストーリーが完了した」と判断する。

- データの参照先: 会話履歴 (主)、フェーズフロー上の作業なら `docs/working/<title>/要件定義.md` のユーザーストーリー一覧も補助的に参照
- 起点: 各ストーリーの完了サイン (依頼者の合意発話 / push 直前 / 完了基準達成 等) を main が観測したとき + 依頼者の終わり発話 (「終わり」「お疲れさま」「一旦切ろう」等) を観測したとき
- 判定: main が文脈から「対象ストーリー一覧」と「各完了状態」を読み取り、すべて完了なら促す

観測手段の具体は本設計で固定せず、main session の判断に委ねる (= 運用結果次第で観測モデル自体を見直す余地あり、改善はふりかえり skill 自身で拾う想定)。

### 発火条件 (description に書く)

- セッション内全ストーリー完了で発火
- 依頼者の終わり発話 (作業を区切る発話) で発火
- 依頼者が「ふりかえろう」「retrospective」と明示したときに発火
- 同セッション内で 1 度 skip されたら再発火しない

### 提案 → 承認フロー

1. main session が観測モデルに基づき「ストーリー完了したようなのでふりかえりしませんか?」と依頼者に提案 (この時点で skill 本文はまだ load されていない)
2. 依頼者の返答を判定 (細かい判定境界は main session の judgement に委ねる):
   - 承認 → main が skill 本文を load、Step 1 (情報収集) から進行
   - 拒否 → skill 本文 load せず、短く引き下がる、同セッション内で再発火しない (= skip 状態)
3. 依頼者が明示的に呼び出した場合 (= skill 名指定 / 「やっぱりふりかえり」等) は skip 状態を上書きして発火

E2E 観測可能なのは「ふりかえりを促す発話が出た」まで。発火判定の精度検証は通常の E2E では困難 (skill ベンチマーク領域)。

## 3. 進行プロセス

skill load 後、本文に従って以下の Step を回す。

**既存フレームワーク (KPT / YWT / Fun-Done-Learn 等) を採用しない理由**: ふりかえりの本質は「共同で気づきを引き出し、改善案を作る」ことで、xp-harness のペアプロ哲学・対話駆動と直接整合する独自プロセスで足りる。フレームワークの「型」を持ち込むと、Step 間の責務 (= 情報収集 / 深掘り / アクション) が混在しがちで、ふりかえりの本質に直接効きにくい。

### Step 1: 情報収集

- 軸: **よかった点 / 伸びしろ** の 2 軸 (基本)、依頼者指示で変更可
- 進め方: main が会話履歴・観察から先にリストを出す → 依頼者が追加 / 削除 / 修正
- 注意: アンカリング防止のため、本文に「依頼者は自分の実感を優先して書き換えてよい」と明示

### Step 2: 深掘りトピック選択

- リストから依頼者が深掘りしたいトピックを選ぶ

### Step 3: 深掘り分析

- **main session が自己分析として答えを出す**: main session が「なんで?」の答え (= 作業者の根本理由) を先に出し、依頼者がフィードバック。依頼者に「なんで?」を投げるのではない (= 作業者 = main session が原因を知っているはず、依頼者は外側のチェック)
- **1 問ずつ進める**: 1 つの「なんで?」 + 1 つの自己分析の答え + 依頼者のフィードバック を 1 サイクル
- 目安 5 回前後、状況判断で打ち切り可 (= 5 Whys の慣習に倣う目安、状況に応じて柔軟に増減)
- **一気に 5 回出さない (NG)** — 読み手の負担と方向ミスの手戻りを避ける (dialogue-principles の「1 問ずつ」と整合)
- **単一の軸で打ち切らない** — 1 つの「なんで?」連鎖で根本に到達したと判断する前に、他の要因 (別角度からの「なんで?」) も探る (強制ではなく充実度の推奨)

### Step 4: アクション or 分析

- アクションを出すか、分析だけで終わる
- **アクション 0 件 OK** (= 「現時点で出ない」も価値ある情報、(なし) と明示記録)

### Step 5: 次へ

- 別のトピックを深掘りしたい場合 Step 2 に戻る
- 終了する場合は記録に進む

## 4. 成果物 (記録) の設計

### 記録先

- パス: `docs/working/<title>/ふりかえり.md`
- 単位: 1 `<title>` ディレクトリ = 1 ファイル
- 累積方式: 複数セッションを 1 ファイルに追記 (時系列順)

`<title>` は要件定義 / 基本設計 と同じディレクトリ規約 (= ケバブケース)。

### フォーマット

```markdown
# <title> ふりかえり

## 2026-05-17: [このふりかえりのタイトル]

### サマリ
[1-3 行、今回のふりかえりの位置付け]

### トピック一覧
よかった点:
- ...

伸びしろ:
- ...

### 深掘り 1: [深掘りしたトピックの名前]
#### 経緯
[作業 / 議論の流れ]

#### 分析
[原因 / 課題 / 構造的問題]

#### アクション
[次に試すこと、0 件なら (なし)]

### 深掘り 2: [別のトピックの名前]
...
```

### 記録の規則

- `## YYYY-MM-DD: [タイトル]` で 1 セッションのふりかえりを区切る
- 「トピック一覧」は Step 1 の結果 (= 全リスト、深掘りしなかったものも含む)
- 「深掘り N」section は Step 2-4 で深掘りしたトピックごとに作る。深掘りしなかったトピックは現れない
- 深掘り番号 N はセッション内 (= 各日付セクション内) でリセット、1 から開始
- アクション 0 件は「(なし)」と明示
- 複数セッション分は時系列順に追記

## 6. 完了の定義

要件定義.md §3 Done と整合する E2E 観測条件 (checklist 形式):

- [ ] main session が完了サインを観測し、ふりかえりを促す発話を出す
- [ ] 依頼者が明示同意した場合、Step 1 (情報収集) の発話が出る
- [ ] 依頼者が深掘り対象を選択するか、深掘りスキップの発話が成立する
- [ ] 依頼者が明示拒否した場合、main session が短く引き下がり、同セッション内で再発火しない
- [ ] 最終的に Step 5 のループを抜け、`docs/working/<title>/ふりかえり.md` への書き出しに進む
- [ ] 書き出される内容がフォーマット通り (タイトル / サマリ / トピック一覧 / 深掘り N [経緯 / 分析 / アクション])
- [ ] アクション 0 件のときは「(なし)」と明示される
- [ ] 1 `<title>` 内で複数セッション分が累積記録される (時系列順に追記、深掘り番号はセッション内でリセット)

発火判定の精度検証は通常の E2E では困難 (skill ベンチマーク領域)。完了基準として E2E で観測可能なのは「ふりかえりを促す発話が出た」「依頼者の返答に応じた挙動が起きた」までと位置づける。
