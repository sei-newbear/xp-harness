# ふりかえり skill — 要件定義

> このドキュメントは進行中の設計メモ。実装後もそのまま残す（振り返り・PR レビューで参照される）。
> 削除のタイミング・要否は依頼者または外側の仕組みで別途判断する。

ROADMAP TODO 25 (Tier 1) に対応する skill の要件定義。

## 1. 背景と動機

xp-harness を使う中で開発者が日々気付く改善点を拾うサイクルが回っていない。skill / agent / instruction の不備に気付いても、その場で言語化されないまま流れ、ROADMAP や改善実装に繋がらない。

使い手 (依頼者本人 + 利用者 (consumer) 含む) からのフィードバックを継続的に harness 改善に繋げるため、ふりかえりを「自発的に促す」仕組みで構造化したい。「気付いた時に拾う」を人頼みではなく仕組みで担保する。

ただし、現状は利用者の範囲が顔の見える人々に限られるため、改善案が project ローカルに残れば、改修者への伝達は直接対話で成立する前提とする。skill の責務は project ローカルでの記録までとし、伝達経路は skill 内に作らない。

ふりかえりの責務は XP の retrospective と同じく **改善案 (action) を出して記録するまで**。その場で skill / agent / instruction を修正することはこの skill の責務外 (= 改善実装は別 skill / agent / 人間判断の仕事)。

## 2. ユーザーストーリー

- 開発者が、作業の気づきを継続的に拾うために、セッション内のすべてのストーリーが完了したタイミングで自発的にふりかえりを促され、改善案を出して記録できる

## 3. Done（完了基準）

- main session が、会話履歴 / `docs/working/<title>/要件定義.md` 等の文脈から「セッション内のすべてのストーリーが完了した」と判断したタイミングで、自発的にふりかえりを促す (観測手段の具体は基本設計で決める。運用結果次第で観測モデル自体を見直す余地あり、改善はふりかえり skill 自身で拾う想定)
- 自発的に促すときは 1 回提案する。依頼者が「やらない」と返したら引き下がる (押し付けない、`dialogue-principles` と整合)
- ふりかえり対話を通れば完了とする (改善案 0 件 / やらない選択も完了扱い)
- 改善案は使い手環境の project ローカルにドキュメント形式で残る (具体パス・命名は基本設計で決める)
- 改善案の中身は「分析結果」から「改善のアクション」までの幅を持つ。項目の具体は基本設計で決める
- フレームワーク選択責務は基本設計で決める (要件レベルで固定しない)
- 発火判定の精度検証は通常の E2E では困難 (skill ベンチマーク領域)。完了基準として E2E で観測可能なのは「ふりかえりを促す発話が出た」までと位置づける
- 依頼者が「ふりかえろう」と明示した場合の発火経路は skill 機構 (description) で自然に成立する想定。確実に呼びたいときは skill を直接指定すればよい。要件としては別途詰めない

## 4. スコープ外

1. 改善案の実装そのもの (skill / agent / instruction の修正は別 skill / agent / 人間判断に委譲)
2. ふりかえりフレームワークの固定 (KPT 等を skill 内部で決め打ちしない)
3. 改修者への伝達経路全般 (現状は利用者の範囲が顔の見える人々に限られるため、口頭・直接対話で扱う前提。skill 内に経路を作らない。配布規模が広がったら将来検討課題)
4. 複数セッションをまたいだ集計・傾向分析 (傾向分析画面のようなもの)
5. 「やらない」が連発したときに skill 側で問いかけ方を自動チューニング等のメタ介入
6. 既存 skill (`define-requirements` / `slice-tdd` / 他) の改修 (main session が文脈から判断する観測モデルを採るため、既存 skill 改修の必要性は薄い。基本設計で必要と判明したら別 TODO として切り出す)
