### 深掘り 1: 調査報告が実況地続きで読みにくい

#### 経緯
apm 挙動調査で、「検証するね → bash の生出力（巨大、例: 11KB の AGENTS.md 全ダンプ）→ 解釈」を 1 メッセージに地続きで並べた。結論が冒頭に区切られず、過程を追わないと要点に届かない報告が多かった（記録に残る事実）。

#### 分析
根は 2 つ（単線で打ち切らない）:

1. **構造的（取り決め側、主因）**: 「実況と報告を分ける／結論→詳細→証拠」という報告品質の規律が `harness-verification` skill に閉じている。この skill は「xp-harness の skill / subagent を検証する」ときに発火する設計で、今回は apm という外部ツールの調査だったため description にマッチせず発火せず、報告品質を当てるトリガーが無かった。規律自体は「調査・検証の報告全般」に効く汎用物なのに特定 skill に埋まっているのが穴。
2. **振る舞い（従属、一部推測）**: 規律を思い出せても調査に乗って「実況のまま報告」の楽な流れに乗った面がある（`dialogue-principles` 規律5「機械的処理に流れない」の失敗）。1 を直せば減る従属要因。

#### アクション
- 「調査スパイク skill 新設」カード（`kanban/`）を backlog → tier1 に昇格（別々の調査で同じ痛みが 2 回顕在化 = 痛み顕在化の条件が固まった。1 回目は AgentTeam 調査スパイク）。
- 同カードに設計論点を追記: **報告品質の規律（実況と報告を分ける／結論→詳細→証拠）を `harness-verification` と新 skill でどう single-source にするか**（規律装置最小注入・重複回避）。
- 実装は当該カード着手時（define-requirements から）。ここでは実装しない。

### 深掘り 2: 調査の網羅性の穴 —「裏付けで閉じ、反証で開かない」型（自己分析、依頼者は外側チェックで大枠一致）

#### 経緯
compile を最初に調べたとき「改修者向け内容が漏れ込むか（漏れ込む方）」だけ確認して「問題なし」で閉じ、「利用者の既存ファイルを上書きするか（壊す方）」を見なかった。依頼者の「既存 AGENTS.md がある場合は？」で初めてテストし、データ破壊が判明。＝ 取り決めでなく依頼者に拾われた（偶然救済、次は再現しない）。

#### 分析
表面の見落としでなく、型としての片肺。3 つの伸びしろ（name 初回誤結論・compile 漏れのみ・compile 上書き未検証）が全て「一つの肯定的な角度を確認したら結論を出す」で共通。根:

- **「問いに答える」ために調べ、「判断の危険を潰す」ために調べていない**。起点に「何を決めるための調査か」と「その判断が壊れる道の全集合（失敗モード・壊す方の副作用・利用者の現実の初期状態）」を置いていない。
- 最初に思いついた枠を一つ**裏付けたら閉じる**。反証（覆しにいく）で開かない。
- 皮肉: `harness-verification` に既に「反証可能性」の規律があるのに、そこに閉じていて調査に効かなかった（深掘り 1 の報告書式と同じ「一箇所に集約されていない」問題）。
- 調査は常に空ディレクトリから実行し、利用者の現実の初期状態（既存資産あり）を再現しなかった。

#### アクション
- 調査スパイク skill の核を「問いに答える」でなく「**判断の危険を潰す**」に据える。開始時に (1) どの判断のための調査か (2) その判断が壊れる道の列挙（失敗モード・壊す方の副作用・利用者の現実の初期状態） (3) 各モードを反証しにいく → 全部潰れて初めて完了。→ 調査スパイクカードに追記。
- 「反証可能性」も「報告書式」と同様、`harness-verification` と新 skill で一箇所に集約する設計論点として記録。
- 実装は当該カード着手時。ここでは実装しない。
