### 深掘り 1: 調査報告が実況地続きで読みにくい

#### 経緯
apm 挙動調査で、「検証するね → bash の生出力（巨大、例: 11KB の AGENTS.md 全ダンプ）→ 解釈」を 1 メッセージに地続きで並べた。結論が冒頭に区切られず、過程を追わないと要点に届かない報告が多かった（記録に残る事実）。

#### 分析
根は 2 つ（単線で打ち切らない）:

1. **構造的（取り決め側、主因）**: 「実況と報告を分ける／結論→詳細→証拠」という報告品質の規律が `harness-verification` skill に閉じている。この skill は「xp-harness の skill / subagent を検証する」ときに発火する設計で、今回は apm という外部ツールの調査だったため description にマッチせず発火せず、報告品質を当てるトリガーが無かった。規律自体は「調査・検証の報告全般」に効く汎用物なのに特定 skill に埋まっているのが穴。
2. **振る舞い（従属、一部推測）**: 規律を思い出せても調査に乗って「実況のまま報告」の楽な流れに乗った面がある（`dialogue-principles` 規律5「機械的処理に流れない」の失敗）。1 を直せば減る従属要因。

#### アクション
- 「調査スパイク skill 新設」カード（`kanban/`）を backlog → tier1 に昇格（別々の調査で同じ痛みが 2 回顕在化 = 痛み顕在化の条件が固まった。1 回目は AgentTeam 調査スパイク）。
- 同カードに設計論点を追記: **報告品質の規律（実況と報告を分ける／結論→詳細→証拠）を `harness-verification` と新 skill でどう single-source にするか**（規律装置最小注入・重複回避）。
- 実装は当該カード着手時（define-requirements から）。ここでは実装しない。
