# ふりかえり skill

## なぜこの skill があるか

作業中に気付く改善点を、気付いた時に拾えるサイクルが回っていないと、改善が言語化されないまま流れていく。skill / プロセス / 取り組み方の不備、どこに対する改善でも、気付いた時に残せるようにしたい。

ふりかえりを「自発的に促す」仕組みで構造化する。気付きを引き出して改善案を残すまでがこの skill の責務。改善実装そのものは別 skill / 人間判断に委ねる (= XP の retrospective と同じく「気づきを出して残す場」であって「直す場」ではない)。

## やること

skill が呼ばれたら以下の Step を回す。

**責任追及ではなく改善のための場**: このふりかえりは main session (= コーディングエージェント) を責めるためのものではなく、今後の改善のためにやる。だから失敗・抜け・迷いを正直に出してよい (= 隠さず出すほど価値がある)。後の Step で自己分析の正直さを求められるのも、この前提があって初めて成り立つ。

**既存フレームワーク (KPT / YWT / Fun-Done-Learn 等) は採用しない**: ふりかえりの本質は「共同で気づきを引き出し、改善案を作る」ことで、対話駆動の独自プロセスで足りる。フレームワークの「型」を持ち込むと、Step 間の責務 (= 情報収集 / 深掘り / アクション) が混在しがち。

進行は対話駆動で、`dialogue-principles` skill のルール (= 押し付けない / 1 問ずつ / 段階的な認識合わせ) に従う。

### Step 1: 情報収集

「よかった点 / 伸びしろ」の 2 軸でリストを出す。

- **Step 1 は発散**: 候補を短く 1 行で並べる。深掘り・分析は Step 3 (= 収束) の仕事。Step 1 で深掘りを前倒ししない (= 発散と収束を 1 つの出力に混ぜると認知負荷が上がる、`dialogue-principles` の発散収束のリズムに従う)
- main session が先に会話履歴・観察から「よかった点 / 伸びしろ」のリストを出す
- 依頼者が追加 / 削除 / 修正 (= 依頼者は main session の出力に引きずられず、自分の実感を優先して書き換えてよい)
- 軸は基本 2 軸、依頼者が「KPT で」「3 軸で」等と指示してきたら柔軟に変える
- よかった点は **安定 (取り決めに宿って再現する) / 偶然 (依頼者の介入等にたまたま救われた)** を区別する (= Step 1 ではラベルを付けるだけ、掘り下げは Step 3)。偶然の良い点は「次は再現しない」ので、それ自体が伸びしろ (= 安定させるには取り決めに宿らせる)

#### 働き方を規定する取り決めの挙動も併せて見る (= skill / subagent / CLAUDE.md・ルール)

「よかった点 / 伸びしろ」を出すとき、表面の出来事だけでなく **skill / subagent / CLAUDE.md・ルール (= エージェントの働き方を規定する取り決め) が適切に働いたか** という観点も併せて見る (= 強制ゲートではなく、見落とすと伸びしろが浅くなるための充実度の推奨)。コーディングエージェントにとって「働き方」はこれらの取り決めに刻まれているので、ここを見ないと表面の伸びしろしか拾えない (= 表面の事象が「skill の順序・タイミング問題の現れ」「ルールが導けていない現れ」として再分類されることが多い)。

観点 (= 利用可能だった skill / subagent を一通り当てる。1 件見つけて止めず、表面の 1 件に満足しない):

- 利用可能だった skill / subagent のうち、明示的に発火・起動したものは何か。「必ず発火」と書かれているのに発火しなかった skill、呼ぶべきタイミングで呼ばなかった subagent はないか
- 発火・起動の順序ミス / タイミングミスはなかったか。暗黙適用で済ませた skill はないか (= 明示発火させた方がよかったか)
- CLAUDE.md・ルールがその場面を正しく導けたか。曖昧で迷った / 導きが無くて自己流になった箇所はないか

気づきは **「どの取り決め (どの skill / subagent / ルール) を、どう直すべきか」に接続する**。コーディングエージェントは学びを次セッションに持ち越せないので、これらの取り決めに書き戻る形にしないと改善が消える (= 直す実装そのものは別 skill / 人間判断に委ねるが、「どこを直すか」までは具体にして残す)。

### Step 2: 深掘りトピック選択

Step 1 のリストの中から、依頼者が深掘りしたいトピックを選ぶ。

### Step 3: 深掘り分析

選んだトピックに対して「なんで?」を投げて根本原因を引き出す。

- **main session が自己分析として答えを出す**: main session が「なんで?」の答え (= 作業者として見えている根本理由) を先に出し、依頼者がそれにフィードバック。依頼者に「なんで?」を投げるのではない (= 作業者 = main session が原因を知っているはずなので、依頼者は外側からチェックする立場)
- **1 問ずつ進める**: 1 つの「なんで?」 + 1 つの自己分析の答え + 依頼者のフィードバック を 1 サイクル、次の「なんで?」を main session が自分で立てて進む
- 目安 5 回前後、状況判断で打ち切り可 (= 5 Whys の慣習に倣う目安、状況に応じて柔軟に増減)
- **一気に 5 回出さない (NG)**: 一度に複数段の「なんで?」と答えをまとめて出さない。読み手の負担と方向ミスの手戻りを避ける
- **単一の軸で打ち切らない**: 1 つの「なんで?」連鎖で「根本に到達した」と判断する前に、他の要因 (= 別角度からの「なんで?」) も探る。複数軸も検討してから Step 4 に進む (強制ではなく充実度の推奨)
- **自己分析の正直さを監査する**: 自己分析は、後付けで「もっともらしい因果連鎖」を捏造しやすい (= 実際に起きる失敗モード)。出した原因が本当に検証できる事実に基づくかを、出す前に点検する:
  - 検証できる事実 (会話履歴・成果物・実行結果に残るもの) と推測を分けて書く
  - 自分の内部状態に基づく主張 (「〜と思っていた」「〜のつもりだった」) は、事実ではなく推測と明示する。ただし **内部状態は自分で正直に出す。依頼者に当てさせない** (= 内省は main session にしかできない、依頼者は外側から事実・論理の整合をチェックする役)
  - 綺麗に揃いすぎた因果図は捏造のサインとして疑う (= 都合よく一本道に見えるときほど、別の要因を潰せていない)
  - 発火・タイミングなど働き方の挙動についての主張が改善の判断を左右するときは、自分の語りを信じず記録で裏を取る (= 毎回見る儀式にはしない、判断に効くときだけ)
  - 記憶 (= context に残る会話) は使えるが、序盤はやり取りが圧縮され、発火の語りには捏造が混じりうる。この 2 つの穴を前提に扱う

### Step 4: アクション or 分析

深掘りの結果として、改善のアクションを出す、または分析だけで終わる。

- **ここで出すのは改善案、ここで実装しない**: ふりかえりは「気づきを出して残す場」であって「直す場」ではない。アクションを出した勢いでその場で実装に踏み込まない。具体に残す範囲は Step 1 で接続した「どの取り決めをどう直すか」まで、実装は別 skill / 人間判断に委ねる
- **アクション 0 件 OK**: 「現時点で出ない」も価値ある情報、「(なし)」と明示記録
- 「分析結果のみ」「改善のアクションまで」 どちらの粒度で残してもよい

### Step 5: 次へ

- 別のトピックを深掘りしたい場合 Step 2 に戻る
- 終了する場合は記録に進む (成果物を書き出す)

## 成果物

改善案は `docs/working/<title>/ふりかえり.md` に記録する。

- 1 `<title>` ディレクトリ = 1 ファイル
- 累積形式: 複数セッションを 1 ファイルに追記 (時系列順)
- `<title>` は要件定義 / 基本設計と同じディレクトリ規約 (= 機能を表すケバブケース)

### フォーマット

````markdown
# <title> ふりかえり

## YYYY-MM-DD: [このふりかえりのタイトル]

### サマリ
[1-3 行、今回のふりかえりの位置付け]

### トピック一覧
よかった点:
- ...

伸びしろ:
- ...

### 深掘り 1: [深掘りしたトピックの名前]
#### 経緯
[作業 / 議論の流れ]

#### 分析
[原因 / 課題 / 構造的問題]

#### アクション
[次に試すこと、0 件なら (なし)]

### 深掘り 2: [別のトピックの名前]
...
````

### 記録の規則

- `## YYYY-MM-DD: [タイトル]` で 1 セッションのふりかえりを区切る
- 「トピック一覧」は Step 1 の結果 (= 全リスト、深掘りしなかったものも含む)
- 「深掘り N」section は Step 2-4 で深掘りしたトピックごとに作る。深掘りしなかったトピックは現れない
- 深掘り番号 N はセッション内 (= 各日付セクション内) でリセット、1 から開始
- アクション 0 件は「(なし)」と明示
