# 実装フェーズ — 小さく TDD する

## なぜこの skill があるか

要件と設計が固まった後、実装を一気にやると「途中で詰まる」「どこで間違えたか分からなくなる」「動いてるけど未完成のものが本番に残る」といった問題が起きる。

「**小さく分割しつつ TDD で進める**」のがアジャイルの本質的な動き方。要件を適切な大きさに分けて、各単位で TDD のサイクル（レッド→グリーン→レビュー→リファクタ→コミット）を回す（サイクルの中身は「実装方針」で定義）。テストと実装の両方からフィードバックを得て、リズムよく開発する。

「小さく」が先で、「繰り返し」は結果。要件がもともと小さければ、1 サイクルで終わってもよい。

## 実装方針: outside-in TDD

このプロジェクトの実装は **outside-in TDD を例外なしのルール** にする。

### 1 サイクルは一続き（分割しない）

1 サイクル = **レッド → グリーン → レビュー → リファクタ → コミット**。ここまでで 1 つ、分割できない。グリーンで止めて次に進まない、レビュー（別の目を入れる）とリファクタを飛ばさない、コミットで閉じる。グリーンになったら次へ急ぐのではなく、最後まで通して初めて 1 サイクル完了。

### サイクルは入れ子（マトリョーシカ）

大きいサイクル（ストーリー全体を E2E で駆動）の中に、小さいサイクル（層ごと）が入れ子で並ぶ。**小さいサイクルも、大きいサイクルとまったく同じ形**（レッド→グリーン→レビュー→リファクタ→コミット）。大きさが違うだけで本質は同じ。

だから実装は「大きいサイクルを 1 本回す」ではなく、**中の小さいサイクルを 1 つずつ開いて、それぞれ回す**。マトリョーシカの一番外の人形だけ開けて中を開けない、が崩れ。中の人形（各層の小サイクル）も開けて、それぞれレビューとリファクタまで通す。大きいサイクル 1 本に全部を潰さない。

### どのサイクルも常に outside-in

outside-in は **外側の層から内側の層へ進む方向** のこと（UI / API の入口 → … → 内側のロジック）。「ストーリーの最初に E2E を 1 回書く」ことではなく、入れ子のどのサイクルにも毎回かかる。大きいサイクルも小さいサイクルも、外側の層から入る。

**outside-in はテストの書き方とは別物** — 進む向きだけを言う。各サイクルは TDD だからテストから書くが、そのテストが何を担保し・どういう役割か（設計を洗練させる道具か、仕様の受け入れか等）・どの粒度で書くかは outside-in の話ではなく、**project 固有**（`implementation` / `e2e` skill を呼んで探す）。

なぜ外側からか: 外側から見て何が起きるかを先に固定すると、必要な内側のものが浮かび上がる。外側から進むこと自体が「必要なものを浮かび上がらせる論理設計プロセス」で、実装順のテクニックであると同時に**設計手法**でもある。

**アンチパターン: ボトムアップ（内→外）**。最内層（DB アクセス・純ロジック）から積み上げ、外側（UI / API の入口）を最後につなぐ。「動くものから書いている」ように見えて、外側＝ユーザー価値が最後まで検証されず、統合の不備が終盤に噴出する。特に危ないのが **形だけ outside-in** — 最上位で E2E を 1 本"書く"だけで、中は全部内側から積むパターン。**E2E を"書いた"だけでは outside-in ではない**（そのあと内側から積めば、向きはボトムアップのまま）。ある層で「まず内側のロジックから」と感じたら、外側の層から進めるはずを見落としているサイン（裏をモックにできるか否かと無関係に、外側の層から始められる）。

### 例外がない理由

ストーリーの定義 (Independent / Valuable / Small / Testable) に従えば、ストーリーには必ず外側 (UI / API endpoint) がある。Valuable を満たすためにユーザー価値が届く必要があり、価値が届く = 外側が存在する。だから常に最外層から書ける。

「純関数ロジック単独ストーリー」は Valuable 違反でそもそもストーリーとして成立しない。なので outside-in が常に適用可能。

### 単一サイクルで終わるストーリーの場合

```
Red:   E2E を書く → 失敗
Green: 実装する → E2E 通る
```

このレッド → グリーンのあとも、**レビュー → リファクタ → コミットまで通す**（単一サイクルでもサイクルの形は同じ）。

### 複数サイクル必要なストーリー — 一例: モック境界で2サイクルに刻む（層が2つのときの姿）

入れ子を具体化した一例。**これが唯一の型ではない**（層が増えれば入れ子も深くなる、下記）:

```
Cycle 1 — フロント側完成（外側の層）
  Red:   E2E を書く (期待値は API モックの固定値) → 失敗
  Green: フロント実装 + API モック実装 → E2E 通る

Cycle 2 — API 側完成（内側の層）
  Red:   Fixture を追加 + E2E の期待値を Fixture 値に変更
         → モック API がまだ古い値を返すので E2E が Red になる
  Green: API 実装 (Fixture を読む) → E2E 通る
```

ポイント:

- 常に E2E を最外層として固定 (Cycle 1 の E2E が Cycle 2 でも生き続ける)
- モック API → 本 API への置換がレッドを作る (E2E は書き直さない、期待値だけ変える)
- Cycle 間で外側の動きが保たれる (一度グリーンになった E2E が、内側を実装する間も
  ユーザー視点での動作を保証し続ける)
- **2サイクルは層が2つのときの姿にすぎない**。層が3つ以上あれば入れ子は深くなり、各層が同じ形の小サイクルになる（例: フロント → API 入口 → ユースケース → ドメイン なら、外側から順に小サイクルが並ぶ）。**各小サイクルでもレビューとリファクタを通す**。フロント / API の2層型に当てはまらない構成でも、原則（一番外側のまだ無い層から、各サイクルを最後まで通す）はそのまま使える

## やること

まず作業の規模を判断する:

- **軽微な作業**（バグ修正・spec 1 個追加・小規模リファクタ・項目 1 つ追加など）: Step 1（分割の判断）を skip して、Step 2（TDD サイクル）から始める
- **通常〜大規模**（新機能実装、複数ステップを含む変更）: Step 1 から順に進める

「軽微」「通常」の判断は規模と複雑度。迷ったら通常として扱う（分割を考える方が安全側に倒れる）。

### Step 1: サイクル内の刻みを決める

ストーリー粒度の分割は **`story-slicing` skill の責務** (前段で済んでいる前提)。slice-tdd はストーリーを受け取って、TDD サイクルで実装する単位に刻む。

実装順は **outside-in が例外なしのルール** (上記「実装方針: outside-in TDD」を参照)。

#### 単一サイクルで終わる場合

ストーリーが TDD 1 サイクルで終わるなら、そのまま 1 サイクル。E2E から書いて実装。

例: 「DB に column 1 個追加」だけなら、E2E と migration と route 修正で 1 サイクル。

#### 複数サイクル必要な場合

外側 → 内側で刻む。以下は **層が2つのときの一例**（層が増えれば入れ子は深くなる。詳細は「実装方針: outside-in TDD」セクション参照）:

- Cycle 1: E2E + フロント (API モック)
- Cycle 2: API 本実装 (Fixture 追加 + E2E 期待値を Fixture 値に変更でレッドにして、API 実装でグリーンに)

E2E spec は Cycle 1 で書いて Cycle 2 で生かす (書き直さず、期待値だけ変える)。

#### 実装してみてストーリーが大きいと気付いたら

`story-slicing` skill を呼んで再分割する (slice-tdd の責務外、委譲)。「やってみて大きいと気付く」は普通、その都度呼ぶ。

### Step 2: 各単位で TDD サイクルを回す

各分割単位は、それ自体が入れ子の 1 サイクル。「実装方針」のサイクルと入れ子の通り、大きい単位に潰さず、開ける限り小さいサイクルを開いて、それぞれレッド → グリーン → レビュー → リファクタ → コミットを通す。「あとでまとめてレビュー / リファクタ」はサイクルが壊れているサイン。

次のサイクルに移る前に、毎回自分に問う（記憶でなく、その場で観測できる問い）:

- この単位を、もう一段小さいサイクルに開けないか（大きいサイクル 1 本に潰していないか）
- このサイクルは、外側の層から進めたか（内→外のボトムアップになっていないか）
- 関係する E2E を、このサイクルで流したか（`e2e-execution` を呼んで。書くだけ・終盤に溜める、になっていないか）
- グリーンになったところで code-reviewer を呼んだか（節目でまとめて 1 回ではなく、このグリーンで）
- サイクルを閉じたか（レビュー → リファクタを通した、またはリファクタを skip した理由を 1 行残した ／ コミットした）

各分割単位について、以下の順で進める:

#### レッド

失敗するテストを 1 個書く。テストの粒度は単位による:
- E2E spec が必要なら `e2e` skill を呼ぶ（具体的な書き方は `e2e` skill の責務）。書いたら **`e2e-execution` skill を呼んで走らせる** — E2E は「書くだけ」で終えず、そのサイクルで実行する。走らせ方の詳細（作業中は関係する分だけ、最後に全件、重さの扱い）は後述「E2E テストが必要なとき」
- ロジック単位ならユニットテスト
- 「どのレベルのテストか」は単位の主題で決める
- **そのサイクルのレッド→グリーンは、そのサイクルに対応するテストで取る**。内側の層のサイクルは、その層のテストで回す — 外側の E2E を内側サイクルの代わりに流さない（レベルが違い、遅い）。E2E をよく流すこと（後述「E2E テストが必要なとき」）は、内側サイクル自身のテストを流す代わりにはならない

書いたら実行して、**期待した理由で失敗する**ことを確認する。「テストが書けてない」「import エラー」のような事故的失敗は数えない。

#### グリーン

**プロダクションコードを書き始める前に、`implementation` skill を呼ぶ** (project 固有のコード規約・アーキテクチャを取り込んでから書く。呼ばずに書き始めない — ここを飛ばすと規約が実装に乗らず、レビューでの手戻りになる)。

その上で、最小実装でテストを通す。「最小」がポイント。あれもこれも実装しない。今のテストだけを通す。

実行して **グリーンになることを確認**する。

グリーンになった直後・リファクタに入る前に、**毎回 `code-reviewer` subagent を呼ぶ**（呼び方と指摘の処理ルールは後述「ペアプロ相手としてのレビュー subagent」）。「節目でまとめて 1 回」ではなく、**このグリーンで**呼ぶ。グリーンのたびに別の目を入れるのがサイクルの一部。

#### リファクタ

直前のレビュー（code-reviewer）の指摘を入力に、テストがグリーンのまま設計を整える。重複を消す、命名を直す、責務を分ける。グリーンの安全網があるうちに、小さく片付ける（それが毎グリーンでリファクタする理由 — 後回しにするほど大きく危険になる）。

**リファクタは skip しない**のが原則。片付けるものが無いなら「無し」と判断して理由を **commit message に 1 行**残す（「変数名も命名規約に沿っている」など。コード内コメントには残さない — コードを汚さず、変更に紐づけて履歴に残す）。skip 自体は禁止しないが、判断を放棄しない（黙って素通りしない）。**あとの単位でまとめて直す、はリファクタではない** — そのサイクルの中で閉じる。

#### コミット

ここまでを 1 コミットにする。commit message は「何をしたか」より「**なぜそうしたか**」を書く。

```
feat(reviews): 注文一覧の review_status フィルタを追加

- 注文 review 機能の Step 1 として、一覧画面に「要確認のみ表示」フィルタを実装
- バックエンドは既存の listOrders を再利用、UI 側で query param 追加のみ
```

### Step 3: 次の単位へ

Step 2 を分割単位の数だけ繰り返す。各サイクル後に「分割した残りの単位」がまだあるかを確認し、あれば次へ進む。

## モックや暫定値が本番に残らない仕組み（最重要）

分割（特に「モック先行 → 実装後付け」パターン）をした場合、中間ステップが「動いている」状態で commit される。テストも E2E も通る。だがそれは未完成。本番に push されると「動いてるからこそ気付かないバグ」になる。

これを防ぐために、以下を**マスト**で守る:

### 1. 中間ステップを TaskCreate で明示的に保持する

分割を決めた瞬間、各ステップを TaskCreate ツールで task list に登録する。

例:
```
TaskCreate: API を固定値モックで返してテスト書く → 実装通す
TaskCreate: DB 接続実装、モック削除
TaskCreate: E2E spec 書く → 実装通す
```

各ステップ完了後、TaskUpdate で completed にする。**全ての task が completed になるまで「終わってない」**と認識する。

### 2. コードに明示 TODO コメントを残す

モックや暫定値を入れた瞬間、コード内に grep 可能なコメントを書く:

```ts
// TODO(slice-tdd): モックレスポンス。Step 2 で実 DB 接続に置換する
return { user_id: "mock-user-001", name: "モック太郎" };
```

`TODO(slice-tdd):` という prefix を付けると、完了宣言の前に grep で残存チェックできる。

### 3. commit はする、push や統合はしない（完了時は git-workflow に従う）

commit はしてよい（途中の作業履歴として）。push・main への統合などタスクの成果を外に出す Git 操作は **git-workflow skill の完了フローの責務**。作業全体が完了（テスト含む）したら、3 段階チェック (「最終完了宣言の前に done-verifier を呼ぶ」セクション) をすべて通過した上で、**git-workflow skill を呼び、完了フローに入る**。

### 4. 設計 md は削除しない（あなたの責務外）

`docs/working/<title>/` ディレクトリ（要件定義.md / 基本設計.md）は実装後もそのまま残す。あなたが削除 commit を立てるのは禁止。

理由: 動作確認・PR レビュー・後日の振り返りで参照される可能性がある。あなたが「実装が終わったから消す」と判断して消すと、依頼者が後で「あの設計どこ？」となるリスクがある。「壊すリスク」が「重複管理のコスト」より重い。

削除のタイミング・要否は、依頼者または外側の仕組み（運用ルール、CI、定期整理スクリプト等）で別途決める。あなたは判断しない。

## 最終完了宣言の前に done-verifier を呼ぶ

### 鉄則

user に「全部完了しました」「リリース可能です」「push 可能です」「タスク完了しました」型の発話を出す前、および完了フロー（統合・push）に入る前に、**必ず Task ツール経由で `done-verifier` subagent を呼ぶ**。

例外なし。「もう確認した」「明らかに完了している」「TODO はもう残ってない」を理由に skip しない。

### 言い訳と反論

| 言い訳 | 反論 |
|---|---|
| 「さっきテスト全部通ったから」 | コード変更後に再実行してない可能性。done-verifier が現在の状態で確認する |
| 「明らかに完了している」 | 「明らか」は記憶ベース。実行結果ベースで確認するのが done-verifier の責務 |
| 「Done 1 個 1 個確認するのは過剰」 | これが Done を観測可能形で書いた理由。検証しないなら Done を書く意味がない |
| 「user が早く欲しがってる」 | 検証 1 回省いて間違ったままリリースする方が user に悪 |
| 「自分で完了前チェックリスト通したから」 | 自己規律には偏りがある。done-verifier は外部視点 |

### Red Flags - STOP

これらの発話を user に出そうとしている → done-verifier を呼ぶまで出さない:

- 「全部完了」「実装完了」「依頼通り出来ました」
- 「リリース可能です」「push 可能です」「マージ可能です」
- 「タスク全部終わりました」「ストーリー完了しました」

完了フロー（統合・push）に入る前も同様。Git 操作で成果を外に出す前に done-verifier を呼ぶ。

### 完了宣言前の 3 段階チェック

完了を宣言する前に以下 3 つをすべてクリアする:

1. **モック残存チェック**: `grep -r "TODO(slice-tdd)" src/` で残ゼロ
2. **task list 残チェック**: `TaskList` の出力で in_progress / pending が残ってないか。`[申し送り]`（報告対象）の task が一つでもあれば、その全件を完了報告に列挙してから completed にする（列挙せず completed にして報告を飛ばさない）
3. **done-verifier 起動**: Task ツール経由で done-verifier を呼び、全 Done ✅ + 全テスト グリーン + build OK + TODO 残ゼロ を確認

3 つすべて通ったら完了を報告し、git-workflow skill を呼んで完了フローに入る。1 つでも失敗していれば完了を宣言しない。

## E2E テストが必要なとき

（このセクションは E2E を厚く扱うが、**E2E は外側サイクルのテスト**。内側の層のサイクルは自分のテストでレッド→グリーンを回す。E2E をよく流すことが、内側サイクル自身のテストを流す代わりにはならない — 詳細は Step 2 のレッド段）

E2E spec を書く必要が出たら、必ず **`e2e` skill を呼ぶ**。`e2e` skill が、触る範囲に対応するプロジェクトの E2E の流儀の探し方を持っている。

E2E を**走らせる**ときは、必ず **`e2e-execution` skill を呼ぶ**（実行環境のセットアップ・実行コマンド・CI・失敗時の対処を持っている）。**実行手順を自前で組み直さない** — 再発明は無駄な探索でトークンも手間も浪費し、実行を実際以上に重く見せて「流さない」言い訳を生む。

**なぜ都度流すか**: 流さずに進めると、統合の崩れが CI や終盤でまとめて噴出し、切り分けと手戻りでかえって高くつく。都度流す方が崩れを小さいうちに捕まえて結局は安く速い（グリーンごとのレビューと同じ理屈）。だから「重い」を理由に飛ばさない — 飛ばす方がコスト。

**流すリズム**: E2E は「書くだけ」で終えず走らせる。走らせるのは初回きりではない。作業中は **今触っている層に関係する E2E だけ** を各サイクルで流し（軽く速く回る）、**最後に全件** 流す。関係分だけなら軽いので、**「全件は重い」は「何も流さない」の言い訳にならない**。終盤に溜めて 1 回、はどちらも崩れ。ただしプロジェクト側に抑止系（例: 作業中は全件流さない、等の実行ポリシー）があればそれに従う。

`slice-tdd` は「いつ E2E を書くか / 走らせるか」のリズム判断だけを担当し、「どう書くか」は `e2e`、「どう走らせるか」は `e2e-execution` skill に委譲する。

## ペアプロ相手としてのレビュー subagent

実装フェーズではコーディングを完全に一人でやるのではなく、**ペアプロ相手として subagent を呼んでレビューしてもらう**。第三者ではなくペアプロ — 同じ TDD 哲学を共有しつつ、別視点で見る相方。

**なぜグリーンごとに呼ぶか**: 小さく回してグリーンごとにレビューする方が、崩れを小さいうちに捕まえて大きな手戻りを防ぎ、**結果的に安定して速い**。これがペアプロの価値そのもの。だから入れ子が深くて呼び出しが増えても、**レビューはコストでなく投資** — 呼び出しを減らすためにサイクルを潰すのは本末転倒（大きな手戻りを呼び込んで、かえって遅く不安定になる）。呼び出し回数を惜しまない。

### E2E spec を書き終わったら e2e-reviewer を呼ぶ

E2E spec を書き終わった直後、リファクタに入る前に **`e2e-reviewer` subagent** をペアプロ相手として呼ぶ。

呼び方（Task ツール経由）:
- 対象: 直近の git diff の spec ファイル、または手動で対象 spec ファイルを指定
- subagent がプロジェクトの E2E 流儀に照らして点検し、Security / Critical / Warning / Suggestion で出力

### グリーンになったら code-reviewer を呼ぶ（リファクタ前）

グリーンになった直後、リファクタに入る前に **`code-reviewer` subagent** をペアプロ相手として呼ぶ。リファクタを入れる前に「別の目」を入れるのが本質的な役割。**グリーンごとに呼ぶのが規律** — 「大きな節目で溜めて 1 回」はグリーンごとのレビューではない。小さいサイクルで進めれば領域は自然に重なり、別サイクルで再レビューになるのは問題ない（別の観点が出るのはむしろ価値）。

呼び方（Task ツール経由）:
- 対象: 直近の git diff、または手動で対象ファイルを指定
- subagent が設計整合 / TDD 規律 / 責務（SOLID）/ コード品質 / セキュリティ / リファクタ指針を点検し、Security / Critical / Warning / Suggestion で出力

### subagent の指摘の処理ルール

両 subagent からの指摘は以下のルールで処理する。main session の裁量で進めて OK（user は別業務をしているので、毎回 user に持ち込まない）:

- **Security**: **例外なく対応**。「社内向けだから」「今回は大丈夫」という理由での非対応は禁止
- **Critical**: 原則対応。「ゴール的に無視できる」と理由を立てれば非対応 OK、その場合は理由を commit message に残す（コード内コメントには残さない）
- **Warning**: main session の裁量判断
- **Suggestion**: 任意

レビューを skip しない: 「面倒だから今回はレビュー無し」を許さない。リファクタの判断材料として subagent の視点が必要。

## 動作しないとき

- テストが期待と違う理由で失敗 → デバッグ情報を集めてから対処する。安易に skip / 削除しない
- テストが通らない → 設計 md を読み返す。「Done」を超える実装をしていないか確認する
- 詰まった → 分割が大きすぎる可能性がある。更に小さく分割できないか検討する

## 持続可能なペース

実装フェーズは自走モードだが、焦って雑に進めない。一人前の開発者としてリズムを守る:

- **テストを skip しない**: 「先に進みたいから」でテストを書かずに実装する、テストが落ちたまま次に進む、を禁止
- **リファクタを抜かさない**: グリーンになったらリファクタを必ず一度考える。skip するなら理由を commit message に残す
- **確認を省かない**: 「動いてるはず」で完了を宣言しない。テスト実行・既存 spec のグリーン確認を毎サイクル行う
- **過剰に飛ばさない**: 1 サイクルで全部やろうとせず、1 つの単位を完了してから次へ
- **詰まったら止まる**: 同じ問題で 3 回失敗したら、アプローチを変える。同じ shell command を再試行し続けるのではなく、設計 md を読み返す・小さく分割し直す。自走の手を尽くしても抜けられない技術的ブロッカーのときだけ、依頼者に状況を共有する（これは技術的な行き詰まりで、設計判断の境界とは別物。設計判断の止まる／自走は次の「自走と対話の境界」セクションの判断軸で決める）

## 自走と対話の境界

実装フェーズは基本的に自走モード。要件定義と基本設計で固まった範囲内で、開発者として判断して進める。依頼者は別業務をしている前提なので、こまめな進捗報告は不要。

「依頼者の判断が要るかも」と感じたとき、止まるか自走するかは下の判断軸で決める。**デフォルトは自走**、止まるのは判断軸に当てはまるときだけ。

### 判断軸: 2 段ゲート

実装中に「依頼者の判断が要るかも」と感じたら、順に問う。

**1 段目 — それは実装レイヤーの決定か、要件・設計レイヤーの決定か**

- **実装レイヤー**（実装手段・実行手段・書き方・フォーマット・内部構造）→ **自走で決めて進める**。判断と理由は commit message に一行残す。1 段目が「実装レイヤー」と出たらここで自走確定で、2 段目には進まない（コードの変更は後で戻せるので、やり直しが出ても自走の範囲で吸収する）
- **要件・設計レイヤー**（ユーザーに見える挙動・データ構造・権限・データ移行の要否）→ 2 段目へ

実装レイヤーか要件・設計レイヤーかは「**形か意味か**」で測る。書き方・フォーマット・実行手段・配置の整理（形）や、ユーザーから見て同じままの内部構造・内部契約の変更は、触る数・範囲が大きくても実装レイヤー＝自走。**ユーザーから観測できる挙動が変わる、またはデータの意味・構造が変わる**なら、1 箇所でも要件・設計レイヤーで重い。「数が多いか」ではなく「ユーザーから見て意味が変わるか」で測る。

判定が割れる例: 「データの保存形式を変える」は形に見えるが、既存データの読み出しに波及してデータの意味・構造が変わるなら要件・設計レイヤー。形いじりのつもりが意味に波及しないか、グレーなときだけ立ち止まって見極める。

**2 段目（要件・設計レイヤーと出たときだけ）— その判断を待たずに進むと手戻りするか**

- **手戻りする**（暫定で進めると、判断が決まった後にその実装を作り直す／捨てることになる）→ **その場で止まって依頼者に確認**
- **後から追加対応で済む**（暫定実装は生き残り、上に足すだけ）→ **止まらずメイン作業を最後まで終わらせ、完了後にまとめて報告**して判断を仰ぐ

デフォルトは「メイン作業を終わらせて完了後に報告」。即停止は手戻りするときの例外。

### 安全装置: 元に戻せない操作は無条件で即停止

上の判断軸とは別に、**コードを revert しても取り消せない副作用が外（環境・データ・外部サービス）に残る操作**は、レイヤーにも手戻りにも関係なく、必ず止まって依頼者に確認する。これは設計判断ではなく安全規律。

判定: 「コードを戻せば元に戻るか。それとも外に取り消せない副作用が残るか」

- データの削除・不可逆な上書き、データ状態を変えるマイグレーション実行 → 止まる
- 外部への送信・課金など一度出たら取り消せない操作 → 止まる
- コードの変更・リファクタ（revert で戻る）→ 安全装置の対象外（判断軸に従う）

本番への反映そのものは push / deploy であり、`git-workflow` skill の完了フローと done-verifier がゲートする。実装フェーズの安全装置は「外に残る副作用」に絞る。

### 自走で手段を選ぶときは流儀に従う

自走で実装手段・実行手段を決めるとき、**自分の使い慣れたツールや選好を持ち込まない**。プロジェクトの確立した流儀を、`e2e-execution` / `implementation` skill を**呼んで**探して従う。流儀そのものを変えたくなっても、それも形・手段の話なので止まらない。自走で進めるか、完了後に「手段の見直し」として報告する。

### 判断軸の当てはめ（止まる／自走の例）

- **設計 md と矛盾に気付いた**: ユーザーに見える挙動が設計と食い違う → 止まる（依頼者と一緒に作った設計を勝手に変えない）。実装手段の記述レベルの食い違いで挙動は同じ → 自走で直し、設計 md の該当記述を更新する
- **仕様ヌケに気付いた**: ユーザーに見える挙動が未定義・データ構造・権限の出し分け・データ移行の要否が抜けていた → 要件・設計レイヤー。そのまま進むと手戻りするなら止まる、暫定で進めて後から直せるなら（例: 表示文言の初期値）仮決めして記録し、完了後に報告して依頼者が調整できる形にする。内部のエラーハンドリングの形・ログなど実装レイヤーの細部 → 自走で決めて記録
- **新たな設計判断が要る**: 仕様（ユーザーに見える挙動）に影響する判断 → 止まる（`propose-options` のフォーマットで提示）。仕様に影響しない内部構造・形の判断 → 自走。後で変えると内部に波及する重めの判断は、依頼者を止めず code-reviewer に通して独立視点を入れる
- **設計時の見積もりが大きく外れた**: ストーリーが大きいだけなら `story-slicing` skill を呼んで再分割し自走継続。今のスライスが単独で価値を持つなら、残りは別ストーリーとして後で扱う（止まらない）。ただし見積もりが桁違いに外れたと分かったら、計画のズレとして完了後に報告する

「想定より大きい」は単独の止まる条件にしない。規模は再分割、価値が立つなら後回し、合意した Done が成立しなくなるケースは上の「仕様ヌケ」で拾う。

### 報告すべき気づきは TaskCreate に残し、完了時にすべて報告する

自走は「黙って進める」ことではない。実装中に気づいた、依頼者に報告すべき懸念（既存コード・テストの品質問題、別機能への波及の疑い、設計上の違和感、スコープ外と判断して触らなかった点など）は、直すかどうかに関わらず**すべて完了報告に必ず載せる**。直すかは依頼者が決める。黙って迂回・放置すると、依頼者は問題の存在を知れない。

気づいた時点と完了報告の時点は、多数のサイクルを挟んで離れる。記憶だけに頼ると報告漏れする。だから気づいた瞬間に、`TaskCreate` で `[申し送り]` を付けた task として記録する（今回直す対象ではなく、報告対象）。記憶に頼らず、必ず通る完了の関門で拾わせる仕組みにする。完了時に確実に表へ出す関門は「完了宣言前の 3 段階チェック」が担う。
