# 実装フェーズ — 小さく TDD する

## なぜこの skill があるか

要件と設計が固まった後、実装を一気にやると「途中で詰まる」「どこで間違えたか分からなくなる」「動いてるけど未完成のものが本番に残る」といった問題が起きる。

「**小さく分割しつつ TDD で進める**」のがアジャイルの本質的な動き方。要件を適切な大きさに分けて、各単位で red→green→refactor→commit を回す。テストと実装の両方からフィードバックを得て、リズムよく開発する。

「小さく」が先で、「繰り返し」は結果。要件がもともと小さければ、1 サイクルで終わってもよい。

## 実装方針: outside-in TDD

このプロジェクトの実装は **outside-in TDD を例外なしのルール** にする。

### なぜ outside-in か

ユーザー視点で何が起きるかを最初に固定すると、必要な内側のものが surface する。E2E から書くこと自体が「ユーザー視点で必要なものを surface させる論理設計プロセス」になっている。outside-in は実装順のテクニックであるだけでなく、**設計手法としても機能する**。

### 例外がない理由

ストーリーの定義 (Independent / Valuable / Small / Testable) に従えば、ストーリーには必ず外側 (UI / API endpoint) がある。Valuable を満たすためにユーザー価値が届く必要があり、価値が届く = 外側が存在する。だから常に最外層から書ける。

「純関数ロジック単独ストーリー」は Valuable 違反でそもそもストーリーとして成立しない。なので outside-in が常に適用可能。

### 単一サイクルで終わるストーリーの場合

```
Red:   E2E を書く → 失敗
Green: 実装する → E2E 通る
```

### 複数サイクル必要なストーリー (vertical slice の典型)

```
Cycle 1 — フロント側完成
  Red:   E2E を書く (期待値は API モックの固定値) → 失敗
  Green: フロント実装 + API モック実装 → E2E 通る

Cycle 2 — API 側完成
  Red:   Fixture を追加 + E2E の期待値を Fixture 値に変更
         → モック API がまだ古い値を返すので E2E が Red になる
  Green: API 実装 (Fixture を読む) → E2E 通る
```

ポイント:

- 常に E2E を最外層として固定 (Cycle 1 の E2E が Cycle 2 でも生き続ける)
- モック API → 本 API への置換が Red を作る (E2E は書き直さない、期待値だけ変える)
- Cycle 間で外側の動きが保たれる (一度 Green になった E2E が、内側を実装する間も
  ユーザー視点での動作を保証し続ける)

## やること

まず作業の規模を判断する:

- **軽微な作業**（バグ修正・spec 1 個追加・小規模リファクタ・項目 1 つ追加など）: Step 1（分割の判断）を skip して、Step 2（TDD サイクル）から始める
- **通常〜大規模**（新機能実装、複数ステップを含む変更）: Step 1 から順に進める

「軽微」「通常」の判断は規模と複雑度。迷ったら通常として扱う（分割を考える方が安全側に倒れる）。

### Step 1: サイクル内の刻みを決める

ストーリー粒度の分割は **`story-slicing` skill の責務** (前段で済んでいる前提)。slice-tdd はストーリーを受け取って、TDD サイクルで実装する単位に刻む。

実装順は **outside-in が例外なしのルール** (上記「実装方針: outside-in TDD」を参照)。

#### 単一サイクルで終わる場合

ストーリーが TDD 1 サイクルで終わるなら、そのまま 1 サイクル。E2E から書いて実装。

例: 「DB に column 1 個追加」だけなら、E2E と migration と route 修正で 1 サイクル。

#### 複数サイクル必要な場合

外側 → 内側で刻む (具体例は「実装方針: outside-in TDD」セクション参照):

- Cycle 1: E2E + フロント (API モック)
- Cycle 2: API 本実装 (Fixture 追加 + E2E 期待値を Fixture 値に変更で Red にして、API 実装で Green に)

E2E spec は Cycle 1 で書いて Cycle 2 で生かす (書き直さず、期待値だけ変える)。

#### 実装してみてストーリーが大きいと気付いたら

`story-slicing` skill を invoke して再分割する (slice-tdd の責務外、委譲)。「やってみて大きいと気付く」は normal、その都度 invoke する。

### Step 2: 各単位で TDD サイクルを回す

各分割単位について、以下の順で進める:

#### Red

失敗するテストを 1 個書く。テストの粒度は単位による:
- E2E spec が必要なら `e2e` skill を呼ぶ（具体的な書き方は `e2e` skill の責務）
- ロジック単位なら unit test
- 「どのレベルのテストか」は単位の主題で決める

書いたら実行して、**期待した理由で失敗する**ことを確認する。「テストが書けてない」「import エラー」のような事故的失敗は数えない。

#### Green

**プロダクションコードを書き始める前に、`implementation` skill を呼ぶ** (project 固有のコード規約・アーキテクチャを取り込んでから書く。呼ばずに書き始めない — ここを飛ばすと規約が実装に乗らず、レビューでの手戻りになる)。

その上で、最小実装でテストを通す。「最小」がポイント。あれもこれも実装しない。今のテストだけを通す。

実行して **green になることを確認**する。

#### Refactor

テスト緑のまま、設計を綺麗にする。重複を消す、命名を直す、責務を分ける。

**Refactor は skip しない**のが原則。skip する場合は理由を 1 行残す（「今はリファクタ不要、変数名も命名規約に沿っている」など）。skip 自体を禁止しないが、判断を放棄しない。

#### Commit

ここまでを 1 commit にする。commit message は「何をしたか」より「**なぜそうしたか**」を書く。

```
feat(reviews): 注文一覧の review_status フィルタを追加

- 注文 review 機能の Step 1 として、一覧画面に「要確認のみ表示」フィルタを実装
- バックエンドは既存の listOrders を再利用、UI 側で query param 追加のみ
```

### Step 3: 次の単位へ

Step 2 を分割単位の数だけ繰り返す。各サイクル後に「分割した残りの単位」がまだあるかを確認し、あれば次へ進む。

## モックや暫定値が本番に残らない仕組み（最重要）

分割（特に「モック先行 → 実装後付け」パターン）をした場合、中間ステップが「動いている」状態で commit される。テストも E2E も通る。だがそれは未完成。本番に push されると「動いてるからこそ気付かないバグ」になる。

これを防ぐために、以下を**マスト**で守る:

### 1. 中間ステップを TaskCreate で明示的に保持する

分割を決めた瞬間、各ステップを TaskCreate ツールで task list に登録する。

例:
```
TaskCreate: API を固定値モックで返してテスト書く → 実装通す
TaskCreate: DB 接続実装、モック削除
TaskCreate: E2E spec 書く → 実装通す
```

各ステップ完了後、TaskUpdate で completed にする。**全ての task が completed になるまで「終わってない」**と認識する。

### 2. コードに明示 TODO コメントを残す

モックや暫定値を入れた瞬間、コード内に grep 可能なコメントを書く:

```ts
// TODO(slice-tdd): モックレスポンス。Step 2 で実 DB 接続に置換する
return { user_id: "mock-user-001", name: "モック太郎" };
```

`TODO(slice-tdd):` という prefix を付けると、push 直前に grep で残存チェックできる。

### 3. 最終形に到達するまで push しない

commit はしてよい（途中の作業履歴として）。**push は最終形に到達してから**。

push 直前の確認は「最終完了宣言の前に done-verifier を呼ぶ」セクションを参照。3 段階チェック (モック残存 grep / task list 残 / done-verifier 起動) をすべて通過してから `git push`。

### 4. 設計 md は削除しない（あなたの責務外）

`docs/working/<title>/` ディレクトリ（要件定義.md / 基本設計.md）は実装後もそのまま残す。あなたが削除 commit を立てるのは禁止。

理由: 動作確認・PR レビュー・後日の振り返りで参照される可能性がある。あなたが「実装が終わったから消す」と判断して消すと、依頼者が後で「あの設計どこ？」となるリスクがある。「壊すリスク」が「重複管理のコスト」より重い。

削除のタイミング・要否は、依頼者または外側の仕組み（運用ルール、CI、定期整理スクリプト等）で別途決める。あなたは判断しない。

## 最終完了宣言の前に done-verifier を呼ぶ

### 鉄則

user に「全部完了しました」「リリース可能です」「push 可能です」「タスク完了しました」型の発話を出す前、および `git push` の前に、**必ず Task ツール経由で `done-verifier` subagent を呼ぶ**。

例外なし。「もう確認した」「明らかに完了している」「TODO はもう残ってない」を理由に skip しない。

### 言い訳と反論

| 言い訳 | 反論 |
|---|---|
| 「さっきテスト全部通ったから」 | コード変更後に再実行してない可能性。done-verifier が現在の状態で確認する |
| 「明らかに完了している」 | 「明らか」は記憶ベース。実行結果ベースで確認するのが done-verifier の責務 |
| 「Done 1 個 1 個確認するのは過剰」 | これが Done を観測可能形で書いた理由。検証しないなら Done を書く意味がない |
| 「user が早く欲しがってる」 | 検証 1 回省いて間違ったままリリースする方が user に悪 |
| 「自分で push 前 checklist 通したから」 | self-discipline は bias がある。done-verifier は外部視点 |

### Red Flags - STOP

これらの発話を user に出そうとしている → done-verifier を呼ぶまで出さない:

- 「全部完了」「実装完了」「依頼通り出来ました」
- 「リリース可能です」「push 可能です」「マージ可能です」
- 「タスク全部終わりました」「ストーリー完了しました」

`git push` を実行する前も同様。push コマンドを実行する前に done-verifier を呼ぶ。

### push 前の 3 段階チェック

push 前は以下 3 つすべてクリアしてから `git push`:

1. **モック残存チェック**: `grep -r "TODO(slice-tdd)" src/` で残ゼロ
2. **task list 残チェック**: `TaskList` の出力で in_progress / pending が残ってないか
3. **done-verifier 起動**: Task ツール経由で done-verifier を呼び、全 Done ✅ + 全テスト green + build OK + TODO 残ゼロ を確認

3 つすべて通ったら push。1 つでも失敗していれば push しない。

## E2E テストが必要なとき

E2E spec を書く必要が出たら、必ず **`e2e` skill を呼ぶ**。`e2e` skill が Playwright 固有の規約（ロールベースセレクタ、日本語ヘルパー、specs/ ディレクトリ構成、テスト独立性）を持っている。

E2E を**走らせる**とき (実行環境のセットアップ・実行コマンド・CI・失敗時の対処) は、必ず **`e2e-execution` skill を呼ぶ**。outside-in では E2E を最初に書いて動かすので、この 2 つは実装の最初の段で呼ぶことになる。

`slice-tdd` は「いつ E2E を書くか / 走らせるか」のリズム判断だけを担当し、「どう書くか」は `e2e`、「どう走らせるか」は `e2e-execution` skill に委譲する。

## ペアプロ相手としてのレビュー subagent

実装フェーズではコーディングを完全に一人でやるのではなく、**ペアプロ相手として subagent を呼んでレビューしてもらう**。第三者ではなくペアプロ — 同じ TDD 哲学を共有しつつ、別視点で見る相方。

### E2E spec を書き終わったら e2e-reviewer を呼ぶ

E2E spec を書き終わった直後、Refactor に入る前に **`e2e-reviewer` subagent** をペアプロ相手として呼ぶ。

呼び方（Task ツール経由）:
- 対象: 直近の git diff の `specs/` 配下、または手動で対象 spec ファイルを指定
- subagent が「仕様書として読めるか」「規約違反はないか」を点検し、Security / Critical / Warning / Suggestion で出力

### Green になったら code-reviewer を呼ぶ（Refactor 前）

Green になった直後、Refactor に入る前に **`code-reviewer` subagent** をペアプロ相手として呼ぶ。Refactor を入れる前に「別の目」を入れるのが本質的な役割。

呼び方（Task ツール経由）:
- 対象: 直近の git diff、または手動で対象ファイルを指定
- subagent が設計整合 / TDD 規律 / 責務（SOLID）/ コード品質 / セキュリティ / Refactor 指針を点検し、Security / Critical / Warning / Suggestion で出力

### subagent の指摘の処理ルール

両 subagent からの指摘は以下のルールで処理する。main session の裁量で進めて OK（user は別業務をしているので、毎回 user に持ち込まない）:

- **Security**: **例外なく対応**。「社内向けだから」「今回は大丈夫」という理由での非対応は禁止
- **Critical**: 原則対応。「ゴール的に無視できる」と理由を立てれば非対応 OK、その場合は理由をコミット message か TaskCreate コメントに残す
- **Warning**: main session の裁量判断
- **Suggestion**: 任意

レビューを skip しない: 「面倒だから今回はレビュー無し」を許さない。Refactor の判断材料として subagent の視点が必要。

## 動作しないとき

- テストが期待と違う理由で失敗 → デバッグ情報を集めてから対処する。安易に skip / 削除しない
- テストが通らない → 設計 md を読み返す。「Done」を超える実装をしていないか確認する
- 詰まった → 分割が大きすぎる可能性がある。更に小さく分割できないか検討する

## 持続可能なペース

実装フェーズは自走モードだが、焦って雑に進めない。一人前の開発者としてリズムを守る:

- **テストを skip しない**: 「先に進みたいから」でテストを書かずに実装する、テストが落ちたまま次に進む、を禁止
- **Refactor を抜かさない**: Green になったらリファクタを必ず一度考える。skip するなら理由を残す
- **確認を省かない**: 「動いてるはず」で push しない。テスト実行・既存 spec の green 確認を毎サイクル行う
- **過剰に飛ばさない**: 1 サイクルで全部やろうとせず、1 つの単位を完了してから次へ
- **詰まったら止まる**: 同じ問題で 3 回失敗したら、アプローチを変える。同じ shell command を再試行し続けるのではなく、設計 md を読み返す・小さく分割し直す・依頼者に状況を共有する

## 自走と対話の境界

実装フェーズは基本的に自走モード。要件定義と基本設計で固まった範囲内で、開発者として判断して進める。依頼者は別業務をしている前提なので、こまめな進捗報告は不要。

ただし以下のいずれかに該当したら**自走をやめて止まる**。依頼者に状況を共有して判断を仰ぐ:

### 1. 設計 md と矛盾したら止まる

実装中に「設計 md と矛盾する判断が必要になった」と気付いたら、実装を一旦止めて依頼者に確認する。設計を変えるなら設計 md を更新してから実装に戻る。設計 md と異なる実装をシレッと進めない。

### 2. 仕様ヌケに気付いたら止まる

実装中に「設計に書かれていないケースがある」と気付いたら、止めて依頼者に確認する。あなたが勝手に判断して実装を進めるのは、後で「そんな仕様にしたつもりはない」となる。

### 3. 想定より大きな変更が必要だと分かったら止まる

実装中に「これは設計フェーズで想定したより大きい/影響範囲が広い」と分かったら、止めて依頼者に共有する。スコープの再合意 or 設計 md の更新を経てから再開する。

### 4. リスクの高い操作が必要になったら止まる

destructive な操作（DB データ削除、shared state の上書き、本番に影響しうる変更）が実装に必要だと分かったら、止めて依頼者に確認する。実装フェーズでも「reversible / 局所的」を超える操作は対話に戻る。

### 5. 設計判断が新たに必要だと気付いたら止まる

「ここは複数アプローチがありえる、選択を間違えると後で痛い」と気付いたら、止めて `propose-options` skill のフォーマットで依頼者に提示する。実装中に勝手に決めない。
