# E2E テストの流儀

このスキルは E2E テスト (Playwright 想定) を書く・直すときに参照する。流儀をここに集約してある。

skill 名は `e2e` というインターフェース名にしてあるが、**ここでは E2E は Playwright で書く前提**。中身は Playwright 固有の規約（API の使い方、セレクタ、`test.use` 等）を前提にしている。他のフレームワーク（Cypress / Vitest browser mode など）を使う consumer は、自前で同名 skill を `.apm/skills/e2e/SKILL.md` に置いて last-installed-wins で全体 override するか、別 skill 名 (例: `e2e-cypress`) を新規作成する。

## 大前提：テストは仕様書

ここに置くテストは「動作確認スクリプト」ではなく **仕様書** として機能することを目指している。非エンジニア（テナントオペレーション担当・デザイナー・PdM）が読んで「何をして何が起きるか」がわかるべき。

この前提がすべてのルールの根拠になっている。迷ったら「これを読んだ非エンジニアが意図を追えるか？」で判断する。

## ルール

### 1. セレクタはユーザー視点で

要素の取得は **ロールベースのセレクタ** を基本にする。

優先順位:

1. `getByRole('button', { name: '注文作成' })` — ユーザーが認識する役割と文言
2. `getByLabel('ユーザーコード')` / `getByText('スタンバイ中')` — 画面上の見える文字
3. `getByTestId(...)` — どうしても上記で表現できないとき
4. `page.locator('#some-id')` — **最終手段**。なぜ ID でしか取れないか説明できないなら採用しない

理由: ID セレクタは実装詳細に結合する。リファクタで簡単に壊れるし、テストを読んでも「ユーザーが何を見て何を押したか」が分からない。仕様書としての価値がなくなる。

### 2. 操作はヘルパー関数（日本語名）に切り出す

テスト本体は **ステップを並べただけ** に見える状態を保つ。Playwright API の生呼び出しはヘルパー関数に隠蔽する。

ヘルパーは `e2e/steps/` 配下に置く。**関数名は日本語で意図を表現** する。

良い例（`e2e/specs/注文作成フロー.spec.ts`）:

```ts
test.use(ログインユーザーを指定する(taro));

test('未注文のtaroがログインすると注文作成注文が完了し、自動的にスタンバイ画面へ戻る', async ({ page }) => {
  await 指定したテナントの管理画面をセットアップする(page, { テナント: store-001 });
  await 注文画面を開く(page);
  await 指定したユーザーの注文作成確認画面が表示される(page, { ユーザー: taro });
  await 注文作成ボタンをタップする(page);
  await 指定したユーザーの注文作成完了画面が表示される(page, { ユーザー: taro });
  await スタンバイ画面に自動復帰する(page);
});
```

悪い例（同じシナリオを生で書いたもの）:

```ts
test('注文作成注文', async ({ page }) => {
  await page.goto('/setup/STORE-001/...');
  await page.waitForURL('/');
  await page.evaluate(() => navigator.mediaDevices.getUserMedia({ video: true }));
  await expect(page.locator('#confirm-modal')).toBeVisible();
  await page.click('#confirm-btn');
  await expect(page.locator('.success-toast')).toContainText('注文作成しました');
  await page.waitForSelector('#standby-screen');
});
```

下のコードは「Playwright を使って何をしているか」は読めるが、「業務として何が起きているか」は読めない。

ヘルパーを新しく作る基準:
- テスト本体に Playwright API（`page.click`, `page.locator`, `expect(page...)`, `page.route`, `page.waitForTimeout` など）が出てきたら、それはヘルパーに昇格させるサイン
- 既存ヘルパーで似たことをしているものがないか必ず確認してから新規作成する
- **ヘルパーは嘘をつかない**：引数として受け取る値は実装内で必ず使う（`期待ファイル ↔ 引数のユーザー` の検証など）。「ドキュメンテーションのためだけの引数」を持つヘルパーは作らない。語り口を強化したいならテスト名・後続のアサーションヘルパーで補う

既存ヘルパーの一覧 (project ごとに異なる、以下は example):
- `e2e/steps/認証.ts` — 認証情報の差し込み
- `e2e/steps/管理画面端末.ts` — セットアップ・注文画面遷移
- `e2e/steps/注文操作.ts` — ボタン操作
- `e2e/steps/画面状態.ts` — 画面表示の検証
- `e2e/steps/DB操作.ts` — 前提状態のDB注入・クリア
- `e2e/steps/test-fixture.ts` — test fixture 生成

### 3. 前提（GIVEN）と行動（WHEN/THEN）を describe / test で分離する

すべてのテストは BDD の三段構造で書く:

| 構造 | 書く場所 | 役割 |
|---|---|---|
| **GIVEN**（前提） | `test.describe` の名前 + `beforeEach` | 環境（管理画面がテナント X として登録済み）と人物状態（DB 上の注文履歴）。テストの主題ではない、所与の世界 |
| **WHEN**（行動） | `test` 本体 | 登場人物が起こす行動（ログインする、ボタンをタップする） |
| **THEN**（検証） | `test` 本体 | 行動の結果として何が見えるか・記録されるか |

書き方:

```ts
test.use(ログインユーザーを指定する(taro));

test.describe('注文承認フロー', () => {
  test.describe('store-001の管理画面で30分前に注文作成注文しているtaroが', () => {
    // GIVEN: 環境と人物状態を作る
    test.beforeEach(async ({ page }) => {
      await 指定したテナントの管理画面をセットアップする(page, { テナント: store-001 });
      await 指定したユーザーの注文履歴をクリアする({ ユーザー: taro });
      await 指定したユーザーに過去の注文作成注文を仕込む({ ユーザー: taro, 何分前: 30 });
    });

    // WHEN/THEN: 行動と検証だけ
    test('ログインすると注文承認注文が完了し、自動的にスタンバイ画面へ戻る', async ({ page }) => {
      await 注文画面を開く(page);
      await 指定したユーザーの注文承認確認画面が表示される(page, { ユーザー: taro });
      await 注文承認ボタンをタップする(page);
      await 指定したユーザーの注文承認完了画面が表示される(page, { ユーザー: taro });
      await スタンバイ画面に自動復帰する(page);
    });
  });
});
```

レポート:
```
注文承認フロー > store-001の管理画面で30分前に注文作成注文しているtaroが > ログインすると注文承認注文が完了し、自動的にスタンバイ画面へ戻る
```

3 つを連結すると **完結した日本語の仕様文** になるのが目印。

#### 命名の文法

- **describe（GIVEN）**: 名詞句で終わる、人物または環境を主語にした句
  - 例: `store-001の管理画面で30分前に注文作成注文しているtaroが`
  - 例: `store-002の管理画面で19時間前に注文作成注文しているjiroが`
  - 例: `未登録のタブレットで`
- **test（WHEN/THEN）**: 動詞句で始まり、結果まで言い切る
  - 例: `ログインすると注文承認注文が完了し、自動的にスタンバイ画面へ戻る`
  - 例: `「了解」をタップするとスタンバイ画面に戻る`

#### 「主題か前提か」の判定

同じ操作（例: `指定したテナントの管理画面をセットアップする`）が、テストの主題なら **WHEN として test 本体に**、前提として在ってほしいだけなら **GIVEN として beforeEach に** 配置する。`デバイスセットアップ` のテストではセットアップ自体が主題なので test 本体、`注文作成フロー` ではセットアップは前提なので beforeEach、というように同じ操作が文脈で位置を変える。

#### テスト独立性

`workers: 1` / `fullyParallel: false` で順次実行されるが、**実行順序に依存しない** こと。

- 前提となる DB 状態は必ず `beforeEach` で **自分で作る**（他テストの副作用に頼らない）
- 注文履歴のような可変データは `指定したユーザーの注文履歴をクリアする` でクリアしてから投入
- `afterEach` だけでは不十分（前のテストが afterEach 前に落ちると次のテストが汚染状態でスタートする）。**前提状態の構築は必ず beforeEach で**
- `stores` / `employees` のような静的マスタは globalSetup で投入済み。テスト側で再投入しなくてよい

理由: テストを単独で動かしたとき、CI で順序が変わったとき、特定テストだけ skip したときに、それでも仕様として正しく動くことを保証する。実際に `注文作成フロー → クールダウン` の順だけ落ちる、という気づきにくい依存が発生したことがある（注文作成フローが残した 'in' 注文でシステムが注文承認を提案する状態になり、クールダウンテストが期待する注文作成確認画面と食い違う）。

検証法: 怪しいテストは `npx playwright test e2e/specs/<疑問のテスト>.spec.ts` 単独で 1 回、`npx playwright test e2e/specs/<別のテスト>.spec.ts e2e/specs/<疑問のテスト>.spec.ts` の順で 1 回実行して両方通れば独立。

### 4. ファイル命名・配置

- spec ファイルは `e2e/specs/` 配下に置く
- ファイル名: `<シナリオ名>.spec.ts`（例: `注文作成フロー.spec.ts`）。連番は付けない（依存関係ではないので）
- 1 ファイル＝ 1 シナリオを基本に。`describe` でグループ化、`test` で個別ケース
- 登場人物（テナント・管理者・ユーザー）は `e2e/data/` に定義済みのものを使う。テスト中に新しい人物を作らない（仕様書の登場人物は限定したほうが読みやすい）
- `e2e/data/` は **e2e のテストデータの唯一のソース**（product 側 `scripts/seed.sql` への依存はゼロ）。`e2e/data/DB初期化.ts` が global-setup から呼ばれて静的マスタを INSERT OR IGNORE で投入する。新しい登場人物を増やすときは値オブジェクトを定義し、必要なら DB初期化.ts にも対応する INSERT を追加する

### 5. 認証情報はファイル単位

```ts
test.use(ログインユーザーを指定する(taro));
```

をファイル先頭に書く。**同一ファイル内のテストは同じユーザーの認証情報を共有する** のが前提（test fixture がファイル単位で差し替わる仕組みのため）。

別のユーザーの認証情報が必要なシナリオは別ファイルに分ける。

### 6. 時間に関わるテストは仮想時計を試す（ただし注意点あり）

数秒の `waitForTimeout` を含むテストは、まず `仮想時計を有効にする(page)` ＋ `仮想時計を進める(page, { ミリ秒 })` で置き換えられないか検討する。仮想時計（Playwright の `page.clock`）は `Date.now` / `setInterval` / `setTimeout` / `requestAnimationFrame` を仮想時間で動作させる。

ただし以下のケースで **仮想時計が機能しないことがある**:

- `setInterval` が画面状態をガードしている（例: `if (screen !== "standby") return;`）かつ、別の処理（別処理 → 提案画面遷移など）と画面状態が衝突する
- `await externalDetector.detect(input)` のようなネイティブ・ブラウザ実装が絡む待機

実際に通知ローテーション（3 秒間隔の setInterval、screen=standby ガード付き）を仮想時計化しようとして、別処理が頻繁に screen=processing へ遷移するため通知 setInterval が毎回スキップされる、という現象に当たった。**仮想時計を入れて挙動が変わったら、リアル時間（`実時間で待機する`）にフォールバックする** こと。

## 困ったときの参照先

- 仕様書として書けているかの生きた例: `e2e/specs/注文作成フロー.spec.ts`
- 実行コマンド・ディレクトリ構造の詳細: `e2e/README.md`
- Playwright 設定（webServer, globalSetup の挙動など）: `playwright.config.ts`, `e2e/global-setup.ts`

## 新しいシナリオを追加する手順

1. **登場人物の用意**: `e2e/data/` に必要なユーザー・テナントが揃っているか確認
2. **ヘルパーの用意**: `e2e/steps/` に必要なヘルパーが揃っているか確認。なければ既存ヘルパーの粒度に合わせて追加（日本語名で意図を表現）
3. **ファイル作成**: `e2e/specs/<シナリオ名>.spec.ts` を新規作成
4. **ログインユーザー指定**: ファイル先頭で `test.use(ログインユーザーを指定する(対象ユーザー))`
5. **外側 describe を書く**: 機能・領域名で `test.describe('<機能名>', () => { ... })`
6. **内側 describe を書く**: GIVEN を名詞句で表現する `test.describe('<環境 + 人物状態を記述する句>', () => { ... })`
   - 例: `store-001の管理画面で30分前に注文作成注文しているtaroが`
   - テスト内容によらず、すべてのテストはこの内側 describe の中に置く（前提なしの場合は `未登録のタブレットで` のような最小状態でも describe を作る）
7. **beforeEach に GIVEN を集める**: 内側 describe の `test.beforeEach(async ({ page }) => { ... })` に
   - 管理画面のセットアップ（`指定したテナントの管理画面をセットアップする`）— セットアップが主題なら例外的に test 本体に置く
   - 注文履歴のクリア（独立性のため必須）
   - 必要なら `指定したユーザーに過去の注文作成注文を仕込む` で人物状態を仕込む
8. **test 本体は WHEN/THEN だけ**: `test('<動詞句で結果まで言い切る>', ...)` の中身は、登場人物の行動と画面の検証のみ。`page.*` や直接の DB 操作・セットアップは出てこない
9. **テスト名チェック**: `<機能名> > <GIVEN 句> > <WHEN/THEN 句>` を連結して読み、自然な日本語の仕様文になることを確認
10. **実行**: `npm test` で全件 green を確認
11. **独立性チェック**: `npx playwright test e2e/specs/<書いたテスト>.spec.ts` 単独で通る + 別テストの後に走らせても通る、を確認
