# 本番自走セッション分析 — 認証基盤の作り替え（2 リポジトリ横断・約 4,000 行・20 コミット）

sandbox 検証ではなく、既に走った本番の実運用セッションを「あるべき振る舞い」に照らして事後分析した記録。harness-verification 本文「本番の実運用セッションを見る場合」に沿い、理想を slice-tdd 本文から立てて transcript に照合した。

## 条件（再現レシピ）

- sandbox: なし（本番実運用セッションの事後分析）
- 対象セッションの素性:
  - main session モデル: claude-sonnet-5
  - 全経過 約 7.2 時間 / うち自走 約 4.8 時間 / 人間待ち 約 2.4 時間（人間の介入は生涯 5 回：初回依頼・段取り承認・「終わらせて」の一言・完了前の AskUserQuestion 回答・最後の質問）
  - 内容: 既存認証基盤を複数テナント対応へ作り替え。API（Go・層構造）と FE（TypeScript）の 2 リポジトリ + E2E の 3 箇所横断
  - commit prefix: refactor 1 / feat 大半 / docs・test 少数（＝純リファクタでなく「作り替え + 新規外側の追加」の混在）
  - 実装は 27 タスクに分解 → 20 コミット、途中 auto-compaction が 1 回（コンテキスト最大 約 93 万トークン → 圧縮）
- 分析手段（transcript 解析）:
  - session jsonl を python で解析。`tool_use` と対応する `tool_result` のタイムスタンプ差で各ツールの所要を測り、カテゴリ分け（レビュー待ち / テスト実行 / build / DB / git / 調査 / 残差＝実装・思考）
  - commit 実行時刻を境界にサイクルを区切り、サイクル単位で時間内訳を出す。180 秒超のツール外アイドルを人間待ちとして各サイクルから除外
  - subagent の `<session>/subagents/agent-*.jsonl` から code-reviewer / e2e-reviewer の全報告本文を抽出し、severity と指摘の型で分類
- 落とし穴（実際に踏んだ／記録として残す）:
  - **AskUserQuestion の回答フォーマットは 2 種類**。`"...have been answered"` と `"The user answered: ..."` の両方があり、片方だけを検出条件にすると人間の介入を取りこぼす（push 承認を「無断 push」と誤判定した）。**tool_result の user 側介入は両フォーマットで拾う**
  - **権限プロンプト（push 等の許可）は transcript に残らない**。記録に出ないものを「やらなかった」と断定しない（harness-verification 手順4 の「出力に出ないことは起きなかったと断定しない」の実例）
  - **日付をまたいだ commit 混入**。commit 境界でサイクルを切るとき、対象日の終端より後の commit を拾うと巨大な区間が出る。終端で切る

## 結果

### 1. 時間内訳（自走 約 287 分）

| カテゴリ | 割合 | 備考 |
|---|---|---|
| main の思考・生成 | 約 49% | 約 4,000 行の設計・記述の実体 |
| subagent 待ち | 約 41% | code-reviewer 19 回で大半。うち subagent が実際にコマンドを叩いていたのは一部で、残りは読解・判断 |
| コマンド実行（テスト/build/DB/git/調査） | 約 10% | **テスト実行そのものは全体の約 6.5%（約 19 分）だけ** |
| コンパクション処理 | 約 1% | 1 回・約 3.6 分 |

要点: 時間を作るのは「実行」でなく「読み書きと判断」。テストを速くしても縮まない。単一で最大は code-reviewer の待ち（自走の約 31%）だが、これはオーバーヘッドでなく作業そのもの（後述）。

**フェーズ別の配分（作る→確かめるの移り）**:

| フェーズ | 実装/思考 | レビュ待ち | テスト |
|---|---|---|---|
| BE 実装 | 55% | 30% | 4% |
| FE 実装 | 53% | 33% | 8% |
| E2E + 仕上げ | 53% | 25% | 14% |

実装/思考とレビュ待ちの比率はフェーズをまたいでほぼ一定（リズムが崩れていない）。テスト比率が後半で上がるのは検証テールの正常形（空転ではない）。

### 2. サイクル構造（入れ子・outside-in）

理想（slice-tdd 本文）: 大サイクル（ストーリーを最外 E2E で駆動）の中に層ごとの小サイクルが入れ子。各サイクルは red→green→レビュー→リファクタ→commit の 5 点セット。向きは外→内（契約先行）。

観測:
- **5 点セット: PASS（模範的）**。全 commit が直前に code-reviewer を通過（レビュー飛ばし 0）。リファクタ後にテスト再実行して green 維持を確認する痕跡あり。リファクタ不要時は commit message に理由を残す。red は「正しい理由で失敗」を各ユニットで都度確認（import エラー等を red に数えない）。push 前に done-verifier、E2E 後に e2e-reviewer も規律どおり
- **入れ子: 半分**。1 サイクル内の多層 red-green ミニサイクル（domain→usecase→endpoint の各層で test-first）は全サイクルで踏めており「大サイクル 1 本に潰す」崩れは無い。だが**ストーリー全体を包む最外 E2E 大サイクルが存在しない**。真の最外 browser E2E は全実装完了後に後付けされた
- **outside-in の向き: FAIL（最大の逸脱）**。宣言レベルで「（内側の）データ→ロジック→入口の順に実装します」「内部層完成、次に top-level」と、slice-tdd が名指しするボトムアップに該当。新規 API・新規画面という「新しい外側が生まれたサイクル」でも内側から積んだ。外側テストを先に書けたのは 1 サイクルのみ（＝性質でなく、向きを問う発火がその瞬間に落ちた）

**作り替え（リアーキ）性を差し引いた判定**: 既存内部構造の作り替えサイクル（外側契約が既存で E2E が守る）は最内から入っても違反として弱く、差し引ける。差し引けないのは新規 API・新規画面（新しい外側が生まれた）のサイクル。したがって outside-in 破れは「スライス全体」でなく「新規外側サイクル」に限局。なお slice-tdd 本文は新機能ストーリー前提で、作り替え/リアーキ時の適用が薄い（＝ここを skill に言語化する余地はあるが、優先度は高くないと依頼者判断）。

### 3. レビュー指摘の性質（20 レビュー・66 件）

- **繰り返しは 4 型・約 14 件のみ**（無関係スナップショット混入 ×3、pure function 未抽出でテスト欠落〔FE〕×4-5、逐語重複の集約時期判断 ×3、コメント追従漏れ ×3）。残り約 50 件はそのサイクル固有の一点物。「繰り返しっぽい」印象は量（20 回 × 平均 3.3 件）と軽さ（Suggestion が過半）が作る
- **skill カバー可能性**:
  - (A) 規約に書けば防げる: スナップショット混入（commit 前の diff 点検・対象明示 add）、**FE の pure function 抽出 + 直接テスト**（＝この対象プロジェクトには FE のコード規約 skill が存在しない穴。後半で減らない指摘はこの穴に落ちる）
  - (B) レビューでしか捕まらない〔＝価値の核〕: 設計書の Done 照合、front+back 横断の race、情報漏洩の文字列一致、dead code の到達不能証明。**Critical/Security の実体はすべてここ**。一般ルール化不能
  - (C) 既に規約にあるのに守られなかった: 否定パスの DB 不変裏取り（プロジェクト規約に明文）、生プリミティブの値オブジェクト化（ドメイン規約の赤信号）、暫定値の本番残存（slice-tdd の既存規律）。＝「書けば守られる」が成り立たない実証データ。規約追加でなく発火・遵守の側の課題
- **結論**: レビューは無駄でも過剰でもない。空振りは 19 回中 1 回。(A)/(C) を規律装置に最小注入して機械が得意な捕捉を機械に任せ、reviewer は (B) のペアプロ相手として残す、という xp-harness の切り分けがそのまま裏取りされた

### 4. 派生論点 — 並列化（AgentTeam）の芽（保留、次の検討材料）

「小タスクに分けたのだから独立分は並列で回せばコンテキストも減る」という問いが出た。今回のデータで見えた両面:

- 慎重材料: 今回の 27 タスクは 1 ストーリーを層で刻んだ**相互依存**の実装単位（後タスクが前タスクの型・契約に依存）。並列に切ると各コンテキストが契約を推測で埋め、統合時に不整合が噴出しうる（＝ボトムアップのアンチパターンが警告するもの）。並列化の正しい単位はタスクでなく**独立ストーリー**（INVEST の I）
- 慎重材料: 今回の最重要指摘（横断 race・重複の 3 サイクル追跡）は**全体を 1 コンテキストで見たから**出た。コンテキストを絞る並列化と横断的な捕捉は正面からトレードオフ
- 賛成材料: コンテキスト圧の痛みが数字で顕在化（最大 約 93 万トークン → 1 回 compaction）。philosophy の 4 象限で AgentTeam は象限4（痛み未明確）だが、この痛みは象限を動かす材料。ただし lead 固定の配布は「中央集権は最上位性能を超えない」とズレるため、入れるならペアプロ的に弱める形

判定: 「独立ストーリー群の並列」は追う価値、「1 ストーリーの層タスクの並列」は今回データ上むしろ危険。追うなら philosophy を通した正式検討に載せる。

## この検証自体の手法メモ（次回のため）

- 本番セッションは「理想を先に立てる」が最重要。今回は slice-tdd 本文から outside-in / 入れ子 / 5 点セットの 3 点を lens として確定してから照合した
- 時間・介入の復元は tool_use↔tool_result のペアと commit 境界で機械的に出せる。ただし上の「落とし穴」3 点（AskUserQuestion 2 フォーマット・権限プロンプト不記録・日付またぎ commit）を必ず踏まえる
- 指摘の分類・サイクル復元は独立した 2 分析として並行実行できる（今回 subagent 2 本に分けた）
