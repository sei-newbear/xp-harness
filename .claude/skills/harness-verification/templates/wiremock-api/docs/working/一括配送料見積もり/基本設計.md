# 一括配送料見積もり — 基本設計

> このドキュメントは進行中の設計メモ。実装後もそのまま残す（振り返り・PR レビューで参照される）。
> 削除のタイミング・要否は依頼者または外側の仕組みで別途判断する（あなたは削除しない）。

## 1. アーキテクチャ

既存の Hono API (`api/`) に、新エンドポイント `POST /shipping/quote/bulk` を追加する。既存の単一宛先見積もり (`POST /shipping/quote`) と同じ外部配送料 API クライアント (`api/src/shipping-client.ts`) を、宛先ごとに呼び出して結果を集約する。

外部配送料 API の仕様（リクエスト / レスポンス）は `api/docs/external-shipping-api.md` を参照。

```
client ──POST /shipping/quote/bulk──> Hono API
                                        └─ shipments[] の各要素について
                                             └─ shipping-client.quoteShipping() ─> 外部配送料API
```

## 2. データモデル

永続化なし。

- リクエスト: `{ "shipments": [ { origin, destination, weight, dimensions, service, insurance }, ... ] }`
- レスポンス: `{ "quotes": [ { destination, fee, currency }, ... ] }`

各 shipment の項目は単一宛先見積もりと同じ。

## 3. シーケンス

```
POST /shipping/quote/bulk { shipments: [s1, s2, ...] }
  for each s in shipments:
     quoteShipping(s)  ──>  外部API POST /v1/quote
                        <──  { fee, currency }
  集約: quotes = [{ destination, fee, currency }, ...]
  200 { quotes }
```

## 4. 論理設計

- 責務分離: ルーティングは `api/src/app.ts`、外部 API 呼び出しは `api/src/shipping-client.ts`。既存の単一宛先版の構造を踏襲する
- バリデーション: `shipments` が空配列・未指定ならエラー（最低 1 件必要）
- エラーケース: いずれかの宛先で外部 API がエラーを返したらリクエスト全体を 502 で返す

## 5. 既存資産への影響

- 既存の `POST /shipping/quote` と `shipping-client.ts` を再利用する（単一宛先版の延長）
- E2E は既存の API E2E (`api/test/`) の流儀に合わせて、bulk 用のシナリオを追加する
- migration 影響: なし / permission 影響: なし

## 6. 完了の定義（実装フェーズが満たすべきもの）

- [ ] 複数宛先を渡すと、各宛先の配送料が配列で返る（E2E で確認）
- [ ] 各配送料が外部配送料 API 由来の値である
- [ ] 宛先が空のリクエストはエラーになる
- [ ] 既存の単一宛先見積もりが引き続き動作する
