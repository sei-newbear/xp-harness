# 最終形 (= 理想像、北極星) の議論経緯

SKILL.md の「最終形 (= 理想像、北極星)」section がどういう議論から生まれたかの記録。section 本文は結論の凝縮なので、根拠・反例・退けた案・マトリックスの象限別の既定値はこちらに残す (2026-07 の依頼者と main session の対話で確定)。

## 背景 — なぜこの整理を始めたか

きっかけは 2 つ:

1. 依頼者が「タスクの難易度・影響度・規模に応じて、harness がどう立ち向かうか (対話の深さ・分割・任せ方) をマトリックスで整理したい」と持ちかけた
2. その背景に、人間の介在ポイントを減らして自動で改善しながら回し続ける開発の潮流 (いわゆるループエンジニアリング) への関心があった。依頼者はそれを「XP のチームのような振る舞い」と見ていて、xp-harness をそこに近づけたい。具体像は「ストーリーを出すところは一緒にやるが、ストーリーが出たら自律的にやる」

なお本文書では、チームにおける利用者の基本の役割を**プロダクトオーナー**と書く (利用者は選択でエンジニア / ドメインエキスパートとしてモブにも入る — 役割は固定でない)。

## 難易度と影響度は別レバー (束ねない)

当初案は「縦軸 = 難易度または影響度、横軸 = 規模」だったが、難易度と影響度は harness の別の装置に対応するので束ねない、で合意した:

- **難易度**が高い → 認識ズレ・手戻りのリスクが高い → 効くのは**事前の対話の深さ** (要件定義・基本設計を対話的にやる、調査スパイクを挟む)
- **影響度**が大きい → 間違えたときの被害が大きい → 効くのは**事後の検証・統合の厳しさ** (reviewer subagent、done-verifier、直接統合でなく PR を通す)

束ねると同じ処方を出してしまう反例ペア:

- 認証設定の 1 行変更 = 難易度低・影響度大 → 対話的な設計は不要だが、出口の門 (レビュー・検証) は厚くすべき
- 隔離されたモジュール内の複雑なアルゴリズム = 難易度高・影響度小 → 対話や調査は要るが、統合は軽くてよい

先行事例: 改修者向け CLAUDE.md の「利用者影響がある変更は PR を通す、ない変更は main へ直接統合」は、影響度で統合の厳しさだけを切り替えるルール (対話の深さは変えていない)。影響度はすでに独立レバーとして運用されていた。

## マトリックス (難易度 × 規模) — 関与の既定値

- 縦軸 = **難易度**: ループの前提 (認識と合否基準) を作るのに対話・調査がどれだけ要るか。「難しい」とは合否基準をまだ書けない状態のこと
- 横軸 = **規模**: ループをいくつに割るか。ストーリーを小さくするのは、1 周の射程を人間の介在なしで走り切れる長さに縮める行為

| | 規模小 | 規模大 |
|---|---|---|
| **難易度高** | 要件定義→基本設計を対話。未知が残るなら調査スパイクで既知に変える。分割は規模のレバーなので難易度には効かない (小さくて難しいタスクの難しさは、分割で消えない不確実性であることが多い) | 複数ストーリーを縛る決定 (アーキ・データモデル・契約) は最初に対話で固め、1 ストーリーに閉じる設計は各ループに委ねる。別セッションに渡すなら失敗点は引き継ぎの自己完結性に移る |
| **難易度低** | 完全に任せる (define-requirements の発火除外リスト = 再現条件が明確なバグ修正・定型作業、の一般化) | 要件定義とストーリー分割まで対話し、個別ストーリーの設計は任せる。規模が大きいときの真のリスクは横断的な一貫性で、それは分割の対話そのものがマクロ設計を兼ねることでカバーされる |

**これは既定値であって規則ではない。** 関与の深さは境界で利用者が選ぶ — 難しくても「今回は任せる」も、易しくても「ドメインが絡むから一緒にやる」も選べる。harness の役割は、この既定値を材料に選択肢を見せること。

## ループが成立する条件 — なぜ XP なのか

自動で改善しながら回る仕組みには「小さい単位」と「機械で判定できる合否基準」が要る。合否基準のないループは、それっぽい成果物を量産するだけで収束しない。

XP のプラクティスは元々、人間のチームがフィードバックループを速く回すための工学 — 小さいストーリーは 1 周の射程を縮めるため、受け入れテストと観測可能な Done は周回ごとの合否判定のため。この本質は**回す主体が人間でもエージェントでも変わらない**。xp-harness の既存部品 (観測可能な Done、story-slicing、done-verifier、引き継ぎの自己完結性検証) がそのままループの部品になるのは偶然ではなく、本質 (XP) に向けて作ってきた必然の帰結。ループエンジニアリングが XP に似て見えるのは、あちらが独立に同じ本質へ収束したから。

## 不変条件の変遷 — プロダクトオーナーは何を握るか

議論の中で 2 回言い直された:

1. 最初の案: 「すべてのストーリーの入口 (Why) と出口 (受け入れ) にプロダクトオーナーが居る」を不変条件とする
2. 修正: それだとチームが自分で仕事を発見して動く余地が消える。実際の XP チームも、顧客がすべての作業単位を発行するわけではない (バグ報告を拾う、壊れた窓を直す、小さな改善を自分で始める)。顧客が握るのは方向性と優先順位であって、作業の発生源の独占ではない
3. 確定: **「プロダクトオーナーが握るのは方向性と、大きい・後戻りしにくい決定。全作業がプロダクトオーナーを経由することは求めない」**

これが放任と区別される成立条件は 2 つ:

- **方向性が成果物に宿ること**: 意図が会話の中にしか無いと、チームは何を基準に仕事を選んでいいか分からず、自律が当て推量になる。かんばん (優先順位つきのバックログ) がこの器
- **チームの動きが観測可能なこと**: 自律で動いた結果をプロダクトオーナーが受け入れ・ふりかえりで観測でき、方向を直せる。「握る」= 全部に事前承認を出すことではなく、方向を示し・結果を見て・直せる状態を保つこと

チームが自分で仕事を発見するとき、マトリックスは「そのまま進めてよいか、プロダクトオーナーに上げるべきか」の判定にも同じ形で効く (規模が大きい・方向性に触れるなら上げる、小さければ完結してよい。影響度が大きければ出口の門を厚く)。

## 退けた構図 — 「介在を境界に固定する」

議論の途中で「人間はストーリーの境界に居て、ループの内側は無人で回る」という構図 (介在の境界固定) を立てたが、退けた。人間を境界に**固定する**のはチームではなく発注者と請負の構図で、XP のチームはメンバーの持ち場を固定しない — プロダクトオーナーがドメインエキスパート / エンジニアとしてモブに入る余地を残す。

「介在を減らす」こと自体は目的ではない。**介在の深さを境界で選べること**が目的で、選んだ結果として内側が無人のストーリー (= ループで回る) も、モブで回るストーリーも、同じチームの二つの回り方として存在する。「フル自走 (要件も受け入れも人間なしで回す) には倒さない」(判断軸 5) は現時点の運用でも変わらない — 顧客の意図を運ぶ装置のないループは収束しないから。ただし不変なのは「意図を運ぶ装置が要る」ことであって、「装置が人間である」ことではない (次の「理想像自体の暫定性」を参照)。

## 理想像自体の暫定性

理想像の確定時に依頼者が明示した前提: **この理想像は変わりうるもの**として扱う。固定なのは上位の目的で、それは xp-harness という名前が背負っている **XP の目的そのもの — 変化を抱擁すること** (変更のコストを低く保ち続けることで、学びと市場の変化をソフトウェアに反映し続け、ビジネス価値を出し続けられる状態を保つ。内部品質はその前提条件)。依頼者の言い換えでは「品質が高く、変更に強く、ビジネスを成長させられる開発」。理想像はこの目的に対する「今見えている範囲での最善」に過ぎない。人間がプロダクトオーナーを担うのもこの判断の一部で、将来エージェントがプロダクトオーナー役を担い、人間の介在ポイントが一切無くなる可能性も排除しない。

つまり階層は「目的 (固定) > 理想像 (暫定の指針) > 規律・装置 (手段)」。これは XP 自身の「価値 > 原則 > プラクティス」の三層と同型で、プラクティスを目的に照らして入れ替えてよいのと同じことを、理想像のレイヤーでも行う。理想像を根拠に判断するとき、理想像そのものが目的に照らして更新されうることを忘れない。
