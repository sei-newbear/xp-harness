# 引き継ぎ可能な自己完結ドキュメントの skill と subagent

## 完了 (2026-07-07)

実装・マージ・リリース済み。`handoff-docs` skill と `handoff-verifier` subagent を新設し、define-requirements / basic-design に自己完結性の集約と引き継ぎ点検動線を追加。PR #15 マージ、**v0.12.0** でリリース。sandbox 実走で全動線を検証 (handoff-docs 発火 → 自己完結ドキュメント → handoff-verifier 文脈ゼロ点検 → 引き継ぎの一言、会話由来の却下案が実ファイルに残ることも確認)。責務境界は philosophy の 4 象限を通し「XP の進め方の規律 (引き継ぎ可能性の保証) 」として xp-harness に取り込む判断で確定。

## 背景 / Why

エージェントが「別セッションへ引き継ぐ前提の成果物」(要件定義・基本設計・引き継ぎ書・PRD・README 等) を作ると、書いた本人の主観で「完成・自己完結」と宣言してしまいがち。書き手自身の会話の文脈が "のり" になって、ドキュメントに書かれていない前提を脳内で補完するため、本人には完成して見える。だから書き手も、文脈を持ったままの自己点検も、穴を見つけられない。

想定読者は人間ではなくエージェント。人が読む前提を外すと「簡潔さより文脈の完全性、長くてよい」に前提が変わる。人間向けの読みやすさを意識して情報を削ると、別セッションで前提・判断の経緯が欠落し、設計意図とズレた実装に流れる。

これには **書き手側** (自己完結に書かせる) と **点検側** (文脈ゼロで引き継ぎ可能性を検証する) の両面がある。片方だけでは価値が立たない (書かせても点検しないと穴に気づけない / 点検だけでは書き手を導けない)。両者は対で、ユーザーストーリーとして分割できない。この 2 つを 1 案件として扱う (元は tier1 の 2 カードだった: 「成果物の引き継ぎ可能性を保証する」= 点検側、「設計ドキュメントをエージェント向けに自己完結化」= 書き手側。統合済み)。

## 効いたやり方 (実セッションで実証)

「文脈ゼロ」のサブエージェントを立て、成果物だけ (会話履歴なし) を渡して次をやらせる:

1. 成果物だけを頼りに下流タスクに着手できるか
2. 矛盾・抜け・曖昧点はどこか
3. 「文脈ゼロの別セッションが、設計確認で止まらず進められるか」の Yes/No

そして直して再実行、Yes になるまで回す。

実セッションでは、これが「完了」として出てしまうはずだった具体的な穴を捕まえた:

- セクション間でフィールドの置き場が食い違っていた
- 必須の列挙集合が書かれていなかった
- 仕様が上書き層に散らばっていて再構成に 4 手順の脳内パッチが要った

1 回目 = No、修正後 = Yes。肝は「点検者が会話の文脈をゼロにする (成果物だけ渡す)」こと。実際の引き継ぎ条件を再現できるのはそれだけだから。

これは「done-verifier (コードが Done を達成したか確認する)」ではない。文脈ゼロの読み手にとっての「ドキュメントの自己完結性」= 引き継ぎ可能性を点検するもの。

## 決まっている方向性 (今日の到達点)

要件定義はまだ。今日固まったのは方向性のみ:

- 成果物は **新規の横断 skill 1 つ + 新規 subagent 1 つ** のセット。
- **skill の守備範囲**: 要件定義.md / 基本設計.md に閉じない。「別セッションのエージェント (や人) が読んで進める前提の成果物」全般 — 引き継ぎ書・PRD・README なども含む。define-requirements / basic-design 本文の改修に閉じる話ではない (ドキュメント全般が対象なので独立した横断 skill にする)。
- skill が担うのは **(a) 書き手側の規律** (別セッションのエージェントが読んで自走できる自己完結ドキュメントの書き方) と **(b) 点検 subagent を呼ぶ動線** (いつ呼ぶか / Yes になるまでどう回すか) の両方を束ねる。
- **subagent** は文脈ゼロの点検側。既存 reviewer 群 (done-verifier / pre-implementation-reviewer 等) とは責務が別で、「引き継ぎ可能性 = ドキュメントの自己完結性」に限定する。
- 想定読者はエージェント → 長さより文脈の完全性を優先 (Why / 決定の根拠 / 前提 / 制約 / スコープ境界 / 却下案とその理由 等を漏れなく)。

## 再開時の起点

- 要件定義フェーズから入る (define-requirements)。Why は上記でほぼ固まっているので、Done / スコープ / skill と subagent の責務境界を詰める。
- philosophy の 4 象限 + 責務境界を通して、取り込むか / 上流フィードバックに留めるかを判断する。
- 取り込む場合の設計論点:
  - 文脈ゼロ subagent をどう実現するか (会話履歴を渡さず成果物だけ渡す呼び出し方)。
  - 「Yes になるまで回す」ループを誰が回すか (main session / subagent 自身)。
  - 既存 reviewer subagent 群との責務切り分け (= done-verifier とは別物、引き継ぎ可能性に限定)。
  - skill が (a) 書き手規律 と (b) 点検動線 を 1 つに束ねる構造で SRP を破らないか。
- テストする段 (skill / subagent ができた後) で、この案件自体をドッグフーディングで引き継ぐ引き継ぎ書を作る (= 本 skill / subagent の実地検証を兼ねる)。skill も subagent も無い今の段階で引き継ぎ書を先に作るのは順序が逆なのでやらない。
- 改修は skill-design-style の改修フローに従い、skill-reviewer に通す。
