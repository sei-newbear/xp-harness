# slice-tdd: バックエンド入口の受け渡し契約を先に固定する

## 背景 / Why

v0.11.0 の書き直しで、冷えた別セッションがバックエンドを契約先行で設計するようになった（usecase のテストを先に、依存性の逆転で port を先に定義）。ただし **route↔usecase の受け渡し契約**（リクエストがどう入って usecase にどう渡るか）は E2E 頼りで、route コードは最後に書かれた。理想は、この入口の受け渡し契約も先に固定してから内側に進むこと（依頼者の指摘）。失敗ではなく refinement。

## 状況

大枠（バックエンド内部の契約先行）は解消済み。これは「もう一段外側＝入口の受け渡し契約も先に」の詰め。route を E2E で担保する戦略（層のテスト戦略は project 依存）とも両立させる必要がある。

## 再開時の起点

- slice-tdd の outside-in 節で「入口（呼び出し側から usecase への受け渡し）の契約も先に決める」を、**層名を焼かず役割ベース**で表現できるか検討（route/REST 等の具体名はルールに入れない、例のみ）。
- 過剰注入にならないよう最小で。効いたかは compose sandbox の冷えた再検証で事後評価。
