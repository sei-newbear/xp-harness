# ふりかえり skill 発火検知の既存 skill フック

## 背景 / Why

ふりかえり skill の発火検知は「main session が context から『セッション内のすべてのストーリーが完了した』と判断する」段階的観測モデルを採用し、既存 skill にはフックを入れていない。ただし実運用で文脈判断が安定せず「拾い損ね / 早すぎる発火」が発生する場合、既存 skill にフックを入れる選択肢が残る (機能拡張 / 責務境界)。

## 状況

ふりかえり skill の basic-design で、ふりかえり skill の発火検知は **「main session が context (会話履歴 / docs/working/<title>/要件定義.md 等) から『セッション内のすべてのストーリーが完了した』と判断する」段階的観測モデル** を採用した。既存 skill (`define-requirements` / `slice-tdd` / 他) には今回フックを入れない (= 要件のスコープ外 #6 と整合)。

ただし、実運用で main session の文脈判断が安定せず「拾い損ね / 早すぎる発火」が発生する場合、既存 skill にフックを入れる選択肢が残る。本 TODO はその参照点。

## 再開時の起点

2. 不十分と判明したら、フック方式を検討:
   - `define-requirements` でストーリー一覧を明示的な data として書き出す

   - その他観測手段
3. 既存 skill の責務範囲を超えないか確認 (= フックが skill の責務を膨らませないか)
