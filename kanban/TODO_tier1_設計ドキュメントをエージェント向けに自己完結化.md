# 設計ドキュメントをエージェント向けに自己完結化

## 背景 / Why
要件定義.md / 基本設計.md (define-requirements / basic-design の成果物) は、別セッションのコーディングエージェント (slice-tdd) が読んで実装を進める前提。読み手は人間ではなくエージェントなので、簡潔さより「コンテキストが漏れなく self-contained であること」を優先したい。人間向けの読みやすさを意識して情報を削ると、別セッションで前提・判断の経緯が欠落し、設計意図とズレた実装に流れる。

要点: 人間が読まない前提なので長くてよい。むしろ別セッションでそのまま動くだけのコンテキスト (Why / 決定の根拠 / 前提 / 制約 / スコープ境界) を漏れなく書き起こす方向に倒す。

## 状況
- 現状の define-requirements / basic-design 本文が、成果物の「想定読者 = 別セッションのコーディングエージェント」「長さより文脈の完全性を優先」を明示しているか未確認。
- 人間向けの簡潔さに最適化している記述があれば、エージェント向けの自己完結性とトレードオフになっている可能性。

## 再開時の起点
- まず `.claude/skills/define-requirements/SKILL.md` と `.claude/skills/basic-design/SKILL.md` 本文の成果物書式に関する記述を読み、想定読者と長さ方針がどう書かれているか確認する。
- 「別セッションのエージェントが読んで自走できる」ために漏らしてはいけないコンテキスト項目 (Why / 決定根拠 / 前提 / 制約 / スコープ境界 / 却下案とその理由 等) を洗い出し、成果物テンプレに反映するか検討する。
- philosophy の「対話駆動で固めた前提を持って自走する」と整合させる (= 対話で固めた前提を成果物に漏れなく落とすのが、別セッション自走の前提)。
- 改修は skill-design-style の改修フローに従い、skill-reviewer に通す。
