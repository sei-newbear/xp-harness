# 軽量依頼で要件固めをスキップ確認する

## 背景 / Why
軽量なタスクや、すでに内容が固まっている依頼に対しても `define-requirements` が発火し、Why / Done / スコープを問われて「さくっと」進まない。依頼者の体感として、軽い依頼には重い。

## 状況
- `define-requirements` の発火対象は広い（自然な口調の依頼・md まとめ・項目追加削除など幅広く発火）。これは「依頼者の解決策が真の課題に合っているか検証する価値がある」ため意図的に広く取っている。
- ただし軽量・確定済みの依頼まで一律にフルの要件ヒアリングが走ると、価値に対して対話コストが重い。
- 一方で「軽そうだから」とエージェントが独断で要件定義を skip するのは危険（簡単そうで難しいことがある）。skip 判断を独断にしない安全弁が要る。

## 論点 / 落としどころ候補
- 軽量と判断したら、要件定義をフルで回すのではなく「要件を固めなくていいか」を依頼者に**ヒアリングして確認**を取る（勝手に skip しない・勝手にフル発火もしない）。
- 「軽量」の判定を感覚語でなく観測可能な問いで書けるか（例: 期待動作が既に明確 / 影響範囲が閉じている 等）。既存の「発火しない 3 条件」との境界も整理する。
- 実装先の候補: `define-requirements` skill の description（発火条件）か本文（発火後に軽量と気付いたら確認に倒す振る舞い）か。philosophy の「対話と自走の境界」との整合も見る。

## 再開時の起点
`define-requirements` の発火条件（description）と「発火しない 3 条件」を読み直し、軽量依頼で確認に倒す動線をどこに置くか（description か本文か）を propose-options で複数案化するところから。
