# git-workflow: 自走でセッション開始時の git fetch が飛ぶ

## 背景 / Why

ある実プロジェクトの自走実装を検証した際、git-workflow の「セッション開始時に `git fetch origin` を必ず最初に叩く」が効かず、stale なローカル状態のまま作業を始めた。結果、origin が先に進んでいたことに気づかず、追加するファイル（DB migration など連番で衝突しうるもの）が既存と番号衝突。依頼者の指摘で初めて発覚した。自走で同期確認の規律が落ち、しかも harness 側でなく依頼者が捕捉した点が問題。

## 状況

- セッション開始時に branch / status は確認したが、`git fetch` は実行していない（初 fetch は依頼者の指摘後）。
- git-workflow 本文は「fetch しないと古い情報のまま up to date を返す」「fetch を必ず最初に」と明記しているのに、自走で飛んだ。
- 結果、最新化前の作業で連番ファイルが衝突し、振り直しの手戻り。

## 再開時の起点

- git-workflow のセッション開始同期（fetch を最初に）が、自走でも確実に効く形になっているか確認。「最新化してから着手」を自走が飛ばさせない手当てを検討。
- 規律の文言だけで効いていないなら、なぜ自走で落ちるか（観測可能な前提確認に落とせるか）を見る。
