# 成果物の引き継ぎ可能性を保証する

## 背景 / Why
エージェントが「別セッションへ引き継ぐ前提の成果物」(設計書・仕様・要件・PRD・README 等) を作ると、書いた本人の主観で「完成・自己完結」と宣言してしまいがち。書き手自身の会話の文脈が "のり" になって、ドキュメントに書かれていない前提を脳内で補完するため、本人には完成して見える。だから書き手も、文脈を持ったままの自己点検も、穴を見つけられない。

これは「done-verifier (コードが Done を達成したか確認する)」ではない。文脈ゼロの読み手にとっての「ドキュメントの自己完結性」= 引き継ぎ可能性を点検するもの。

## 効いたやり方 (実セッションで実証)
「文脈ゼロ」のサブエージェントを立て、成果物だけ (会話履歴なし) を渡して次をやらせる:
1. 成果物だけを頼りに下流タスクに着手できるか
2. 矛盾・抜け・曖昧点はどこか
3. 「文脈ゼロの別セッションが、設計確認で止まらず進められるか」の Yes/No

そして直して再実行、Yes になるまで回す。

実セッションでは、これが「完了」として出てしまうはずだった具体的な穴を捕まえた:
- セクション間でフィールドの置き場が食い違っていた
- 必須の列挙集合が書かれていなかった
- 仕様が上書き層に散らばっていて再構成に 4 手順の脳内パッチが要った

1 回目 = No、修正後 = Yes。

## 提案
「引き継ぎ可能性 (handoff readiness)」を見る一級の harness パターン / サブエージェント。文脈ゼロで成果物だけを渡して点検し、Yes になるまで回す。エージェントが「別セッションで動く自己完結ドキュメント」と宣言する直前のガードとして、あるいは任意の点検として使える。

肝は「点検者が会話の文脈をゼロにする (成果物だけ渡す)」こと。実際の引き継ぎ条件を再現できるのはそれだけだから。

## 状況
- Claude Code (harness) 開発チームへのフィードバック / 機能アイデアとして起票。
- 関連: tier1 の「設計ドキュメントをエージェント向けに自己完結化」は **書き手側** (define-requirements / basic-design に自己完結な成果物を書かせる) の改善。本カードは **点検側** (文脈ゼロ subagent で引き継ぎ可能性を検証する) の機構で、対になる。両者をセットで設計すると噛み合う。
