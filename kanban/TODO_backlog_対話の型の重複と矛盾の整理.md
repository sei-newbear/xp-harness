# 「対話の型」の重複と矛盾の整理 (dialogue-principles と各フェーズ skill)

## 背景 / Why

`dialogue-principles` と `define-requirements` / `basic-design` に、対話の進め方の規律が二重・三重に書かれている。規律装置最小注入 (規律は最小限の文書で注入する、重複させない) に反する状態。

さらに悪いことに、重複しているだけでなく **既定値が逆向きに矛盾している箇所がある**:

- `define-requirements` / `basic-design` の「対話の型 (proactive な認識合わせ)」: 「質問はクローズな選択肢で出すのを基本に。依頼者の負担が小さい」
- `dialogue-principles` の「議論はチャット、クローズな選択は質問ツール」: 「議論・自由記述で考えを引き出したい場面はチャットで応答を待つ。クローズな選択肢に縛ると議論が浅くなる。**迷ったらチャット側に倒す**」

## 状況

これまでは要件対話の入口で dialogue-principles が発火していなかったため、矛盾が表に出ていなかった。2026-07-20 の description 純化で「対話モードに入る地点で dialogue-principles を呼ぶ」動線を足した結果 (`docs` 不要、経緯は `.claude/skills/harness-verification/references/2026-07-20_description純化.md`)、**要件定義・基本設計の対話中は両方が必ず context に同居する**ようになった。つまり重複が増えたのではなく、潜在していた矛盾が毎回発現する形に変わった。

矛盾した既定値を同時に読むと、main session がどちらに倒すかを毎回その場で組み立て直すことになる。

## 再開時の起点

1. 矛盾している 2 つの既定値のうち、どちらが xp-harness の意図かを決める (philosophy の「対話は共創を目指す」に照らすと、議論を選択肢に押し込まない dialogue-principles 側が筋に見えるが、要件定義の「1 問ずつ・依頼者の負担を小さく」も実運用で効いている。単純にどちらかを消す話ではない可能性がある)
2. 決めた既定値を single source (= `dialogue-principles`) に集約し、各フェーズ skill 側は「呼ぶ」だけを残す方向で整理する。ただしフェーズ固有の型 (要件定義のセクション単位の approval gate、基本設計のトピック単位の合意) は各 skill に残す必要があるので、何が「フェーズ固有」で何が「対話一般」かの切り分けが本題
3. 整理後は harness-verification で対話フェーズの振る舞いが劣化していないか確認する (段階的開示・1 問ずつが崩れないか)
