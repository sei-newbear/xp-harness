# 調査スパイク skill 新設

## 背景 / Why

要件定義・基本設計・実装にはフェーズ skill があるが、調査/スパイクという作業クラスには持ち家が無い。そのため main session は調査をストーリーの型に流し込み、「問いに答える」をゴールに走ってしまう。スパイクは成果物が情報そのもので、次工程の入力に引き継がれて初めて価値になる — Done は「問いに答えた」でなく「答えが次工程の入力として使える形で引き継がれている」まで含む必要がある。

AgentTeam 調査スパイク (2026-07-15〜16) で実際に起きた痛み: 完了の早すぎる宣言 2 回、調査記録が会話の時系列追記 (チャット報告の控え) になり用語割れが発生、引き継ぎ点検 (handoff-verifier) は依頼者に促されて実施。経緯は `docs/working/AgentTeam調査スパイク/ふりかえり.md` の深掘り 1。

## 状況

skill が持つべきものの候補 (要件定義フェーズで固める):

- 調査の開始時に固めること: 問い (何を潰すか)、出力の宛先 (誰の入力になるか)、Done (引き継ぎ可能な知見)
- 調査記録の書式 (時系列追記でなく、読み手起点の構成。用語の固定)
- 仕上げの手順: handoff-docs を呼ぶ (引き継ぎ文書として仕上げる)、handoff-verifier で文脈ゼロ点検
- スパイクは XP 由来の実践なので、配布側 skill (`.apm/skills/`) になりうる (責務境界は要件定義で判断)

**2 回目の顕在化 (2026-07-18, Codex対応セッション)**: apm 挙動の調査 (実質スパイク) で、報告が「実況 → 生出力 → 解釈」の地続きになり読みにくかった (`docs/working/Codex対応/ふりかえり.md` 深掘り 1)。AgentTeam 調査スパイクに続き、別々の調査で同じ痛みが 2 回。backlog → tier1 に昇格した根拠。

設計論点 A (報告書式): 「実況と報告を分ける／結論 → 詳細 → 証拠」という報告品質の規律は、現状 `harness-verification` skill 本文に閉じている (発火は「xp-harness の skill/subagent 検証」時のみ)。調査スパイク全般に効かせるには、この報告書式の規律を **`harness-verification` と新 skill でどう single-source にするか** を決める (規律装置最小注入・重複回避)。

設計論点 B (調査の網羅性 = 反証で開く): Codex対応セッションの深掘り 2 で、調査が「問いに答える」で閉じ、「判断の危険を潰す」まで開かない片肺が見えた (裏付けを一つ取ったら閉じ、壊す方の副作用・利用者の現実の初期状態を反証しにいかない。3 回同型を再演)。skill の核を **「問いに答える」でなく「判断の危険を潰す」** に据え、開始時に (1) どの判断のための調査か (2) その判断が壊れる道の列挙 (失敗モード・壊す方の副作用・利用者の現実の初期状態) (3) 各モードを反証しにいく → 全部潰れて完了、を要求する。`harness-verification` に既にある「反証可能性」の規律も、報告書式と同様に **新 skill と一箇所に集約** する (詳細は `docs/working/Codex対応/ふりかえり.md` 深掘り 2)。

棄却済みの代替案: handoff-docs の description の例示に「調査記録」を足す (例示リストの増殖は「例示に無いものは対象外」の読みを強化するため筋が悪い)。

## 移設メモ (main-instructions 再構成から, 2026-07-18)

`main.instructions.md` のゼロベース再構成 (`docs/working/main-instructions再構成/`) で、「新しいライブラリ・ツールは思想・ワークフローを調べてから使う」の規律が、常時 instruction には常時性が弱い (導入時のみ) と判定された。ただし「闇雲に使わせない」ために消したくないため、当面 instruction に B 芯として仮置きしてある。**この調査スパイク skill を新設する際に、この規律を取り込み、instruction 側の芯を移設 (または routing 化) する。**「ツール・ライブラリ導入時に、その思想・作法を調べてから使う (framework と戦わない)」は調査の一種で、この skill の責務に収まる見込み。
