# ふりかえり skill 強化: 人間側プロンプトの改善案も出す

## 背景 / Why

現状の retrospective skill は観測対象を **エージェントの働き方** (= skill / subagent / CLAUDE.md・ルールが適切に働いたか、および純粋な推論ミス・良い振る舞い) に置いている。一方、ペアプロでは依頼者側の伝え方 (= プロンプト) も改善余地を持つ — 曖昧な依頼・前提の欠落・誤誘導があるとエージェントが空回りする。

ふりかえりで「エージェント側の直し」だけでなく「**依頼者側のプロンプトの改善案**」も surface させ、次の依頼の質を上げたい、というのが依頼者の意向 (tier1 指定、2026-07-14 レビュー)。

## 状況

`.apm/skills/retrospective/SKILL.md` の「観測」段は、取り決め (skill / subagent / ルール) の発火・中身と、ハーネスに絡まない推論ミス / 良い振る舞いを見るが、**依頼者のプロンプト自体を改善対象として明示していない**。アクション段も「どの取り決めをどう直すか」に接続する形で、直す先が取り決め (= エージェント側) に閉じている。

### 要件定義フェーズで潰す論点 (= このカードは要件確定ではない)

着手前に define-requirements で以下を固める:

1. **スコープ**: 「依頼者のプロンプト改善案を出す」は xp-harness の責務 (= XP の進め方の規律) に収まるか。ペアプロの相互改善として筋が通るか、それとも「中身の流儀 / 人間コーチング」として配布対象外か (philosophy「責務境界」で判定)
2. **プロンプト依存との整合**: philosophy「振る舞いは skill / 規約に宿らせる (プロンプト依存にしない)」との関係。プロンプト改善を促すことが、この軸と衝突しないか整理する (= 恒久的な振る舞いは取り決めに宿す、が原則。プロンプト改善は「その場の依頼の質」であって恒久振る舞いではない、という切り分けが要るか)
3. **迎合の裏返しにならないか**: 依頼者に改善を促すのが、責任の押し付け・上から目線にならない出し方 (= 正直さ・共創の構えと両立する書式)
4. **観測段への埋め込み方**: 既存の 2 軸 (よかった点 / 伸びしろ) に人間側の観点をどう足すか。独立軸にするか、伸びしろの一部にするか
5. **アクションの残し先**: エージェント側は取り決めに書き戻すが、人間側プロンプト改善の残し先はどこか (= 次セッションに持ち越せない問題は人間側にはない。都度のふりかえり.md 記録で足りるか)

## 再開時の起点

1. define-requirements skill を呼び、上記論点 (特に #1 スコープ, #2 プロンプト依存との整合) を対話で潰す
2. 固まったら retrospective skill の「観測」「アクション」段に人間側プロンプトの観点を追加
3. skill-reviewer に通す
