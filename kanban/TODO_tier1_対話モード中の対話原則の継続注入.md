# 対話モード中の対話原則の継続注入

## 背景 / Why

`dialogue-principles` は skill として 1 回発火すると薄れ、長い対話で規律（特に段階的開示）が効かなくなる。段階的開示が落ちるとエージェントが分析結果を一度に全部出し、人間が大量の出力を読まされて対話が破綻する ── 人間側の「読む量の限界」を守るのが対話原則の必要不可欠な役割。

対話原則はエージェントの base デフォルト（省略バイアス・許可取り・全部出す）の**逆をいく**ので、1 回注入では上書きが持続せず、**強い継続支援**がないと守れない、という性質がある（dialogue-principles 本文自身が「system prompt の省略バイアスを上書きする ★最重要」と言っている）。

実データ（2026-07-18 の本番セッション分析セッション）: 常時 context に乗っている `philosophy` / `skill-design-style` ですら違反が起きた（改修者の disposition を配布側にパッチしようとした責務境界ミス、案のスコープずらし、一気に話しすぎ）。**静的に先頭へ埋め込むだけでは salience が直近会話に負ける**。

## 状況

- 「要件定義のモードゲート」設計メモ（`docs/working/要件定義のモードゲート/設計メモ.md`）が、モードを `define-requirements`（配布物）に入れる話をしている。ただし現状モードは **"合意による共有理解"** であって、持続するフラグ（機械可読な状態）ではない
- 継続注入を「対話モードなら」で発火させるには、まず **モードを持続状態にする** 必要がある。だから本カードはモードゲートの**上物**として乗る（モードを"合意"から"状態"に一歩進める）

## 再開時の起点

- **モードの持続状態化**: 対話モードを保持し、モード合意ゲートで設定・自走合意で解除。モードゲート設計メモに紐付けて設計する
- **対話モード中の継続注入**: 対話モードの間、毎ターン `dialogue-principles` を再注入する支援機構（**矯正でなく気づかせ**、ブロックしない）。静的埋め込みより recency で salience が勝つのが狙い
- **設計論点（濃淡）**: 「毎ターン全文注入」はコンテキスト肥大になりうる → "凝縮したコアを毎ターン／全文はゲートで" のように濃淡をつけるか。philosophy「支援 hook は失敗モードが大きい節目に限る・乱立を避ける」と折り合わせる
- **温度感（過大評価しない）**: これは**緩和であって根絶ではない**。常時あっても違反は起きうる（上記実データ）ので、ペアプロ（人・reviewer subagent）の捕捉は残す前提。ただし recency により現状の静的埋め込みよりは上振れが期待できる
- 着手時は通常のフェーズゲート（`git-workflow` → 要件が固まっていなければ `define-requirements`）を通す。本カードは出発点であって要件確定ではない
