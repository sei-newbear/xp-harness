# AgentTeam 構成の導入 (ストーリー並行実行)

## 背景 / Why

調査スパイク (`docs/working/AgentTeam調査スパイク/調査記録.md`) で構想の成立を実測確認済み。親セッション (依頼者と対話) がストーリーを分割し、ストーリーごとの担当エージェントが worktree 分離で並行実装し、担当の配下でレビュアー (サブエージェント) に規律注入が効く — この構成を xp-harness の harness 機構 (skill / instruction) として導入する。

スコープ注記: 「AgentTeam 構成」はカード名だが、AgentTeam のチーム機構そのものを使うかは未確定 (並行実行だけなら通常サブエージェントの並行で足りることを調査で確認済み)。何を導入するかはチーム機構の採否を含めて要件定義で決める。用語の対応 (親セッション = lead、担当エージェント = チームメンバーまたは通常サブエージェント、レビュアー = サブエージェント) は調査記録の冒頭を参照。

## 状況

実測済みの土台: 担当エージェントを汎用型で立てれば、CLAUDE.md + skill 発火の規律経路が親セッションと同様に効き、slice-tdd / レビュアー呼び / done-verifier / 止まる判断まで再現される (通し・並行の両方で確認済み)。

設計で決めること (調査で積み上がった論点):

- 委任規約をどこに宿らせるか: ゴールの渡し方、「レビュアーはサブエージェントとして呼ぶ (チームメンバーにするとプリロードが黙って落ちる)」、worktree 指定、自走範囲の前置き
- 並行 E2E の実行環境分離 (ポート衝突の実痛みを確認済み)
- worktree の寄せ先: git-workflow の外部配置規律 (許可プロンプトが都度出る) か、組み込みの自動分離 (repo 内 `.claude/worktrees/`、プロンプトなし) か
- チーム機構 (常駐・担当同士の直接対話・共有タスクリスト) を使うか、通常サブエージェントの並行で始めるか (並行実行だけなら後者で足りることを確認済み)

## 再開時の起点

1. `docs/working/AgentTeam調査スパイク/調査記録.md` の「制約を構想に照らした最終整理」「本格導入判断への論点」を読み込む
2. 要件定義フェーズから始める (define-requirements で Why / Done を固める)
