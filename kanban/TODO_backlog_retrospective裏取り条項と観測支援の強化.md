# retrospective の裏取り条項と観測支援の強化

## 背景 / Why

AgentTeam 調査スパイクのふりかえり (2026-07-17) 中に、main session が handoff-docs の description を確認せず「取り決めにどこにも規定が無い」を「事実」ラベルで出す失敗が起きた (自分の発火失敗を問わずに済む、都合のいい因果)。retrospective skill の裏取り条項は「発火・タイミングなど挙動の主張」と例示されており、「取り決めの内容・有無」という主張クラスを覆っていなかった。

構造的な発見: context に常時載っている description は既読感を作るが、確認の代わりにならない (Read したものへの主張は正確、載っているだけのものへの主張が外れた、という対照がセッション内で取れた)。経緯は `docs/working/AgentTeam調査スパイク/ふりかえり.md` の深掘り 2。

## 状況

やることの候補 (着手時に精査):

- retrospective skill の裏取り条項を「改善の判断に効く主張全般 (取り決めの内容・有無を含む)」に拡張
- 「『無い』という否定主張は、context の記憶でなくファイル実体を Read / grep してから事実として出す」を明記
- 観測の「利用可能だった skill / subagent / ルールを一通り当てる」を支える一覧スクリプト (board.sh 型、読み取り専用) の検討。スクリプトは繰り返し作業の手助けの道具として正当 (毎回のふりかえりで発生する反復作業)。設計論点: クロスエージェント対応 — skill の配置パスはエージェントごとに違うため、skill 同梱スクリプトにするか、道具非依存の指示に留めるか

## 再開時の起点

1. `docs/working/AgentTeam調査スパイク/ふりかえり.md` の深掘り 2 を読む
2. retrospective skill 本文 (`.apm/skills/retrospective/SKILL.md`) の「通底する構え」「観測」節を読み、条項の直し方を skill-design-style に沿って設計する
